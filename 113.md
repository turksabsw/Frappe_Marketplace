# Task 113 - Content Snapshot Field UX Improvement Plan

## Executive Summary

The Moderation Case DocType in the `tradehub_compliance` app contains a `content_snapshot` field that captures a JSON representation of content at the time a moderation case is created. While the underlying auto-capture mechanism is well-implemented, the field itself has several UX and data-integrity issues: it uses a raw JSON fieldtype (rendering unformatted text), lacks `read_only` enforcement (allowing accidental edits to audit evidence), has no dedicated timestamp field for when the snapshot was taken, and provides no formatted display for moderators reviewing case content. This plan proposes a comprehensive set of improvements to make the snapshot field immutable, visually informative, and audit-compliant.

## Current State Analysis

### Field Definition (moderation_case.json)

The `content_snapshot` field is defined in the Moderation Case DocType JSON at line 258-263:

```json
{
  "fieldname": "content_snapshot",
  "fieldtype": "JSON",
  "label": "Content Snapshot",
  "description": "Snapshot of content at time of case creation (for audit purposes)"
}
```

**Key observations:**

| Property | Current Value | Issue |
|----------|--------------|-------|
| `fieldtype` | `JSON` | Renders as raw JSON text in a textarea -- not user-friendly for moderators |
| `read_only` | **Not set** (defaults to `0`) | Snapshot can be manually edited, compromising audit integrity |
| `label` | "Content Snapshot" | Adequate but could be more descriptive |
| `description` | "Snapshot of content at time of case creation (for audit purposes)" | Functional but lacks guidance on what the snapshot contains |
| `hidden` | Not set | Field is visible, which is correct for moderator review |

### Field Location in Form Layout

The field sits in the **Content** tab, within the `content_section` ("Content Under Review") section. It appears after `content_created_at` in the field order (line 29 of field_order array), which places it at the bottom of the content information area -- a logical position.

### Auto-Snapshot Mechanism (moderation_case.py)

The `capture_content_snapshot()` method (lines 99-161) is called during `before_insert()` and operates as follows:

1. **Trigger condition**: Only runs when `content_type` and `content_id` are set AND `content_snapshot` is empty
2. **Base snapshot fields**: `name`, `doctype`, `captured_at` (as string of `now_datetime()`)
3. **Title resolution**: Iterates through `["title", "name", "listing_name", "store_name", "seller_name"]` to find a title
4. **Creation date capture**: Copies `content_doc.creation` to both snapshot and `content_created_at` field
5. **Status capture**: Includes `content_doc.status` if available
6. **Content-type-specific fields**:
   - **Listing**: `description`, `base_price`, `selling_price`, `seller`, `category`
   - **Review**: `rating`, `review_text`, `reviewer`, `listing`
   - **Storefront**: `store_name`, `tagline`, `seller`
7. **Owner resolution**: Iterates through `["seller", "owner", "user", "reviewer", "created_by"]` to identify content owner
8. **Storage**: Serialized via `json.dumps(snapshot)` into the field
9. **Error handling**: Catches all exceptions and logs to `frappe.log_error` with title "Moderation Case Snapshot Error"

### Identified Issues

1. **No immutability enforcement**: Any user with write permission on Moderation Case (System Manager, Marketplace Admin, Marketplace Moderator) can edit the snapshot contents, which undermines its value as audit evidence.

2. **Raw JSON display**: The `JSON` fieldtype renders as a plain textarea with raw JSON text. Moderators must mentally parse nested key-value pairs, making case review slow and error-prone.

3. **No dedicated snapshot timestamp**: The `captured_at` timestamp is embedded inside the JSON blob. There is no separate, queryable field for when the snapshot was taken, making it impossible to filter or report on snapshot timing via standard Frappe queries.

4. **No snapshot versioning**: If content changes after the case is created, there is no mechanism to re-capture or compare snapshots. The single capture happens only in `before_insert()`.

5. **Missing content types**: The snapshot logic only handles `Listing`, `Review`, and `Storefront` content types. Other potential moderation targets (e.g., seller profiles, messages, user-generated media) fall through to the base-only snapshot with just `name`, `doctype`, and `captured_at`.

6. **No data validation**: The snapshot data is never validated against a schema. Corrupted or manually altered JSON could break downstream consumers.

## Proposed Design / Architecture

### 1. Field Type Change: JSON -> Code (with JSON mode)

**Recommendation**: Change fieldtype from `JSON` to `Code` with `options='JSON'`.

**Rationale**:

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| `JSON` (current) | Native JSON type | Raw textarea, no highlighting, no formatting | Reject (current) |
| `HTML` | Can render formatted content | Loses raw JSON access; requires building HTML from JSON on every save | Reject |
| `Code` with `options='JSON'` | Syntax highlighting, collapsible blocks, read-only mode works naturally, preserves raw JSON access | Slightly larger field widget | **Recommended** |
| `Small Text` / `Long Text` | Simple text display | No highlighting, no JSON awareness | Reject |

The `Code` fieldtype with `options='JSON'` provides:
- Syntax-highlighted JSON display in the Frappe Desk form
- Native `read_only` support that disables the code editor
- Proper JSON formatting and indentation in the editor
- Compatibility with existing `json.dumps()` storage in Python

**Proposed field definition:**

```json
{
  "fieldname": "content_snapshot",
  "fieldtype": "Code",
  "options": "JSON",
  "label": "Content Snapshot (Immutable)",
  "description": "Read-only snapshot of the original content at the time this moderation case was created. This data is preserved for audit purposes and cannot be modified.",
  "read_only": 1
}
```

### 2. Label and Description Improvements

**Current:**
- Label: `"Content Snapshot"`
- Description: `"Snapshot of content at time of case creation (for audit purposes)"`

**Proposed:**
- Label: `"Content Snapshot (Immutable)"`
- Description: `"Read-only snapshot of the original content at the time this moderation case was created. This data is preserved for audit purposes and cannot be modified."`

The word "Immutable" in the label immediately communicates to moderators that this field should not (and cannot) be altered. The expanded description clarifies the purpose and sets expectations.

### 3. read_only: 1 Enforcement

Add `"read_only": 1` to the field definition. This prevents all users (including System Manager via the form) from editing the snapshot content through the UI.

**Additional server-side enforcement** should be added in `moderation_case.py` `validate()` method to prevent API-level tampering:

```python
def validate_snapshot_immutability(self):
    """Prevent modification of content snapshot after initial capture."""
    if not self.is_new() and self.content_snapshot:
        old_snapshot = frappe.db.get_value(
            "Moderation Case", self.name, "content_snapshot"
        )
        if old_snapshot and old_snapshot != self.content_snapshot:
            frappe.throw(
                _("Content Snapshot cannot be modified after case creation. "
                  "This field is immutable for audit purposes.")
            )
```

This provides defense-in-depth: UI-level `read_only` prevents accidental edits, while server-side validation prevents programmatic tampering.

### 4. Snapshot Timestamp Field Addition

Add a new dedicated `Datetime` field to store the snapshot capture time as a first-class, queryable field:

```json
{
  "fieldname": "snapshot_timestamp",
  "fieldtype": "Datetime",
  "label": "Snapshot Captured At",
  "read_only": 1,
  "description": "Timestamp when the content snapshot was captured (auto-set on case creation)"
}
```

**Field order placement**: Insert `snapshot_timestamp` immediately before `content_snapshot` in the `field_order` array, so moderators see when the snapshot was taken before viewing its contents.

Updated field order excerpt:
```json
"content_created_at",
"snapshot_timestamp",
"content_snapshot",
```

**Python update** in `capture_content_snapshot()`:

```python
# At the end of the snapshot capture block, after self.content_snapshot = json.dumps(snapshot):
self.snapshot_timestamp = now_datetime()
```

The `captured_at` key should still be kept inside the JSON blob for self-contained audit evidence, but the separate field enables:
- List view filtering by snapshot time
- Standard Frappe reports on snapshot age
- SLA calculations relative to snapshot time

### 5. Enhanced Auto-Snapshot Mechanism

#### 5a. Pretty-printed JSON Storage

Currently `json.dumps(snapshot)` produces a compact single-line JSON string. Change to:

```python
self.content_snapshot = json.dumps(snapshot, indent=2, ensure_ascii=False, default=str)
```

- `indent=2`: Human-readable formatting in the Code field
- `ensure_ascii=False`: Preserves Turkish characters (e.g., seller names with "S", "c", "g")
- `default=str`: Handles non-serializable types (dates, Decimals) gracefully

#### 5b. Extended Content Type Support

Add snapshot field mappings for additional content types that may be moderated:

```python
# Additional content type handlers in capture_content_snapshot()
elif self.content_type == "Seller Profile":
    for field in ["seller_name", "company_name", "email", "status", "verification_status"]:
        if hasattr(content_doc, field):
            snapshot[field] = getattr(content_doc, field)
elif self.content_type == "Message":
    for field in ["subject", "content", "sender", "recipient"]:
        if hasattr(content_doc, field):
            snapshot[field] = getattr(content_doc, field)
```

#### 5c. Snapshot Metadata Enrichment

Add metadata fields to the snapshot for better audit context:

```python
snapshot = {
    "_meta": {
        "snapshot_version": "1.0",
        "captured_at": str(now_datetime()),
        "captured_by": frappe.session.user,
        "source_doctype": self.content_type,
        "source_name": content_doc.name,
    },
    "data": {
        # ... actual content fields ...
    }
}
```

Wrapping in `_meta` and `data` keys provides a clear schema that future consumers can rely on, and the `snapshot_version` field allows backward-compatible schema evolution.

### 6. Client-Side JSON Renderer

While the `Code` fieldtype with `options='JSON'` provides syntax highlighting, a richer visual renderer can be added via a client script to display the snapshot as a formatted, human-readable card.

**Approach**: Add a custom HTML section below the Code field that renders the JSON as a structured summary.

**Implementation location**: `moderation_case.js` (client script)

```javascript
// Pseudocode for moderation_case.js
frappe.ui.form.on('Moderation Case', {
    refresh: function(frm) {
        if (frm.doc.content_snapshot) {
            render_snapshot_preview(frm);
        }
    }
});

function render_snapshot_preview(frm) {
    try {
        let snapshot = JSON.parse(frm.doc.content_snapshot);
        let html = build_snapshot_html(snapshot);
        // Render below the content_snapshot field
        $(frm.fields_dict.content_snapshot.wrapper)
            .find('.snapshot-preview').remove();
        $(frm.fields_dict.content_snapshot.wrapper)
            .append('<div class="snapshot-preview">' + html + '</div>');
    } catch (e) {
        // If JSON is invalid, show warning
        console.warn('Invalid snapshot JSON:', e);
    }
}

function build_snapshot_html(snapshot) {
    let rows = '';
    let data = snapshot.data || snapshot;  // Support both v1.0 schema and legacy

    for (let [key, value] of Object.entries(data)) {
        if (key.startsWith('_')) continue;  // Skip meta keys
        let display_key = frappe.model.unscrub(key);  // Convert snake_case to Title Case
        let display_value = format_snapshot_value(key, value);
        rows += `<tr><td class="text-muted">${display_key}</td><td>${display_value}</td></tr>`;
    }

    return `
        <div class="snapshot-card" style="margin-top: 10px; border: 1px solid var(--border-color); border-radius: var(--border-radius); padding: 15px;">
            <h6 class="text-muted" style="margin-bottom: 10px;">
                <i class="fa fa-camera"></i> Content Snapshot Preview
            </h6>
            <table class="table table-bordered table-sm">
                <tbody>${rows}</tbody>
            </table>
        </div>
    `;
}

function format_snapshot_value(key, value) {
    if (value === null || value === undefined) return '<span class="text-muted">N/A</span>';
    if (key.includes('price')) return frappe.format(value, {fieldtype: 'Currency'});
    if (key.includes('date') || key.includes('creation') || key.includes('_at'))
        return frappe.datetime.str_to_user(value);
    if (typeof value === 'object') return `<code>${JSON.stringify(value)}</code>`;
    return frappe.utils.escape_html(String(value));
}
```

This renderer provides:
- A visual card below the raw JSON field showing key-value pairs in a table
- Proper formatting of currency, date, and other typed values
- XSS protection via `frappe.utils.escape_html()`
- Graceful fallback if JSON is invalid
- Support for both the proposed v1.0 schema (`_meta`/`data`) and legacy flat schema

### Data Migration Considerations

Changing from `JSON` to `Code` fieldtype:

- **Database column**: Both `JSON` and `Code` fieldtypes store data as `longtext` in MariaDB. The underlying column type does **not** change, so no data loss occurs during migration.
- **Existing data**: All existing `content_snapshot` values (raw JSON strings) will display correctly in the `Code` field with JSON highlighting.
- **Pretty-printing existing data**: A one-time migration script could reformat existing compact JSON to indented format:

```python
# One-time migration (run via bench console or as a patch)
import json
import frappe

cases = frappe.get_all(
    "Moderation Case",
    filters={"content_snapshot": ["is", "set"]},
    fields=["name", "content_snapshot"],
    limit_page_length=0
)

for case in cases:
    try:
        data = json.loads(case.content_snapshot)
        pretty = json.dumps(data, indent=2, ensure_ascii=False, default=str)
        frappe.db.set_value("Moderation Case", case.name, "content_snapshot", pretty, update_modified=False)
    except (json.JSONDecodeError, TypeError):
        pass  # Skip invalid entries

frappe.db.commit()
```

### Adding snapshot_timestamp to Existing Records

Existing records do not have a `snapshot_timestamp` value. A backfill script can extract it from the JSON:

```python
for case in cases:
    try:
        data = json.loads(case.content_snapshot)
        captured_at = data.get("captured_at") or data.get("_meta", {}).get("captured_at")
        if captured_at:
            frappe.db.set_value(
                "Moderation Case", case.name,
                "snapshot_timestamp", captured_at,
                update_modified=False
```

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Data loss during field type change** | Low | High | Both `JSON` and `Code` use `longtext` column in MariaDB -- no actual column type change occurs. Verify with `bench migrate --dry-run` before applying. |
| **Existing compact JSON display** | Medium | Low | Compact JSON still renders correctly in `Code` field (just not indented). Run optional migration script to pretty-print existing data. |
| **Snapshot immutability blocking legitimate corrections** | Low | Medium | System Manager can bypass via `frappe.db.set_value()` directly if an emergency correction is needed. The validation only blocks form/API saves, not direct DB writes. |
| **Client-side renderer XSS** | Low | High | Use `frappe.utils.escape_html()` for all rendered values. Never use `innerHTML` with unescaped user content. |
| **Performance impact of pretty JSON** | Low | Low | `indent=2` adds whitespace but `longtext` column handles it easily. Typical snapshot is under 1KB even with indentation. |
| **Turkish character encoding** | Medium | Medium | `ensure_ascii=False` preserves Turkish characters natively. Test with seller names containing "S", "c", "g", "I", "i", "O", "U" characters. |
| **Backward compatibility of schema change** | Low | Medium | Wrapping snapshot in `_meta`/`data` structure is optional. Renderer supports both legacy flat format and new structured format. Introduce gradually. |
