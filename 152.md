## 1. Executive Summary

This document defines the **Variant Request Flow** for the TradeHub B2B Marketplace. It introduces a new `Variant Request` DocType that allows **authorized sellers** to request new product variants from brand owners. When multiple sellers request the same variant, the system aggregates demand signals to help brand owners prioritize which variants to add to their catalog.

### Key Deliverables

1. **Variant Request DocType** — A new standalone DocType (naming_series `VRQ-.YYYY.-.#####`) for sellers to formally request new product variants with structured attribute fields (color, size, material, packaging)
2. **Authorization Prerequisite** — Only sellers with an active Brand Gating record (status=Approved) for the brand can create variant requests
3. **Multi-Seller Demand Aggregation** — Matching criteria to identify "same variant" requests from different sellers, with demand signal display (request_count, requesting_sellers, first_requested_date)
4. **Brand Owner Management Panel** — Dashboard for brand owners to view, filter, and bulk-manage incoming variant requests
5. **Auto-Catalog Addition on Approval** — Step-by-step flow from approval to automatic Product Variant creation
6. **Notification Flow** — Comprehensive notifications for request lifecycle events and demand threshold triggers
7. **Multi-Tenant Isolation** — All data scoped via tenant field inherited from requesting seller

## 3. Variant Request DocType

### 3.1 Overview

The `Variant Request` DocType allows authorized sellers to formally request that a brand owner add a new product variant to the catalog. Key principles:

1. **Authorization-Gated** — Only sellers with Brand Gating status=Approved can create requests
2. **Structured Attributes** — Variant details use the same attribute fields as Product Variant (color, size, material, packaging) for 1:1 mapping on approval
3. **Demand Aggregation** — Multiple sellers requesting the "same" variant are linked via a `demand_group_key` for aggregated demand signals
4. **Brand Owner Managed** — The brand owner (via `owner_seller` on Brand) or System Manager reviews and approves/rejects requests
5. **Auto-Catalog on Approval** — Approved requests automatically create a Product Variant record

### 3.3 Status Workflow

```
                    ┌──────────┐
                    │  Draft   │  (Seller is editing, not yet submitted)
                    └────┬─────┘
                         │ submit()
                         v
                    ┌──────────┐
                    │ Pending  │  (Submitted, awaiting review)
                    └────┬─────┘
                         │ brand owner/admin picks up
                         v
                  ┌──────────────┐
                  │ Under Review │  (Being evaluated)
                  └──────┬───────┘
                    ┌────┴────┐
                    v         v
             ┌──────────┐ ┌──────────┐
             │ Approved │ │ Rejected │
             └──────────┘ └──────────┘
                  │
                  v
          (Auto-creates Product Variant)

  At any point before Approved/Rejected:
             ┌───────────┐
             │ Cancelled │  (Seller withdraws request)
             └───────────┘
```

**Status definitions:**

| Status | Color | Description | Who Sets |
|--------|-------|-------------|----------|
| Draft | gray | Seller is preparing the request, not yet visible to brand owner | Seller (auto on create) |
| Pending | blue | Submitted and awaiting review by brand owner/admin | System (on submit action) |
| Under Review | orange | Brand owner/admin has begun reviewing the request | Brand owner / System Manager |
| Approved | green | Request approved — triggers auto-catalog addition | Brand owner / System Manager |
| Rejected | red | Request rejected with reason | Brand owner / System Manager |
| Cancelled | gray | Seller withdrew the request before approval/rejection | Seller |

**Valid status transitions:**

```python
VALID_TRANSITIONS = {

    "Approved":     [],  # Terminal state

}
```

**Note:** Brand owners access variant requests through `has_permission` hooks (see Section 3.7), not through a separate role. When a brand owner views the Variant Request list, the permission_query_conditions function includes records where the brand's `owner_seller` matches their Seller Profile.

### 3.7 Permission Model — Brand Owner Access

```python
def has_permission(doc, ptype, user):
    """
    Custom permission check for Variant Request.

    Access rules:
    1. System Manager: Full access to all requests
    2. Seller (owner): Full access to own requests
    3. Brand owner: Read + Write access to requests for their brand
    """
    if "System Manager" in frappe.get_roles(user):
        return True

    # Get user's Seller Profile
    seller = get_seller_profile_for_user(user)
    if not seller:
        return False

    # Owner access: seller who created the request
    if doc.requesting_seller == seller:
        return True

    # Brand owner access: owner_seller on the Brand
    if ptype in ("read", "write"):
        brand_owner = frappe.db.get_value("Brand", doc.brand, "owner_seller")
        if brand_owner == seller:
            return True

    return False


def get_permission_query_conditions(user):
    """
    SQL conditions for Variant Request list view.

    Returns conditions that show:
    - Requests created by the current seller (own requests)
    - Requests for brands where current seller is owner_seller
    - All requests for System Manager
    """
    if "System Manager" in frappe.get_roles(user):
        return ""

    seller = get_seller_profile_for_user(user)
    if not seller:
        return "1=0"

    return """(
        `tabVariant Request`.requesting_seller = '{seller}'
        OR `tabVariant Request`.brand IN (
            SELECT name FROM `tabBrand`
            WHERE owner_seller = '{seller}'
        )
    )""".format(seller=frappe.db.escape(seller))
```

## 4. Authorization Prerequisite — Brand Gating Check

### 4.1 Rule: Only Authorized Sellers Can Create Variant Requests

A seller MUST have an active Brand Gating record with `authorization_status = "Approved"` for the specified brand before they can create a Variant Request. This is enforced at two levels:

#### Server-Side Enforcement (validate hook)

```python
def validate_seller_authorization(self):
    """
    Ensure requesting seller is authorized for the brand via Brand Gating.

    Only sellers with Brand Gating status=Approved can create variant requests.
    This prevents unauthorized sellers from even submitting requests.
    """
    if not self.requesting_seller or not self.brand:
        return

    # System Manager can bypass authorization check (for testing/admin)
    if "System Manager" in frappe.get_roles():
        return

    # Check for active Brand Gating record
    brand_gating = frappe.db.get_value(
        "Brand Gating",
        {
            "brand": self.brand,
            "seller": self.requesting_seller,
            "authorization_status": "Approved",
            "is_active": 1
        },
        ["name", "authorization_type", "valid_to"],
        as_dict=True
    )

    if not brand_gating:
        frappe.throw(
            _("Seller {0} is not authorized to sell brand {1}. "
              "Please obtain Brand Gating approval first via Brand Authorization Request.").format(
                self.seller_name or self.requesting_seller,
                self.brand_name or self.brand
            ),
            title=_("Brand Authorization Required")
        )

    # Check validity period
    if brand_gating.valid_to and getdate(brand_gating.valid_to) < getdate(today()):
        frappe.throw(
            _("Brand authorization for {0} has expired on {1}. "
              "Please renew your Brand Gating authorization.").format(
                self.brand_name or self.brand,
                brand_gating.valid_to
            ),
            title=_("Authorization Expired")
        )

    # Store the Brand Gating reference
    self.brand_gating = brand_gating.name
```

## 5. Multi-Seller Demand Aggregation

### 5.1 Problem Statement

When multiple sellers independently request the same variant (e.g., "Red XL Cotton" for the same product), the brand owner should see aggregated demand — not individual duplicates. This requires:

1. A matching algorithm to determine when two requests are for the "same" variant
2. A grouping key that links related requests
3. A signal display format showing aggregated demand

### 5.2 Demand Group Key Generation

The `demand_group_key` is a normalized, deterministic string that groups equivalent variant requests. It is generated from the parent product and normalized attribute values.

**Key format:**
```
{parent_product}:C={normalized_color}|S={normalized_size}|M={normalized_material}|P={normalized_packaging}
```

**Example:**
```
SKU-00042:C=RED|S=XL|M=COTTON|P=BOX
```

#### Normalization Algorithm

```python
def generate_demand_group_key(self):
    """
    Generate a normalized key for demand aggregation.

    Steps:
    1. Normalize each attribute value (uppercase, trim, transliterate Turkish chars)
    2. Build deterministic key from parent_product + sorted normalized attributes
    3. Only include attributes that have values
    """
    parts = [self.parent_product or ""]

    # Normalize and collect attribute pairs
    attr_parts = []
    for prefix, field in [
        ("C", self.requested_color),
        ("S", self.requested_size),
        ("M", self.requested_material),
        ("P", self.requested_packaging),
    ]:
        if field:
            normalized = self.normalize_attribute_value(field)
            attr_parts.append(f"{prefix}={normalized}")

    if attr_parts:
        parts.append("|".join(attr_parts))

    self.demand_group_key = ":".join(parts)


def normalize_attribute_value(self, value):
    """
    Normalize an attribute value for matching purposes.

    Steps:
    1. Strip whitespace
    2. Convert to uppercase
    3. Transliterate Turkish characters (Ç→C, Ş→S, Ğ→G, İ→I, Ö→O, Ü→U)
    4. Remove non-alphanumeric characters (except dash and underscore)
    5. Collapse multiple spaces/dashes

    Examples:
    - "Kırmızı" → "KIRMIZI"
    - "Red"     → "RED"
    - "  Navy Blue  " → "NAVY-BLUE"
    - "X-Large" → "X-LARGE"
    """
    if not value:
        return ""

    result = value.strip().upper()

    # Turkish character transliteration
    turkish_map = {
        'Ç': 'C', 'Ş': 'S', 'Ğ': 'G', 'İ': 'I', 'Ö': 'O', 'Ü': 'U',
        'ç': 'C', 'ş': 'S', 'ğ': 'G', 'ı': 'I', 'ö': 'O', 'ü': 'U',
    }
    for tr_char, en_char in turkish_map.items():
        result = result.replace(tr_char, en_char)

    # Replace whitespace with dash, remove special characters
    result = re.sub(r'\s+', '-', result)
    result = re.sub(r'[^A-Z0-9\-_]', '', result)
    result = re.sub(r'-+', '-', result)  # collapse multiple dashes

    return result.strip('-')
```

### 5.3 Matching Strategy: Exact Match on Normalized Key

**Decision: Exact matching on normalized keys (NOT fuzzy matching)**

**Rationale:**
- Fuzzy matching introduces ambiguity (how fuzzy is "close enough"?)
- The normalization step already handles case differences, whitespace, and Turkish character variants
- For B2B marketplace, attribute values tend to be structured (e.g., "XL", "Red", "Cotton") not free-text descriptions
- False positives in aggregation are worse than false negatives (merging "Red" with "Rose" would be confusing)

**What normalization handles:**
| Input A | Input B | Normalized | Match? |
|---------|---------|------------|--------|
| "Red" | "red" | RED | Yes |
| "Kırmızı" | "KIRMIZI" | KIRMIZI | Yes |
| "X-Large" | "X Large" | X-LARGE | Yes |
| "Navy Blue" | "navy blue" | NAVY-BLUE | Yes |
| "  Cotton  " | "Cotton" | COTTON | Yes |

**What normalization does NOT handle (by design):**
| Input A | Input B | Match? | Reason |
|---------|---------|--------|--------|
| "Red" | "Kırmızı" | No | Different languages — requires admin/brand owner to manually merge |
| "XL" | "Extra Large" | No | Abbreviation vs full form — would need synonym mapping |
| "100% Cotton" | "Cotton" | No | Qualifier difference — semantically different |

**Future enhancement:** An optional synonym mapping table (e.g., "XL" = "Extra Large", "Red" = "Kırmızı") could be added later if needed, managed per brand or globally. This is out of scope for v1.

### 5.4 Demand Signal Calculation

When a Variant Request is submitted (status changes from Draft to Pending), the system:

1. Generates the `demand_group_key`
2. Queries all existing Variant Requests with the same key (excluding Cancelled/Rejected)
3. Updates the aggregation fields on ALL requests in the group

```python
def update_demand_aggregation(self):
    """
    Update demand signal fields for all requests in the same demand group.

    Called on:
    - Request submission (Draft → Pending)
    - Request cancellation (any → Cancelled)
    - Request rejection (any → Rejected)
    """
    if not self.demand_group_key:
        self.generate_demand_group_key()

    if not self.demand_group_key:
        return

    # Get all active requests in this demand group
    group_requests = frappe.get_all(
        "Variant Request",
        filters={
            "demand_group_key": self.demand_group_key,
            "status": ("not in", ["Cancelled", "Rejected"]),
            "name": ("!=", "")  # include all
        },
        fields=["name", "requesting_seller", "seller_name", "request_date"],
        order_by="request_date asc"
    )

    # Calculate aggregation values
    request_count = len(group_requests)
    seller_names = list(set(r.seller_name for r in group_requests if r.seller_name))
    first_requested = group_requests[0].request_date if group_requests else None

    # Update all requests in the group
    for req in group_requests:
        frappe.db.set_value("Variant Request", req.name, {
            "demand_request_count": request_count,
            "demand_sellers_summary": ", ".join(sorted(seller_names)),
            "demand_first_requested": first_requested
        }, update_modified=False)
```

## 7. Auto-Catalog Addition on Approval

### 7.1 Overview

When a Variant Request is approved, the system automatically creates a `Product Variant` record. This eliminates manual data entry and ensures the variant is immediately available in the catalog.

### 7.2 Step-by-Step Flow

```
Step 1: Brand owner/admin changes Variant Request status → "Approved"
         │
         v
Step 2: validate() → validate_status_transition()
         │  Checks: previous status was "Pending" or "Under Review"
         v
Step 3: on_update() detects status changed to "Approved"
         │
         v
Step 4: create_product_variant() method called
         │
         ├─── 4a: Check if variant already exists (duplicate check)
         │         Query Product Variant with same parent + normalized attributes
         │         If exists → link to existing, skip creation, log message
         │
         ├─── 4b: Build Product Variant document
         │         Map fields: requested_color → color, requested_size → size, etc.
         │         Set parent product, seller, tenant from SKU Product
         │         Set status = "Active"
         │
         ├─── 4c: Insert the new Product Variant
         │         product_variant.insert(ignore_permissions=True)
         │         Auto-generates variant_code and variant_name
         │
         ├─── 4d: Link back to Variant Request
         │         Set created_variant = product_variant.name
         │         Set created_variant_code = product_variant.variant_code
         │         Log creation details in auto_creation_log
         │
         └─── 4e: Update all requests in demand group
                   If demand_group_key exists, link ALL approved requests
                   to the same created Product Variant
         │
         v
Step 5: Send notification to requesting seller(s)
         │  "Your variant request has been approved! Variant {code} is now in the catalog."
         v
Step 6: Update demand aggregation counts
         │  Recalculate counts excluding this approved request
         v
Done
```

### 7.3 Implementation: create_product_variant()

```python
def create_product_variant(self):
    """
    Auto-create a Product Variant from an approved Variant Request.

    This is called from on_update() when status changes to 'Approved'.
    Skipped if self.flags.skip_variant_creation is True (for bulk group approvals).
    """
    if self.flags.get("skip_variant_creation"):
        return

    # Already created?
    if self.created_variant:
        return

    log_lines = []

    try:
        # Step 4a: Check for existing variant with same attributes
        existing = self._find_existing_variant()
        if existing:
            self.created_variant = existing
            log_lines.append(f"Linked to existing variant: {existing}")
            frappe.db.set_value("Variant Request", self.name,
                "auto_creation_log", "\n".join(log_lines),
                update_modified=False)
            return

        # Step 4b: Build the Product Variant document
        variant_doc = frappe.new_doc("Product Variant")
        variant_doc.sku_product = self.parent_product
        variant_doc.status = "Active"

        # Map variant attributes from request
        variant_doc.color = self.requested_color or ""
        variant_doc.size = self.requested_size or ""
        variant_doc.material = self.requested_material or ""
        variant_doc.packaging = self.requested_packaging or ""

        # Map custom attributes
        if self.custom_attribute_1_label:
            variant_doc.attribute_1_label = self.custom_attribute_1_label
            variant_doc.attribute_1_value = self.custom_attribute_1_value
        if self.custom_attribute_2_label:
            variant_doc.attribute_2_label = self.custom_attribute_2_label
            variant_doc.attribute_2_value = self.custom_attribute_2_value

        # Copy reference image as variant thumbnail if provided
        if self.reference_image:
            variant_doc.variant_thumbnail = self.reference_image

        log_lines.append("Building Product Variant from request attributes...")

        # Step 4c: Insert the Product Variant
        variant_doc.insert(ignore_permissions=True)
        log_lines.append(f"Created Product Variant: {variant_doc.name}")
        log_lines.append(f"Variant Code: {variant_doc.variant_code}")
        log_lines.append(f"Variant Name: {variant_doc.variant_name}")

        # Step 4d: Link back to this request
        self.created_variant = variant_doc.name

        # Step 4e: Link all requests in demand group to same variant
        if self.demand_group_key:
            self._link_demand_group_to_variant(variant_doc.name)
            log_lines.append(f"Linked demand group to variant: {self.demand_group_key}")

    except Exception as e:
        log_lines.append(f"ERROR: {str(e)}")
        frappe.log_error(
            title=f"Variant Request Auto-Creation Failed: {self.name}",
            message=frappe.get_traceback()
        )

    # Save log
    self.auto_creation_log = "\n".join(log_lines)
    frappe.db.set_value("Variant Request", self.name, {
        "created_variant": self.created_variant,
        "auto_creation_log": self.auto_creation_log
    }, update_modified=False)


def _find_existing_variant(self):
    """
    Check if a Product Variant already exists with matching attributes.

    Returns the variant name if found, None otherwise.
    """
    filters = {
        "sku_product": self.parent_product,
        "status": ("!=", "Discontinued"),
    }

    # Build attribute filters (only for non-empty values)
    if self.requested_color:
        filters["color"] = self.requested_color
    if self.requested_size:
        filters["size"] = self.requested_size
    if self.requested_material:
        filters["material"] = self.requested_material
    if self.requested_packaging:
        filters["packaging"] = self.requested_packaging

    existing = frappe.db.get_value("Product Variant", filters, "name")
    return existing


def _link_demand_group_to_variant(self, variant_name):
    """
    Link all approved requests in the same demand group to the created variant.
    """
    approved_in_group = frappe.get_all(
        "Variant Request",
        filters={
            "demand_group_key": self.demand_group_key,
            "status": "Approved",
            "created_variant": ("is", "not set"),
            "name": ("!=", self.name)
        },
        pluck="name"
    )

    for req_name in approved_in_group:
        frappe.db.set_value("Variant Request", req_name, {
            "created_variant": variant_name
        }, update_modified=False)
```

### 7.5 Edge Cases

| Scenario | Handling |
|----------|----------|
| Variant already exists for same attributes | Link to existing variant instead of creating duplicate; log message |
| Parent product is archived between request and approval | `validate_parent_product_status()` blocks approval |
| Brand Gating revoked between request and approval | Approval still proceeds — the variant belongs to the brand owner's catalog, not the requesting seller |
| Multiple requests approved simultaneously | First `create_product_variant()` creates; subsequent ones detect existing and link |
| Product Variant creation fails (e.g., validation error) | Error logged in `auto_creation_log`, admin notified, request stays Approved but `created_variant` is empty |

```python
def send_notification(self, event):
    """
    Send notification based on event type.

    Uses Frappe's built-in notification system (frappe.sendmail + System Notification).
    """
    recipients = self._get_notification_recipients(event)
    subject, message = self._get_notification_content(event)

    if not recipients:
        return

    # Create System Notification (in-app)
    for user in recipients:
        frappe.get_doc({
            "doctype": "Notification Log",
            "for_user": user,
            "type": "Alert",
            "document_type": "Variant Request",
            "document_name": self.name,
            "subject": subject,
            "email_content": message,
        }).insert(ignore_permissions=True)

    # Send email notification
    frappe.sendmail(
        recipients=recipients,
        subject=subject,
        message=message,
        reference_doctype="Variant Request",
        reference_name=self.name,
    )


def _get_notification_recipients(self, event):
    """Determine notification recipients based on event type."""
    recipients = []

    if event in ("submitted", "cancelled"):
        # Notify brand owner
        brand_owner_seller = frappe.db.get_value("Brand", self.brand, "owner_seller")
        if brand_owner_seller:
            owner_user = frappe.db.get_value("Seller Profile", brand_owner_seller, "user")
            if owner_user:
                recipients.append(owner_user)

        # Also notify System Managers for submitted
        if event == "submitted":
            system_managers = frappe.get_all(
                "Has Role",
                filters={"role": "System Manager", "parenttype": "User"},
                pluck="parent"
            )
            recipients.extend(system_managers[:5])  # Limit to 5 admins

    elif event in ("under_review", "rejected"):
        # Notify requesting seller
        seller_user = frappe.db.get_value("Seller Profile", self.requesting_seller, "user")
        if seller_user:
            recipients.append(seller_user)

    elif event == "approved":
        # Notify requesting seller
        seller_user = frappe.db.get_value("Seller Profile", self.requesting_seller, "user")
        if seller_user:
            recipients.append(seller_user)

        # Notify all sellers in demand group
        if self.demand_group_key:
            group_sellers = frappe.get_all(
                "Variant Request",
                filters={
                    "demand_group_key": self.demand_group_key,
                    "status": ("in", ["Pending", "Under Review"]),
                    "requesting_seller": ("!=", self.requesting_seller)
                },
                pluck="requesting_seller"
            )
            for seller in set(group_sellers):
                user = frappe.db.get_value("Seller Profile", seller, "user")
                if user:
                    recipients.append(user)

    elif event == "demand_threshold":
        # Notify brand owner
        brand_owner_seller = frappe.db.get_value("Brand", self.brand, "owner_seller")
        if brand_owner_seller:
            owner_user = frappe.db.get_value("Seller Profile", brand_owner_seller, "user")
            if owner_user:
                recipients.append(owner_user)

    return list(set(recipients))  # Deduplicate


def _get_notification_content(self, event):
    """Generate notification subject and message based on event."""
    variant_desc = " / ".join(filter(None, [
        self.requested_color, self.requested_size,

    templates = {
```

### 8.3 Demand Threshold Notifications

Demand threshold notifications are triggered when the `demand_request_count` crosses specific thresholds:

```python
DEMAND_THRESHOLDS = [3, 5, 10, 25, 50]

def check_demand_threshold_notification(self):
    """
    Check if demand count has crossed a notification threshold.

    Only sends notification if the count just crossed the threshold
    (i.e., the previous count was below the threshold).
    """
    current_count = self.demand_request_count or 0

    for threshold in DEMAND_THRESHOLDS:
        if current_count >= threshold and (current_count - 1) < threshold:
            # Just crossed this threshold
            self.send_notification("demand_threshold")
            break  # Only one threshold notification at a time
```

```python
# For license information, please see license.txt

"""
Variant Request DocType for Trade Hub B2B Marketplace.

This module implements the Variant Request DocType which allows authorized
sellers to request new product variants from brand owners. Key features include:
- Authorization-gated creation (Brand Gating status=Approved required)
- Multi-seller demand aggregation via normalized demand_group_key
- Auto-catalog addition on approval (creates Product Variant)
- Brand owner management with bulk operations
- Multi-tenant data isolation via tenant field (inherited from Seller Profile)
"""

import re

import frappe
from frappe import _
from frappe.model.document import Document
from frappe.utils import cint, today, getdate, now_datetime


class VariantRequest(Document):
    """
    Variant Request DocType for the Trade Hub B2B marketplace.
    """

    def before_insert(self):
        """Set defaults before inserting a new request."""
        self.set_default_fields()

    def validate(self):
        """Validate request data before saving."""
        self.validate_seller_authorization()
        self.validate_product_brand_match()
        self.validate_parent_product_status()
        self.validate_at_least_one_attribute()
        self.validate_status_transition()
        self.validate_tenant_isolation()
        self.generate_demand_group_key()

    def on_update(self):
        """Actions after request is updated."""
        if self.has_value_changed("status"):
            self.handle_status_change()

    # =========================================================================
    # INITIALIZATION
    # =========================================================================

    def set_default_fields(self):
        """Set default field values."""
        if not self.request_date:
            self.request_date = today()
        if not self.status:
            self.status = "Draft"

    # =========================================================================
    # VALIDATION METHODS
    # =========================================================================

    def validate_at_least_one_attribute(self):
        """At least one variant attribute must be specified."""
        if not any([
            self.requested_color,
            self.requested_size,
            self.requested_material,
            self.requested_packaging
        ]):
            frappe.throw(
                _("At least one variant attribute (color, size, material, or packaging) "
                  "must be specified")
            )

    def validate_status_transition(self):
        """Validate status transition is allowed."""
        if not self.has_value_changed("status"):
            return

        old_status = self.get_doc_before_save()
        if not old_status:
            return
        old_status = old_status.status

        VALID_TRANSITIONS = {
            "Draft":        ["Pending", "Cancelled"],
            "Pending":      ["Under Review", "Rejected", "Cancelled"],
            "Under Review": ["Approved", "Rejected", "Cancelled"],
            "Approved":     [],
            "Rejected":     [],
            "Cancelled":    [],
        }

        allowed = VALID_TRANSITIONS.get(old_status, [])
        if self.status not in allowed:
            frappe.throw(
                _("Cannot change status from {0} to {1}. "
                  "Allowed transitions: {2}").format(
                    old_status, self.status,
                    ", ".join(allowed) or "None (terminal state)"
                )
            )

    # validate_seller_authorization() — see Section 4.1
    # validate_product_brand_match() — see Section 4.2
    # validate_parent_product_status() — see Section 4.3
    # validate_tenant_isolation() — follows Product Variant pattern

    # =========================================================================
    # STATUS CHANGE HANDLING
    # =========================================================================

    def handle_status_change(self):
        """Handle side-effects of status changes."""
        if self.status == "Pending":
            self.update_demand_aggregation()
            self.send_notification("submitted")

        elif self.status == "Under Review":
            self.reviewed_by = frappe.session.user
            self.send_notification("under_review")

        elif self.status == "Approved":
            self.reviewed_by = self.reviewed_by or frappe.session.user
            self.review_date = self.review_date or today()
            self.create_product_variant()
            self.update_demand_aggregation()
            self.send_notification("approved")

        elif self.status == "Rejected":
            self.reviewed_by = self.reviewed_by or frappe.session.user
            self.review_date = self.review_date or today()
            self.update_demand_aggregation()
            self.send_notification("rejected")

        elif self.status == "Cancelled":
            self.update_demand_aggregation()
            self.send_notification("cancelled")

    # =========================================================================
    # DEMAND AGGREGATION
    # =========================================================================

    # generate_demand_group_key() — see Section 5.2
    # normalize_attribute_value() — see Section 5.2
    # update_demand_aggregation() — see Section 5.4
    # check_demand_threshold_notification() — see Section 8.3

    # =========================================================================
    # AUTO-CATALOG CREATION
    # =========================================================================

    # create_product_variant() — see Section 7.3
    # _find_existing_variant() — see Section 7.3
    # _link_demand_group_to_variant() — see Section 7.3

    # =========================================================================
    # NOTIFICATIONS
    # =========================================================================

    # send_notification() — see Section 8.2
    # _get_notification_recipients() — see Section 8.2
    # _get_notification_content() — see Section 8.2

    # =========================================================================
    # TENANT ISOLATION
    # =========================================================================

    def validate_tenant_isolation(self):
        """
        Validate multi-tenant isolation.

        Follows same pattern as Product Variant:
        - Tenant is inherited from requesting seller
        - System Manager can access all tenants
        - Other users can only access their own tenant's data
        """
        if not self.tenant and self.requesting_seller:
            self.tenant = frappe.db.get_value(
                "Seller Profile", self.requesting_seller, "tenant"
            )

        if not self.tenant:
            frappe.throw(_("Variant Request must belong to a tenant"))

        if "System Manager" in frappe.get_roles():
            return

        from tradehub_core.tradehub_core.utils.tenant import get_current_tenant
        current_tenant = get_current_tenant()

        if current_tenant and self.tenant != current_tenant:
            frappe.throw(
                _("Access denied: You can only access variant requests in your tenant")
            )
```

## 10. Multi-Tenant Isolation

### 10.1 Tenant Field Design

Following the established pattern from Product Variant and Brand Gating:

| Aspect | Implementation |
|--------|---------------|
| **Tenant source** | `fetch_from=requesting_seller.tenant` (inherited from Seller Profile) |
| **Read-only** | Yes — sellers cannot change tenant |
| **In standard filter** | Yes — enables admin filtering by tenant |
| **Validation** | `validate_tenant_isolation()` checks current user's tenant matches document tenant |
| **System Manager bypass** | System Manager can access all tenants |

### 10.2 Permission Query Conditions

```python
def get_permission_query_conditions(user):
    """
    SQL conditions for multi-tenant list filtering.

    Combines tenant isolation with brand owner access.
    """
    if "System Manager" in frappe.get_roles(user):
        return ""

    from tradehub_core.tradehub_core.utils.tenant import get_current_tenant
    current_tenant = get_current_tenant()

    seller = get_seller_profile_for_user(user)

    conditions = []

    if current_tenant:
        conditions.append(
            f"`tabVariant Request`.tenant = '{frappe.db.escape(current_tenant)}'"
        )

    if seller:
        # Own requests
        conditions.append(
            f"`tabVariant Request`.requesting_seller = '{frappe.db.escape(seller)}'"
        )
        # Brand owner requests
        conditions.append(
            f"""`tabVariant Request`.brand IN (
                SELECT name FROM `tabBrand` WHERE owner_seller = '{frappe.db.escape(seller)}'
            )"""
        )

    if conditions:
        return "(" + " OR ".join(conditions) + ")"

    return "1=0"  # No access
```

|----------|----------|

```python
@frappe.whitelist()
def get_demand_groups(brand=None, min_count=1):
    """
    Get aggregated demand groups for variant requests.

    Args:
        brand: Optional brand filter
        min_count: Minimum request count to include (default 1)

    Returns:
        list: Demand groups with aggregated data
    """
    filters = {
        "status": ("in", ["Pending", "Under Review"]),
        "demand_group_key": ("is", "set"),
    }
    if brand:
        filters["brand"] = brand

    groups = frappe.db.sql("""
        SELECT
            demand_group_key,
            brand,
            brand_name,
            parent_product,
            parent_product_name,
            requested_color,
            requested_size,
            requested_material,
            requested_packaging,
            COUNT(*) as request_count,
            GROUP_CONCAT(DISTINCT seller_name ORDER BY seller_name) as sellers,
            MIN(request_date) as first_requested,
            SUM(estimated_demand_quantity) as total_estimated_demand
        FROM `tabVariant Request`
        WHERE status IN ('Pending', 'Under Review')
          AND demand_group_key IS NOT NULL
          AND demand_group_key != ''
          {brand_filter}
        GROUP BY demand_group_key
        HAVING request_count >= %(min_count)s
        ORDER BY request_count DESC, first_requested ASC
    """.format(
        brand_filter=f"AND brand = %(brand)s" if brand else ""
    ), {
        "brand": brand,
        "min_count": cint(min_count)
    }, as_dict=True)

    return groups
```

```javascript
                            fieldname: "reason", label: __("Rejection Reason"),
                            fieldtype: "Select", reqd: 1,
                            options: ["Variant Already Exists", "Insufficient Demand",
                                      "Brand Policy", "Technical Infeasibility",
                                      "Duplicate Request", "Other"]
                        },
                        { fieldname: "notes", label: __("Notes"), fieldtype: "Small Text" }
                    ],
                    primary_action: function() {
                        frappe.call({
                            method: "tradehub_catalog.tradehub_catalog.doctype.variant_request.variant_request.reject_variant_request",
                            args: {
                                name: frm.doc.name,
                                reason: d.get_value("reason"),
                                notes: d.get_value("notes")
                            },
                            callback: function() { d.hide(); frm.reload_doc(); }
                        });
                    }
                });
                d.show();
            }, __("Actions"));
        }

        // Show demand signal indicator
        if (frm.doc.demand_request_count > 1) {
            frm.dashboard.add_indicator(
                __("{0} sellers requesting this variant", [frm.doc.demand_request_count]),
                frm.doc.demand_request_count >= 5 ? "red" : "orange"
            );
        }

        // Show created variant link
        if (frm.doc.created_variant) {
            frm.dashboard.add_indicator(
                __("Variant Created: {0}", [frm.doc.created_variant_code || frm.doc.created_variant]),
                "green"
            );
        }
    },

    requesting_seller: function(frm) {
        frm.set_value("brand", "");
        frm.set_value("parent_product", "");
    },

    brand: function(frm) {
        frm.set_value("parent_product", "");
    }
});
```

### 13.2 Daily Scheduled Task: Recalculate Demand Aggregation

```python
def recalculate_demand_aggregation():
    """
    Daily job to recalculate demand aggregation for all active demand groups.

    This handles edge cases where counts might drift (e.g., if a request
    was cancelled but the aggregation wasn't updated due to a bug).
    """
    # Get all unique demand group keys with active requests
    groups = frappe.db.sql("""
        SELECT DISTINCT demand_group_key
        FROM `tabVariant Request`
        WHERE demand_group_key IS NOT NULL
          AND demand_group_key != ''
          AND status IN ('Pending', 'Under Review', 'Draft')
    """, as_dict=True)

    for group in groups:
        key = group.demand_group_key

        # Recalculate
        requests = frappe.get_all(
            "Variant Request",
            filters={
                "demand_group_key": key,
                "status": ("not in", ["Cancelled", "Rejected"])
            },
            fields=["name", "seller_name", "request_date"],
            order_by="request_date asc"
        )

        count = len(requests)
        sellers = ", ".join(sorted(set(r.seller_name for r in requests if r.seller_name)))
        first_date = requests[0].request_date if requests else None

        # Update all requests in group
        for req in requests:
            frappe.db.set_value("Variant Request", req.name, {
                "demand_request_count": count,
                "demand_sellers_summary": sellers,
                "demand_first_requested": first_date,
            }, update_modified=False)

    frappe.db.commit()
```

## 14. Edge Cases and Error Handling

### 14.1 Duplicate Request Detection

```python
def validate_duplicate_request(self):
    """
    Check if the seller already has an active request for same variant on same product.

    Active = status in (Draft, Pending, Under Review, Approved)
    """
    if not self.requesting_seller or not self.parent_product:
        return

    filters = {
        "requesting_seller": self.requesting_seller,
        "parent_product": self.parent_product,
        "status": ("not in", ["Cancelled", "Rejected"]),
        "name": ("!=", self.name or ""),
    }

    # Check exact attribute match
    if self.requested_color:
        filters["requested_color"] = self.requested_color
    if self.requested_size:
        filters["requested_size"] = self.requested_size
    if self.requested_material:
        filters["requested_material"] = self.requested_material
    if self.requested_packaging:
        filters["requested_packaging"] = self.requested_packaging

    existing = frappe.db.get_value("Variant Request", filters,
        ["name", "status"], as_dict=True)

    if existing:
        frappe.throw(
            _("You already have an active variant request ({0}, status: {1}) "
              "for the same product and attributes. "
              "Please wait for the existing request to be processed.").format(
                existing.name, existing.status
            )
        )
```

### 14.2 Edge Case Matrix

| # | Scenario | Handling | Severity |
|---|----------|----------|----------|
| 1 | Seller's Brand Gating is revoked after request creation | Existing requests remain active; validation only checks on create/validate, not on approval | Low |
| 2 | Parent product archived after request submission | `validate_parent_product_status()` blocks on every save; admin must manually reject | Medium |
| 3 | Brand owner changes between request and approval | New owner sees existing requests; permission query uses current `owner_seller` | Low |
| 4 | Two sellers request same variant simultaneously | Both get individual requests; demand aggregation links them after submission | Low |
| 5 | Variant Request approved but Product Variant creation fails | Error logged in `auto_creation_log`; request stays Approved but `created_variant` is empty; admin can retry | Medium |
| 6 | Brand Gating expired between request submission and review | Request remains valid; only validated on initial creation | Low |
| 7 | Same seller submits request for variant that already exists as Product Variant | `_find_existing_variant()` links to existing variant on approval instead of creating duplicate | Low |
| 8 | Bulk approve with one request in wrong status | Per-request error handling; failed requests listed in response; others proceed | Low |
| 9 | Demand group has requests from multiple tenants | Aggregation is cross-tenant by design; brand owner sees all requests for their brand | Design |
| 10 | Request cancelled after it was the only one in demand group | Demand count drops to 0; remaining group members (if any) get updated counts | Low |
