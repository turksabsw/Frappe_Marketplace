## 1. Overview

### 1.1 Problem Statement

In a B2B marketplace where multiple buyers compete for the same inventory and sellers need protection against non-paying buyers, temporal controls are essential to maintain platform integrity. Without protection mechanisms:

- **Buyers can hold inventory hostage** by starting checkout and never completing payment, blocking other buyers from purchasing.
- **Sellers lose revenue** when buyers abandon orders after stock has been reserved, especially during high-demand periods.
- **Price changes go unnoticed** by buyers who leave items in carts for days, leading to disputes when they finally checkout at an outdated price.
- **Abusive buyers** repeatedly place orders and cancel them, disrupting seller operations, wasting warehouse picking/packing resources, and gaming promotions.
- **Stale carts accumulate** with out-of-stock or deactivated products, causing checkout failures and poor buyer experience.

### 1.2 Scope

This document defines:

1. **Buyer-side protection** - Stock reservation at checkout, countdown timer, stock release on expiry, price change detection
2. **Seller-side protection** - No pre-checkout stock lock, payment deadlines, auto-cancellation, quantity limits, abuse detection
3. **Cart health check** - Scheduled job for periodic cart validation (stock, price, product status)
4. **Cart Protection Settings DocType** - Platform-wide configuration with per-seller overrides
5. **Stock Reservation DocType** - Temporary inventory hold records
6. **Buyer Abuse Log DocType** - Tracking cancellation patterns and abuse incidents
7. **Scheduler integration** - hooks.py additions for automated temporal checks

### 1.4 Design Principles

1. **No stock lock before checkout** - Browsing and adding to cart does NOT reserve inventory. Stock is only reserved when the buyer explicitly initiates checkout.
2. **Time-bounded reservations** - All stock reservations have a finite, configurable duration. No indefinite holds.
3. **Seller autonomy** - Sellers can override platform defaults for reservation duration, payment deadlines, and quantity limits.
4. **Graceful degradation** - When reservations expire, buyers receive clear warnings and can re-reserve if stock is still available.
5. **Abuse detection, not punishment** - The system detects patterns and escalates to human review rather than auto-banning buyers.
6. **Atomic operations** - Stock reservation and release use database transactions to prevent race conditions.

## 2. Buyer-Side Protection

### 2.1 Checkout Stock Reservation

#### 2.1.1 Reservation Trigger

Stock reservation is triggered when the buyer transitions the cart from `Active` to `Checkout` status via `initiate_checkout()` (defined in 123.md).

```
Trigger Point: Cart.status changes from "Active" to "Checkout"
              (i.e., buyer clicks "Proceed to Checkout")

Pre-conditions:
  1. Cart has at least one item
  2. Cart is not expired
  3. No existing active (non-expired, non-cancelled) Checkout Session for this Cart
  4. All items pass stock availability check (qty <= available_stock)

Reservation Flow:
  1. For each Cart Line item:
     a. Check current available stock for the item's warehouse
     b. If available_stock >= cart_line.qty:
        - Create Stock Reservation record
        - Deduct reserved_qty from available stock (logical hold, not physical)
     c. If available_stock < cart_line.qty:
        - FAIL: Return error with item details and available quantity
        - Offer buyer to adjust quantity to available amount
  2. Set Cart.checkout_started_at = now()
  3. Set Cart.status = "Checkout"
  4. Create Checkout Session with expires_at = now() + reservation_duration
  5. Return reservation confirmation with countdown timer value
```

#### 2.1.3 Available Stock Calculation with Reservations

The available stock calculation must account for active reservations:

```
available_stock(item, warehouse) =
    actual_stock(item, warehouse)                     # From ERPNext Bin / Stock Ledger
  - SUM(Stock Reservation.reserved_qty)               # All active reservations
    WHERE Stock Reservation.item_code = item
    AND Stock Reservation.warehouse = warehouse
    AND Stock Reservation.status = "Active"
```

**SQL Query for Available Stock:**

```sql
SELECT
    bin.actual_qty - COALESCE(res.total_reserved, 0) AS available_qty
FROM `tabBin` bin
LEFT JOIN (
    SELECT item_code, warehouse, SUM(reserved_qty) AS total_reserved
    FROM `tabStock Reservation`
    WHERE status = 'Active'
    GROUP BY item_code, warehouse
) res ON res.item_code = bin.item_code AND res.warehouse = bin.warehouse
WHERE bin.item_code = %(item_code)s
AND bin.warehouse = %(warehouse)s
```

**Python Method:**


#### 2.1.4 Reservation Concurrency Handling

When multiple buyers attempt to reserve the same item simultaneously:

```
Strategy: Pessimistic Locking with SELECT ... FOR UPDATE

1. BEGIN TRANSACTION
2. SELECT actual_qty FROM `tabBin`
   WHERE item_code = :item AND warehouse = :warehouse
   FOR UPDATE
3. SELECT SUM(reserved_qty) FROM `tabStock Reservation`
   WHERE item_code = :item AND warehouse = :warehouse AND status = 'Active'
   FOR UPDATE
4. Calculate available = actual_qty - total_reserved
5. IF available >= requested_qty:
     INSERT Stock Reservation (status='Active', reserved_qty=requested_qty, ...)
     COMMIT
   ELSE:
     ROLLBACK
     RAISE InsufficientStockError(available_qty=available)
```

**Race Condition Prevention:**

- `FOR UPDATE` locks prevent two concurrent transactions from both seeing the same available stock
- If Transaction A reserves 50 of 100 available, Transaction B will see only 50 available
- Lock timeout: 5 seconds (configurable). If lock cannot be acquired, return a retry-able error

### 2.2 Countdown Timer

#### 2.2.1 Timer Configuration

The countdown timer tells the buyer how long their stock reservation will last. The timer is visible on the checkout page and decrements in real-time.

```
Timer Duration Resolution:
  1. Check seller-specific override: Seller Cart Protection Override.checkout_reservation_minutes
  2. If no seller override: Use Cart Protection Settings.checkout_reservation_minutes (platform default)
  3. If no platform setting: Hard-coded fallback = 30 minutes

Per-Seller Resolution (multi-seller cart):
  When a cart has items from multiple sellers with different reservation durations:
  - Each Seller Cart Group has its OWN timer based on the seller's override
  - The checkout UI shows per-group timers
  - The overall checkout session expires at MIN(all group timers)
    (i.e., the earliest expiring group determines the session timeout)
```

#### 2.2.2 Timer Display

**Frontend Implementation Plan:**

```javascript
    //   Orange: 2-5 minutes remaining
    //   Red:    < 2 minutes remaining (pulse animation)

    //   First warning:  5 minutes remaining -> toast notification
    //   Final warning:  2 minutes remaining -> modal dialog with "Extend" option
    //   Expiry:         0 seconds           -> redirect to cart with expiry message

```

**Timer Extension:**

- Buyers can request a one-time extension when the timer warning appears
- Extension duration = 50% of original duration (e.g., 30min timer -> 15min extension)
- Maximum extensions per checkout: configurable (default: 1)
- Extension requires re-checking stock availability for all reserved items
- If stock is no longer available during extension attempt, the specific item's reservation fails

### 2.3 Stock Release on Expiry

#### 2.3.1 Expiry Detection

Stock reservations are released when the checkout timer expires. Detection happens via two mechanisms:

1. **Scheduled job** (`check_expired_reservations`) - Runs every 5 minutes to catch all expired reservations
2. **On-demand check** - When the buyer interacts with the checkout page, the API checks if the session has expired

#### 2.3.2 Release Process

```
Trigger: Stock Reservation.expires_at <= now() AND Stock Reservation.status = "Active"

Release Flow:
  1. Identify all expired active reservations (batch query)
  2. For each expired reservation:
     a. Set status = "Expired"
     b. Set released_at = now()
     c. Set notes = "Auto-released: reservation timer expired"
     d. The reserved_qty is now available for other buyers
  3. Group expired reservations by Checkout Session
  4. For each affected Checkout Session:
     a. If ALL reservations in the session are expired:
        - Set Checkout Session.status = "Expired"
        - Set Cart.status = "Active" (revert from Checkout)
        - Clear Cart.checkout_started_at
     b. If SOME reservations expired (multi-seller):
        - Mark expired seller groups as "Expired" in Checkout Session
        - Notify buyer about which seller groups lost reservation
        - Buyer can still complete checkout for non-expired groups
  5. Send notification to buyer (in-app + optional email)
```

#### 2.3.3 Buyer Notification on Expiry

```
Notification Template: "checkout_reservation_expired"

Subject: "Your checkout reservation has expired"

Body:
  "Your checkout reservation for the following items has expired:
   - [Item Name] x [Qty] from [Seller Name]
```

### 2.4 Price Change Detection

#### 2.4.1 Price Snapshot at Reservation

When stock is reserved, the current price is captured:

```
On Stock Reservation creation:
  reservation.reserved_price = get_current_price(item_code, seller, buyer)
  reservation.currency = cart.currency
```

#### 2.4.2 Price Change Check

Price changes are detected at two points:

1. **During checkout** - When the buyer clicks "Pay" for a seller group
2. **By cart health check** - Scheduled job that periodically checks all active reservations

```
Price Check Algorithm:
  1. For each active Stock Reservation:
     current_price = get_current_price(reservation.item_code, reservation.seller, reservation.buyer)
     if current_price != reservation.reserved_price:
       reservation.current_price = current_price
       reservation.price_changed = 1
       reservation.save()
  2. Group price-changed reservations by Cart / Checkout Session
  3. Calculate impact:
     price_difference = (current_price - reserved_price) * reserved_qty
     If price_difference > 0: "Price increased by {amount}"
     If price_difference < 0: "Price decreased by {amount} - you save more!"
  4. Notify buyer with option to:
     a. Accept new price and continue checkout
     b. Cancel checkout and return to cart
```

#### 2.4.3 Price Change Policy

```
Policy Options (configured in Cart Protection Settings):

1. "block_on_increase" (default):
   - Price increase: Block payment, show warning, require buyer acknowledgment
   - Price decrease: Auto-apply new lower price, show savings notification
   - Recalculate cart totals after acknowledgment

2. "always_use_reserved_price":
   - Honor the price at time of reservation regardless of changes
   - Seller absorbs any increase; buyer does not benefit from decrease
   - Best for B2B contracts with quoted prices

3. "always_use_current_price":
   - Always charge current price at time of payment
   - Show price change notification but do not block
   - Recalculate cart totals automatically
```

## 3. Seller-Side Protection

### 3.1 No Stock Lock Before Checkout

**Principle:** Adding items to a cart does NOT reserve or lock any inventory. Stock is only held when the buyer explicitly initiates checkout.

```
Cart Operations (NO stock reservation):
  - add_to_cart(item, qty)        -> No reservation. Just adds to Cart Line.
  - update_cart_item(item, qty)   -> No reservation. Just updates Cart Line.qty.
  - remove_from_cart(item)        -> No reservation to release.

Checkout Operations (WITH stock reservation):
  - initiate_checkout()           -> Creates Stock Reservation for all cart items
  - cancel_checkout()             -> Releases all Stock Reservations
  - complete_payment()            -> Converts Stock Reservations to "Consumed"
```

**Rationale:**

- In B2B marketplaces, buyers often build carts over days or weeks before purchasing
- Locking stock at cart-add time would paralyze inventory for other buyers
- This matches the pattern used by AliExpress, Amazon, and Alibaba

### 3.2 Post-Order Payment Deadline

#### 3.2.1 Payment Deadline Mechanism

After an order is placed with an offline payment method (Bank Transfer, Letter of Credit, etc.), the buyer has a configurable deadline to complete payment. If payment is not received by the deadline, the order is automatically cancelled.

```
Payment Deadline Resolution:
  1. Check seller-specific override: Seller Cart Protection Override.payment_deadline_hours
  2. If no seller override: Use Cart Protection Settings.payment_deadline_hours (platform default)
  3. If no platform setting: Hard-coded fallback = 24 hours

Applicable Order Statuses:
  - "Await Payment" (primary - order placed, payment pending)
  - "Pending" (if order requires manual confirmation before payment)

NOT Applicable When:
  - Payment method is instant (Credit Card, Digital Wallet) - payment is already processing
  - Order status is "Payment Received" or beyond
  - Order is B2B with Credit Terms (separate payment schedule from 117.md)
```

### 3.3 Auto-Cancellation on Payment Timeout

#### 3.3.1 Auto-Cancel Job

A scheduled job checks for orders that have exceeded their payment deadline:

```python
# tradehub_commerce/tasks/cart_protection.py

def auto_cancel_overdue_orders():
    """Cancel orders where payment deadline has passed.

    Runs: Every 15 minutes (via scheduler_events cron)
    Scope: All Marketplace Orders with:
        - status IN ("Await Payment", "Pending")
        - payment_deadline IS NOT NULL
        - payment_deadline < NOW()
        - auto_cancel_scheduled = 1

    Actions per order:
        1. Set status = "Cancelled"
        2. Set payment_overdue = 1
        3. Set cancellation_reason = "Payment deadline exceeded"
        4. Set cancelled_by = "System"
        5. Release any active Stock Reservations for this order
        6. Update Sub Order statuses
        7. Notify buyer: "Order cancelled due to non-payment"
        8. Notify seller: "Order {id} auto-cancelled - payment not received within {hours}h"
        9. Log to Buyer Abuse Log (for pattern tracking)
    """
```

#### 3.3.2 Grace Period

Before auto-cancellation, a grace period warning is sent:

```
Grace Period Flow:
  1. At 75% of deadline (e.g., 18h of 24h):
     - Send "Payment Reminder" notification to buyer
     - Subject: "Payment reminder for order {order_id}"
     - Body: "You have {remaining_hours} hours to complete payment."

  2. At 90% of deadline (e.g., 21.6h of 24h):
     - Send "Final Payment Warning" notification to buyer
     - Subject: "Urgent: Order {order_id} will be cancelled in {remaining_hours} hours"
     - Body: "This is your final reminder. Complete payment now to avoid cancellation."

  3. At 100% of deadline:
     - Execute auto-cancellation
     - No additional warning - buyer was already notified twice
```

### 3.4 Maximum Quantity Limits

#### 3.4.1 Per-Item Quantity Limits

Sellers can set maximum order quantities to prevent inventory hoarding:

```
Quantity Limit Resolution:
  1. Check item-level limit: Listing.max_order_qty (if set on the listing)
  2. Check seller-level limit: Seller Cart Protection Override.default_max_qty_per_item
  3. Check platform limit: Cart Protection Settings.default_max_qty_per_item
  4. If no limit set: No restriction (unlimited)

Enforcement Points:
  - add_to_cart(): Validate qty <= max_qty_per_item
  - update_cart_item(): Validate new qty <= max_qty_per_item
  - initiate_checkout(): Final validation of all item quantities
```

#### 3.4.2 Cart-Level Limits

```
Cart Limits (from Cart Protection Settings or Seller override):

  1. max_items_per_cart:
     - Maximum number of distinct line items in a single cart
     - Default: 100
     - Enforced at add_to_cart()

  2. max_total_qty_per_cart:
     - Maximum total quantity across all items in a cart
     - Default: 10000
     - Enforced at add_to_cart() and update_cart_item()

  3. max_carts_per_buyer:
     - Maximum number of active (non-expired, non-converted) carts per buyer
     - Default: 5
     - Enforced at create_cart()

  4. max_reservations_per_buyer:
     - Maximum number of concurrent stock reservations per buyer
     - Default: 3 (i.e., buyer can have at most 3 active checkout sessions)
     - Enforced at initiate_checkout()
```

### 3.5 Buyer Abuse Detection

#### 3.5.1 Abuse Patterns

The system monitors for the following buyer behavior patterns:

| Pattern | Description | Detection Method | Default Threshold |
|---------|-------------|-----------------|-------------------|
| **Repeat Cancellation** | Buyer repeatedly places orders and cancels | Count cancellations in rolling window | 3 cancellations in 30 days |
| **Payment Timeout** | Buyer repeatedly lets payment deadlines expire | Count auto-cancellations in rolling window | 3 timeouts in 30 days |
| **Reservation Hoarding** | Buyer reserves stock and lets it expire without payment | Count expired reservations in rolling window | 5 expired reservations in 7 days |
| **Cart Bombing** | Buyer creates excessive carts with large quantities | Count carts created + total qty in rolling window | 10 carts with >1000 total qty in 24 hours |
| **Dispute Abuse** | Buyer opens excessive disputes as leverage | Count disputes in rolling window | 5 disputes in 60 days |

#### 3.5.3 Abuse Detection Algorithm

```python
def detect_buyer_abuse(buyer, event_type, event_details):
    """Check if a buyer's action triggers abuse detection.

    Called after:
        - Order cancellation (by buyer)
        - Payment deadline expiry (auto-cancel)
        - Stock reservation expiry
        - Cart creation (when excessive)
        - Dispute creation

    Algorithm:
        1. Get the buyer's abuse detection thresholds:
           a. Check buyer-specific overrides (if whitelisted as high-volume buyer)
           b. Check Cart Protection Settings platform defaults
        2. Query incident count in rolling window:
           SELECT COUNT(*) FROM (
               -- Count relevant events based on event_type
               -- e.g., for Repeat Cancellation:
               SELECT name FROM `tabMarketplace Order`
               WHERE buyer = :buyer
               AND status = 'Cancelled'
               AND cancelled_by_role = 'Buyer'
               AND modified >= DATE_SUB(NOW(), INTERVAL :window_days DAY)
           )
        3. If count >= threshold:
           a. Create Buyer Abuse Log entry
           b. Determine severity:
              - Low:      count = threshold (just crossed)
              - Medium:   count = threshold * 1.5
              - High:     count = threshold * 2
              - Critical: count = threshold * 3 OR financial impact > threshold
           c. Apply immediate action based on severity:
              - Low:      Log only, notify marketplace admin
              - Medium:   Log + send warning to buyer
              - High:     Log + restrict checkout (require admin approval)
              - Critical: Log + suspend buyer account pending review
           d. Notify affected sellers
        4. Return abuse detection result
    """
```

#### 3.5.4 False Positive Handling

To prevent legitimate high-volume B2B buyers from being falsely flagged:

```
False Positive Mitigation:
  1. Buyer Whitelist:
     - Platform admins can whitelist specific buyers (Buyer Abuse Whitelist)
     - Whitelisted buyers have elevated thresholds (e.g., 10x default)
     - Applied to verified B2B accounts with good payment history

  2. Buyer Score Integration:
     - Factor in buyer's historical completion rate:
       completion_rate = completed_orders / total_orders
     - If completion_rate > 90%, increase thresholds by 2x
     - If completion_rate > 95%, increase thresholds by 3x

  3. Legitimate Cancellation Reasons:
     - Track cancellation reasons; "seller_requested" and "item_unavailable"
       do not count toward buyer abuse
     - Only "buyer_changed_mind", "found_cheaper", and no-reason cancellations count

  4. Review Before Action:
     - Severity "Low" and "Medium" never auto-restrict buyers
     - Only "High" and "Critical" trigger checkout restrictions
     - All restrictions require admin review within 24 hours
     - If not reviewed in 24h, restriction is auto-lifted (safe default)
```

```python
def cart_health_check():
    """Comprehensive health check for all active carts.

    Frequency: Hourly
    Performance Target: < 60 seconds for up to 50,000 active carts

    Checks performed on each active cart:

    1. OUT-OF-STOCK CHECK:
       For each Cart Line:
         current_stock = get_available_stock(item_code, warehouse)
         if current_stock <= 0:
           -> Flag item as out_of_stock
           -> Add to notification queue

    2. LOW-STOCK WARNING:
       For each Cart Line:
         if current_stock < cart_line.qty AND current_stock > 0:
           -> Flag item as insufficient_stock
           -> Record available_qty
           -> Add to notification queue

    3. PRICE CHANGE CHECK:
       For each Cart Line:
         current_price = get_current_price(item_code, seller)
         if current_price != cart_line.unit_price:
           -> Flag item with price_changed
           -> Record old_price and new_price
           -> Add to notification queue

    4. DEACTIVATED PRODUCT CHECK:
       For each Cart Line:
         listing = frappe.get_value("Listing", cart_line.listing, "status")
         if listing != "Active":
           -> Flag item as product_unavailable
           -> Record reason (deactivated, deleted, suspended)
           -> Add to notification queue

    5. SELLER STATUS CHECK:
       For each unique seller in cart:
         seller_status = frappe.get_value("Seller Profile", seller, "status")
         if seller_status NOT IN ("Active", "Verified"):
           -> Flag all items from this seller
           -> Add to notification queue

    6. ABANDONED CART DETECTION:
       if cart.last_activity < NOW() - INTERVAL abandoned_cart_hours HOUR:
         AND cart.abandoned_at IS NULL:
           -> Set cart.abandoned_at = NOW()
           -> Set cart.status = "Abandoned"
           -> Queue recovery email (if Cart Protection Settings.send_recovery_email = 1)

    Notification Aggregation:
       - Group all issues per cart into a single notification
       - Template: "cart_health_issues"
       - Channels: in-app notification, optional email
       - Don't notify for the same issue twice within 24 hours (dedup)
    """
```

### 6.3 Override Resolution Logic

```python
def get_protection_setting(setting_name, seller=None):
    """Resolve a protection setting value with seller override support.

    Resolution order:
        1. If seller is specified, check Seller Cart Protection Override
           - Find row where seller = given seller AND enabled = 1
           - If row exists AND the specific setting value > 0 (or non-empty for Select):
             Return seller override value
        2. Return platform default from Cart Protection Settings

    Args:
        setting_name: Field name in Cart Protection Settings
                      (e.g., "checkout_reservation_minutes", "payment_deadline_hours")
        seller: Optional Seller Profile name for per-seller lookup

    Returns:
        The resolved setting value (seller override or platform default)

    Example:
        # Platform default: 30 minutes
        # Seller "SLRP-00042" override: 45 minutes
        get_protection_setting("checkout_reservation_minutes")
        # Returns: 30

        get_protection_setting("checkout_reservation_minutes", seller="SLRP-00042")
        # Returns: 45

        get_protection_setting("checkout_reservation_minutes", seller="SLRP-00099")
        # Returns: 30 (no override for this seller)
    """
    settings = frappe.get_single("Cart Protection Settings")

    if seller:
        for override in settings.seller_overrides:
            if override.seller == seller and override.enabled:
                override_value = getattr(override, setting_name, None)
                if override_value:
                    # For Int fields, 0 means "use platform default"
                    if isinstance(override_value, int) and override_value == 0:
                        break
                    # For Select fields, empty string means "use platform default"
                    if isinstance(override_value, str) and not override_value:
                        break
                    return override_value
                break

    return getattr(settings, setting_name)
```

### 6.4 Multi-Seller Cart Override Resolution

When a cart contains items from multiple sellers, each with different override settings:

```
Scenario: Cart with items from Seller A (override: 45min) and Seller B (no override, platform: 30min)

Checkout Reservation:
  - Seller A's items: reserved for 45 minutes
  - Seller B's items: reserved for 30 minutes
  - Checkout Session expires_at: MIN(45, 30) = 30 minutes from now
  - However, Seller A's specific Stock Reservations expire in 45 minutes
  - The UI shows per-group timers:
    "Seller A: 44:59 remaining"
    "Seller B: 29:59 remaining"
  - Overall session timer: "Session expires in: 29:59"

Payment Deadline (post-order):
  - Each Marketplace Order (one per seller) gets its own deadline
  - Seller A's order: 48-hour deadline (if seller override)
  - Seller B's order: 24-hour deadline (platform default)
  - Independent auto-cancellation per order

Quantity Limits:
  - Per-item limits are seller-specific
  - Cart-level limits use the most restrictive applicable limit
```

### 6.5 Validation Rules

```python
def validate_seller_override(doc, method=None):
    """Validate seller override values against platform min/max boundaries.

    Called on Cart Protection Settings save.

    Rules:
        - checkout_reservation_minutes must be between min_reservation_minutes
          and max_reservation_minutes (platform settings)
        - payment_deadline_hours must be between min_payment_deadline_hours
          and max_payment_deadline_hours
        - No duplicate sellers in the override table
        - Seller must exist and be active
    """
    settings = doc  # Cart Protection Settings instance
    seen_sellers = set()

    for override in settings.seller_overrides:
        # No duplicate sellers
        if override.seller in seen_sellers:
            frappe.throw(
                f"Duplicate seller override for {override.seller_name or override.seller}. "
                "Each seller can have at most one override row."
            )
        seen_sellers.add(override.seller)

        # Validate reservation duration bounds
        if override.checkout_reservation_minutes:
            if override.checkout_reservation_minutes < settings.min_reservation_minutes:
                frappe.throw(
                    f"Seller {override.seller_name}: Reservation duration "
                    f"({override.checkout_reservation_minutes} min) is below platform minimum "
                    f"({settings.min_reservation_minutes} min)."
                )
            if override.checkout_reservation_minutes > settings.max_reservation_minutes:
                frappe.throw(
                    f"Seller {override.seller_name}: Reservation duration "
                    f"({override.checkout_reservation_minutes} min) exceeds platform maximum "
                    f"({settings.max_reservation_minutes} min)."
                )

        # Validate payment deadline bounds
        if override.payment_deadline_hours:
            if override.payment_deadline_hours < settings.min_payment_deadline_hours:
                frappe.throw(
                    f"Seller {override.seller_name}: Payment deadline "
                    f"({override.payment_deadline_hours} hrs) is below platform minimum "
                    f"({settings.min_payment_deadline_hours} hrs)."
                )
            if override.payment_deadline_hours > settings.max_payment_deadline_hours:
                frappe.throw(
                    f"Seller {override.seller_name}: Payment deadline "
                    f"({override.payment_deadline_hours} hrs) exceeds platform maximum "
                    f"({settings.max_payment_deadline_hours} hrs)."
                )
```

## 9. Edge Cases

### 9.1 Timer Edge Cases

#### 9.1.1 Checkout at 29:59 of Reservation

**Scenario:** Buyer starts the checkout process, fills in shipping/billing details, and clicks "Pay" at 29:59 of a 30-minute reservation (1 second before expiry).

**Handling:**
```
Payment Submission Validation:
  1. At payment submission time, check:
     IF Stock Reservation.expires_at > NOW():
       -> Proceed with payment processing
       -> Lock the reservation (prevent expiry during payment processing)
       -> Set reservation.status = "Processing" (new transitional state)
     ELSE:
       -> Return error: "Your checkout reservation has expired"
       -> Redirect buyer to cart with offer to re-reserve

  2. "Processing" state behavior:
     - The check_expired_reservations job SKIPS reservations in "Processing" state
     - Processing state has its own timeout: 5 minutes (for payment gateway response)
     - If payment succeeds within 5 min: status -> "Consumed"
     - If payment fails or times out: status -> "Expired", release stock
```

**Key Principle:** Once the buyer clicks "Pay" before expiry, the system honors the reservation through the payment processing window. The payment gateway response time should not penalize the buyer.

#### 9.1.2 Payment at 23:59:59 of Deadline

**Scenario:** Buyer submits payment via bank transfer at 23:59:59 of a 24-hour deadline. The payment webhook arrives 2 minutes later (at 24:01:59).

**Handling:**
```
Payment Deadline with Gateway Check:
  1. auto_cancel_overdue_orders job finds the order at 24:01:00
  2. Before cancelling, if check_payment_gateway_before_cancel = 1:
     a. Query payment gateway for pending transactions matching this order
     b. If gateway reports "processing" or "pending" transaction:
        -> DO NOT cancel
        -> Set order.notes = "Auto-cancel deferred: payment in progress at gateway"
        -> Extend deadline by 1 hour (grace for gateway processing)
     c. If gateway reports no pending transaction:
        -> Proceed with auto-cancellation
  3. When the webhook arrives at 24:01:59:
     a. If order was not cancelled: process normally
     b. If order was cancelled: attempt to reverse cancellation
        -> If stock is still available: reinstate order, apply payment
        -> If stock is sold: refund payment, notify buyer
```

### 9.2 Concurrent Reservation Conflicts

**Scenario:** Two buyers both have items from the same seller in checkout. Buyer A's reservation expires, freeing 50 units. Simultaneously, Buyer B tries to add 50 units of the same item to their checkout.

**Handling:**
```
1. Buyer A's reservation expires:
   - check_expired_reservations sets reservation status = "Expired"
   - 50 units become available

2. Buyer B's add to checkout request:
   - get_available_stock() now shows 50 units available
   - Create new Stock Reservation for Buyer B
   - Using SELECT FOR UPDATE to prevent race conditions

3. If both happen at the exact same instant:
   - Database lock (FOR UPDATE) serializes the operations
   - First transaction to complete gets the stock
   - Second transaction sees updated availability
```

### 9.3 Abuse Detection False Positives

**Scenario:** A legitimate bulk B2B buyer places 5 orders per week, and occasionally cancels 1-2 due to production schedule changes. Over 30 days, they have 4 cancellations (exceeding the default threshold of 3).

**Handling:**
```
False Positive Mitigation Path:
  1. Buyer Abuse Log created with severity = "Low" (just crossed threshold)
  2. System checks buyer's completion rate:
     completed_orders = 16 (in 30 days)
     cancelled_orders = 4
     completion_rate = 16/20 = 80%
  3. Since completion_rate < 90%, no threshold multiplier applied
  4. BUT: check cancellation reasons:
     - 2 cancellations: reason = "production_schedule_change" (legitimate)
     - 1 cancellation: reason = "seller_requested_cancel"
     - 1 cancellation: reason = "buyer_changed_mind"
  5. Only 1 cancellation counts toward abuse (buyer_changed_mind)
  6. 1 < threshold of 3 -> Dismiss the abuse log as false positive
  7. Auto-resolve with note: "False positive: 3 of 4 cancellations have legitimate reasons"
```

### 9.4 Multi-Seller Cart with Staggered Expiry

**Scenario:** Cart has items from 3 sellers. Seller A's reservation (15 min) expires, but Seller B (30 min) and Seller C (45 min) are still active.

**Handling:**
```
Partial Expiry Flow:
  1. Seller A's reservations expire at T+15:
     - Release Seller A's stock reservations
     - Mark Seller A's Checkout Seller Group as "Expired"
     - Notify buyer: "Items from Seller A are no longer reserved"
     - Checkout Session remains active (not all groups expired)

  2. Buyer options:
     a. Complete checkout for Seller B and C only (remove Seller A items)
     b. Re-reserve Seller A items (if stock still available)
        - Creates new Stock Reservations for Seller A
        - Timer resets for Seller A's group
     c. Cancel entire checkout

  3. If buyer does nothing:
     - Seller B's reservations expire at T+30
     - Seller C's reservations expire at T+45
     - When ALL groups expire: Checkout Session -> "Expired"
     - Cart reverts to "Active"
```

### 9.5 Stock Becomes Unavailable During Reservation

**Scenario:** Buyer has 100 units reserved. The seller manually adjusts stock in ERPNext from 150 to 50 (stock correction), making the physical stock less than the reservation.

**Handling:**
```
Stock Change During Active Reservation:
  1. Stock correction reduces actual_qty from 150 to 50
  2. Active reservation holds 100 units
  3. Available stock = 50 (actual) - 100 (reserved) = -50 (negative!)
  4. This is a critical state - actual stock cannot fulfill reservation

  Detection:
  - cart_health_check (hourly) detects negative available stock
  - OR: A Stock Ledger Entry trigger checks for over-reservation

  Resolution:
  - Notify buyer: "Stock for [item] has been reduced. Only 50 of your 100
    reserved units may be available."
  - Notify seller: "Warning: Active stock reservation exceeds physical stock
    for [item]. Reservation: 100, Actual: 50."
  - Do NOT auto-cancel the reservation (seller may replenish)
  - Flag the reservation for admin review
  - If buyer proceeds to pay: validate stock again at payment time
    - If actual < reserved: allow partial fulfillment or cancel
```

### 9.6 Timezone Considerations

```
All times stored in UTC:
  - Stock Reservation.reserved_at: UTC
  - Stock Reservation.expires_at: UTC
  - Marketplace Order.payment_deadline: UTC
  - Cart.checkout_started_at: UTC

Frontend display:
  - Convert UTC to buyer's local timezone for display
  - Countdown timer uses client-side clock (synced with server)
  - Timer accuracy: re-sync with server every 60 seconds to prevent drift

Scheduler jobs:
  - Run on server timezone (UTC recommended)
  - All comparisons use NOW() in UTC
```

## 13. Performance Considerations

### 13.1 Scheduler Job Performance

| Job | Frequency | Target Duration | Key Optimization |
|-----|-----------|----------------|------------------|
| `check_expired_reservations` | Every 5 min | < 10 seconds | Batch UPDATE with indexed status + expires_at query |
| `auto_cancel_overdue_orders` | Every 15 min | < 30 seconds | Indexed payment_deadline query, batch processing |
| `cart_health_check` | Hourly | < 60 seconds | Batch stock checks, price cache, notification dedup |
| `daily_abuse_detection` | Daily | < 5 minutes | Aggregate SQL queries, incremental processing |
| `cleanup_expired_reservations` | Daily | < 30 seconds | Batch DELETE with date range |

### 13.2 Stock Availability Query Caching

```
Strategy: Short-TTL cache for available stock queries

  - Cache key: f"available_stock:{item_code}:{warehouse}"
  - TTL: 30 seconds (stock changes should reflect quickly)
  - Invalidation: On Stock Reservation create/update/delete
  - Implementation: frappe.cache().set_value() / get_value()

  Why 30s TTL:
    - Prevents stampede on hot items (many buyers checking at once)
    - Short enough that stock changes are reflected quickly
    - Reduces database load during peak checkout times
```

### 13.3 Reservation Lock Contention

```
Mitigation for High-Contention Items:

  1. Short lock duration: SELECT FOR UPDATE with 5-second timeout
  2. Retry with backoff: If lock fails, retry after 100ms, 200ms, 400ms (max 3 retries)
  3. Queue fallback: If all retries fail, enqueue the reservation request
     and notify the buyer of a brief delay
  4. Monitoring: Track lock wait times and contention frequency
     - Alert if average lock wait > 1 second
     - Alert if contention rate > 10% of reservation attempts
```

## 16. Glossary

| Term | Definition |
|------|-----------|
| **Stock Reservation** | A temporary hold on inventory during checkout, preventing other buyers from purchasing the same stock. Does not physically move stock. |
| **Checkout Timer** | A countdown displayed to the buyer showing how long their stock reservation will last. |
| **Payment Deadline** | The maximum time allowed for a buyer to complete payment after an order is placed with an offline payment method. |
| **Auto-Cancellation** | Automatic order cancellation when the payment deadline is exceeded without payment. |
| **Cart Health Check** | A periodic scheduled job that validates cart contents against current stock, prices, and product status. |
| **Buyer Abuse Detection** | An automated system that monitors buyer behavior patterns (cancellations, timeouts, etc.) and flags potential abuse. |
| **Cart Protection Settings** | A singleton configuration DocType containing all platform-wide temporal protection settings. |
| **Seller Cart Protection Override** | A child table row that overrides platform defaults for a specific seller. |
| **Price Snapshot** | The price of an item captured at the time of stock reservation, used for price change detection. |
| **Reservation Hoarding** | An abuse pattern where a buyer repeatedly reserves stock without completing payment, blocking other buyers. |
| **Cart Bombing** | An abuse pattern where a buyer creates excessive carts with large quantities to disrupt the marketplace. |
| **Grace Period** | Warning notifications sent before auto-cancellation (at 75% and 90% of the payment deadline). |
| **Timer Extension** | A one-time (configurable) option for buyers to extend their checkout reservation when the timer is about to expire. |
| **Override Resolution** | The logic that determines which setting value to use: seller-specific override or platform default. |
| **False Positive** | A legitimate buyer behavior incorrectly flagged as abuse by the detection system. |
