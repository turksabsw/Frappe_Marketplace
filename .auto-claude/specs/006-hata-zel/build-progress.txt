### E. KEY ISSUES SUMMARY

1. **Field type inconsistency**: Marketplace Shipment uses Select (weight_unit, dimension_unit) while Shipment uses Link to UOM (weight_uom, dimension_uom). Needs migration.

2. **Field name mismatch bug**: marketplace_shipment.js set_query references weight_uom/dimension_uom/volume_uom but JSON has weight_unit/dimension_unit. All 3 calls silently fail.

3. **Data vs Link inconsistency**: Shipment volume_uom is Data type while weight_uom and dimension_uom are Link type. Needs migration to Link.

4. **No set_query on catalog/seller UOM fields**: 12+ Link→UOM fields across Listing, SKU, SKU Product, PIM Product, PIM Product Variant, Buy Box Entry have no set_query filters. Users see ALL UOM entries.

5. **Default value inconsistency across DocTypes**:
   - Weight UOM: "kg" (Shipment), "Kg" (Listing/SKU/PIM)
   - Dimension UOM: "cm" (Shipment/PIM), "Centimeter" (Listing/Listing Variant), "Cm" (SKU Product)
   - Stock UOM: "Nos" (Listing), "Unit" (SKU Product/Buy Box Entry)

6. **Hardcoded units in labels**: Shipping Rule, Logistics Provider, Shipping Rate Tier, Shipping Zone Rate all have units hardcoded in field labels (e.g., "Per KG Rate", "Max Weight (KG)") with no dynamic UOM selection.

7. **Fragile name-based filtering**: Existing set_query filters use hardcoded UOM name lists. Any new UOM added to the system won't appear in filtered dropdowns.

### C. SELLER BALANCE PYTHON CLASS - KEY PATTERNS

**Balance Operations (all use db_set for atomic updates):**
1. add_earnings(amount, order_name, description) → pending_balance += amount
2. release_to_available(amount, escrow_name, description) → pending -= amount, available += amount
3. deduct_commission(amount, order_name, description) → lifetime_commissions += amount
4. process_refund(amount, order_name, description) → available -= amount (then pending if insufficient)
5. hold_funds(amount, reason) → available -= amount, held += amount
6. release_held_funds(amount, release_to, description) → held -= amount, available += amount
7. apply_adjustment(amount, reason, adjustment_type) → available += amount, lifetime_adjustments += amount

**Payout Lifecycle:**
1. can_payout() → checks status, suspension, payment issues, threshold, IBAN
2. request_payout(amount) → available -= amount, reserved += amount
3. complete_payout(amount, reference) → reserved -= amount, lifetime_payouts += amount
4. fail_payout(amount, reason) → reserved -= amount, available += amount + sets payment issue flag

**Validation:** validate_seller, validate_balances (no negative), validate_iban (TR format + MOD-97), validate_payout_settings
**Transaction logging:** _log_transaction() writes to JSON field (recent_transactions) - keeps last 100 entries
**Cache:** clear_cache on update (seller_balance:{name} and seller_balance_seller:{seller})
**Factory:** get_or_create_seller_balance(seller) - auto-creates from Seller Profile data
**Scheduled job:** process_scheduled_payouts() - finds eligible balances and requests payouts

**KEY INSIGHT FOR AD WALLET:**
- The wallet uses BOTH a child table (transaction_records → Seller Balance Transaction) AND a JSON field (recent_transactions) for transaction history
- The JSON field is deprecated in favor of the child table but both exist
- All balance mutations use db_set() for atomic single-field updates (not save())
- get_or_create pattern is essential - wallet auto-creates on first access
- Permission checks in API endpoints: session.user == seller.user OR has_permission

### G. CRITICAL DIFFERENCES FOR AD SYSTEM

1. **Wallet direction reversed:** Seller Balance receives money (earnings), Ad Wallet spends money (ad costs). Flow is: seller deposits → ad spend deducts → balance decreases.

2. **No payout needed:** Ad Wallet doesn't need payout methods/IBAN/scheduling. Instead needs: deposit/top-up methods, auto-pause on zero balance, minimum balance warnings.

3. **Campaign requires approval:** Unlike marketing Campaigns (seller can create freely), Ad Campaigns should require admin approval before going Active (fraud prevention).

4. **Real-time budget tracking:** Ad Campaigns need real-time budget checks per click/impression, not just aggregate tracking like marketing Campaigns.

5. **Auction integration:** Ad Campaign Items need bid amounts and quality scores, which marketing Campaign doesn't have.

6. **No discount mechanics:** Ad Campaigns don't have discount_type/discount_value. Instead they have bid_amount, pricing_model (CPC/CPM/CPT), and placement targeting.
