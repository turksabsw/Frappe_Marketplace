### E. KEY ISSUES SUMMARY

1. **Field type inconsistency**: Marketplace Shipment uses Select (weight_unit, dimension_unit) while Shipment uses Link to UOM (weight_uom, dimension_uom). Needs migration.

2. **Field name mismatch bug**: marketplace_shipment.js set_query references weight_uom/dimension_uom/volume_uom but JSON has weight_unit/dimension_unit. All 3 calls silently fail.

3. **Data vs Link inconsistency**: Shipment volume_uom is Data type while weight_uom and dimension_uom are Link type. Needs migration to Link.

4. **No set_query on catalog/seller UOM fields**: 12+ Link→UOM fields across Listing, SKU, SKU Product, PIM Product, PIM Product Variant, Buy Box Entry have no set_query filters. Users see ALL UOM entries.

5. **Default value inconsistency across DocTypes**:
   - Weight UOM: "kg" (Shipment), "Kg" (Listing/SKU/PIM)
   - Dimension UOM: "cm" (Shipment/PIM), "Centimeter" (Listing/Listing Variant), "Cm" (SKU Product)
   - Stock UOM: "Nos" (Listing), "Unit" (SKU Product/Buy Box Entry)

6. **Hardcoded units in labels**: Shipping Rule, Logistics Provider, Shipping Rate Tier, Shipping Zone Rate all have units hardcoded in field labels (e.g., "Per KG Rate", "Max Weight (KG)") with no dynamic UOM selection.

7. **Fragile name-based filtering**: Existing set_query filters use hardcoded UOM name lists. Any new UOM added to the system won't appear in filtered dropdowns.
