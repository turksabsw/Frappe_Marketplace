## 1. Executive Summary

This document defines the Seller Multi-Location System for the TradeHub B2B Marketplace. The system enables sellers to manage multiple physical locations (warehouses, stores, branches, showrooms) and integrates location selection into shipment creation, order fulfillment, stock management, and delivery note generation.

### Current State

The **Location Item** child DocType already exists in `tradehub_core` and is referenced by the `Seller Profile` DocType via a `locations` Table field. However, the existing implementation has several gaps:

1. **Location Item** uses only 5 location types (`Warehouse`, `Store`, `Distribution Center`, `Pickup Point`, `Return Center`) — missing `Branch` and `Showroom` required by the spec.
2. **Marketplace Shipment** (`tradehub_logistics`) currently populates origin address from the seller's **primary address fields** (`address_line_1`, `city`) — it does NOT use the `locations` child table at all.
3. **Sub Order** (`tradehub_commerce`) has no field to specify which seller location should fulfill the order.
4. There is no validation enforcing that exactly one location is marked as default per seller.
5. No ERPNext Warehouse integration exists for stock management per location.

### Proposed Changes

| Area | Change Type | Description |
|------|------------|-------------|
| **Location Item** (tradehub_core) | Extend | Add `Branch` and `Showroom` to `location_type` options; add `erpnext_warehouse` Link field for stock integration; add `country` Link field |
| **Seller Profile** (tradehub_seller) | Enhance | Add `validate()` logic for default location constraint; add helper methods for location lookup |
| **Marketplace Shipment** (tradehub_logistics) | Modify | Add `fulfillment_location` field; update `copy_addresses_from_sub_order()` to use selected location |
| **Sub Order** (tradehub_commerce) | Extend | Add `fulfillment_location` field for seller to select shipping origin |
| **Listing** (tradehub_seller) | Extend | Add `ships_from_location` field linking to a specific seller location |

## 4. Seller Profile Integration

### 4.1 Existing Integration (No Change)

The Seller Profile already has the `locations_section` Section Break and the `locations` Table field (options: `Location Item`). No structural changes are needed to the form layout.

### 4.2 Validation Logic — Default Location Constraint

**Rule:** Exactly one location must be marked as `is_default = 1` if the seller has any locations. Zero locations is valid (new sellers may not have locations yet).

**Implementation in `seller_profile.py`:**

```python
def validate(self):
    # ... existing validations ...
    self.validate_default_location()

def validate_default_location(self):
    """Ensure exactly one active location is marked as default."""
    if not self.locations:
        return

    active_locations = [loc for loc in self.locations if loc.is_active]
    if not active_locations:
        frappe.throw(
            _("At least one location must be active if locations are defined."),
            title=_("Location Validation Error")
        )

    default_locations = [loc for loc in active_locations if loc.is_default]

    if len(default_locations) == 0:
        frappe.throw(
            _("Exactly one location must be marked as default. "
              "Please set one location as the default."),
            title=_("Default Location Required")
        )
    elif len(default_locations) > 1:
        frappe.throw(
            _("Only one location can be the default. "
              "Found {0} default locations: {1}").format(
                len(default_locations),
                ", ".join([loc.location_name for loc in default_locations])
            ),
            title=_("Multiple Default Locations")
        )
```

### 4.3 Helper Methods for Location Lookup

Add the following utility methods to `seller_profile.py`:

```python
def get_default_location(self):
    """Get the default Location Item for this seller."""
    for loc in self.locations:
        if loc.is_default and loc.is_active:
            return loc
    return None

def get_fulfillment_locations(self):
    """Get all active locations that can fulfill orders."""
    return [
        loc for loc in self.locations
        if loc.is_active and loc.can_fulfill_orders
    ]

def get_return_locations(self):
    """Get all active locations that accept returns."""
    return [
        loc for loc in self.locations
        if loc.is_active and loc.can_accept_returns
    ]

def get_location_by_name(self, location_name):
    """Get a specific location by its name."""
    for loc in self.locations:
        if loc.location_name == location_name and loc.is_active:
            return loc
    return None

def get_location_by_idx(self, idx):
    """Get a specific location by its child table index."""
    for loc in self.locations:
        if loc.idx == idx and loc.is_active:
            return loc
    return None
```

## 5. Shipment Origin Integration (tradehub_logistics)

### 5.1 Problem Statement

When a `Marketplace Shipment` is created for a `Sub Order`, the origin address is currently populated from the seller's primary address fields on Seller Profile. This means:

- All shipments from a multi-location seller show the same origin
- Carrier pickup scheduling always references the primary address
- Shipping cost calculations may be inaccurate (wrong origin city)
- Tracking ETAs may be wrong (distance from actual fulfillment location)

### 5.2 Proposed Changes to Marketplace Shipment

#### New Field: `fulfillment_location`

Add a new field to `Marketplace Shipment` to reference the seller's fulfillment location:

```json
{
  "description": "Seller location from which this shipment originates",
  "fieldname": "fulfillment_location_idx",
  "fieldtype": "Int",
  "label": "Fulfillment Location Index",
  "hidden": 1,
  "description": "Index of the Location Item in Seller Profile locations table"
},
{
  "description": "Name of the fulfillment location (for display)",
  "fieldname": "fulfillment_location_name",
  "fieldtype": "Data",
  "label": "Fulfillment Location",
  "read_only": 1,
  "description": "Auto-populated from the selected Location Item"
}
```

**Design Decision — Why not a Link field?**

`Location Item` is a child table (`istable: 1`), which means its records don't have stable, user-friendly names. Child table rows are identified by `name` (random hash) and `idx` (position). We store the `idx` and `location_name` for reference. The actual address data is **copied** to the shipment's origin fields at creation time (denormalized), ensuring the shipment record is self-contained even if the seller later modifies or removes the location.

#### Updated `copy_addresses_from_sub_order()` Method

```python
def copy_addresses_from_sub_order(self):
    """Copy addresses from the linked Sub Order, using fulfillment location if available."""
    if not self.sub_order:
        return

    sub_order = frappe.get_doc("Sub Order", self.sub_order)

    # Copy seller reference
    if not self.seller:
        self.seller = sub_order.seller

    # Copy recipient info from sub order shipping address
    if not self.recipient_name:
        self.recipient_name = sub_order.shipping_address_name or sub_order.buyer_name
    if not self.recipient_phone:
        self.recipient_phone = sub_order.shipping_phone or sub_order.buyer_phone
    if not self.destination_address_line1:
        self.destination_address_line1 = sub_order.shipping_address_line1
    if not self.destination_address_line2:
        self.destination_address_line2 = sub_order.shipping_address_line2
    if not self.destination_city:
        self.destination_city = sub_order.shipping_city
    if not self.destination_state:
        self.destination_state = sub_order.shipping_state
    if not self.destination_postal_code:
        self.destination_postal_code = sub_order.shipping_postal_code
    if not self.destination_country:
        self.destination_country = sub_order.shipping_country or "Turkey"

    # Get sender info from seller profile — USE LOCATION if available
    if self.seller and not self.sender_name:
        seller_profile = frappe.get_doc("Seller Profile", self.seller)
        self.sender_name = seller_profile.seller_name

        # Determine fulfillment location
        fulfillment_loc = None

        # Priority 1: Use location specified on Sub Order
        if hasattr(sub_order, 'fulfillment_location_idx') and sub_order.fulfillment_location_idx:
            fulfillment_loc = seller_profile.get_location_by_idx(
                sub_order.fulfillment_location_idx
            )

        # Priority 2: Use seller's default location
        if not fulfillment_loc:
            fulfillment_loc = seller_profile.get_default_location()

        # Apply location address to shipment origin
        if fulfillment_loc:
            self.fulfillment_location_idx = fulfillment_loc.idx
            self.fulfillment_location_name = fulfillment_loc.location_name
            self.sender_phone = fulfillment_loc.phone or seller_profile.contact_phone

            if not self.origin_address_line1:
                self.origin_address_line1 = fulfillment_loc.street_address
            if not self.origin_address_line2:
                self.origin_address_line2 = fulfillment_loc.building_info
            if not self.origin_city:
                self.origin_city = fulfillment_loc.city_name
            if not self.origin_postal_code:
                self.origin_postal_code = fulfillment_loc.postal_code
            if not self.origin_country:
                self.origin_country = fulfillment_loc.country or "Turkey"
        else:
            # Fallback to primary address (backward compatibility)
            self.sender_phone = seller_profile.contact_phone
            if hasattr(seller_profile, 'address_line_1') and seller_profile.address_line_1:
                if not self.origin_address_line1:
                    self.origin_address_line1 = seller_profile.address_line_1
                if not self.origin_city and hasattr(seller_profile, 'city'):
                    self.origin_city = seller_profile.city
                if not self.origin_country:
                    self.origin_country = "Turkey"
```

### 5.3 Return Shipment Location Handling

When a return shipment is created via `create_return_shipment()`, the return destination should be the seller's designated return location (a location with `can_accept_returns = 1`), not necessarily the original fulfillment location.

**Updated `create_return_shipment()` logic:**

```python
def create_return_shipment(self, reason=None):
    """Create a return shipment for this delivery."""
    if self.status != "Delivered":
        frappe.throw(_("Can only create return for delivered shipments"))

    # Determine return destination (seller's return location)
    return_location = None
    if self.seller:
        seller_profile = frappe.get_doc("Seller Profile", self.seller)
        return_locations = seller_profile.get_return_locations()
        if return_locations:
            return_location = return_locations[0]  # Use first available return location
        elif seller_profile.get_default_location():
            return_location = seller_profile.get_default_location()

    # Build return destination from seller location or original origin
    if return_location:
        dest_address1 = return_location.street_address
        dest_address2 = return_location.building_info
        dest_city = return_location.city_name
        dest_postal = return_location.postal_code
        dest_country = return_location.country or "Turkey"
        dest_phone = return_location.phone
    else:
        # Fallback: swap original origin as return destination
        dest_address1 = self.origin_address_line1
        dest_address2 = self.origin_address_line2
        dest_city = self.origin_city
        dest_postal = self.origin_postal_code
        dest_country = self.origin_country
        dest_phone = self.sender_phone

    return_shipment = frappe.get_doc({
        "doctype": "Marketplace Shipment",
        "sub_order": self.sub_order,
        "seller": self.seller,
        "is_return": 1,
        "return_reason": reason,
        "original_shipment": self.name,
        "carrier": self.carrier,
        "shipment_type": "Standard",
        # Origin = buyer address (where return ships FROM)
        "sender_name": self.recipient_name,
        "sender_phone": self.recipient_phone,
        "origin_address_line1": self.destination_address_line1,
        "origin_address_line2": self.destination_address_line2,
        "origin_city": self.destination_city,
        "origin_state": self.destination_state,
        "origin_postal_code": self.destination_postal_code,
        "origin_country": self.destination_country,
        # Destination = seller's return location
        "recipient_name": self.sender_name,
        "recipient_phone": dest_phone,
        "destination_address_line1": dest_address1,
        "destination_address_line2": dest_address2,
        "destination_city": dest_city,
        "destination_postal_code": dest_postal,
        "destination_country": dest_country,
        # Copy package details
        "package_count": self.package_count,
        "total_weight": self.total_weight,
        "weight_unit": self.weight_unit,
        "description": f"Return: {self.description or ''}"
    })

    return_shipment.insert()
    return {
        "status": "success",
        "return_shipment": return_shipment.name,
        "return_shipment_id": return_shipment.shipment_id
    }
```

## 6. Delivery Note Integration

### 6.1 Context

In the ERPNext ecosystem, a **Delivery Note** is a document that records items shipped to a customer. For TradeHub, the `Marketplace Shipment` serves as the primary shipment tracking document, but ERPNext Delivery Notes may be generated for ERP integration (especially for sellers who use ERPNext for accounting).

### 6.2 Delivery Note ↔ Location Mapping

When a Delivery Note is created from a Sub Order:

1. The `set_warehouse` field on Delivery Note items should reference the ERPNext Warehouse linked to the fulfillment location (`Location Item.erpnext_warehouse`).
2. The source address fields on the Delivery Note should be populated from the fulfillment location.

**Integration approach:**

```python
def create_delivery_note_from_sub_order(sub_order_name):
    """Create an ERPNext Delivery Note from a Sub Order with location awareness."""
    sub_order = frappe.get_doc("Sub Order", sub_order_name)
    seller_profile = frappe.get_doc("Seller Profile", sub_order.seller)

    # Determine fulfillment location
    fulfillment_loc = None
    if hasattr(sub_order, 'fulfillment_location_idx') and sub_order.fulfillment_location_idx:
        fulfillment_loc = seller_profile.get_location_by_idx(
            sub_order.fulfillment_location_idx
        )
    if not fulfillment_loc:
        fulfillment_loc = seller_profile.get_default_location()

    # Get ERPNext Warehouse from location
    warehouse = None
    if fulfillment_loc and fulfillment_loc.erpnext_warehouse:
        warehouse = fulfillment_loc.erpnext_warehouse

    delivery_note = frappe.get_doc({
        "doctype": "Delivery Note",
        "customer": sub_order.buyer,
        "company": frappe.db.get_default("company"),
        "set_warehouse": warehouse,
        "items": [
            {
                "item_code": item.item_code,
                "qty": item.qty,
                "rate": item.rate,
                "warehouse": warehouse
            }
            for item in sub_order.items
        ]
    })

    return delivery_note
```

## 7. Stock Management Integration

### 7.1 Approach: ERPNext Warehouse Mapping

TradeHub does not have its own inventory management system — it relies on ERPNext's native Warehouse and Stock Entry DocTypes for stock tracking. The integration strategy is to link each `Location Item` to an ERPNext `Warehouse` via the proposed `erpnext_warehouse` field.

### 7.2 Stock Query by Location

```python
@frappe.whitelist()
def get_stock_by_seller_location(seller, item_code=None):
    """
    Get stock levels at each seller location.

    Args:
        seller: Seller Profile name
        item_code: Optional — filter by specific item

    Returns:
        list: Stock levels per location
    """
    seller_profile = frappe.get_doc("Seller Profile", seller)
    result = []

    for loc in seller_profile.locations:
        if not loc.is_active or not loc.erpnext_warehouse:
            continue

        filters = {"warehouse": loc.erpnext_warehouse}
        if item_code:
            filters["item_code"] = item_code

        stock = frappe.db.sql("""
            SELECT
                item_code,
                actual_qty,
                reserved_qty,
                (actual_qty - reserved_qty) as available_qty
            FROM `tabBin`
            WHERE warehouse = %(warehouse)s
            {item_filter}
        """.format(
            item_filter="AND item_code = %(item_code)s" if item_code else ""
        ), {
            "warehouse": loc.erpnext_warehouse,
            "item_code": item_code
        }, as_dict=True)

        result.append({
            "location_name": loc.location_name,
            "location_type": loc.location_type,
            "location_code": loc.location_code,
            "city_name": loc.city_name,
            "warehouse": loc.erpnext_warehouse,
            "is_default": loc.is_default,
            "stock": stock
        })

    return result
```

### 7.3 Stock Validation Before Fulfillment

Before a seller selects a fulfillment location for an order, the system should check if that location has sufficient stock:

```python
def validate_location_stock(seller, location_idx, items):
    """
    Validate that a fulfillment location has sufficient stock for order items.

    Args:
        seller: Seller Profile name
        location_idx: Index of Location Item in seller's locations table
        items: List of dicts with item_code and qty

    Returns:
        dict: Validation result with insufficient items if any
    """
    seller_profile = frappe.get_doc("Seller Profile", seller)
    location = seller_profile.get_location_by_idx(location_idx)

    if not location:
        return {"valid": False, "error": "Location not found"}

    if not location.erpnext_warehouse:
        # No warehouse linked — skip stock validation
        return {"valid": True, "warning": "No warehouse linked, stock not verified"}

    insufficient = []
    for item in items:
        available = frappe.db.get_value(
            "Bin",
            {"warehouse": location.erpnext_warehouse, "item_code": item["item_code"]},
            "(actual_qty - reserved_qty)"
        ) or 0

        if available < item["qty"]:
            insufficient.append({
                "item_code": item["item_code"],
                "required": item["qty"],
                "available": available,
                "shortage": item["qty"] - available
            })

    return {
        "valid": len(insufficient) == 0,
        "insufficient_items": insufficient,
        "warehouse": location.erpnext_warehouse,
        "location_name": location.location_name
    }
```

### 7.4 Stock Transfer Between Locations

When stock needs to be moved between seller locations, a Stock Entry (Material Transfer) should reference both source and target warehouses:

```python
def create_inter_location_transfer(seller, from_location_idx, to_location_idx, items):
    """
    Create a stock transfer between two seller locations.

    Args:
        seller: Seller Profile name
        from_location_idx: Source Location Item idx
        to_location_idx: Destination Location Item idx
        items: List of dicts with item_code and qty

    Returns:
        dict: Stock Entry details

    seller_profile = frappe.get_doc("Seller Profile", seller)
    from_loc = seller_profile.get_location_by_idx(from_location_idx)
    to_loc = seller_profile.get_location_by_idx(to_location_idx)

    if not from_loc or not to_loc:
        frappe.throw(_("Invalid location specified"))
    if not from_loc.erpnext_warehouse or not to_loc.erpnext_warehouse:

        "stock_entry_type": "Material Transfer",
        "items": [
            {

                "t_warehouse": to_loc.erpnext_warehouse
            }
            for item in items
        ]
    })

    stock_entry.insert()
    stock_entry.submit()

    return {
        "status": "success",
        "stock_entry": stock_entry.name,
        "from_location": from_loc.location_name,
```

```python
def auto_assign_fulfillment_location(sub_order):
    """
    Auto-assign fulfillment location when Sub Order is created or when seller has one location.

    Called from Sub Order's before_insert or when seller accepts the order.
    """
    if not sub_order.seller:
        return

    seller_profile = frappe.get_doc("Seller Profile", sub_order.seller)
    fulfillment_locs = seller_profile.get_fulfillment_locations()

    if len(fulfillment_locs) == 1:
        # Only one fulfillment location — auto-assign
        loc = fulfillment_locs[0]
        sub_order.fulfillment_location_idx = loc.idx
        sub_order.fulfillment_location_name = loc.location_name
    elif len(fulfillment_locs) > 1:
        # Multiple locations — assign default, seller can change later
        default_loc = seller_profile.get_default_location()
        if default_loc and default_loc.can_fulfill_orders:
            sub_order.fulfillment_location_idx = default_loc.idx
            sub_order.fulfillment_location_name = default_loc.location_name
    # If no locations defined, leave fields empty (backward compatibility)
```

## 9. Default Location Constraint Validation

### 9.1 Rules

1. **Zero locations allowed:** A Seller Profile with no Location Items in the `locations` table is valid (e.g., new seller before setup).
2. **Exactly one default:** If the `locations` table has rows where `is_active = 1`, exactly one must have `is_default = 1`.
3. **First location auto-default:** When the first Location Item is added, it is automatically marked as default.
4. **Delete protection:** Deleting the default location requires setting another location as default first, or it becomes the last location (in which case deletion is allowed, resulting in zero locations).
5. **Deactivation cascades:** Deactivating the default location requires choosing a new default among remaining active locations.

### 9.2 Server-Side Validation

```python
# In seller_profile.py

def validate_default_location(self):
    """Enforce exactly-one-default constraint on locations."""
    if not self.locations:
        return

    active_locations = [loc for loc in self.locations if loc.is_active]

    if not active_locations:
        # All locations are inactive — clear is_default on all
        for loc in self.locations:
            loc.is_default = 0
        return

    defaults = [loc for loc in active_locations if loc.is_default]

    if len(defaults) == 0:
        # No default set — auto-assign first active location
        active_locations[0].is_default = 1
        frappe.msgprint(
            _("'{0}' has been set as the default location.").format(
                active_locations[0].location_name
            ),
            indicator="blue"
        )
    elif len(defaults) > 1:
        # Multiple defaults — keep only the first one
        for i, loc in enumerate(defaults):
            if i > 0:
                loc.is_default = 0
        frappe.msgprint(
            _("Multiple default locations detected. "
              "Keeping '{0}' as the default.").format(defaults[0].location_name),
            indicator="orange"
        )
```

### 9.3 Client-Side Enforcement

The client-side script (described in Section 4.5) provides instant UX feedback:
- When a user checks `is_default` on a location, all other locations are immediately unchecked.
- This prevents the server-side validation from ever rejecting the form under normal usage.

### 10.1 Existing Pattern

The `Marketplace Shipment` also has a `tenant` field (Link → Tenant).

### 10.2 Location Isolation

Since `Location Item` is a **child table** of `Seller Profile`, it inherits tenant isolation automatically through its parent document. There is no need for a separate `tenant` field on Location Item — the parent Seller Profile's tenant determines access.

**Key constraint:** Any API that exposes location data must filter by the calling user's tenant:

```python
@frappe.whitelist()
def get_seller_locations(seller):
    """Get all active locations for a seller — with tenant check."""
    # Verify caller has access to this seller's tenant
    seller_profile = frappe.get_doc("Seller Profile", seller)

    # Frappe's permission system will throw PermissionError
    # if the user doesn't have access to this Seller Profile
    # based on User Permission rules tied to the tenant field.

    locations = []
    for loc in seller_profile.locations:
        if loc.is_active:
            locations.append({...})
    return locations
```

### 10.3 Cross-Tenant Prevention

- A seller cannot view or select locations from another tenant's seller.
- Marketplace Shipment creation validates that the `seller` field on the shipment belongs to the same `tenant`.
- Sub Order's `fulfillment_location_idx` is validated against the seller's actual locations, preventing index manipulation.

```python
# In marketplace_shipment.py validate()

def validate_tenant_isolation(self):
    """Ensure shipment respects tenant boundaries."""
    if self.seller and self.tenant:
        seller_tenant = frappe.db.get_value("Seller Profile", self.seller, "tenant")
        if seller_tenant != self.tenant:
            frappe.throw(
                _("Seller {0} does not belong to tenant {1}").format(
                    self.seller, self.tenant
                )
            )
```

## 14. Edge Cases & Error Handling

### 14.1 Seller with No Locations

**Scenario:** Seller has no Location Items defined.
**Handling:** Backward compatibility — use primary address fields from Seller Profile (`address_line_1`, `city`, `country`). All new code paths include a fallback to primary address.

### 14.2 Default Location Deleted

**Scenario:** Seller deletes the default location while it's referenced by pending Sub Orders.
**Handling:**
- Sub Orders that already have `fulfillment_location_idx` set retain the denormalized `fulfillment_location_name`.
- New shipments from those Sub Orders will detect the invalid idx and fall back to the new default location (or primary address).
- Validation on Seller Profile prevents deleting the only remaining location if active orders reference it. (Soft constraint — warning, not blocking.)

### 14.3 All Locations Deactivated

**Scenario:** Seller deactivates all locations.
**Handling:**
- `validate_default_location()` clears `is_default` on all locations.
- Fulfillment falls back to primary address.
- Consider: automatically set seller to "Vacation" mode if all locations are deactivated and active orders exist.

### 14.4 Location Changed After Sub Order Created

**Scenario:** Seller modifies a location's address after a Sub Order references it by idx.
**Handling:**
- Marketplace Shipment reads location data at **creation time** (denormalized copy).
- If a shipment already exists, its origin fields are already set — location changes don't retroactively affect existing shipments.
- For pending Sub Orders without a shipment yet, the updated location address will be used when the shipment is created.

### 14.5 Location Index Collision

**Scenario:** Location Item idx changes due to row reordering in the child table.
**Handling:**
- Frappe child table idx is positional and can change if rows are reordered.
- **Mitigation:** Use `location_code` as a secondary identifier. If `fulfillment_location_idx` doesn't resolve to a valid location, fall back to matching by `location_code` or `location_name`.
- **Long-term:** Consider adding a stable `location_id` (UUID) field to Location Item for reliable referencing.

### 14.6 Multi-Tenant Location Spoofing

**Scenario:** A malicious API call tries to set `fulfillment_location_idx` to a location that doesn't belong to the seller.
**Handling:**
- The idx is validated against the seller's actual `locations` child table during `copy_addresses_from_sub_order()`.
- If the idx doesn't match any location in the seller's profile, the system falls back to the default location.
- Frappe's `has_permission()` check on Seller Profile prevents cross-tenant data access.

## 16. Performance Considerations

### 16.1 Query Patterns

- **Seller locations are cached** as part of the Seller Profile document cache. No extra DB queries needed when accessing locations through the parent document.
- **Stock queries** hit the ERPNext `Bin` table which has indexes on `(warehouse, item_code)`.
- **Fulfillment location lookup** is an in-memory operation on the Seller Profile document — no DB round-trip.

### 16.2 Denormalization Strategy

Origin address fields on Marketplace Shipment are **denormalized copies** of the location data. This is intentional:
- Shipment records must be self-contained for carrier API integration.
- Historical shipment records must not change if a location is modified.
- This matches the existing pattern where destination address is already denormalized from the Sub Order.

### 16.3 Indexing

No new database indexes are needed:
- Location Items are accessed via their parent document (Seller Profile), not queried independently.
- `fulfillment_location_idx` on Sub Order/Marketplace Shipment is an Int used for in-document lookup, not for SQL joins.
