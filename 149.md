# Task 149: RFQ to RFQ Quote Auto-Transfer Planning Document

## Overview

This document specifies the automatic transfer of RFQ (Request for Quotation) fields to RFQ Quote records. When a seller creates a quote against an RFQ, the system must auto-populate RFQ reference data into the quote as **read-only fields**, leaving only seller-specific fields (price, delivery, notes) editable. This eliminates manual re-entry, reduces errors, and ensures data consistency between the RFQ and its quotes.

## 7. `fetch_from` vs Server-Side Copy Analysis

### 7.1 Transfer Mechanism Options

| Method | How It Works | Pros | Cons |
|--------|-------------|------|------|
| **`fetch_from`** | Frappe auto-fetches value from linked document when link field is set. Defined in DocType JSON. | Zero code required; declarative; built-in Frappe feature; runs on both client and server | Only works for direct Link field references; cannot transform data; values fetched at creation time only; no cross-table fetch |
| **Server-side copy** (Python hook) | Custom Python code in `before_insert` or `before_save` hook copies values programmatically | Full control; can transform data; can copy child table rows; handles complex logic | Requires custom code; must maintain manually; doesn't auto-refresh on link change |
| **Client script** (`frappe.ui.form.on`) | JavaScript runs on form events to copy values | Immediate UI feedback; can trigger on field change | Client-only (bypassed by API calls); unreliable for data integrity |

### 7.2 Recommended Strategy: Hybrid Approach

**Primary: `fetch_from` for all simple field transfers**
- All single-value fields that map directly from RFQ to RFQ Quote should use `fetch_from`
- This is the established pattern already in use (see `rfq_title`, `rfq_buyer`, `seller_name`, `tenant`)
- `fetch_from` fields must have `read_only: 1` to prevent manual editing

**Secondary: Server-side hook for child table auto-population**
- When a seller creates a new RFQ Quote and selects the `rfq` link, the system should auto-create RFQ Quote Item rows -- one for each RFQ Item in the parent RFQ
- This **cannot** be done with `fetch_from` (which only handles scalar fields, not table-to-table mapping)
- Implement in `rfq_quote.py` via `before_insert` or a whitelisted API method

#### Step D: Server-Side Hook for Child Table Auto-Population

```python
# In rfq_quote.py

class RFQQuote(Document):
    def before_insert(self):
        self.auto_populate_items_from_rfq()

    def auto_populate_items_from_rfq(self):
        """Auto-create RFQ Quote Item rows from parent RFQ Items."""
        if not self.rfq:
            return

        # Skip if items already exist (e.g., duplicate/amend)
        if self.items and len(self.items) > 0:
            return

        rfq_doc = frappe.get_doc("RFQ", self.rfq)

        for rfq_item in rfq_doc.items:
            self.append("items", {
                "rfq_item": rfq_item.name,
                "item_name": rfq_item.item_name,
                "item_description": rfq_item.description,
                "qty": rfq_item.quantity,
                "uom": rfq_item.unit,
                "rfq_qty": rfq_item.quantity,
                "rfq_specifications": rfq_item.specifications,
                "rfq_target_price": rfq_item.target_price,
            })
```

> **Note:** The `items` Table field does not yet exist on RFQ Quote. It must be added as part of this implementation (`fieldtype: Table`, `options: RFQ Quote Item`).

## 10. Edge Cases

### 10.1 RFQ with Missing/Partially Filled Fields

**Scenario:** Buyer creates an RFQ without filling optional fields (budget_min, budget_max, delivery_date, delivery_location, etc.).

**Handling:**
- `fetch_from` naturally returns `None`/empty for unfilled fields -- no special handling needed
- RFQ Quote form should use `depends_on` to conditionally show sections only when data exists:
  ```
  depends_on: eval:doc.rfq_budget_min || doc.rfq_budget_max
  ```
- All new auto-transferred fields are optional (`reqd: 0`), so empty values are valid
- The seller form should gracefully show "Not specified" or hide empty sections

### 10.2 RFQ Amendment After Quote Exists

**Scenario:** Buyer amends the RFQ (changes description, quantity, delivery date, etc.) after one or more quotes have been submitted.

**Handling:**
- **`fetch_from` values are static** -- they are captured at the time the RFQ Quote is created and do NOT auto-update when the source RFQ changes
- This is **intentional**: the quote was made based on the original RFQ terms; changing those terms retroactively would invalidate the seller's pricing
- **Amendment workflow:**
  1. When an RFQ is amended, its status should change (e.g., to a new version or "Amended" status)
  2. Existing quotes against the original RFQ should be flagged with a notice: "The parent RFQ has been amended since this quote was created"
  3. Sellers should be notified and given the option to submit a revised quote against the amended RFQ
  4. The `revision_count` and `last_revised_at` fields on RFQ Quote track seller revisions
- **Implementation suggestion:** Add a server hook on RFQ save that checks for existing quotes and sends a notification:
  ```python
  # In rfq.py on_update hook
  def notify_sellers_of_amendment(self):
      quotes = frappe.get_all("RFQ Quote",
          filters={"rfq": self.name, "status": ["not in", ["Rejected", "Withdrawn"]]},
          fields=["name", "seller"])
      for quote in quotes:
          # Send notification to seller about RFQ amendment
          pass
  ```

### 10.3 RFQ Items Table Empty

**Scenario:** RFQ has no items in its child table (only uses parent-level quantity/description).

**Handling:**
- The `auto_populate_items_from_rfq` hook checks `rfq_doc.items` and skips if empty
- The seller can manually add RFQ Quote Item rows or use legacy parent-level fields (`item_name`, `qty`, `unit_price` on RFQ Quote)
- The legacy single-item fields on RFQ Quote (`item_name`, `item_description`, `qty`, `uom`, `unit_price`, `total_price`) serve this use case

### 10.4 Seller Tries to Edit Read-Only Fields

**Scenario:** Seller attempts to modify auto-populated read-only fields via API call (bypassing form UI).

**Handling:**
- `read_only: 1` in DocType JSON prevents UI editing but does NOT block API writes
- Add a `validate` hook in `rfq_quote.py` to enforce read-only on server side:
  ```python
  def validate(self):
      if not self.is_new():
          self.validate_read_only_fields()

  def validate_read_only_fields(self):
      """Prevent modification of auto-populated fields after creation."""
      if self.has_value_changed("rfq"):
          frappe.throw("Cannot change the linked RFQ after quote creation")
      # Additional read-only field checks as needed
  ```

### 10.5 Multiple Quotes from Same Seller for Same RFQ

**Scenario:** A seller submits multiple quotes for the same RFQ.

**Handling:**
- Business rule decision: Allow or prevent?
- **Recommended:** Allow, but only one active quote per seller per RFQ. Previous quotes should be auto-withdrawn when a new one is submitted
- Add a `before_insert` check:
  ```python
  existing = frappe.get_all("RFQ Quote",
      filters={"rfq": self.rfq, "seller": self.seller,
               "status": ["not in", ["Rejected", "Withdrawn"]]})
  if existing:
      frappe.throw("You already have an active quote for this RFQ. "
                   "Please revise the existing quote instead.")
  ```

### 10.6 Currency Mismatch Between RFQ and Quote

**Scenario:** RFQ specifies currency as TRY but seller quotes in USD.

**Handling:**
- Auto-populate `currency` on RFQ Quote from `rfq.currency` via `fetch_from`
- The seller can change currency if multi-currency quoting is allowed
- If strict currency matching is required, add a validation:
  ```python
  if self.currency != frappe.db.get_value("RFQ", self.rfq, "currency"):
      frappe.throw("Quote currency must match RFQ currency")
  ```

### 10.7 Stale fetch_from Data

**Scenario:** Over time, the RFQ's buyer profile data changes (e.g., buyer_name is updated), but old quotes still show the stale fetched value.

**Handling:**
- This is expected Frappe behavior -- `fetch_from` captures a snapshot at link-set time
- For display purposes, the quote form should show "as of quote creation"
- If real-time data is needed, use a virtual field or API call instead of `fetch_from`

## 12. Data Flow Sequence

```
1. Buyer creates RFQ with items, requirements, budget, deadline
   └─> RFQ saved with status "Draft"

2. Buyer publishes RFQ
   └─> RFQ status -> "Published", published_at set

3. Seller opens "Create Quote" for this RFQ
   └─> New RFQ Quote form opens
   └─> Seller selects rfq Link field = RFQ name
   └─> fetch_from auto-populates: rfq_title, rfq_buyer, rfq_code,
       rfq_description, rfq_category, rfq_deadline, rfq_quantity,
       rfq_unit, rfq_budget_min, rfq_budget_max, rfq_delivery_date,
       rfq_delivery_location, rfq_buyer_name, rfq_buyer_company,
       rfq_requires_nda, rfq_allow_partial, rfq_require_all_items

4. before_insert hook fires
   └─> auto_populate_items_from_rfq() creates RFQ Quote Item rows
       for each RFQ Item, pre-filling: rfq_item, item_name,
       item_description, qty, uom, rfq_qty, rfq_specifications,
       rfq_target_price

5. Seller fills in editable fields:
   └─> Parent: price, currency, delivery_days, valid_until, terms, notes
   └─> Per item: unit_price, qty (if different), lead_time_days,
       delivery_date, availability_status, specifications_response, notes

6. Seller submits quote
   └─> validate() runs: checks read-only fields, checks duplicates
   └─> submitted_at set, status = "Submitted"
   └─> RFQ.quote_count incremented

7. (Optional) Buyer amends RFQ
   └─> Existing quotes flagged with amendment notice
   └─> Sellers notified to revise their quotes
```
