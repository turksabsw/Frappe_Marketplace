# Task 119 - Seller & Buyer Profile/Metrics System Plan

## 1. Overview

This document defines the comprehensive profile and metrics tracking system for both sellers and buyers on the TradeHub marketplace platform. It covers:

- **Seller Metrics DocType extension**: Adding Order Defect Rate (ODR), late shipment rate, and Account Health Score (0-1000) to the existing `Seller Metrics` DocType.
- **Buyer Profile DocType extension**: Adding feedback rate, return rate, and payment pattern tracking to the existing `Buyer Profile` DocType.
- **Seller tier system**: Defining Bronze, Silver, Gold, Platinum, and Top Rated tiers using the existing `Seller Tier` DocType.
- **Buyer tier system**: Defining New, Regular, VIP, and Premium tiers using the existing `Buyer Level` DocType.
- **Detailed Seller Ratings (DSR)**: A new child DocType for granular 1-5 star ratings across four categories.
- **Bidirectional feedback system**: An eBay-style model where buyers rate sellers AND sellers rate buyers.
- **Scheduled jobs**: Periodic automatic recalculation of all metrics.

### Existing DocTypes Referenced

| DocType | App | Current State |
|---------|-----|---------------|
| `Seller Metrics` | tradehub_seller | Has order, review, activity metrics. **Missing**: ODR, late_shipment_rate, account_health_score |
| `Buyer Profile` | tradehub_core | Has statistics tab with total_orders, total_spent, avg_order_value. **Missing**: feedback_rate, return_rate, buyer_level link |
| `Seller Tier` | tradehub_seller | Full tier structure with requirements, benefits, progression. Ready for Bronze-Top Rated population |
| `Buyer Level` | tradehub_core | Level/threshold/benefits structure. Ready for New-Premium population |
| `Seller Score` | tradehub_seller | Detailed score tracking with 6 components, weights, penalties/bonuses (0-100 scale) |

## 2. Reference Models

### 2.1 Amazon Seller Performance Model

Amazon uses three primary seller health metrics:

| Metric | Amazon Name | TradeHub Equivalent | Range |
|--------|-------------|---------------------|-------|
| Order Defect Rate | ODR | `order_defect_rate` | 0-100% (target < 1%) |
| Late Shipment Rate | LSR | `late_shipment_rate` | 0-100% (target < 4%) |
| Pre-Fulfillment Cancel Rate | PFCR | `cancellation_rate` (existing) | 0-100% (target < 2.5%) |
| Account Health Score | AHS | `account_health_score` | 0-1000 |

**Amazon ODR components**: Negative feedback rate + A-to-z Guarantee claim rate + chargeback rate.

**Amazon Account Health Score (0-1000)**: Composite score combining ODR, LSR, PFCR, valid tracking rate, policy violations. Score thresholds: 200+ = Healthy, 100-199 = At Risk, 0-99 = Unhealthy/Critical.

### 2.2 eBay Seller Performance Model

eBay uses a **bidirectional feedback** system and **Detailed Seller Ratings (DSR)**:

| Feature | eBay Implementation | TradeHub Plan |
|---------|---------------------|---------------|
| Feedback direction | Buyer rates seller AND seller rates buyer | Same bidirectional model |
| Rating categories | Item as described, Communication, Shipping time, Shipping cost | Product accuracy, Communication, Shipping speed, Price/Value |
| Star scale | 1-5 stars per category | 1-5 stars per category |
| Seller levels | Below Standard, Standard, Above Standard, Top Rated | Bronze, Silver, Gold, Platinum, Top Rated |
| Feedback visibility | Public with anonymity options | Public with anonymity for buyers |

### 2.3 Alibaba Trade Assurance Model

| Feature | Alibaba Implementation | TradeHub Adoption |
|---------|------------------------|-------------------|
| Response rate | Shown on profile, target > 90% | `avg_response_time_hours` (existing) |
| Transaction level | Based on GMV tiers | Mapped to seller tier via `min_total_sales_amount` |
| Trade Assurance badge | Visual trust indicator | Tier badge via `Seller Tier.badge_icon` |

## 4. Business Logic & Formulas

### 4.1 Account Health Score Calculation (0-1000)

The Account Health Score (AHS) is a weighted composite metric modeled after Amazon's system. New sellers start with a score of 1000.

```
AHS = 1000 - (ODR_penalty + LSR_penalty + CR_penalty + policy_penalty + tracking_penalty)

Where:
  ODR_penalty    = order_defect_rate * 300  (max 300 points)
  LSR_penalty    = late_shipment_rate * 200 (max 200 points)
  CR_penalty     = cancellation_rate * 150  (max 150 points)
  policy_penalty = policy_violations * 50   (max 200 points, capped at 4 violations)
  tracking_penalty = (100 - valid_tracking_rate) * 1.5 (max 150 points)

AHS = max(0, AHS)  // Floor at 0
```

**Account Health Status thresholds**:

| Score Range | Status | Action |
|-------------|--------|--------|
| 600-1000 | Healthy | No action required |
| 300-599 | At Risk | Warning notification sent, performance plan suggested |
| 1-299 | Critical | Account under review, listing restrictions may apply |
| 0 | Suspended | Account suspended pending review |

### 4.2 Order Defect Rate (ODR) Calculation

```
ODR = (negative_feedback_orders + disputed_orders + chargeback_orders) / total_evaluated_orders * 100

Where:
  negative_feedback_orders = orders that received 1-2 star overall_rating in Seller Feedback
  disputed_orders = orders with linked Dispute records (status != Resolved in Seller Favor)
  chargeback_orders = orders with chargebacks (from Payment Entry)
  total_evaluated_orders = orders completed in the evaluation period (last 90 days)

Edge case: If total_evaluated_orders = 0, ODR = 0 (new sellers are not penalized)
```

### 4.3 Late Shipment Rate Calculation

```
late_shipment_rate = late_shipped_orders / total_shipped_orders * 100

Where:
  late_shipped_orders = orders where actual_ship_date > promised_ship_date
  total_shipped_orders = orders that were shipped in the evaluation period

Edge case: If total_shipped_orders = 0, late_shipment_rate = 0
```

### 4.4 Positive Feedback Percentage

```
positive_feedback_pct = positive_feedback_count / total_feedback_count * 100

Where:
  positive_feedback_count = Seller Feedback records with feedback_type = 'Positive'
  total_feedback_count = All Seller Feedback records with status = 'Active'

Edge case: If total_feedback_count = 0, positive_feedback_pct = 0 (displayed as "No feedback yet")
```

### 4.5 Buyer Metrics Calculations

```
return_rate = returned_orders / total_orders * 100
feedback_rate = orders_with_feedback / total_orders * 100
dispute_rate = disputes_initiated / total_orders * 100
cancellation_rate = cancelled_orders / total_orders * 100

Where:
  returned_orders = Return Request records linked to buyer (status = Approved/Completed)
  orders_with_feedback = Seller Feedback records linked to buyer
  disputes_initiated = Dispute records initiated by buyer
  cancelled_orders = Marketplace Order records cancelled by buyer

Edge case: If total_orders = 0, all rates = 0, payment_pattern = 'New'
```

### 4.6 DSR Average Calculation

DSR averages are calculated per category across all `Seller Feedback` records for a seller:

```
dsr_avg_product_accuracy = AVG(dsr_product_accuracy) WHERE dsr_product_accuracy > 0
dsr_avg_communication = AVG(dsr_communication) WHERE dsr_communication > 0
dsr_avg_shipping_speed = AVG(dsr_shipping_speed) WHERE dsr_shipping_speed > 0
dsr_avg_price_value = AVG(dsr_price_value) WHERE dsr_price_value > 0

These averages are stored in a summary view, not on the Seller Metrics record directly.
Rolling calculation window: last 12 months of feedback.
```

### 4.7 Bidirectional Feedback Rules

1. **Timing**: Buyer can leave feedback within 60 days of order completion. Seller can leave feedback within 60 days of order completion.
2. **One feedback per order**: Each party can leave exactly one feedback per order (revisions allowed but counted).
3. **No self-rating**: System validates `seller != buyer` on the linked profiles. Orders are always between different parties by design.
4. **Anonymity**: Buyer feedback on `Seller Feedback` can be anonymous (`is_anonymous = 1`). Seller feedback on `Buyer Feedback` is never anonymous (seller identity is known to buyer).
5. **Revision policy**: Feedback can be revised within 30 days of initial submission. `was_revised` flag is set and `revision_count` incremented. Original feedback is preserved in version history (Frappe's track_changes).
6. **Removal**: Only System Manager can remove feedback (set status to `Removed`). Feedback with `status = Removed` is excluded from all calculations.
7. **Seller response**: Sellers can respond publicly to buyer feedback via `seller_response` field. Response does not affect ratings.

### 4.8 Tier Evaluation Logic

**Seller Tier Evaluation** (runs daily via `tier_tasks`):
1. For each active seller, fetch latest `Seller Score.overall_score` and `Seller Metrics` record.
2. Iterate through tiers from highest (`TOP_RATED`) to lowest (`BRONZE`).
3. A seller qualifies for a tier if ALL minimum requirements are met and NO maximum thresholds are exceeded.
4. If the new tier differs from current tier:
   - **Upgrade**: Apply immediately. Notify seller. Log as Comment on Seller Profile.
   - **Downgrade**: Check `downgrade_grace_days`. If grace period has not elapsed since first detection, skip. If grace period has elapsed, downgrade. Notify seller.

**Buyer Level Evaluation** (runs daily via new `buyer_level_tasks`):
1. For each active buyer, calculate total_spent and order_count within `evaluation_period_months`.
2. Compare against `Buyer Level` thresholds using `qualification_type`:
   - `Purchase Amount`: Compare `total_spent` against `threshold_value`
   - `Order Count`: Compare `total_orders` against `min_order_count`
   - `Combined`: Both conditions must be met
3. Apply upgrade/downgrade with grace period logic identical to seller tiers.

## 5. Scheduled Jobs

### 5.1 Seller Metrics Recalculation

**Job name**: `tradehub_seller.tasks.recalculate_seller_metrics`
**Frequency**: Daily (`scheduler_events.daily`)
**Cron expression**: `0 2 * * *` (02:00 AM daily)

**What it calculates**:
- `order_defect_rate` (ODR)
- `late_shipment_rate`
- `valid_tracking_rate`
- `positive_feedback_pct`
- `account_health_score` and `account_health_status`
- `defect_count`
- `dispute_rate`
- `chargeback_rate`
- `order_fulfillment_rate`
- Updates to existing metrics: `cancellation_rate`, `return_rate`, `on_time_delivery_rate`, `avg_rating`, `positive_review_rate`, `complaint_rate`, `avg_response_time_hours`, `repeat_customer_rate`

**Data sources read from**:
- `Marketplace Order` / `Sub Order` - order data, fulfillment dates, shipment dates
- `Seller Feedback` - feedback types, DSR ratings
- `Dispute` - dispute records
- `Payment Entry` - chargeback data
- `Return Request` - return data
- `Listing` - listing counts

**Data targets written to**:
- `Seller Metrics` - all metric fields updated

**Estimated runtime**: ~50ms per seller. For 10,000 sellers: ~8 minutes.

**Error handling**:
```python
def recalculate_seller_metrics():
    sellers = frappe.get_all("Seller Profile", filters={"status": "Active"}, fields=["name"])
    for seller in sellers:
        try:
            _recalculate_for_seller(seller.name)
        except Exception as e:
            frappe.log_error(
                message=f"Metrics recalculation failed for seller {seller.name}: {str(e)}",
                title="Seller Metrics Recalculation Error"
            )
    frappe.db.commit()
```

**Data targets written to**:
- `Buyer Profile` - metric fields updated

**Estimated runtime**: ~30ms per buyer. For 100,000 buyers: ~50 minutes.

**Data sources read from**:

- `Buyer Level` - threshold definitions

**Data targets written to**:
- `Buyer Profile.buyer_level` - updated when level changes

**Error handling**: Same per-record pattern.

### 5.6 Job Execution Order & Locking

To prevent data inconsistency from concurrent job execution, the following execution order is enforced via cron scheduling:

```
02:00 AM - recalculate_seller_metrics  (seller metrics first)
02:30 AM - kpi_tasks                   (KPIs use fresh metrics)
03:00 AM - recalculate_buyer_metrics   (buyer metrics)
03:30 AM - tier_tasks                  (seller tier evaluation uses fresh scores)
04:00 AM - buyer_level_tasks           (buyer level evaluation uses fresh metrics)
```

Each job acquires a Frappe `lock` using `frappe.utils.caching.redis_cache` to prevent overlapping runs:

```python
if frappe.cache.get_value("seller_metrics_recalc_lock"):
    frappe.log_error("Seller metrics recalculation already in progress", "Scheduler Lock")
    return
frappe.cache.set_value("seller_metrics_recalc_lock", 1, expires_in_sec=3600)
try:
    # ... recalculation logic ...
finally:
    frappe.cache.delete_value("seller_metrics_recalc_lock")
```
