# Task 118: Favorites & Wishlist System Planning Document

## Overview

This document specifies the Favorites & Wishlist System for the TradeHub marketplace platform. The system provides an Instagram + Amazon-style experience where buyers can organize products into unlimited named wishlists, quick-favorite products/sellers/stores with a single tap, share lists with other users via invite links, and receive price-drop and low-stock notifications. The feature introduces **three new DocTypes** (User Wishlist, Wishlist Item, Quick Favorite) and leverages the existing **Notification Template** infrastructure (which already supports `Price Change Alert` and `Low Stock Alert` event types) for automated alerts. Social proof counters aggregate favorite/wishlist data for display on product and seller pages.

| target_type Value | Resolves To | Example Reference |
|-------------------|-------------|-------------------|
| `Product` | Product DocType | `PRD-2026-00042` |

### 3.3 Uniqueness Constraint

A compound unique constraint must be enforced: **one favorite per user per target**. This is implemented via a server-side `validate` hook:

```python
def validate(self):
    existing = frappe.db.exists("Quick Favorite", {
        "user": self.user,
        "target_type": self.target_type,
        "target_reference": self.target_reference,
        "name": ("!=", self.name)
    })
    if existing:
        frappe.throw(_("You have already favorited this item."))
```

## 4. Unlimited List Creation Plan

### 4.1 Design Principles

- Users can create **unlimited** wishlists with no system-imposed cap
- Each user has exactly **one default wishlist** (the "quick add" target)
- If a user has no wishlists, a default wishlist named "My Wishlist" is created on first use
- List names must be unique per user but can duplicate across users

### 4.2 Default Wishlist Auto-Creation

When a buyer performs their first wishlist action (quick-add from a product page), the system checks for an existing default wishlist:

```
1. User clicks "Add to Wishlist" on a product
2. System checks: frappe.db.exists("User Wishlist", {"user": user, "is_default": 1})
3. If no default exists:
   a. Create User Wishlist with list_name="My Wishlist", is_default=1, privacy="Private"
   b. Add the product as a Wishlist Item child row
4. If default exists:
   a. Add the product to the existing default wishlist
5. Show confirmation toast: "Added to My Wishlist"
```

### 4.3 Wishlist Management UI Actions

| Action | Description | Implementation |
|--------|-------------|----------------|
| Create List | New named wishlist with privacy selection | Dialog form with list_name, description, privacy, cover_image |
| Rename List | Change wishlist name | Inline edit on list_name field |
| Delete List | Remove wishlist and all items | Confirm dialog -> `frappe.delete_doc("User Wishlist", name)` |
| Change Privacy | Toggle between Private/Shared/Public | Select dropdown on wishlist detail page |
| Set as Default | Mark as the quick-add target | Toggle check -> server unsets other defaults |
| Duplicate List | Copy all items to a new list | Server-side: create new wishlist + copy child rows |

### 4.4 Performance Considerations

- **Pagination:** Wishlists display items with pagination (default: 20 items per page)
- **Lazy Loading:** Item images load on scroll to prevent large initial payloads
- **Count Caching:** `item_count` on User Wishlist is updated on child add/remove to avoid count queries

## 5. Cross-List Item Move Plan

### 5.1 Supported Operations

| Operation | Source | Destination | Behavior |
|-----------|--------|-------------|----------|
| **Copy** | Wishlist A | Wishlist B | Item appears in both lists (duplicate allowed) |
| **Move** | Wishlist A | Wishlist B | Item removed from source, added to destination |
| **Bulk Copy** | Wishlist A (selected items) | Wishlist B | Multiple items copied at once |
| **Bulk Move** | Wishlist A (selected items) | Wishlist B | Multiple items moved at once |

### 5.2 Implementation Approach

Cross-list operations are handled via a whitelisted server-side API:

```python
@frappe.whitelist()
def move_wishlist_items(source_wishlist, target_wishlist, item_names, operation="move"):
    """
    Move or copy items between wishlists.

    Args:
        source_wishlist: Name of the source User Wishlist
        target_wishlist: Name of the target User Wishlist
        item_names: JSON list of Wishlist Item names (child row names)
        operation: "move" or "copy"
    """
    # 1. Validate user owns both wishlists
    # 2. For each item in item_names:
    #    a. Read child row from source
    #    b. Append duplicate row to target (copy all fields)
    #    c. If operation == "move", delete from source
    # 3. Update item_count on both wishlists
    # 4. Save both wishlists
```

### 5.3 Conflict Handling

- **Duplicate product in target:** Allowed. The same product can appear in multiple wishlists and even multiple times in the same wishlist (different seller stores or different notes).
- **Source wishlist deleted during operation:** Transaction rollback; return error message.
- **Target wishlist at capacity:** No capacity limit exists, so this is not an issue.

## 6. Shared List Invite Link Mechanism

### 6.1 Share Token Generation

When a wishlist's privacy is set to `Shared` or `Public`, a unique share token is generated:

```python
import uuid

def generate_share_token(wishlist_doc):
    if not wishlist_doc.share_token and wishlist_doc.privacy in ("Shared", "Public"):
        wishlist_doc.share_token = str(uuid.uuid4()).replace("-", "")[:16]
        wishlist_doc.share_url = f"/wishlist/shared/{wishlist_doc.share_token}"
```

### 6.2 Privacy Level Behavior

| Privacy Level | Visible To | Access Method | Share Link |
|---------------|------------|---------------|------------|
| **Private** | Owner only | Direct access via "My Wishlists" | Not available |
| **Shared** | Owner + invited users | Invite link with token | `{site}/wishlist/shared/{token}` |
| **Public** | Everyone | Browse/search on marketplace | `{site}/wishlist/public/{token}` |

### 6.3 Invite Flow (Shared Lists)

```
1. Owner creates wishlist with privacy = "Shared"
2. System generates share_token and share_url
3. Owner copies share_url and sends to friends/colleagues
4. Recipient opens share_url:
   a. If logged in: wishlist is displayed read-only, with option to "Follow" or "Copy to My Wishlists"
   b. If guest: redirect to login, then display wishlist
5. Owner can also invite by email:
   a. Enter comma-separated emails in shared_with field
   b. System sends notification email with share_url to each address
   c. Invited users see the wishlist in their "Shared With Me" section
```

### 6.4 Shared List Permissions

| Action | Owner | Invited User | Public Viewer |
|--------|-------|--------------|---------------|
| View items | Yes | Yes | Yes (public only) |
| Add items | Yes | No | No |
| Remove items | Yes | No | No |
| Edit notes/priority | Yes | No | No |
| Copy entire list | Yes | Yes | Yes |
| Copy single item | Yes | Yes | Yes |
| Delete list | Yes | No | No |
| Change privacy | Yes | No | No |

### 6.5 Token Security

- Tokens are 16-character hexadecimal strings (collision probability negligible)
- Changing privacy from `Shared`/`Public` to `Private` invalidates the share URL (server returns 404)
- Token is regenerated if the owner explicitly clicks "Reset Share Link"
- No rate limiting on share link access (read-only, low risk)

### 7.3 Server-Side Query

```python
@frappe.whitelist(allow_guest=True)
def get_public_wishlists(search=None, sort_by="modified", page=1, page_size=20):
    """
    Fetch public wishlists with optional search and pagination.
    """
    filters = {"privacy": "Public", "item_count": (">", 0)}
    if search:
        filters["list_name"] = ("like", f"%{search}%")

    wishlists = frappe.get_all(
        "User Wishlist",
        filters=filters,
        fields=["name", "list_name", "user_full_name", "description",
                "cover_image", "item_count", "created_date", "modified"],
        order_by=f"{sort_by} DESC",
        start=(page - 1) * page_size,
        page_length=page_size
    )
    return wishlists
```

### 7.4 Anti-Abuse Measures

- Empty public wishlists (item_count = 0) are excluded from browse results
- Users can report inappropriate public wishlists (links to existing moderation system)
- Public wishlist names are subject to content moderation filters
- Maximum 50 public wishlists per user (to prevent spam)

## 8. Social Proof Counters

### 8.1 Overview

Social proof counters display aggregate favorite/wishlist counts on product pages, seller profile pages, and seller store pages. These counters encourage engagement by showing popularity signals.

### 8.2 Counter Fields on Existing DocTypes

The following fields should be **added to existing DocTypes** (or leveraged if already present):

| DocType | Fieldname | Fieldtype | Label | Read Only | Description |
|---------|-----------|-----------|-------|-----------|-------------|
| Listing | wishlist_count | Int | Wishlist Count | Yes | **Already exists.** Number of times this listing's product has been added to wishlists |
| Listing | favorite_count | Int | Favorite Count | Yes | **New field.** Number of Quick Favorites of type "Product" pointing to this listing's product |
| Product | total_wishlist_count | Int | Total Wishlist Count | Yes | **New field.** Aggregate count across all listings for this product |
| Product | total_favorite_count | Int | Total Favorite Count | Yes | **New field.** Quick Favorite count for this product |
| Seller Profile | favorite_count | Int | Favorite Count | Yes | **New field.** Quick Favorites of type "Seller Profile" pointing to this seller |
| Seller Store | favorite_count | Int | Favorite Count | Yes | **New field.** Quick Favorites of type "Seller Store" pointing to this store |

### 8.3 Counter Update Mechanism

Counters are updated in two ways:

**Real-Time (Event-Driven):**

| Event | Action | Counter Updated |
|-------|--------|-----------------|
| Quick Favorite created (target_type = Product) | Increment | Product.total_favorite_count, Listing.favorite_count |
| Quick Favorite deleted (target_type = Product) | Decrement | Product.total_favorite_count, Listing.favorite_count |
| Quick Favorite created (target_type = Seller Profile) | Increment | Seller Profile.favorite_count |
| Quick Favorite deleted (target_type = Seller Profile) | Decrement | Seller Profile.favorite_count |
| Quick Favorite created (target_type = Seller Store) | Increment | Seller Store.favorite_count |
| Quick Favorite deleted (target_type = Seller Store) | Decrement | Seller Store.favorite_count |
| Wishlist Item added (via child table) | Increment | Product.total_wishlist_count, Listing.wishlist_count |
| Wishlist Item removed (via child table) | Decrement | Product.total_wishlist_count, Listing.wishlist_count |

**Batch Reconciliation (Scheduled):**

A daily scheduled task recalculates all counters from source data to fix any drift caused by edge cases (e.g., direct database edits, failed transactions):

```python
def reconcile_social_proof_counters():
    """
    Daily batch job to reconcile all favorite/wishlist counters.
    Runs at 3:00 AM (configurable).
    """
    # 1. For each Product: count Quick Favorites where target_type="Product" and target_reference=product.name
    # 2. For each Listing: count Wishlist Items where product=listing.product AND seller_store=listing.seller
    # 3. For each Seller Profile: count Quick Favorites where target_type="Seller Profile"
    # 4. For each Seller Store: count Quick Favorites where target_type="Seller Store"
    # 5. Update counters only if they differ (avoid unnecessary writes)
```

## 9. Price-Drop & Low-Stock Notification Plan

### 9.1 Existing Notification Infrastructure

The existing **Notification Template** DocType already includes the following relevant event types in its `event_type` Select field:

- `Price Change Alert` -- triggered when product/listing price changes
- `Low Stock Alert` -- triggered when stock falls below threshold

These event types are ready to use. The plan is to create specific Notification Template records for wishlist-related alerts and build the server-side trigger logic.

### 9.2 Notification Template Records to Create

| Template Code | Template Name | Event Type | Channel | Priority | Subject |
|---------------|---------------|------------|---------|----------|---------|
| `WISHLIST_PRICE_DROP_EMAIL` | Wishlist Price Drop Alert (Email) | Price Change Alert | Email | Normal | `Price drop on {{ product_name }}!` |
| `WISHLIST_PRICE_DROP_PUSH` | Wishlist Price Drop Alert (Push) | Price Change Alert | Push Notification | Normal | `{{ product_name }} price dropped {{ drop_pct }}%` |
| `WISHLIST_PRICE_DROP_INAPP` | Wishlist Price Drop Alert (In-App) | Price Change Alert | In-App | Normal | `Price drop: {{ product_name }}` |
| `WISHLIST_LOW_STOCK_EMAIL` | Wishlist Low Stock Alert (Email) | Low Stock Alert | Email | High | `{{ product_name }} is running low on stock` |
| `WISHLIST_LOW_STOCK_PUSH` | Wishlist Low Stock Alert (Push) | Low Stock Alert | Push Notification | High | `Only {{ stock_qty }} left: {{ product_name }}` |

### 9.3 Template Variables

| Variable | Type | Source | Description |
|----------|------|--------|-------------|
| `product_name` | String | Product.product_name | Name of the product |
| `product_code` | String | Product.product_code | Product identifier |
| `product_url` | String | Computed | Full URL to the product page |
| `original_price` | Currency | Wishlist Item.original_price | Price when user added to wishlist |
| `new_price` | Currency | Listing.selling_price | Current selling price |
| `drop_amount` | Currency | Computed | `original_price - new_price` |

| `low_stock_threshold` | Int | Listing.low_stock_threshold | The threshold that triggered the alert |

| `wishlist_name` | String | User Wishlist.list_name | Name of the wishlist containing the item |
| `wishlist_url` | String | Computed | URL to the user's wishlist |

### 9.4 Price Change Detection Flow

```
Price Change Detection (Triggered by Listing.on_update or scheduled scan):

1. Listing price changes (selling_price updated)
2. Server hook fires: on_update for Listing
3. Query all Wishlist Items where:
   - product = listing.product (or listing = listing.name)
   - is_available = 1
4. For each affected Wishlist Item:
   a. Calculate price_change_pct
   b. Update current_price and price_change_pct fields
   c. If price decreased AND abs(price_change_pct) >= minimum_drop_threshold:
      i.   Check rate-limiting (see Section 9.6)
      ii.  Get the owning User Wishlist -> user
      iii. Send notification via Notification Template (WISHLIST_PRICE_DROP_*)
5. Log notification in Notification Log
```

### 9.5 Low Stock Detection Flow

```
Low Stock Detection (Triggered by Listing.on_update or scheduled scan):

1. Listing stock changes (available_qty updated)
2. Server hook fires: on_update for Listing
3. If available_qty <= low_stock_threshold AND available_qty > 0:
   a. Query all Wishlist Items where product = listing.product
   b. For each affected Wishlist Item:
      i.   Check rate-limiting
      ii.  Get the owning User Wishlist -> user
      iii. Send notification via Notification Template (WISHLIST_LOW_STOCK_*)
4. If available_qty > 0 AND previous_qty was 0 (back in stock):
   a. Query all Wishlist Items where product = listing.product AND is_available = 0
   b. Send back-in-stock notification (WISHLIST_BACK_IN_STOCK_*)
   c. Update is_available = 1 on the Wishlist Item
5. If available_qty == 0:
   a. Update is_available = 0 and availability_note = "Out of stock" on affected Wishlist Items
6. Log notification in Notification Log
```

### 9.6 Rate-Limiting Strategy (Notification Flood Prevention)

To prevent notification floods when prices or stock change frequently (e.g., dynamic pricing, flash sales), the following rate-limiting rules apply:

| Rule | Parameter | Default | Description |
|------|-----------|---------|-------------|
| **Minimum Drop Threshold** | `min_price_drop_pct` | 5% | Only notify if price drop exceeds this percentage |
| **Per-Product Cooldown** | `product_notification_cooldown_hours` | 24 hours | Max one price-drop notification per product per user per 24 hours |
| **Per-User Daily Cap** | `user_daily_notification_cap` | 10 | Max 10 wishlist notifications per user per day |
| **Global Batch Window** | `notification_batch_window_minutes` | 30 minutes | Batch notifications within a 30-minute window into a single digest |
| **Low Stock Cooldown** | `low_stock_cooldown_hours` | 48 hours | Max one low-stock alert per product per user per 48 hours |

### 9.7 Rate-Limiting Implementation

```python
def should_send_notification(user, product, notification_type):
    """
    Check rate-limiting rules before sending a wishlist notification.

    Returns: True if notification should be sent, False if rate-limited.
    """
    # 1. Check per-product cooldown
    last_notification = frappe.db.get_value("Notification Log", {
        "for_user": user,
        "document_type": "Wishlist Item",
        "type": notification_type,
        "subject": ("like", f"%{product}%"),
        "creation": (">", now_datetime() - timedelta(hours=cooldown_hours))
    })
    if last_notification:
        return False

    # 2. Check per-user daily cap
    today_count = frappe.db.count("Notification Log", {
        "for_user": user,
        "document_type": "Wishlist Item",
        "creation": (">", today_start())
    })
    if today_count >= user_daily_notification_cap:
        return False

    return True
```

## 10. Edge Cases

### 10.1 Empty Wishlist

| Scenario | Behavior |
|----------|----------|
| User views a wishlist with zero items | Display an empty state illustration with message: "This wishlist is empty. Browse products to start adding." CTA button: "Browse Products" |
| Public wishlist has zero items | Exclude from public browse results (`item_count > 0` filter). Direct URL access shows empty state with message: "This wishlist has no items yet." |
| User deletes all items from a wishlist | Wishlist remains (not auto-deleted). `item_count` updates to 0. Show empty state. |
| Default wishlist is empty | Same empty state, but with enhanced CTA: "Start shopping to fill your wishlist!" |

### 10.2 Deleted Product in Wishlist

| Scenario | Behavior |
|----------|----------|
| Product status changes to "Archived" or "Inactive" | Wishlist Item.`is_available` set to `0`, `availability_note` = "Product no longer available". Item remains visible but grayed out with strikethrough on name. |
| Product DocType record hard-deleted | Wishlist Item's `product` Link becomes a broken reference. On next wishlist load, detect broken links and set `is_available` = 0, `availability_note` = "Product has been removed". Display with "Removed" badge. |
| Listing deleted but Product exists | If wishlist item referenced a specific Listing, mark as "This seller no longer offers this product". Product link remains valid; user can find alternative sellers. |
| Seller Store suspended/deactivated | Wishlist Item's `seller_store` Link remains valid but seller is inactive. Show notice: "This seller is currently unavailable". |

### 10.3 Notification Flood from Price Changes

| Scenario | Mitigation |
|----------|------------|
| Dynamic pricing engine updates prices every 5 minutes | Per-product cooldown (24h) ensures max 1 notification per product per day |
| Flash sale drops price for 1 hour then reverts | Minimum drop threshold (5%) filters trivial changes; cooldown prevents double-notification |
| Seller bulk-updates prices on 100 products at once | Per-user daily cap (10) prevents inbox flood; remaining items queued for next day or digest |
| Scheduled stock sync triggers many low-stock alerts | Low-stock cooldown (48h) and daily cap prevent mass alerts |
| Price oscillates up and down repeatedly | Only downward price changes trigger notifications; upward changes are silent |

### 10.4 Concurrent Modifications

| Scenario | Behavior |
|----------|----------|
| Two users copy the same shared list simultaneously | Each copy operation is independent; no conflict |
| User adds item while another tab deletes the wishlist | Second operation (add item) fails with "Wishlist not found" error; user sees toast notification |
| User moves item while price update runs | Move operation takes priority (user action); price update will process on next cycle if item exists |

### 10.5 Large Wishlists

| Scenario | Behavior |
|----------|----------|
| User adds 1000+ items to a single wishlist | Pagination (20 items/page) handles display. Performance: child table queries use indexed `parent` field. |
| User has 100+ wishlists | Wishlist list view uses pagination. Default wishlist always appears first. |
| Public browse has 10,000+ public wishlists | Paginated query (20/page) with index on `privacy` and `item_count` fields |

### 10.6 Tenant Isolation

| Scenario | Behavior |
|----------|----------|
| User on Tenant A favorites a product from Tenant B | Allowed only if cross-tenant product visibility is enabled in marketplace settings. If disabled, favoriting is blocked with message. |
| Tenant-specific wishlist queries | All wishlist queries include `tenant` filter when multi-tenancy is active |
