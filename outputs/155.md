# Task 155: Popup Error Detection & Fix - Systematic Audit & Remediation Plan

## 1. Overview

This document defines a systematic methodology for detecting, cataloguing, prioritising, and fixing popup errors across all TradeHub Marketplace DocTypes. Popup errors in Frappe/ERPNext appear as red desk alerts, modal dialogs, or console exceptions that interrupt user workflows on form open, field interaction, or document save.

### Scope

The TradeHub Marketplace comprises **7 custom apps** with **227 DocTypes** total:

| App | DocType Count | Primary Domain |
|-----|--------------|----------------|
| `tradehub_core` | 30 | KYC, buyers, organisations, locations, tenancy |
| `tradehub_seller` | 38 | Seller profiles, listings, SKUs, scoring, balance |
| `tradehub_catalog` | 53 | PIM, products, categories, attributes, media |
| `tradehub_commerce` | 51 | Orders, payments, escrow, commissions, RFQ |
| `tradehub_logistics` | 11 | Shipments, carriers, shipping rules, tracking |
| `tradehub_marketing` | 13 | Campaigns, coupons, group buys, subscriptions |
| `tradehub_compliance` | 31 | Certificates, consent, contracts, reviews, moderation |

### Known Issues from Codebase Analysis

The Phase 1 codebase analysis (subtasks 1-1 through 1-4) already identified several concrete error sources:

1. **`marketplace_shipment.js` references wrong field names** - `set_query('weight_uom', ...)` targets a non-existent field; actual field is `weight_unit` (Select type). All 3 `set_query` calls silently fail.
2. **`shipment.json` `volume_uom` is `Data` not `Link`** - `set_query` on a Data field silently fails. Users see free-text input instead of a filtered dropdown.
3. **598 out of 737 Link fields (81.1%) lack `set_query` filters** - Unfiltered Link fields expose all records across tenants, increasing error probability when users select wrong records.
4. **Inconsistent `tax_id_type` options** - KYC Profile uses `"TC Kimlik No\nVergi No\nPassport"` while Seller Profile/Application uses `"TCKN\nVKN"`. Cross-referencing between these DocTypes can produce mismatches.
5. **Inconsistent UOM default values** - `kg` vs `Kg` vs `Kilogram`, `cm` vs `Cm` vs `Centimeter` across DocTypes. If the default does not match an actual UOM record name, the Link field displays a "not found" error on form load.

### Goals

- [ ] Define a repeatable audit methodology that any developer can execute
- [ ] Catalogue all common error source types with detection signatures
- [ ] Specify a structured error logging format for tracking findings
- [ ] Establish priority tiers with clear classification criteria
- [ ] Produce a complete DocType audit checklist covering all 227 DocTypes
- [ ] Define fix ordering strategy to eliminate highest-impact errors first
- [ ] Plan regression prevention to stop new errors from being introduced

## 2. Systematic Audit Methodology

### 2.1 Audit Process Overview

The audit follows a 5-step process for each DocType:

```
Step 1: Static Analysis     -- Inspect JSON definition and JS client script
Step 2: Permission Scan     -- Check role-based access per user type
Step 3: Form Open Test      -- Open a new form and observe console/desk
Step 4: Interaction Test    -- Click every field, trigger every depends_on
Step 5: Save/Submit Test    -- Attempt save with valid and invalid data
```

### 2.2 Test Matrix

Each DocType must be tested across multiple dimensions:

| Dimension | Values to Test |
|-----------|---------------|
| **User Roles** | System Manager, Seller, Buyer, Guest (if applicable) |
| **Document State** | New (unsaved), Draft, Submitted (if submittable), Amended, Cancelled |
| **Browser** | Chrome (primary), Firefox (secondary) |
| **Data State** | Empty database (no records), Populated database (with sample data) |
| **Tenant Context** | With tenant set, Without tenant set (if applicable) |

### 2.3 Static Analysis Checklist (Per DocType)

Before opening any form in the browser, inspect the DocType definition files:

```
<app>/doctype/<doctype_slug>/<doctype_slug>.json   -- Field definitions
<app>/doctype/<doctype_slug>/<doctype_slug>.js      -- Client script
<app>/doctype/<doctype_slug>/<doctype_slug>.py      -- Server controller
```

**JSON Inspection:**

| Check | What to Look For | Error Type if Violated |
|-------|-----------------|----------------------|
| Link field `options` | Target DocType must exist in the system | `LinkTargetError` - "DocType not found" popup |
| `fetch_from` references | Must follow pattern `link_field.target_field` where both the link field and the target field exist | `FetchFromError` - silent failure or console error |
| `depends_on` expressions | `eval:doc.field_name` must reference fields that exist in this DocType | `DependsOnError` - JS eval exception popup |
| `mandatory_depends_on` | Same as `depends_on` - field references must be valid | `MandatoryDependsOnError` - save-time validation crash |
| `default` values on Link fields | Default value must match an actual record name in the target DocType | `DefaultValueError` - "not found" popup on form load |
| `read_only_depends_on` | Expression must reference valid fields | `ReadOnlyDependsOnError` - JS eval exception |
| `collapsible_depends_on` | Expression must reference valid fields | `CollapsibleDependsOnError` - section rendering failure |
| Child table `options` | Must reference a valid child DocType | `ChildTableError` - table fails to render |
| `set_query` in JS file | Must target actual field names that exist in the JSON | `SetQueryMismatch` - silently fails, no filtering applied |
| `set_query` field types | `set_query` only works on `Link` fields, not `Data` or `Select` | `SetQueryWrongType` - silently fails |

**JS Inspection:**

| Check | What to Look For | Error Type if Violated |
|-------|-----------------|----------------------|
| `frappe.ui.form.on('DocType', ...)` | DocType name must match exactly (case-sensitive) | `JSBindingError` - all event handlers silently fail |
| `frm.set_query('field', ...)` | Field name must exist in DocType JSON | `SetQueryMismatch` - silently fails |
| `frm.set_value('field', ...)` | Field name must exist | `SetValueError` - console error, possible popup |
| `frm.toggle_display('field', ...)` | Field name must exist | `ToggleDisplayError` - console warning |
| `cur_frm.call(...)` or `frm.call(...)` | Server method must exist in the `.py` controller | `ServerMethodError` - red popup "Method not found" |
| `frappe.call({ method: ... })` | Whitelisted API method must exist and be accessible | `APICallError` - red popup or 403/404 |
| Undefined variable references | Variables used before declaration | `JSRuntimeError` - console TypeError, possible popup |
| Missing null checks | `frm.doc.field.property` without checking if field is null | `JSNullError` - TypeError popup |

### 2.5 Automated Scanning Script

A Python script should be developed to automate the static analysis phase. Pseudocode:

```python
import frappe
import json
import os

def audit_all_doctypes():
    """Scan all TradeHub DocType JSONs for potential error sources."""
    apps = [
        'tradehub_core', 'tradehub_seller', 'tradehub_catalog',
        'tradehub_commerce', 'tradehub_logistics', 'tradehub_marketing',
        'tradehub_compliance'
    ]
    findings = []

    for app in apps:
        doctype_dir = frappe.get_app_path(app, app, 'doctype')
        for dt_folder in os.listdir(doctype_dir):
            json_path = os.path.join(doctype_dir, dt_folder, f'{dt_folder}.json')
            if not os.path.exists(json_path):
                continue

            with open(json_path) as f:
                dt_def = json.load(f)

            fieldnames = {f['fieldname'] for f in dt_def.get('fields', [])}

            for field in dt_def.get('fields', []):
                # Check 1: Link field targets
                if field.get('fieldtype') == 'Link':
                    target = field.get('options', '')
                    if target and not frappe.db.exists('DocType', target):
                        findings.append({
                            'doctype': dt_def['name'],
                            'field': field['fieldname'],
                            'error_type': 'LinkTargetError',
                            'message': f'Link target "{target}" does not exist',
                            'priority': 'Critical'
                        })

                # Check 2: fetch_from validity
                fetch_from = field.get('fetch_from', '')
                if fetch_from:
                    parts = fetch_from.split('.')
                    if len(parts) != 2:
                        findings.append({
                            'doctype': dt_def['name'],
                            'field': field['fieldname'],
                            'error_type': 'FetchFromError',
                            'message': f'Invalid fetch_from format: "{fetch_from}"',
                            'priority': 'Medium'
                        })
                    elif parts[0] not in fieldnames:
                        findings.append({
                            'doctype': dt_def['name'],
                            'field': field['fieldname'],
                            'error_type': 'FetchFromError',
                            'message': f'fetch_from references non-existent field: "{parts[0]}"',
                            'priority': 'High'
                        })

                # Check 3: depends_on field references
                for dep_attr in ['depends_on', 'mandatory_depends_on',
                                 'read_only_depends_on', 'collapsible_depends_on']:
                    dep_expr = field.get(dep_attr, '')
                    if dep_expr and dep_expr.startswith('eval:'):
                        # Extract doc.field_name references
                        import re
                        refs = re.findall(r'doc\.(\w+)', dep_expr)
                        for ref in refs:
                            if ref not in fieldnames and ref not in ('name', 'doctype',
                                    'docstatus', 'owner', 'creation', 'modified'):
                                findings.append({
                                    'doctype': dt_def['name'],
                                    'field': field['fieldname'],
                                    'error_type': 'DependsOnError',
                                    'message': f'{dep_attr} references non-existent field: "{ref}"',
                                    'priority': 'High'
                                })

                # Check 4: Default values on Link fields
                if field.get('fieldtype') == 'Link' and field.get('default'):
                    default_val = field['default']
                    target = field.get('options', '')
                    if target and default_val not in ('__user', ''):
                        if not frappe.db.exists(target, default_val):
                            findings.append({
                                'doctype': dt_def['name'],
                                'field': field['fieldname'],
                                'error_type': 'DefaultValueError',
                                'message': f'Default "{default_val}" not found in {target}',
                                'priority': 'Medium'
                            })

    return findings
```

## 3. Error Detection Tools - Detailed Usage

### 3.1 Browser Console Error Capture

**Setup:**

1. Open Chrome DevTools (F12) -> Console tab
2. Enable "Preserve log" to retain errors across page navigations
3. Set filter level to "Errors" and "Warnings"
4. Optionally enable "Log XMLHttpRequests" to see all API calls

**Frappe-Specific Console Patterns:**

| Console Pattern | Meaning | Severity |
|-----------------|---------|----------|
| `Uncaught TypeError: Cannot read properties of undefined` | JS null reference, usually from missing field or broken fetch_from chain | High |
| `frappe.throw()` or `frappe.msgprint()` with red indicator | Server-side validation error surfaced to client | Varies |
| `404 (Not Found)` on `/api/method/...` | Missing or non-whitelisted server method called from JS | High |
| `403 (Forbidden)` on `/api/resource/...` | Permission denied for the current user role | Medium |
| `417 (Expectation Failed)` | Server-side `frappe.throw()` - validation or business logic error | High |
| `409 (Conflict)` | Document modified by another user (stale version) | Low |
| `[Vue warn]` or `[Deprecation]` | Framework-level warnings, usually cosmetic | Low |
| `SyntaxError` in custom script | JS syntax error in the DocType client script | Critical |
| `ReferenceError: X is not defined` | Undefined variable in client script | High |
| `LinkValidationError` | Link field value doesn't match any record in target DocType | High |

**Capture Protocol:**

```javascript
// Inject this in the browser console before testing to capture all errors:
window.__tradehub_errors = [];
window.addEventListener('error', function(e) {
    window.__tradehub_errors.push({
        timestamp: new Date().toISOString(),
        message: e.message,
        source: e.filename,
        line: e.lineno,
        column: e.colno,
        type: 'js_error'
    });
});

// Override frappe.throw to capture application errors:
const _original_throw = frappe.throw;
frappe.throw = function(msg) {
    window.__tradehub_errors.push({
        timestamp: new Date().toISOString(),
        message: typeof msg === 'string' ? msg : msg.message || JSON.stringify(msg),
        type: 'frappe_throw'
    });
    _original_throw.apply(this, arguments);
};

// Dump captured errors:
// JSON.stringify(window.__tradehub_errors, null, 2)
```

## 4. Common Error Source Catalogue

### 4.1 Broken Link Fields

**Description:** A Link field whose `options` (target DocType) does not exist or whose default/fetched value does not match any record.

**Detection Signatures:**
- Console: `LinkValidationError` on save
- Console: Empty dropdown when clicking a Link field (target DocType exists but has no records)
- Error Log: `frappe.exceptions.LinkValidationError`
- Static: `options` value in JSON not matching any installed DocType

**Common Scenarios in TradeHub:**

| Scenario | Example | Impact |
|----------|---------|--------|
| Target DocType not installed | Link to a DocType from an app not in `installed_apps` | Form error popup on render |
| Deleted record referenced | Order references a deleted Seller Profile | Save fails with LinkValidationError |
| Cross-tenant reference | Link field shows records from all tenants, user picks wrong one | Data integrity issue, potential save error |
| Renamed DocType | Link still targets old DocType name | "DocType not found" popup |

**Fix Pattern:**
```python
# Server-side: Ensure Link target exists
if not frappe.db.exists('DocType', field.options):
    frappe.throw(f'Link field {field.fieldname} targets non-existent DocType: {field.options}')

# Client-side: Add set_query to filter valid records
frm.set_query('target_field', function() {
    return {
        filters: {
            'status': 'Active',
            'tenant': frm.doc.tenant || frappe.defaults.get_default('tenant')
        }
    };
});
```

### 4.2 Missing Permissions

**Description:** User role lacks read/write/create permission on the DocType or on a linked DocType, causing popup errors when navigating to forms or attempting to save.

**Detection Signatures:**
- Red popup: "Insufficient Permission for [DocType]"
- Console: `403 Forbidden` on API calls
- Error Log: `frappe.exceptions.PermissionError`
- Network tab: `403` response on `GET /api/resource/DocType/name`

**Common Scenarios in TradeHub:**

| Scenario | Example | Impact |
|----------|---------|--------|
| Missing read on linked DocType | Seller can't read `Commission Plan` referenced from Order | Link field shows nothing, save may fail |
| Missing create on target DocType | User tries to quick-create from Link field "+" button | Red popup "Not permitted to create" |
| Permission Level (permlevel) restriction | `verification_status` at permlevel 1 but role not granted level 1 write | Field appears read-only, user confused |
| Role Profile mismatch | New role added to DocType but not to user's Role Profile | Entire form inaccessible |

**Fix Pattern:**
```python
# In DocType JSON permissions array:
{
    "permissions": [
        {"role": "Seller", "read": 1, "write": 1, "create": 1},
        {"role": "Buyer", "read": 1, "write": 0, "create": 0},
        {"role": "System Manager", "read": 1, "write": 1, "create": 1, "delete": 1}
    ]
}

# For permlevel restrictions:
{
    "fieldname": "verification_status",
    "permlevel": 1
    // Ensure role has perm for permlevel 1:
    // {"role": "System Manager", "permlevel": 1, "read": 1, "write": 1}
}
```

### 4.3 Invalid `fetch_from` References

**Description:** A field's `fetch_from` property references a link field or target field that doesn't exist, causing either silent failure (value not populated) or a console error.

**Detection Signatures:**
- Field value remains empty when it should be auto-populated
- Console: `Cannot read properties of undefined (reading 'field_name')`
- No explicit popup in most cases (silent failure)

**Common Patterns:**

| Pattern | Example | Result |
|---------|---------|--------|
| Link field doesn't exist | `fetch_from: "seller.company_name"` but field `seller` doesn't exist | Silent failure |
| Target field doesn't exist on linked DocType | `fetch_from: "seller_profile.company_name"` but Seller Profile has `store_name` not `company_name` | Silent failure |
| Link field is renamed | Field was `seller` but renamed to `seller_profile`; `fetch_from` not updated | Silent failure |
| Chain fetch (nested) | `fetch_from: "seller_profile.organization.name"` - Frappe doesn't support nested fetch_from | Returns literal string |

**Validation Script:**
```python
def validate_fetch_from(doctype_name):
    """Check all fetch_from references in a DocType are valid."""
    meta = frappe.get_meta(doctype_name)
    errors = []
    fieldnames = {f.fieldname for f in meta.fields}

    for field in meta.fields:
        if not field.fetch_from:
            continue
        parts = field.fetch_from.split('.')
        if len(parts) != 2:
            errors.append(f'{field.fieldname}: Invalid fetch_from format "{field.fetch_from}"')
            continue

        link_field_name, target_field_name = parts

        # Check link field exists
        if link_field_name not in fieldnames:
            errors.append(f'{field.fieldname}: Link field "{link_field_name}" does not exist')
            continue

        # Check link field is actually a Link type
        link_field = meta.get_field(link_field_name)
        if link_field.fieldtype != 'Link':
            errors.append(f'{field.fieldname}: "{link_field_name}" is {link_field.fieldtype}, not Link')
            continue

        # Check target field exists on linked DocType
        target_dt = link_field.options
        if target_dt and frappe.db.exists('DocType', target_dt):
            target_meta = frappe.get_meta(target_dt)
            target_fieldnames = {f.fieldname for f in target_meta.fields}
            # Also check standard fields
            target_fieldnames.update(['name', 'owner', 'creation', 'modified',
                                       'modified_by', 'docstatus'])
            if target_field_name not in target_fieldnames:
                errors.append(
                    f'{field.fieldname}: Field "{target_field_name}" '
                    f'not found in {target_dt}'
                )

    return errors
```

### 4.4 Bad `depends_on` Expressions

**Description:** A field's `depends_on`, `mandatory_depends_on`, `read_only_depends_on`, or `collapsible_depends_on` expression references a field name that doesn't exist, causing a JavaScript evaluation error.

**Detection Signatures:**
- Console: `ReferenceError: [field] is not defined` or `TypeError: Cannot read properties of undefined`
- Form layout breaks (sections not rendering correctly)
- Fields that should be hidden are always visible (or vice versa)
- `mandatory_depends_on` failures cause save-time crashes

**Common Patterns:**

| Pattern | Example | Result |
|---------|---------|--------|
| Typo in field reference | `depends_on: "eval:doc.seller_tpye=='Individual'"` | Field always hidden or always visible |
| Renamed field not updated | `depends_on: "eval:doc.category"` but field renamed to `product_category` | Visibility condition never matches |
| Field from different DocType | Copy-pasted from another DocType with different fields | eval error on form load |
| Complex expression syntax error | `depends_on: "eval:doc.status == 'Active' && doc.type"` - missing comparison | May work but logic is wrong |
| Array method on string | `depends_on: "eval:['a','b'].includes(doc.status)"` - works, but `doc.status.includes(...)` on null crashes | TypeError if field is empty |

**Impact by `depends_on` Type:**

| Attribute | Failure Impact |
|-----------|---------------|
| `depends_on` | Field/section visibility wrong - cosmetic/functional |
| `mandatory_depends_on` | Save fails unexpectedly or required field not enforced |
| `read_only_depends_on` | Field editable when it shouldn't be, or locked when it should be editable |
| `collapsible_depends_on` | Section always expanded or always collapsed |

### 4.5 JavaScript Errors in Client Scripts

**Description:** Runtime errors in DocType `.js` files that cause popups, broken form behavior, or silent failures.

**Detection Signatures:**
- Red popup: "There were errors" with stack trace
- Console: `TypeError`, `ReferenceError`, `SyntaxError`
- Form buttons not working (click handler threw exception)
- Fields not updating on change (event handler threw exception)

**Common Patterns:**

| Pattern | Example | Result |
|---------|---------|--------|
| Wrong DocType name in `frappe.ui.form.on()` | `frappe.ui.form.on('Marketplace Shipments', ...)` (plural) | All handlers silently ignored |
| Missing null check on `frm.doc.field` | `frm.doc.seller_profile.length` when seller_profile is null | TypeError popup |
| `set_query` on wrong field name | `frm.set_query('weight_uom', ...)` but field is `weight_unit` | Silently fails, no filtering |
| `set_query` on non-Link field | `frm.set_query('volume_uom', ...)` but field is Data type | Silently fails |
| Calling undefined method | `frm.custom_method()` that doesn't exist | TypeError popup |
| Async/callback errors | `frappe.call` without error callback; server returns 500 | Unhandled promise rejection |
| `cur_frm` used outside form context | Script loaded but accessed before form is ready | TypeError popup |

**Known Issues in TradeHub:**

| File | Issue | Status |
|------|-------|--------|
| `marketplace_shipment.js` | `set_query('weight_uom', ...)` targets non-existent field (`weight_unit` is the actual field) | Confirmed - 3 broken set_query calls |
| `marketplace_shipment.js` | `set_query('dimension_uom', ...)` targets non-existent field (`dimension_unit` is actual) | Confirmed |
| `marketplace_shipment.js` | `set_query('volume_uom', ...)` field doesn't exist on this DocType at all | Confirmed |
| `shipment.js` | `set_query('volume_uom', ...)` on a Data field, not Link | Confirmed |

### 4.6 Server-Side Hook Errors

**Description:** Python `validate`, `before_save`, `on_update`, `after_insert` hooks that raise unhandled exceptions, producing red popup errors on save.

**Detection Signatures:**
- Red popup after clicking Save: "There were errors"
- Error Log entry with full Python traceback
- `bench --verbose` shows 500 response on `save_doc`

**Common Patterns:**

| Pattern | Example | Result |
|---------|---------|--------|
| Missing import | `from tradehub_core.utils import func` where `func` was moved | ImportError on save |
| Database field mismatch | Python code references `doc.old_field` after schema change | AttributeError |
| API call in validate | External API call fails, exception not caught | Timeout or ConnectionError popup |
| Circular import | Module A imports from Module B which imports from A | ImportError on first use |
| frappe.throw in wrong context | `frappe.throw()` inside a background job | Unhandled exception in worker |

### 4.7 Dynamic Link Field Errors

**Description:** Fields with `fieldtype: "Dynamic Link"` where the referenced doctype field is empty or contains an invalid DocType name.

**Detection Signatures:**
- Empty dropdown with no options on Dynamic Link field
- Console: "DocType [value] not found"
- Save error: LinkValidationError

**Common in TradeHub:**
- `reference_type` + `reference_name` patterns in Escrow Event, Order Event, etc.
- `entity_type` + `entity_name` in Media Asset
- If `reference_type` is set to a value not in the Select options, the Dynamic Link breaks

### 4.8 Child Table Errors

**Description:** Errors in child table (Table field) rendering or interaction, caused by missing child DocType, broken field references within the child, or parent-child field mismatches.

**Detection Signatures:**
- Red popup: "DocType [child_name] not found" on form load
- Empty table that should have rows
- "Add Row" button produces error
- Console: errors during child table rendering

**Common Patterns:**

| Pattern | Example | Result |
|---------|---------|--------|
| Child DocType not installed | `Table` field references `Ad Campaign Item` (not yet created) | Form fails to render table section |
| Child field references parent field that doesn't exist | Child `depends_on: "eval:parent.status=='Active'"` but parent has `workflow_state` not `status` | Child field visibility broken |
| fetch_from in child table | Child field `fetch_from: "product.product_name"` where `product` is in the child row | Works if field exists in child row's Link |

## 5. Error Logging Format Specification

### 5.1 Error Log Entry Schema

Every error discovered during audit must be logged in the following structured format:

```json
{
    "id": "ERR-001",
    "doctype": "Marketplace Shipment",
    "app": "tradehub_logistics",
    "field": "weight_uom",
    "action": "form_open",
    "error_type": "SetQueryMismatch",
    "message": "set_query('weight_uom') targets non-existent field; actual field is 'weight_unit' (Select type). set_query silently fails.",
    "source": "marketplace_shipment.js:setup_uom_filters()",
    "priority": "Medium",
    "detection_method": "static_analysis",
    "user_impact": "UOM dropdown shows all UOMs instead of filtered list",
    "fix_complexity": "Low",
    "fix_description": "Rename set_query target to 'weight_unit' and change field from Select to Link, or remove the set_query call",
    "related_errors": ["ERR-002", "ERR-003"],
    "discovered_at": "2026-03-05",
    "fixed_at": null,
    "verified_at": null
}
```

### 5.2 Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | String | Yes | Unique error identifier, format: `ERR-XXXX` (sequential) |
| `doctype` | String | Yes | Name of the affected DocType |
| `app` | String | Yes | TradeHub app containing the DocType |
| `field` | String | No | Specific field causing the error (null for form-level errors) |
| `action` | Enum | Yes | One of: `form_open`, `field_click`, `field_change`, `save`, `submit`, `cancel`, `amend`, `print`, `workflow_action` |
| `error_type` | String | Yes | Error classification (see Section 4 catalogue) |
| `message` | String | Yes | Human-readable description of the error |
| `source` | String | Yes | File and function/line where the error originates |
| `priority` | Enum | Yes | One of: `Critical`, `High`, `Medium`, `Low` (see Section 6) |
| `detection_method` | Enum | Yes | One of: `static_analysis`, `browser_console`, `error_log`, `manual_testing`, `automated_script` |
| `user_impact` | String | Yes | Description of how this error affects end users |
| `fix_complexity` | Enum | Yes | One of: `Low` (< 30 min), `Medium` (1-4 hours), `High` (1+ days) |
| `fix_description` | String | Yes | Brief description of the required fix |
| `related_errors` | Array | No | IDs of related errors (same root cause) |
| `discovered_at` | Date | Yes | Date when error was found |
| `fixed_at` | Date | No | Date when fix was committed |
| `verified_at` | Date | No | Date when fix was verified in staging |

## 6. Priority Tier Definitions

### 6.1 Critical - Form Won't Open

**Definition:** The DocType form fails to load or render, making the entire DocType unusable.

**Criteria (any one qualifies):**
- [ ] Form produces a red popup error immediately on navigation to `/app/doctype-name/new`
- [ ] Form produces a blank white page or infinite loading spinner
- [ ] JavaScript SyntaxError in the `.js` file prevents the form controller from initializing
- [ ] Link to a non-existent DocType causes "DocType not found" on page load
- [ ] Child table references a non-existent child DocType, preventing form render
- [ ] Permission error prevents all users (including System Manager) from accessing the form

**Impact:** Complete feature blockage. No user can create, view, or edit documents of this type.

**Examples:**
- `frappe.ui.form.on('Non Existent DocType', { ... })` in JS file - form loads but no handlers work
- A `Table` field with `options: "Missing Child Table"` - form fails to render that section
- SyntaxError in client script - entire form JS fails to execute

**SLA Target:** Fix within 4 hours of discovery.

**Count Estimate:** 0-5 across all 227 DocTypes (these tend to be caught quickly in development).

### 6.2 High - Can't Save

**Definition:** The form opens and renders correctly, but the user cannot successfully save or submit the document due to an error.

**Criteria (any one qualifies):**
- [ ] Clicking "Save" produces a red popup error (not a validation message)
- [ ] Server-side `validate` hook raises an unhandled exception
- [ ] Required field is hidden by a broken `depends_on` but still mandatory, causing "mandatory field missing" on save
- [ ] `mandatory_depends_on` references a non-existent field, causing unpredictable mandatory behavior
- [ ] `fetch_from` failure leaves a required field empty, causing save failure
- [ ] Link field default value doesn't exist, causing LinkValidationError on save of new document
- [ ] Unique constraint violation due to wrong autoname/naming pattern

**Impact:** Users can view data but cannot create or update records. Workflow blocked.

**Examples:**
- `mandatory_depends_on: "eval:doc.seller_type=='Individual'"` where `seller_type` doesn't exist - field is always or never mandatory, causing save failures
- Default value `"kg"` on a Link field, but UOM record is named `"Kg"` (case mismatch) - new document save fails
- `validate()` hook calls `frappe.get_doc()` on a deleted record - unhandled DoesNotExistError

**SLA Target:** Fix within 1 business day.

**Count Estimate:** 5-20 across all DocTypes.

### 6.3 Medium - Field-Level Error

**Definition:** The form opens and saves, but individual fields exhibit errors: wrong values fetched, incorrect visibility, broken filtering, or console warnings.

**Criteria (any one qualifies):**
- [ ] `fetch_from` silently fails, leaving a field empty that should be auto-populated
- [ ] `set_query` targets wrong field or non-Link field, so filtering doesn't work (but no popup)
- [ ] `depends_on` expression has a typo, causing wrong field visibility
- [ ] Link field shows all records instead of filtered subset (usable but confusing)
- [ ] Field value not cleared when parent Link field changes
- [ ] Console warning (not error) on field interaction

**Impact:** Users can work around the issue but experience confusion, data quality problems, or extra manual steps.

**Examples:**
- `marketplace_shipment.js` `set_query('weight_uom', ...)` on non-existent field - silently fails, no filtering applied
- `fetch_from: "seller_profile.company_name"` where Seller Profile has `store_name` - field stays empty
- 598 unfiltered Link fields showing all records across tenants

**SLA Target:** Fix within 1 sprint (2 weeks).

**Count Estimate:** 50-200 across all DocTypes (most common category).

### 6.4 Low - Cosmetic / Informational

**Definition:** The error does not affect functionality but creates visual noise, console clutter, or minor UX issues.

**Criteria (any one qualifies):**
- [ ] Console warning that doesn't affect functionality
- [ ] Deprecated API usage warning
- [ ] Minor layout issue (collapsible section always open/closed)
- [ ] Tooltip or help text referencing wrong information
- [ ] Form tour or onboarding step pointing to non-existent element
- [ ] Inconsistent field label capitalization or naming

**Impact:** No functional impact. Professional polish issue.

**Examples:**
- `[Deprecation] frappe.model.std_fields` warning in console
- Section Break `collapsible_depends_on` broken - section always expanded
- Field label says "Seller ID" but field stores Seller Profile name

**SLA Target:** Fix in next maintenance cycle or leave as technical debt.

**Count Estimate:** 20-100 across all DocTypes.

### 6.5 Priority Decision Matrix

When the priority isn't immediately clear, use this decision matrix:

```
                          Can form open?
                         /              \
                       NO                YES
                       |                  |
                    CRITICAL         Can user save?
                                    /              \
                                  NO                YES
                                  |                  |
                                HIGH          Does it affect data?
                                              /              \
                                            YES               NO
                                            |                  |
                                          MEDIUM              LOW
```

### 9.2 Automated Test Suite

Create a test module that programmatically validates all DocTypes:

```python
# tradehub_core/tests/test_doctype_integrity.py

import frappe
import unittest

class TestDocTypeIntegrity(unittest.TestCase):
    """Verify all TradeHub DocType definitions are error-free."""

    TRADEHUB_APPS = [
        'tradehub_core', 'tradehub_seller', 'tradehub_catalog',
        'tradehub_commerce', 'tradehub_logistics', 'tradehub_marketing',
        'tradehub_compliance'
    ]

    def test_all_link_targets_exist(self):
        """Every Link field must reference an installed DocType."""
        for app in self.TRADEHUB_APPS:
            doctypes = frappe.get_all('DocType', filters={'module': app}, pluck='name')
            for dt in doctypes:
                meta = frappe.get_meta(dt)
                for field in meta.fields:
                    if field.fieldtype == 'Link' and field.options:
                        self.assertTrue(
                            frappe.db.exists('DocType', field.options),
                            f'{dt}.{field.fieldname}: Link target "{field.options}" does not exist'
                        )

    def test_all_fetch_from_valid(self):
        """Every fetch_from must reference a valid link_field.target_field path."""
        for app in self.TRADEHUB_APPS:
            doctypes = frappe.get_all('DocType', filters={'module': app}, pluck='name')
            for dt in doctypes:
                meta = frappe.get_meta(dt)
                fieldnames = {f.fieldname for f in meta.fields}
                for field in meta.fields:
                    if not field.fetch_from:
                        continue
                    parts = field.fetch_from.split('.')
                    self.assertEqual(
                        len(parts), 2,
                        f'{dt}.{field.fieldname}: Invalid fetch_from format "{field.fetch_from}"'
                    )
                    self.assertIn(
                        parts[0], fieldnames,
                        f'{dt}.{field.fieldname}: fetch_from link field "{parts[0]}" not found'
                    )

    def test_all_depends_on_fields_exist(self):
        """Every depends_on expression must reference existing fields."""
        import re
        for app in self.TRADEHUB_APPS:
            doctypes = frappe.get_all('DocType', filters={'module': app}, pluck='name')
            for dt in doctypes:
                meta = frappe.get_meta(dt)
                fieldnames = {f.fieldname for f in meta.fields}
                std_fields = {'name', 'doctype', 'docstatus', 'owner',
                              'creation', 'modified', 'modified_by', 'idx',
                              'parent', 'parenttype', 'parentfield'}
                all_fields = fieldnames | std_fields

                for field in meta.fields:
                    for attr in ['depends_on', 'mandatory_depends_on',
                                 'read_only_depends_on', 'collapsible_depends_on']:
                        expr = getattr(field, attr, None)
                        if not expr or not expr.startswith('eval:'):
                            continue
                        refs = re.findall(r'doc\.(\w+)', expr)
                        for ref in refs:
                            self.assertIn(
                                ref, all_fields,
                                f'{dt}.{field.fieldname}: {attr} references '
                                f'non-existent field "{ref}"'
                            )

    def test_no_link_default_mismatch(self):
        """Link field defaults must match actual records or be dynamic."""
        dynamic_defaults = {'__user', '', None}
        for app in self.TRADEHUB_APPS:
            doctypes = frappe.get_all('DocType', filters={'module': app}, pluck='name')
            for dt in doctypes:
                meta = frappe.get_meta(dt)
                for field in meta.fields:
                    if field.fieldtype != 'Link' or not field.default:
                        continue
                    if field.default in dynamic_defaults:
                        continue
                    if field.options:
                        self.assertTrue(
                            frappe.db.exists(field.options, field.default),
                            f'{dt}.{field.fieldname}: Default "{field.default}" '
                            f'not found in {field.options}'
                        )
```
