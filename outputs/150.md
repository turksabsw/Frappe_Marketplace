# Task 150: Buy Box Algorithm Planning Document

## Overview

This document specifies the Buy Box winner algorithm for the TradeHub marketplace. When multiple sellers offer the same SKU Product, the system must determine which seller "wins" the Buy Box -- the primary purchase option displayed to buyers. The algorithm uses a **weighted scoring formula** with **admin-configurable weights** (via the Buy Box Settings DocType), logs every recalculation (via the Buy Box Log DocType), and provides sellers with transparent feedback on their standing. This plan extends the existing **Buy Box Entry** DocType (`autoname: BBX-.#####`) which already stores per-seller-per-product scoring data.

## 2. Scoring Criteria Table

The Buy Box algorithm evaluates each seller's offer on **six** criteria. Each criterion produces a **normalized score** (0.0 to 1.0), which is then multiplied by its configurable weight.

| # | Criterion | Fieldname (Score) | Data Source (Buy Box Entry) | Normalization Method | Default Weight | Weight Range |
|----|-----------|-------------------|-----------------------------|-----------------------|----------------|--------------|
| 1 | Price Competitiveness | price_score | offer_price, currency | Lower is better: `1.0 - (price - min_price) / (max_price - min_price)` | 0.35 | 0.00 - 1.00 |
| 2 | Delivery Speed | delivery_score | delivery_days, seller_on_time_delivery_rate | Lower days is better: `1.0 - (days - min_days) / (max_days - min_days)`, modified by on-time rate | 0.20 | 0.00 - 1.00 |
| 3 | Seller Rating | rating_score | seller_average_rating, seller_total_reviews | Rating/5.0, with review-count confidence: `(rating/5.0) * min(1.0, total_reviews / min_reviews_threshold)` | 0.20 | 0.00 - 1.00 |
| 4 | Stock Reliability | stock_score | stock_available, is_stock_guaranteed, last_stock_update | Available = 1.0, low stock (<10) = 0.5, out = 0.0; bonus for guaranteed stock (+0.1, capped at 1.0); recency penalty if last_stock_update > 7 days | 0.15 | 0.00 - 1.00 |
| 5 | Service Quality | service_score | seller_response_rate, seller_refund_rate | `response_rate * 0.7 + (1.0 - refund_rate) * 0.3` | 0.05 | 0.00 - 1.00 |
| 6 | Seller Tier Bonus | seller_tier_bonus | seller_tier | Tier-based bonus lookup from Buy Box Settings (e.g., Gold=0.10, Silver=0.05, Bronze=0.02, Standard=0.00) | 0.05 | 0.00 - 1.00 |

### Normalization Details

**Price Score Normalization:**
```
Given all active entries for the same sku_product:
  min_price = min(offer_price for all entries)
  max_price = max(offer_price for all entries)

  If min_price == max_price:
    normalized_price = 1.0  (all sellers same price)
  Else:
    normalized_price = 1.0 - (this_price - min_price) / (max_price - min_price)
```

**Delivery Score Normalization:**
```
Given all active entries for the same sku_product:
  min_days = min(delivery_days for all entries)
  max_days = max(delivery_days for all entries)

  If min_days == max_days:
    base_delivery = 1.0
  Else:
    base_delivery = 1.0 - (this_days - min_days) / (max_days - min_days)

  on_time_modifier = seller_on_time_delivery_rate / 100.0
  normalized_delivery = base_delivery * 0.7 + on_time_modifier * 0.3
```

**Rating Score Normalization:**
```
min_reviews_threshold = Buy Box Settings.min_reviews_for_full_confidence (default: 10)
confidence = min(1.0, seller_total_reviews / min_reviews_threshold)
normalized_rating = (seller_average_rating / 5.0) * confidence
```

**Stock Score Normalization:**
```
stock_threshold_low = Buy Box Settings.low_stock_threshold (default: 10)

If stock_available <= 0:
  base_stock = 0.0
Elif stock_available < stock_threshold_low:
  base_stock = 0.5
Else:
  base_stock = 1.0

If is_stock_guaranteed:
  base_stock = min(1.0, base_stock + 0.1)

If last_stock_update is older than 7 days:
  base_stock = base_stock * 0.8  (20% recency penalty)

normalized_stock = base_stock
```

**Service Score Normalization:**
```
normalized_service = (seller_response_rate / 100.0) * 0.7 + (1.0 - seller_refund_rate / 100.0) * 0.3
```

**Seller Tier Bonus Normalization:**
```
tier_bonus_map loaded from Buy Box Settings child table (Buy Box Tier Bonus rows)
normalized_tier_bonus = tier_bonus_map.get(seller_tier, 0.0)
```

## 3. Scoring Formula

### 3.1 Weighted Score Calculation

```
weighted_score = sum(weight_i * normalized_score_i) for i in [price, delivery, rating, stock, service, tier_bonus]
```

**Expanded:**
```
weighted_score = (
    weight_price     * normalized_price_score     +
    weight_delivery  * normalized_delivery_score  +
    weight_rating    * normalized_rating_score    +
    weight_stock     * normalized_stock_score     +
    weight_service   * normalized_service_score   +
    weight_tier      * normalized_tier_bonus
)
```

### 3.2 Weight Constraint

Weights are loaded from the **Buy Box Settings** DocType (see Section 4). The system enforces:

```
sum(all weights) == 1.0
```

If the admin saves weights that do not sum to 1.0, the system **auto-normalizes** them:

```
actual_weight_i = configured_weight_i / sum(all configured_weights)
```

A validation warning is shown when saving Buy Box Settings if weights do not sum to 1.0.

### 3.3 Final Score Scale

The `buy_box_score` stored on the Buy Box Entry is scaled to 0-100:

```
buy_box_score = weighted_score * 100.0
```

The individual component scores (`price_score`, `delivery_score`, `rating_score`, `stock_score`) are also stored as 0-100 values for UI display.

### 3.4 Example Calculation

Given two sellers for the same product:

| Metric | Seller A | Seller B |
|--------|----------|----------|
| offer_price | 100.00 TRY | 120.00 TRY |
| delivery_days | 3 | 5 |
| seller_average_rating | 4.5 | 4.8 |
| seller_total_reviews | 50 | 8 |
| stock_available | 200 | 15 |
| is_stock_guaranteed | Yes | No |
| seller_on_time_delivery_rate | 95% | 88% |
| seller_response_rate | 90% | 85% |
| seller_refund_rate | 2% | 5% |
| seller_tier | Gold | Silver |

**Normalized Scores (Seller A):**
- Price: `1.0 - (100 - 100) / (120 - 100) = 1.0`
- Delivery: `(1.0 - (3-3)/(5-3)) * 0.7 + 0.95 * 0.3 = 0.7 + 0.285 = 0.985`
- Rating: `(4.5/5.0) * min(1.0, 50/10) = 0.9 * 1.0 = 0.9`
- Stock: `1.0 + 0.1 (guaranteed) = 1.0 (capped)`
- Service: `0.9 * 0.7 + (1.0 - 0.02) * 0.3 = 0.63 + 0.294 = 0.924`
- Tier: `0.10 (Gold)`

**Weighted Score (Seller A):**
```
= 0.35 * 1.0 + 0.20 * 0.985 + 0.20 * 0.9 + 0.15 * 1.0 + 0.05 * 0.924 + 0.05 * 0.10
= 0.35 + 0.197 + 0.18 + 0.15 + 0.0462 + 0.005
= 0.9282
buy_box_score = 92.82
```

**Normalized Scores (Seller B):**
- Price: `1.0 - (120 - 100) / (120 - 100) = 0.0`
- Delivery: `(1.0 - (5-3)/(5-3)) * 0.7 + 0.88 * 0.3 = 0.0 + 0.264 = 0.264`
- Rating: `(4.8/5.0) * min(1.0, 8/10) = 0.96 * 0.8 = 0.768`
- Stock: `1.0` (above threshold, no guarantee bonus)
- Service: `0.85 * 0.7 + (1.0 - 0.05) * 0.3 = 0.595 + 0.285 = 0.880`
- Tier: `0.05 (Silver)`

**Weighted Score (Seller B):**
```
= 0.35 * 0.0 + 0.20 * 0.264 + 0.20 * 0.768 + 0.15 * 1.0 + 0.05 * 0.880 + 0.05 * 0.05
= 0.0 + 0.0528 + 0.1536 + 0.15 + 0.044 + 0.0025
= 0.4029
buy_box_score = 40.29
```

**Result:** Seller A wins the Buy Box (92.82 vs 40.29).

## 7. Recalculation Triggers

The Buy Box score must be recalculated when relevant data changes. Each trigger fires a recalculation for the affected `sku_product`.

| # | Trigger Event | DocType | Hook | Detail |
|----|---------------|---------|------|--------|
| 1 | **Price Change** | Buy Box Entry | `on_update` | When `offer_price` is modified, recalculate all entries for the same `sku_product` |
| 2 | **Stock Change** | Buy Box Entry | `on_update` | When `stock_available` changes, recalculate all entries for the same `sku_product` |
| 3 | **Delivery Days Change** | Buy Box Entry | `on_update` | When `delivery_days` is modified, recalculate |
| 4 | **New Seller Added** | Buy Box Entry | `after_insert` | When a new Buy Box Entry is created, recalculate for that `sku_product` |
| 5 | **Seller Entry Removed** | Buy Box Entry | `on_trash` / `after_delete` | When a Buy Box Entry is deleted, recalculate remaining entries |
| 6 | **Entry Status Change** | Buy Box Entry | `on_update` | When `status` changes (e.g., Active -> Suspended), recalculate |
| 7 | **Seller Metrics Update** | Seller Metrics | `on_update` | When Seller Metrics (avg_rating, return_rate, etc.) are recalculated, recalculate all Buy Box entries for that seller |
| 8 | **Seller Profile Change** | Seller Profile | `on_update` | When seller_tier, verification_status, or status changes, recalculate all entries for that seller |
| 9 | **Settings Change** | Buy Box Settings | `on_update` | When admin changes weights or thresholds, recalculate ALL entries across all products |
| 10 | **Manual Trigger** | Buy Box Entry (list view action) | API endpoint | Admin can manually trigger recalculation for a specific product or all products |
| 11 | **Scheduled Daily** | Scheduled Task | `cron` | Daily full recalculation at configured time (Buy Box Settings.batch_recalculate_time) |

### Recalculation Cooldown

To prevent excessive recalculations (e.g., rapid price changes), the system enforces a **cooldown period** per product:

1. When a recalculation is triggered for `sku_product = X`, check if a recalculation was performed within the last `recalculation_cooldown_minutes` (default: 5 minutes).
2. If cooldown is active, **queue** the recalculation to run after cooldown expires.
3. If multiple triggers fire during cooldown, they are **consolidated** into a single recalculation.

### Implementation: Server-Side Hook

```python
# Pseudocode - NOT actual implementation code
# File: buy_box_entry.py

def on_update(self):
    if self.has_value_changed("offer_price") or \
       self.has_value_changed("stock_available") or \
       self.has_value_changed("delivery_days") or \
       self.has_value_changed("status"):
        enqueue_buy_box_recalculation(
            sku_product=self.sku_product,
            trigger_type="Price Change" if self.has_value_changed("offer_price") else "Stock Change",
            trigger_detail=f"Entry {self.name} updated"
        )

def after_insert(self):
    enqueue_buy_box_recalculation(
        sku_product=self.sku_product,
        trigger_type="New Seller",
        trigger_detail=f"New entry {self.name} by {self.seller_name}"
    )
```

## 8. Disqualification Rules

Sellers are disqualified from the Buy Box before scoring begins. Disqualified sellers are still visible in logs but are excluded from winner determination.

| # | Rule | Condition | Configurable | Default |
|----|------|-----------|--------------|---------|
| 1 | **Suspended Seller** | `Buy Box Entry.status == "Suspended"` OR `Seller Profile.status == "Suspended"` | Yes (Buy Box Settings.disqualify_suspended_sellers) | Enabled |
| 2 | **Blocked Seller** | `Seller Profile.status == "Blocked"` | No (always enforced) | Always |
| 3 | **Inactive Entry** | `Buy Box Entry.is_active == 0` OR `Buy Box Entry.status != "Active"` | No (always enforced) | Always |
| 4 | **Expired Entry** | `Buy Box Entry.status == "Expired"` | No (always enforced) | Always |
| 5 | **Unverified Seller** | `Seller Profile.verification_status != "Verified"` | Yes (Buy Box Settings.disqualify_unverified_sellers) | Disabled |
| 6 | **Below-Threshold Rating** | `seller_average_rating < Buy Box Settings.min_rating_threshold` | Yes (threshold configurable) | 0.0 (disabled) |
| 7 | **Excessive Refund Rate** | `seller_refund_rate > Buy Box Settings.max_refund_rate_threshold` | Yes (threshold configurable) | 0.0% (disabled) |
| 8 | **Excessive Delivery Time** | `delivery_days > Buy Box Settings.max_delivery_days_threshold` | Yes (threshold configurable) | 0 (disabled) |
| 9 | **Out of Stock** | `stock_available <= 0` | Yes (Buy Box Settings.require_stock_available) | Enabled |
| 10 | **Vacation Mode** | `Seller Profile.vacation_mode == 1` | No (always enforced) | Always |
| 11 | **Restricted Seller** | `Seller Profile.is_restricted == 1` OR `Seller Profile.can_sell == 0` | No (always enforced) | Always |

### Disqualification Processing Order

1. Apply **always-enforced** rules first (Blocked, Inactive, Expired, Vacation, Restricted)
2. Apply **configurable** rules based on Buy Box Settings
3. Set `is_disqualified = 1` and `disqualification_reason` on the Buy Box Entry
4. Disqualified entries get `buy_box_score = 0`, `is_winner = 0`, and `rank = 0`
5. Log the disqualification in the Buy Box Log `score_breakdown` JSON

### Multiple Disqualification Reasons

If a seller matches multiple disqualification rules, **all** reasons are recorded in `disqualification_reason` as a semicolon-separated list:

```
Suspended seller; Below minimum rating (2.1 < 3.0); Out of stock
```

## 10. Tiebreaker Logic

When two or more sellers have the same `buy_box_score` (within the configured `score_tie_tolerance`), a tiebreaker sequence resolves the winner.

### 10.1 Tie Detection

```
Two scores are considered "tied" if:
  abs(score_a - score_b) <= Buy Box Settings.score_tie_tolerance

Default tolerance: 0.01 (scores within 0.01 points are tied)
```

### 10.2 Tiebreaker Sequence

The system applies tiebreakers in order until one seller is differentiated:

| Priority | Method | Configuration | Logic |
|----------|--------|---------------|-------|
| 1st | **Primary Tiebreaker** | `Buy Box Settings.tiebreaker_method` | See method table below |
| 2nd | **Secondary Tiebreaker** | `Buy Box Settings.secondary_tiebreaker` | Applied if primary also ties |
| 3rd | **Alphabetical Seller Name** | Not configurable (hardcoded fallback) | `seller_name` ascending (deterministic final fallback) |

### 10.3 Tiebreaker Methods

| Method | Logic | Winner Is |
|--------|-------|-----------|
| Lowest Price | Compare `offer_price` | Seller with lowest price |
| Fastest Delivery | Compare `delivery_days` | Seller with fewest delivery days |
| Highest Rating | Compare `seller_average_rating` | Seller with highest rating |
| Most Reviews | Compare `seller_total_reviews` | Seller with most reviews |
| Longest Winner | Compare `winner_since` (earliest non-null) | Current winner retains position (stability preference) |
| Random | `random.choice()` seeded by product name + date | Random selection (changes daily for fairness) |

### 10.4 Example Tiebreaker Scenario

```
Seller A: buy_box_score = 85.24
Seller B: buy_box_score = 85.24
Seller C: buy_box_score = 85.25

Tie tolerance = 0.01

Step 1: A and B are tied (diff = 0.00 <= 0.01). C is also tied with B (diff = 0.01 <= 0.01).
        All three are in the tie group.

Step 2: Apply primary tiebreaker (Lowest Price):
        A.offer_price = 95.00, B.offer_price = 95.00, C.offer_price = 98.00
        A and B still tied; C eliminated from tie.

Step 3: Apply secondary tiebreaker (Highest Rating):
        A.seller_average_rating = 4.5, B.seller_average_rating = 4.7
        B wins.

Result: B is the Buy Box winner.
```

## 11. Edge Cases

### 11.1 Single Seller for Product

**Scenario:** Only one seller has an active Buy Box Entry for a given SKU Product.

**Handling:**
- The single seller is scored against themselves. All normalized scores use self-comparison (since min == max, price_score defaults to 1.0, delivery_score defaults to 1.0).
- Disqualification rules still apply. If the single seller is disqualified, the product has **no Buy Box winner**.
- If the single seller qualifies, they automatically win the Buy Box.
- `rank = 1`, `total_competitors = 1`.
- Buy Box Log records the single-entry calculation.

### 11.2 All Sellers Disqualified

**Scenario:** All sellers for a product are disqualified (e.g., all suspended, all out of stock).

**Handling:**
- No Buy Box winner is set (`is_winner = 0` for all entries).
- All entries have `is_disqualified = 1` with reasons.
- The product page should display a "No seller currently available" message or show the product without a default purchase option.
- Buy Box Log records the calculation with `winner_entry = null`, `winner_seller = null`, `winner_score = 0`, `winner_changed = 1` (if there was a previous winner).
- A notification is optionally sent to the marketplace admin.

### 11.3 Tied Scores (After All Tiebreakers)

**Scenario:** Two sellers have identical scores AND identical values for all tiebreaker criteria.

**Handling:**
- The final fallback is **alphabetical seller name** (ascending), which is always deterministic.
- This ensures a winner is always selected -- the algorithm never fails to produce a result.
- This edge case is extremely unlikely given the number of scoring dimensions.

### 11.4 New Seller with No Metrics

**Scenario:** A newly registered seller creates a Buy Box Entry but has zero reviews, no on-time delivery rate, no response rate.

**Handling:**
- `seller_average_rating = 0`, `seller_total_reviews = 0` -> `rating_score = 0.0` (0 / 5.0 * 0/10 confidence = 0)
- `seller_on_time_delivery_rate = 0` -> delivery score modifier is 0
- `seller_response_rate = 0`, `seller_refund_rate = 0` -> `service_score = 0.0 * 0.7 + 1.0 * 0.3 = 0.3`
- New sellers are inherently disadvantaged in rating and service scores but can compete on **price, delivery speed, and stock**.
- The system does NOT give new sellers a grace period or handicap. They must earn metrics through performance.

### 11.5 Price = 0 or Negative

**Scenario:** A seller enters an offer_price of 0.

**Handling:**
- The Buy Box Entry validation should enforce `offer_price > 0` (required field with min value).
- If somehow a zero price entry exists, it receives `price_score = 1.0` (best possible) which could unfairly advantage it.
- Server-side validation: `validate()` method should raise `frappe.ValidationError` if `offer_price <= 0`.

### 11.6 Concurrent Recalculations

**Scenario:** Two triggers fire simultaneously for the same product (e.g., two sellers update prices at the same time).

**Handling:**
- The cooldown mechanism (Section 7) prevents rapid successive calculations.
- The `enqueue_buy_box_recalculation` function uses Frappe's job queue with a **deduplicated queue name** per product: `buy_box_recalc_{sku_product}`.
- If a job is already queued for the same product, the duplicate is dropped.
- The final recalculation uses the latest data, so intermediate states are safely ignored.

### 11.7 Currency Mismatch

**Scenario:** Sellers offer prices in different currencies (e.g., TRY vs USD).

**Handling:**
- The Buy Box Entry has a `currency` field (default: TRY).
- Price normalization should convert all prices to a common currency using Frappe's `Currency Exchange` DocType.
- If no exchange rate is available, the entry is **flagged** but not disqualified (admin notification).
- Recommendation: Enforce same currency per product category or per tenant via Buy Box Settings (future enhancement).

### 11.8 Seller Tier Deleted or Changed

**Scenario:** A seller's tier is updated or the Seller Tier DocType record is deleted.

**Handling:**
- When `Seller Profile.seller_tier` changes, trigger recalculation (Section 7, trigger #8).
- If the tier in the Buy Box Tier Bonus table is deleted, the tier bonus defaults to `0.0`.
- The system gracefully handles missing tier references with `tier_bonus_map.get(seller_tier, 0.0)`.

## 12. Data Flow Sequence

### 12.1 Recalculation Flow (On Trigger)

```
1. Trigger fires (e.g., price change on Buy Box Entry BBX-00042)
2. System identifies sku_product from the trigger source
3. Check cooldown: Was sku_product recalculated within last N minutes?
   - If yes: Queue for later, exit
   - If no: Proceed
4. Load Buy Box Settings (weights, thresholds, tiebreaker config)
5. Fetch all Buy Box Entry records WHERE sku_product = X AND status = "Active"
6. Phase 1 - Disqualification Check:
   a. For each entry, apply disqualification rules (Section 8)
   b. Mark disqualified entries: is_disqualified = 1, disqualification_reason = "..."
   c. Set disqualified entries: buy_box_score = 0, is_winner = 0, rank = 0
7. Phase 2 - Score Calculation (qualified entries only):
   a. Calculate min/max values for normalization (min_price, max_price, min_days, max_days)
   b. For each qualified entry:
      - Calculate normalized scores for all 6 criteria
      - Calculate weighted_score = sum(weight_i * normalized_score_i)
      - Store buy_box_score = weighted_score * 100
      - Store individual score components (price_score, delivery_score, etc.) * 100
8. Phase 3 - Ranking:
   a. Sort qualified entries by buy_box_score DESC
   b. Check for ties (within score_tie_tolerance)
   c. Apply tiebreaker logic (Section 10)
   d. Assign rank (1, 2, 3, ...)
   e. Set is_winner = 1 for rank 1, is_winner = 0 for all others
   f. Set total_competitors for all entries
9. Phase 4 - Winner Transition:
   a. If winner changed from previous calculation:
      - Update winner_since on new winner
      - Clear winner_since on old winner
      - Set last_winner_change on both entries
10. Phase 5 - Improvement Suggestions:
    a. For non-winning entries, generate improvement_suggestions
    b. Store in the improvement_suggestions field
11. Phase 6 - Logging:
    a. Create Buy Box Log record with full score_breakdown JSON
    b. Record trigger_type, weights_used, winner info
12. Phase 7 - Save:
    a. Save all updated Buy Box Entry records
    b. Update last_score_update = now() on all entries
```

### 12.2 Scheduled Batch Recalculation Flow

```
1. Cron job fires at Buy Box Settings.batch_recalculate_time (default: 02:00)
2. Check Buy Box Settings.enable_batch_recalculate
3. If enabled:
   a. Get list of all distinct sku_product values from Buy Box Entry WHERE status = "Active"
   b. For each sku_product, run the recalculation flow (12.1, steps 4-12)
   c. trigger_type = "Scheduled"
4. Log total products recalculated and duration
```
