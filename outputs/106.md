# Task 106 - i18n Multi-language Support Plan

## Executive Summary

TradeHub is a Turkey-first B2B wholesale marketplace expanding to MENA and Turkic states, built on Frappe v15/ERPNext v15 with 7 custom apps (plus the legacy `tr_tradehub`). A codebase audit reveals that **none of the 7 decomposed tradehub_\* apps** have a `translations/` directory -- only the legacy `tr_tradehub` app has existing translation CSV files (569-line `tr.csv` and 570-line `en.csv`). Across the 7 apps, approximately **703 Python `frappe.throw`/`frappe.msgprint` calls lack `_()` wrappers**, while approximately **1,179 calls already use `_()`** correctly. On the JavaScript side, approximately **464 strings use `__()`** correctly, but roughly **100 `frappe.show_alert`/`frappe.msgprint`/`frappe.throw` calls** use hardcoded strings without `__()`. This plan provides a systematic approach to achieving full i18n coverage across all apps with Turkish (`tr`), English (`en`), Arabic (`ar`), and German (`de`) translation CSVs.

### CSV Format Specification

All translation CSV files must follow this exact format:

```csv
# {App Title} {Language Name} Translations
# Format: source,translated,context (optional)
# Generated for Frappe v15 CSV translation system

# Section: Common Labels
"source_string","translated_string","context (optional)"
```

**Column specification:**

| Column | Required | Description |
|--------|----------|-------------|
| Column 1 (source) | Yes | The exact English string as it appears in `_("...")` or `__("...")` calls, or as the DocType label/description in JSON |
| Column 2 (translated) | Yes | The translated string in the target language |
| Column 3 (context) | No | Optional context hint to disambiguate identical source strings with different meanings |

**Rules:**
- Each value MUST be enclosed in double quotes
- Internal double quotes must be escaped as `""`
- No trailing comma after the last column (Frappe handles both, but consistency matters)
- Lines starting with `#` are treated as comments
- UTF-8 encoding required (especially for Turkish I/i, Arabic RTL characters, German umlauts)
- One CSV per language per app

### `_()` / `__()` Literal-Only Constraint

Frappe's translation extraction relies on **static analysis** of source code. The `_()` and `__()` functions MUST receive **string literals only** -- not variables, not f-strings, not concatenated strings.

**Correct patterns:**

```python
# Python - CORRECT: literal string with .format()
frappe.throw(_("Tenant {0} does not exist").format(tenant_name))

# Python - CORRECT: multi-line literal
frappe.throw(
    _("Cannot create document for tenant {0}. "
      "Please contact your administrator.").format(tenant)
)

# Python - CORRECT: with title parameter
frappe.throw(
    _("Access denied"),
    title=_("Permission Error")
)
```

```javascript
// JavaScript - CORRECT: literal string with .format()
frappe.throw(__('A category cannot be its own parent'));
frappe.msgprint(__('Price cannot be negative'));

// JavaScript - CORRECT: with substitution
frappe.show_alert({
    message: __('Variant {0} created successfully', [variant_name]),
    indicator: 'green'
});
```

**Incorrect patterns (will NOT be extracted for translation):**

```python
# Python - WRONG: f-string
frappe.throw(_(f"Tenant {tenant_name} not found"))

# Python - WRONG: variable
msg = "Access denied"
frappe.throw(_(msg))

# Python - WRONG: concatenation
frappe.throw(_("Error: " + detail_msg))
```

```javascript
// JavaScript - WRONG: template literal
frappe.throw(__(`Category ${name} not found`));

// JavaScript - WRONG: variable
let msg = 'Access denied';
frappe.throw(__(msg));
```

## Turkish I/i Case Sensitivity

### The Problem

Turkish has four distinct "i" characters:

| Character | Unicode | Name | Uppercase |
|-----------|---------|------|-----------|
| i | U+0069 | Latin small letter i | I (U+0130, capital I with dot) |
| I | U+0049 | Latin capital letter I | -- |
| i | U+0131 | Latin small letter dotless i | I (U+0049, standard capital I) |
| I | U+0130 | Latin capital letter I with dot above | -- |

Standard Python `str.lower()` and `str.upper()` use **Unicode default casing**, which does NOT handle Turkish correctly:

```python
# Standard Python (WRONG for Turkish)
"ISTANBUL".lower()  # Returns "istanbul" -- WRONG, should be "istanbul" (with dotless i)
"istanbul".upper()  # Returns "ISTANBUL" -- WRONG, should be "ISTANBUL" (with dotted I)
```

### Impact on Translations

1. **CSV matching**: Frappe's translation lookup is case-sensitive for source strings. A string like `"Invoice"` in the CSV will NOT match `"INVOICE"` in the UI.
2. **String formatting**: Any `.format()` substitution that involves Turkish text must preserve correct casing.
3. **Search/filter**: If users search for translations, Turkish-locale-aware comparison is needed.

### Mitigation Strategy

1. **Source strings in English**: All `_()` and `__()` source strings are in English, so standard casing applies. The Turkish case issue primarily affects the **translated strings** in `tr.csv`.
2. **Do NOT use `.lower()`/`.upper()` on Turkish translations**: If any code normalizes translation strings, use locale-aware functions:

```python
import locale

# Set Turkish locale for case operations
locale.setlocale(locale.LC_CTYPE, 'tr_TR.UTF-8')

# Or use casefold() for case-insensitive comparison (Python 3.x)
# casefold() handles Turkish I correctly in most Python builds
source.casefold() == target.casefold()
```

3. **Translation CSV encoding**: All CSV files MUST be saved as **UTF-8 without BOM** to correctly encode Turkish special characters (I, i, s, c, g, o, u).
4. **Test with Turkish-specific strings**: Include test cases for DocType names and labels that contain "I" (e.g., "Invoice", "Item", "Integration") to verify correct Turkish rendering.

## Dependencies & Prerequisites

### Technical Prerequisites

1. **Frappe v15 translation system**: Already available -- CSV-based translation is the default in v15
2. **No config needed**: Frappe auto-discovers `translations/` directories by convention -- no `hooks.py` changes required
3. **`from frappe import _`**: Must be present in every Python file that uses `_()`
4. **System Settings**: Language dropdown must include `tr`, `en`, `ar`, `de` -- these are Frappe built-in languages

### Operational Prerequisites

1. **Turkish translator**: Native Turkish speaker with B2B domain knowledge for `tr.csv` content
2. **Arabic translator**: Native Arabic speaker for `ar.csv` content (RTL layout considerations)
3. **German translator**: Native German speaker for `de.csv` content
4. **Review process**: Each translated CSV should be reviewed by a second native speaker before deployment

### Version Note

> **IMPORTANT**: Frappe v15 uses the **CSV-based** translation system. Frappe v16+ migrates to **gettext** (`.po`/`.mo` files). All work in this plan targets CSV format. If/when TradeHub upgrades to v16, the CSV files will need to be converted to gettext format using Frappe's migration tools.

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Missing `_()` wrappers cause untranslated strings in production | High | Automated CI check: grep for `frappe.throw` without `_()` |
| Turkish I/i corruption in translations | Medium | UTF-8 encoding enforcement, Turkish locale testing |
| Arabic RTL breaks layout | Medium | Dedicated RTL testing phase, Frappe's built-in RTL support |
| Translation CSV format errors (misquoted strings, encoding issues) | Medium | CSV validation script, `bench get-untranslated` diff check |
| v16 migration breaks CSV translations | Low | Document v15->v16 migration path; Frappe provides conversion tools |
| Performance impact of translation lookups | Low | Frappe caches translations in Redis; negligible overhead |
| Inconsistent translations across apps (same term, different translation) | Medium | Shared glossary document; `en.csv` as canonical reference |
| `frappe._()` vs `_()` inconsistency | Low | Linting rule to enforce `from frappe import _` pattern |

### CI/CD Integration Recommendations

Add these checks to the CI pipeline:

```bash
# Check 1: Verify all frappe.throw/msgprint use _()
grep -rn 'frappe\.throw\|frappe\.msgprint' --include="*.py" | \
  grep -v '_(' | grep -v '#' | grep -v 'log_error' && \
  echo "FAIL: Unwrapped strings found" || echo "PASS"

# Check 2: Verify all JS frappe.throw/msgprint/show_alert use __()
grep -rn 'frappe\.\(throw\|msgprint\|show_alert\)(' --include="*.js" | \
  grep -v '__(' | grep -v '//' && \
  echo "FAIL: Unwrapped JS strings found" || echo "PASS"

# Check 3: Verify translation directories exist
for dir in tradehub_core tradehub_catalog tradehub_seller tradehub_commerce \
           tradehub_compliance tradehub_logistics tradehub_marketing; do
  test -d "frappe-bench/apps/$dir/$dir/translations" || \
    echo "FAIL: Missing translations/ in $dir"
done

# Check 4: Verify CSV format (no BOM, UTF-8)
file frappe-bench/apps/tradehub_*/*/translations/*.csv | grep -v "UTF-8\|ASCII" && \
  echo "FAIL: Non-UTF-8 CSV found" || echo "PASS"
```
