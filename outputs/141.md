# Task 141 — Abonelik Bazli RFQ Goruntulenme Limitleri

## 1. Genel Bakis (Overview)

Bu belge, B2B pazar yeri platformunda abonelik paketlerine bagli aylik RFQ (Teklif Talebi) goruntulenme limitlerini tanimlar. Mevcut durumda tum saticilar, abonelik tier'indan bagimsiz olarak sinirsiz RFQ detay goruntulemesi yapabilmektedir. Bu ozellik ile her abonelik paketine tanimlanmis aylik RFQ goruntulenme limiti uygulanacak, boylece premium abonelik paketleri icin ek deger olusturulacaktir.

**Problem:** Tum abonelik paketlerinde sinirsiz RFQ goruntulenme hakki mevcut — tier farklilastirmasi yok.

**Hedef:** Her abonelik paketine `max_rfq_views` limiti tanimlayarak, satici bazinda aylik RFQ detay goruntulenme sayisini sinirlamak ve yukseltme (upgrade) motivasyonu olusturmak.

**Kapsam:**
- Subscription Package DocType'a yeni alan eklenmesi
- Bagimsiz (standalone) RFQ View Log DocType tanimlanmasi
- Aylik sayac mekanizmasi ve limit kontrol algoritmasi
- Limit-asildiginda kullanici deneyimi (UX) tasarimi
- Plan degisikligi edge case'leri

## 4. Limit Kontrol Mekanizmasi (Limit Check Mechanism)

### 4.1 Goruntulenme Sayisi Hesaplama

Aylik goruntulenme sayisi, veritabani sorgulari ile dinamik olarak hesaplanir — ayri bir sayac alani tutulmaz.

```python
# Pseudocode — RFQ detay sayfasina erisim kontrolu

def check_rfq_view_limit(seller_profile):
    """
    Saticinin aylik RFQ goruntulenme limitini kontrol eder.

    Returns:
        dict: {
            "allowed": bool,
            "current_views": int,
            "max_views": int,
            "remaining": int or "unlimited"
        }
    """
    # 1. Saticinin aktif aboneligini al
    subscription = get_active_subscription(seller_profile)
    if not subscription:
        return {"allowed": False, "reason": "No active subscription"}

    # 2. Abonelik paketindeki max_rfq_views degerini al
    package = frappe.get_doc("Subscription Package", subscription.subscription_package)
    max_views = package.max_rfq_views or 0

    # 3. Sinirsiz kontrolu (0 = sinirsiz)
    if max_views == 0:
        return {
            "allowed": True,
            "current_views": 0,
            "max_views": 0,
            "remaining": "unlimited"
        }

    # 4. Bu ayin basini hesapla
    today = getdate(nowdate())
    month_start = today.replace(day=1)

    # 5. Bu ay icindeki goruntulenme sayisini sorgula
    current_views = frappe.db.count(
        "RFQ View Log",
        filters={
            "seller": seller_profile,
            "viewed_at": [">=", month_start]
        }
    )

    # 6. Limit kontrolu
    allowed = current_views < max_views
    remaining = max(0, max_views - current_views)

    return {
        "allowed": allowed,
        "current_views": current_views,
        "max_views": max_views,
        "remaining": remaining
    }
```

### 4.2 Goruntulenme Kaydi Olusturma

```python
# Pseudocode — RFQ detay goruntulendiginde cagrilacak

def log_rfq_view(seller_profile, rfq_name, user):
    """
    RFQ goruntulenme kaydini olusturur.
    Ayni RFQ icin ayni satici tarafindan tekrar goruntulenme
    aylik sayaci artirmaz (deduplicate).
    """
    today = getdate(nowdate())
    month_start = today.replace(day=1)

    # Ayni RFQ icin bu ay icinde kayit var mi kontrol et
    existing = frappe.db.exists("RFQ View Log", {
        "seller": seller_profile,
        "rfq": rfq_name,
        "viewed_at": [">=", month_start]
    })

    if existing:
        # Tekrar goruntulenme — sayaci artirma, sadece zamani guncelle
        return {"logged": False, "reason": "Already viewed this month"}

    # Yeni goruntulenme kaydi olustur
    view_log = frappe.get_doc({
        "doctype": "RFQ View Log",
        "seller": seller_profile,
        "rfq": rfq_name,
        "user": user,
        "viewed_at": now_datetime(),
        "ip_address": frappe.local.request_ip,
        "user_agent": frappe.get_request_header("User-Agent") or "",
        "subscription": get_active_subscription_name(seller_profile),
        "subscription_package": get_active_package_name(seller_profile)
    })
    view_log.insert(ignore_permissions=True)

    return {"logged": True, "view_log": view_log.name}
```

### 4.3 API Endpoint

```python
# Pseudocode — Whitelisted API method

@frappe.whitelist()
def get_rfq_detail_with_limit_check(rfq_name):
    """
    RFQ detay verisini limit kontrolu ile birlikte dondurur.
    Seller Portal'dan cagirilacak API endpoint.
    """
    user = frappe.session.user
    seller_profile = frappe.db.get_value("Seller Profile", {"user": user}, "name")

    if not seller_profile:
        frappe.throw(_("Seller profile not found"))

    # Limit kontrolu
    limit_status = check_rfq_view_limit(seller_profile)

    if not limit_status["allowed"]:
        return {
            "success": False,
            "limit_reached": True,
            "current_views": limit_status["current_views"],
            "max_views": limit_status["max_views"],
            "upgrade_url": "/subscription/upgrade"
        }

    # Goruntulenme kaydi olustur
    log_rfq_view(seller_profile, rfq_name, user)

    # RFQ detayini dondur
    rfq = frappe.get_doc("RFQ", rfq_name)
    return {
        "success": True,
        "rfq": rfq.as_dict(),
        "remaining_views": limit_status["remaining"]
    }
```

## 5. Aylik Sayac Sifirlama (Monthly Counter Reset)

### Strateji Karsilastirmasi

| Strateji | Avantajlar | Dezavantajlar |
|----------|-----------|---------------|
| **A: Tarih bazli sorgu (Onerilen)** | Ek is gerektirmez, veri kaybi olmaz, daha basit | Tablo buyuyebilir (indeks ile cozulur) |
| B: Zamanlanmis is ile sifirlama | Tablo kucuk kalir | Veri kaybi, ek cronjob gerekli |
| C: Aylik snapshot + arsivleme | Raporlama icin faydali | En karmasik yapi |

### Oneri: Strateji A — Tarih Bazli Sorgu

Goruntulenme sayisi her zaman `viewed_at >= ayin_ilk_gunu` filtresi ile dinamik olarak hesaplanir. Eski kayitlar silinmez — hem raporlama hem de gecmis analizi icin saklanir.

**Performans onlemi:** `(seller, viewed_at)` bileşik indeksi ile aylik sorgular hizli calisir. Tablo buyumesi icin yillik arsivleme cronjob'i planlanabilir (opsiyonel).

- Saticinin aylik goruntulenme sayisi `frappe.cache().hget("rfq_view_count", seller_profile)` ile onbellekte tutulur
- Her yeni goruntulenme kaydinda onbellek guncellenir
- Her ayin 1'inde onbellek otomatik olarak temizlenir
- Cache TTL: 1 saat (stale data onleme)

### 8.3 Izin Kurallari (Permission Rules)

```python
# Pseudocode — RFQ View Log permission query
def get_permission_query(user):
    if "System Manager" in frappe.get_roles(user):
        return ""
    seller = frappe.db.get_value("Seller Profile", {"user": user}, "name")
    if seller:
        return f"`tabRFQ View Log`.seller = '{seller}'"
    return "1=0"  # Diger kullanicilar goremez
```

Ayni satici, ayni RFQ'yu ayni ay icinde birden fazla goruntulediginde yalnizca **bir** goruntulenme sayilir. Bu sayede saticinin ayni talep uzerinde birden fazla inceleme yapabilmesi engellenmez.

## 9. Edge Cases (Ozel Durumlar)

### 9.1 Ay Ortasinda Upgrade (Yukseltme)

**Senaryo:** Satici ay ortasinda Starter (15 limit) paketinden Professional (50 limit) paketine yukseltme yapar. Bu ayda 14 goruntulenme yapmistir.

**Davranis:**
- Yeni limit **aninda** gecerli olur
- Mevcut ay goruntulenme sayisi: 14
- Yeni limit: 50
- Kalan: 50 - 14 = 36 goruntulenme
- Onceki goruntulenme kayitlari silinmez

### 9.2 Ay Ortasinda Downgrade (Dusurme)

**Senaryo:** Satici ay ortasinda Professional (50 limit) paketinden Starter (15 limit) paketine dusurme yapar. Bu ayda 25 goruntulenme yapmistir.

**Davranis:**
- Dusurme genellikle **donem sonunda** gecerli olur (Subscription.downgrade_package() metodu ile uyumlu)
- Eger aninda gecerliyse: Mevcut 25 goruntulenme > yeni limit 15
- Onceden goruntulenmis RFQ'lar erisilebilir kalir (tekrar goruntulenme sayac artirmaz)
- Yeni RFQ goruntulenme hakki olmaz (kalan: 0)
- Dashboard uyarisi: "Goruntulenme limitinizi asmis durumdasiniz (25/15). Yeni ay basinda limit sifirlanacaktir."

### 9.3 Ucretsiz / Deneme (Free / Trial) Paketleri

| Paket Durumu | max_rfq_views | Davranis |
|-------------|---------------|----------|
| Free (Ucretsiz) | 3 | Ayda 3 RFQ goruntulenebilir |
| Trial (Deneme) | 10 | Deneme suresi boyunca aylik 10 RFQ |
| Trial suresi bitti | - | Abonelik durumu "Expired" — hicbir RFQ goruntulenemez |

### 9.4 Abonelik Durumu Grace Period / Suspended

| Abonelik Durumu | RFQ Goruntulenme |
|-----------------|------------------|
| Active | Normal limit uygulanir |
| Trial | Normal limit uygulanir |
| Grace Period | Limit uygulanir, ancak erisim devam eder (api_access_enabled=1) |
| Suspended | Goruntulenme tamamen engellenir (api_access_enabled=0) |
| Cancelled / Expired | Goruntulenme tamamen engellenir |

### 9.5 Zaman Dilimi (Timezone) Sorunlari

- Ay baslangici hesaplamasi **sunucu zaman dilimine** (UTC veya Turkiye saati) gore yapilir
- Tum `viewed_at` Datetime alanlari UTC olarak saklanir
- Saticinin yerel zaman dilimi ile sunucu zaman dilimi arasinda fark olabilir — kabul edilebilir sapma

### 9.6 Toplu Islem ve Race Condition

- Birden fazla tarayici sekmesinden es zamanli RFQ goruntulenme istegi gelebilir
- `log_rfq_view()` fonksiyonu `existing` kontrolu ile deduplicate yapar
- Race condition icin ek onlem: `frappe.db.exists` sorgusu yeterli (last-write-wins kabul edilebilir)

### 9.7 Tablo Buyumesi ve Performans

- Yillik tahmini kayit sayisi: 10,000 satici x 50 goruntulenme = 500,000 kayit
- `(seller, viewed_at)` bileşik indeksi ile aylik sorgu performansi O(log n)
- Opsiyonel: 12 aydan eski kayitlarin arsivlenmesi
