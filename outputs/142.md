# Task 142 — Abonelik Yasam Dongusu Yonetimi (Subscription Lifecycle Management)

## 1. Genel Bakis (Overview)

Bu belge, B2B pazar yeri platformundaki abonelik yasam dongusu yonetimini kapsamli olarak planlar. Mevcut `Subscription` DocType'i (`tradehub_marketing/doctype/subscription/subscription.py`) temel yasam dongusu durumlarini (Draft, Trial, Active, Pending Payment, Grace Period, Suspended, Cancelled, Expired) ve otomatik durum gecislerini (`check_and_transition_status()`) halihazirda icerir. Bu planlama belgesi, mevcut uygulamanin uzerine **kademeli bildirim sistemi**, **merkezi faturalandirma ayarlari**, **askiya alma etki matrisi** ve **admin izleme paneli** ekleyerek abonelik yonetimini olgunlastirir.

**Problem:** Mevcut abonelik sistemi durum gecislerini yonetir ancak:
- Satici/alicilara odeme vade tarihine yaklasma ve gecikme hakkinda **proaktif bildirimler** gonderilmez
- Grace period suresi, bildirim zamanlari ve askiya alma kurallari **merkezi olarak yapilandirilabilir** degildir
- Askiya alinan aboneliklerin platform uzerindeki etkileri resmi olarak dokumante edilmemistir
- Admin'ler abonelik durumlarini toplu olarak izleyebilecek bir **dashboard** 'a sahip degildir

**Hedef:**
- 5 temel durumlu (Active, Past Due, Grace Period, Suspended, Cancelled) durum makinesi tanimlamak
- Vade oncesi ve sonrasi kademeli bildirim sistemi olusturmak
- Merkezi Subscription Billing Settings DocType tanimlamak
- Askiya alma etkilerini tum platform yetenekleri uzerinden belgelemek
- Admin izleme paneli widget'larini tanimlamak

**Mevcut Uygulama ile Iliski:**
Mevcut `subscription.py` dosyasindaki durumlar bu planla eslestirilmistir:

| Mevcut Durum | Bu Plandaki Karsiligi | Aciklama |
|-------------|----------------------|----------|
| Active | Active | Aktif abonelik |
| Pending Payment | Past Due | Odeme bekleniyor / vadesi gecmis |
| Grace Period | Grace Period | Ek sure verilmis |
| Suspended | Suspended | Askiya alinmis |
| Cancelled | Cancelled | Iptal edilmis |
| Draft | (On-kosul) | Henuz aktif edilmemis |
| Trial | (Active varyanti) | Deneme sureci |
| Expired | (Cancelled varyanti) | Suresi dolmus |

### 2.4 Durum Gecis Pseudocode

```python
# Pseudocode — check_and_transition_status() guncellenmis versiyon

def check_and_transition_status(self):
    """
    Zamanlanmis gorev (scheduler) tarafindan cagirilir.
    Mevcut tarihe gore otomatik durum gecislerini uygular.
    """
    today = getdate(nowdate())
    old_status = self.status
    status_changed = False

    if self.status == "Active" and self.end_date:
        if getdate(self.end_date) < today:
            if self.auto_renew:
                # Otomatik yenileme denemesi
                payment_result = attempt_auto_renewal(self)
                if payment_result.success:
                    self.renew(payment_received=True)
                else:
                    self.status = "Past Due"
                    self.last_renewal_attempt = now_datetime()
                    self.failed_renewal_attempts = (self.failed_renewal_attempts or 0) + 1
                    self.renewal_failure_reason = payment_result.error_message
            else:
                self._start_grace_period_internal()
            status_changed = True

    elif self.status == "Past Due":
        days_past_due = date_diff(today, getdate(self.last_renewal_attempt or self.end_date))
        # Admin-yapilandirilabilir: past_due_to_grace_days (varsayilan: 3)
        settings = get_billing_settings()
        if days_past_due >= settings.past_due_to_grace_days:
            self._start_grace_period_internal()
            status_changed = True

    elif self.status == "Grace Period" and self.grace_period_end_date:
        if getdate(self.grace_period_end_date) < today:
            self._suspend_internal(_("Grace period suresi doldu"))
            status_changed = True

    elif self.status == "Suspended":
        # Admin-yapilandirilabilir: auto_cancel_after_suspension_days (varsayilan: 90)
        settings = get_billing_settings()
        if self.suspended_at:
            days_suspended = date_diff(today, getdate(self.suspended_at))
            if days_suspended >= settings.auto_cancel_after_suspension_days:
                self.cancel_subscription(reason="Uzun sureli askida kalma - otomatik iptal")
                status_changed = True

    if status_changed:
        self.save(ignore_permissions=True)
        # Bildirim gonder
        send_status_change_notification(self, old_status, self.status)

    return {
        "old_status": old_status,
        "new_status": self.status,
        "changed": status_changed
    }
```

## 4. Kademeli Bildirim Sistemi (Tiered Notification System)

### 4.1 Bildirim Zaman Cizelgesi

Bildirimler vade tarihine (`end_date` / `next_billing_date`) gore **vade oncesi** ve **vade sonrasi** olarak iki kategoriye ayrilir:

#### Vade Oncesi Bildirimler (Pre-Due)

| Gun | Tetikleyici | Alici | Kanal | Oncelik | Sablon Aciklamasi |
|-----|------------|-------|-------|---------|-------------------|
| -7 gun | `end_date - 7 gun = today` | Abone (Satici/Alici) | E-posta | Normal | **Abonelik Yenileme Hatirlatmasi:** "Aboneliginiz 7 gun icinde sona erecektir. Otomatik yenileme [acik/kapali]. Paket: {paket_adi}, Tutar: {tutar} TL." |
| -3 gun | `end_date - 3 gun = today` | Abone (Satici/Alici) | E-posta + In-app | Yuksek | **Acil Yenileme Hatirlatmasi:** "Aboneliginiz 3 gun icinde sona erecektir. Odeme yonteminizi kontrol edin. Otomatik yenileme [acik/kapali]." |
| 0 gun | `end_date = today` | Abone (Satici/Alici) | E-posta + In-app + SMS (opsiyonel) | Kritik | **Abonelik Vade Tarihi Geldi:** "Aboneliginiz bugun sona ermektedir. Odemeniz alinmazsa hizmetleriniz etkilenebilir." |

#### Vade Sonrasi Bildirimler (Post-Due)

| Gun | Tetikleyici | Alici | Kanal | Oncelik | Sablon Aciklamasi |
|-----|------------|-------|-------|---------|-------------------|
| +1 gun | Odeme vadesi 1 gun gecmis | Abone | E-posta + In-app | Yuksek | **Odeme Gecikti:** "Aboneliginiz icin odemeniz 1 gundur alinamamistir. Lutfen odeme bilgilerinizi guncelleyin. Aboneliginiz su an 'Odeme Bekleniyor' durumundadir." |
| +3 gun | Odeme vadesi 3 gun gecmis | Abone + Admin | E-posta + In-app | Yuksek | **Grace Period Basliyor:** "Odemeniz 3 gundur alinamadi. Aboneliginiz ek sure (grace period) doneminize girmistir. Kalan sure: {grace_period_days} gun." |
| +7 gun | Grace period 4. gunu | Abone | E-posta + In-app | Kritik | **Askiya Alma Uyarisi:** "Odemeniz hala alinamadi. Aboneliginiz {kalan_gun} gun icinde askiya alinacaktir. Urunleriniz deaktive edilecek ve yeni siparis alamayacaksiniz." |
| +14 gun | Grace period sonu (varsayilan) | Abone + Admin | E-posta + In-app + SMS | Kritik | **Abonelik Askiya Alindi:** "Odemeniz {grace_period_days} gun icinde alinamadigi icin aboneliginiz askiya alinmistir. Hizmetlerinizi yeniden aktif etmek icin odeme yapin." |

## 6. Grace Period Mekanizmasi (Grace Period Mechanism)

### 6.1 Grace Period Akisi

```
Past Due suresi doldu (past_due_to_grace_days = 3)
    |
    v
Grace Period Basla
    |
    +---> grace_period_start_date = today
    +---> grace_period_end_date = today + grace_period_days
    +---> status = "Grace Period"
    +---> in_grace_period = 1
    +---> api_access_enabled = 1 (sinirli erisim devam)
    |
    v
Grace Period suresi boyunca:
    - Satici urunlerini gorebilir ancak yeni urun ekleyemez
    - Mevcut siparisler devam eder
    - Yeni siparis alamaz
    - Dashboard'da belirgin uyari gosterilir
    |
    v
Grace Period Sonucu:
    |
    +---+---+
    |       |
  Odeme   Suresi
  Alindi   Doldu
    |       |
    v       v
  Active  Suspended (T9)
```

### 6.2 Grace Period Sirasindaki Kisitlamalar

| Platform Yetenegi | Grace Period'da Durum | Aciklama |
|-------------------|----------------------|----------|
| Urun goruntuleme (kendi urunleri) | Erisim acik | Satici kendi urunlerini gorebilir |
| Yeni urun ekleme | **Engellenmis** | Yeni urun eklenemez |
| Mevcut urun guncelleme | Erisim acik | Mevcut urunler guncellenebilir |
| Siparis alma (yeni) | **Engellenmis** | Yeni siparis alamaz |
| Mevcut siparis yonetimi | Erisim acik | Devam eden siparisler tamamlanabilir |
| RFQ goruntulenme | Erisim acik (limit uygulanir) | 141.md ile uyumlu |
| API erisimi | Erisim acik (sinirli) | `api_access_enabled = 1` |
| Dashboard | Erisim acik | Belirgin uyari banner'i ile |
| Premium ozellikler | **Engellenmis** | Analytics, raporlama vb. |

## 7. Askiya Alma ve Etkileri (Suspension & Effects)

### 7.1 Askiya Alma Etki Matrisi

Grace period suresi doldugunda abonelik askiya alinir. Asagidaki matris, askiya alinan bir aboneligin tum platform yetenekleri uzerindeki etkisini detaylandirir:

|---|-------------------|-------|----------|-------------|-----------|-----------|

| 2 | **Yeni Urun Ekleme** | Izin var | Izin var | **Engel** | **Engel** | **Engel** |
| 3 | **Urun Guncelleme** | Izin var | Izin var | Izin var | **Engel** | **Engel** |
| 4 | **Siparis Alma (yeni)** | Izin var | Izin var | **Engel** | **Engel** | **Engel** |

| 7 | **Teklif Verme (RFQ Response)** | Izin var | Izin var | **Engel** | **Engel** | **Engel** |

### 7.2 Askiya Alma Uygulama Sureci

```python
# Pseudocode — Askiya alma isleminin platform uzerindeki etkileri

def apply_suspension_effects(subscription):
    """
    Abonelik askiya alindiginda tum platform etkilerini uygular.
    """
    seller_profile = subscription.seller_profile

    # 1. Urunleri deaktive et (gizle)
    frappe.db.sql("""
        UPDATE `tabProduct`
        SET is_active = 0, visibility = 'Hidden', suspension_hidden = 1
        WHERE seller = %s AND is_active = 1
    """, seller_profile)

    # 2. API erisimini kapat
    subscription.api_access_enabled = 0

    # 3. Premium profil senkronizasyonu
    sync_premium_profile_api_access(subscription)

    # 4. Yeni siparis kabul mekanizmasini kapat
    frappe.db.set_value("Seller Profile", seller_profile,
                        "accepting_orders", 0)

    # 5. Log kaydi olustur
    frappe.get_doc({
        "doctype": "Subscription Activity Log",
        "subscription": subscription.name,
        "activity_type": "Suspended",
        "details": f"Urunler gizlendi, API erisimi kapatildi",
        "timestamp": now_datetime()
    }).insert(ignore_permissions=True)
```

### 9.2 Bildirim Gonderim Algoritmasi

```python
# Pseudocode — Gunluk bildirim gonderme gorevi

def send_subscription_reminders():
    """
    Her gun calisir. Subscription Billing Settings'ten
    yapilandirilan gunlere gore bildirim gonderir.
    """
    settings = frappe.get_single("Subscription Billing Settings")
    today = getdate(nowdate())

    # 1. Vade oncesi bildirimler
    if settings.enable_pre_due_notifications:
        for reminder in settings.pre_due_reminder_days:
            if not reminder.is_enabled:
                continue

            target_date = add_days(today, abs(reminder.day_offset))
            subscriptions = frappe.get_all("Subscription",
                filters={
                    "status": "Active",
                    "end_date": target_date,
                    f"reminder_{abs(reminder.day_offset)}d_sent": 0
                },
                fields=["name", "subscriber_type", "seller_profile",
                         "buyer_profile", "package_name", "end_date",
                         "current_price"]
            )

            for sub in subscriptions:
                send_reminder_notification(
                    subscription=sub,
                    reminder=reminder,
                    days_until_due=abs(reminder.day_offset)
                )
                # Flag olarak isaretle (tekrar gonderimi onle)
                frappe.db.set_value("Subscription", sub.name,
                    f"reminder_{abs(reminder.day_offset)}d_sent", 1)

    # 2. Vade sonrasi bildirimler
    if settings.enable_post_due_notifications:
        for reminder in settings.post_due_reminder_days:
            if not reminder.is_enabled:
                continue

            target_date = add_days(today, -reminder.day_offset)
            subscriptions = frappe.get_all("Subscription",
                filters={
                    "status": ["in", ["Past Due", "Grace Period"]],
                    "end_date": target_date,
                    f"post_due_reminder_{reminder.day_offset}d_sent": 0
                },
                fields=["name", "subscriber_type", "seller_profile",
                         "buyer_profile", "package_name", "end_date",
                         "outstanding_amount"]
            )

            for sub in subscriptions:
                send_reminder_notification(
                    subscription=sub,
                    reminder=reminder,
                    days_overdue=reminder.day_offset
                )
                frappe.db.set_value("Subscription", sub.name,
                    f"post_due_reminder_{reminder.day_offset}d_sent", 1)
```

### 11.3 Abonelik Durumu Cache Stratejisi

```python
# Abonelik durumu sik sorgulanan bir deger oldugu icin cache kullanilir

CACHE_KEY = "subscription_status:{subscriber_type}:{subscriber_id}"
CACHE_TTL = 300  # 5 dakika

def get_cached_subscription_status(subscriber_type, subscriber_id):
    """Cache'ten abonelik durumunu okur, yoksa DB'den alir ve cache'e yazar."""
    key = CACHE_KEY.format(subscriber_type=subscriber_type, subscriber_id=subscriber_id)
    status = frappe.cache().get_value(key)
    if not status:
        status = frappe.db.get_value("Subscription", {
            "subscriber_type": subscriber_type,
            "seller_profile" if subscriber_type == "Seller" else "buyer_profile": subscriber_id,
            "status": ["in", ["Active", "Trial", "Past Due", "Grace Period"]]
        }, "status")
        frappe.cache().set_value(key, status, expires_in_sec=CACHE_TTL)
    return status

# Durum degisikliklerinde cache temizle
def invalidate_subscription_cache(subscription):
    key = CACHE_KEY.format(
        subscriber_type=subscription.subscriber_type,
        subscriber_id=subscription.seller_profile or subscription.buyer_profile
    )
    frappe.cache().delete_value(key)
```

## 12. Edge Cases (Ozel Durumlar)

### 12.1 Odeme Esnasinda Durum Gecisi

**Senaryo:** Satici Grace Period'da iken odeme yapar, ancak odeme islemi sirasinda scheduler aboneligi askiya alir.

**Cozum:**
- Odeme islemi baslangicinda abonelik kilidlenir (`frappe.lock_doc`)
- Odeme basarili ise durum dogrudan "Active" yapilir (aradaki gecisler atlanir)
- Odeme islemi sirasinda scheduler bu aboneligi atlayacak sekilde yapilandirilir

### 12.2 Birden Fazla Basarisiz Odeme Denemesi

**Senaryo:** Otomatik yenileme 3 kez basarisiz olur, ancak satici farkli bir odeme yontemi ile odeme yapmak ister.

**Cozum:**
- `max_renewal_retries` sayisina ulasilinca otomatik deneme durdurulur
- Satici manuel odeme sayfasindan odeme yapabilir
- Manuel odeme basariliysa dogrudan "Active" durumuna donulur

### 12.3 Mevcut Aboneliklerin Gecisi (Migration)

**Senaryo:** Bu ozellik devreye alindiginda mevcut "Active" durumundaki abonelikler icin ne olur?

**Cozum:**
- Mevcut Active abonelikler oldugu gibi kalir
- Subscription Billing Settings singleton olusturulur, varsayilan degerler set edilir
- Mevcut "Pending Payment" durumundaki abonelikler "Past Due" olarak adlandirilir (migration scripti ile)
- Bildirim flag alanlari (reminder_7d_sent, vb.) mevcut kayitlara eklenir, varsayilan 0

### 12.4 Scheduler Kesilmesi

**Senaryo:** Sunucu arizasi nedeniyle scheduler 3 gun calismazsa, birikimis durum gecisleri ne olur?

**Cozum:**
- `check_subscription_transitions` her calistiginda **tum** aktif abonelikleri kontrol eder
- Gecmis tarihli gecisler toplu olarak uygulanir
- Bildirimler gonderilir ancak "geciken bildirim" notu eklenir
- Log kayitlari gercek tetikleme tarihini degil, islem tarihini kaydeder

### 12.5 Ay Ortasinda Paket Degisikligi ile Fatura Donemi Uyumsuzlugu

**Senaryo:** Satici ayin 15'inde paket degistirir, yeni paketin fatura donemi farklidir.

**Cozum:**
- Upgrade: Aninda gecerli, proration hesaplanir (mevcut `upgrade_package()` ile uyumlu)
- Downgrade: Donem sonunda gecerli (`allow_downgrade_during_period` ayarina bagli)
- Yeni donem tarihleri upgrade/downgrade aninda yeniden hesaplanir

### 12.6 Coklu Abone Tipi (Seller/Buyer/Organization)

**Senaryo:** Ayni kullanici hem Seller hem de Buyer olarak abonelige sahip olabilir.

**Cozum:**
- Her `subscriber_type` icin ayri abonelik kaydi
- Mevcut `validate_subscriber()` zaten ayni tipteki birden fazla aktif aboneligi engelliyor
- Bildirimler her abonelik icin ayri gonderilir
- Dashboard her abonelik tipini ayri sekmede gosterir

### 12.7 Zaman Dilimi ve Gece Yarisi Gecisleri

**Senaryo:** Turkiye saatiyle gece 23:55'te baslatilan odeme islemi, sunucu saatiyle ertesi gune denk gelir.

**Cozum:**
- Tum tarih karsilastirmalari sunucu zaman diliminde (UTC veya Turkiye saati) yapilir
- `end_date` alani Date (tarih) tipindedir — saat bilgisi icermez
- Scheduler gunluk gorevleri sabah erken saatlerde (00:30-03:00 arasi) calistirir
