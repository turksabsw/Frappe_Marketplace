# Task 116 - RBAC/ABAC Role & Permission Architecture Plan

## Executive Summary

TradeHub is a multi-tenant B2B wholesale marketplace built on Frappe v15/ERPNext v15, targeting Turkey-first expansion into MENA and Turkic states. The current permission model relies heavily on Frappe's built-in DocType-centric RBAC with a handful of roles (`System Manager`, `All`, `Seller`, `Buyer`, `Marketplace Admin`, `Marketplace Moderator`, `Marketplace Manager`, `Marka Sahibi`, `Satici Admin`, `Satici`, `Alici Admin`, `Alici Editor`, `Tenant Admin`, `Sales Manager`, `Administrator`) and a fully implemented but **not activated** tenant isolation layer in `tradehub_core/permissions.py`. This document designs a comprehensive 3-tier role hierarchy (Platform Provider, Seller, Buyer), defines a permission taxonomy spanning 7 functional modules, plans ABAC extensions via Frappe's `has_permission` and `permission_query_conditions` hooks, and addresses multi-country regulatory compliance (Turkey KVKK, EU GDPR, AML/KYC, sanctions/embargo). The architecture follows an "80/20 rule": 80% of access control via standard Frappe RBAC roles, 20% via attribute-based extensions (tenant, country, spending limits, risk level).

## Current State Analysis

### 1. Frappe RBAC Foundation

Frappe v15 provides DocType-centric role-based access control:

| Feature | Description |
|---------|-------------|
| **Roles** | Named roles assigned to users (e.g., `System Manager`, `Seller`) |
| **DocPerm** | Permission rules in DocType JSON: role -> CRUD flags per DocType |
| **Permission Levels** | 0-9 field grouping levels for fine-grained field-level access |
| **User Permissions** | Record-level restrictions linking users to specific Link field values |
| **Role Profiles** | Bundle multiple roles into a profile for easy assignment |
| **if_owner** | Permission flag restricting access to records owned by the user |
| **submit/cancel/amend** | Workflow-specific permission flags for submittable DocTypes |

**Limitations for B2B Marketplace:**
- No native attribute-based access (country, spending limit, risk level)
- No delegation or temporary permission assignment
- No approval chain or spending limit enforcement
- No multi-tenant isolation out of the box
- `permission_query_conditions` only affects `frappe.db.get_list()`, NOT `frappe.db.get_all()`

**Key observations:**

1. **Mixed language naming**: Turkish roles (`Marka Sahibi`, `Satici`, `Alici Admin`, `Alici Editor`) coexist with English roles (`Seller`, `Buyer`, `Marketplace Admin`). This creates confusion and should be standardized.

2. **Overly broad `All` role**: Many DocTypes grant read/write to `All`, which means any logged-in user (including unverified buyers) can access seller balances, escrow accounts, listings, etc. This is a significant security gap.

3. **No role granularity for finance**: Escrow Account, Seller Balance, Payment Intent, Commission Plan all use `System Manager` + `All` -- no dedicated finance role.

4. **Missing ABAC activation**: `tradehub_core/permissions.py` has fully implemented `has_tenant_permission()` and `get_tenant_permission_query_conditions()` with `setup_has_permission()` and `setup_tenant_permission_query_conditions()` helper functions that auto-discover tenant-aware DocTypes. However, the permission hooks in **all** `hooks.py` files remain **commented out**.

5. **Organization Member roles exist but are disconnected**: The `Organization Member` child table has a `role` field with options `Admin|Manager|Purchaser|Finance|Member|Viewer`, but these are data-level roles with no connection to Frappe's permission system.

### 3. Tenant Isolation Infrastructure

## Proposed Design / Architecture

### 1. Role Derivation Methodology

Roles should be derived from three dimensions:

```
                    +-----------------+
                    |  ROLE DERIVATION |
                    +--------+--------+
                             |
              +--------------+--------------+
              |              |              |
     +--------v--------+  +-v--------+  +--v-----------+
     |   FUNCTION-BASED |  | PROCESS  |  | ORG HIERARCHY|
     |   (What they do) |  | (Workflow)|  | (Who they    |
     +------------------+  +----------+  |  report to)  |
     | - Catalog mgmt   |  | - Order  |  +--------------+
     | - Finance ops    |  |   flow   |  | - Owner      |
     | - Moderation     |  | - KYC    |  | - Manager    |
     | - Logistics      |  |   review |  | - Staff      |
     | - Support        |  | - Escrow |  | - Viewer     |
     +------------------+  |   release|  +--------------+
                            +----------+
```

**"Company Role" vs "User Role" Distinction:**

| Concept | Definition | Example |
|---------|-----------|---------|
| **Company Role** | The organization's function on the platform | Seller Organization, Buyer Organization, Platform Operator |
| **User Role** | An individual user's function within their company | Seller Staff, Buyer Finance, Platform Moderator |

A single user may hold roles in multiple organizations (e.g., an individual who is both a seller and a buyer), and organizations may operate in multiple countries with different regulatory requirements.

**Country-Based Role Variations:**
Roles remain consistent across countries, but ABAC attributes determine what actions are available:

| Attribute | Turkey | EU/GDPR | MENA |
|-----------|--------|---------|------|
| KYC required for selling | Yes (TCKN/VKN) | Yes (national ID) | Varies by country |
| KVKK consent required | Yes | No (GDPR consent instead) | No |
| E-Invoice capability | e-Fatura/e-Arsiv | Country-specific | Varies |
| Sanctions screening | MASAK list | EU sanctions list | Local + OFAC |
| Data residency | Turkey servers | EU servers | Varies |

### 2. Three Role Groups

#### Group 1: Platform Provider Roles

These roles are held by TradeHub staff who operate the marketplace platform.

| Role Name | Frappe Role ID | Description | Typical Users |
|-----------|---------------|-------------|---------------|
| **Platform Admin** | `Platform Admin` | Full platform administration, tenant management, system configuration | CTO, Head of Operations |
| **Platform Moderator** | `Platform Moderator` | Content moderation, case review, seller/buyer dispute resolution | Moderation team |
| **Platform Support** | `Platform Support` | Customer support, read-only access to most data, can create support tickets | Support agents |
| **Content Reviewer** | `Content Reviewer` | Catalog content review, listing approval, media moderation | Catalog team |

**Replaces**: `Marketplace Admin` -> `Platform Admin`, `Marketplace Moderator` -> `Platform Moderator`, `Marketplace Manager` -> merged into `Platform Admin`

#### Group 2: Seller Roles

These roles are held by users within seller organizations.

| Role Name | Frappe Role ID | Description | Typical Users |
|-----------|---------------|-------------|---------------|
| **Seller Owner** | `Seller Owner` | Full control of seller account, can add/remove team members | Business owner |
| **Seller Manager** | `Seller Manager` | Manage listings, orders, pricing; cannot modify account settings | Operations manager |
| **Seller Staff** | `Seller Staff` | Create/edit listings, process orders, handle shipping | Warehouse staff, catalog editors |
| **Seller Finance** | `Seller Finance` | View financial reports, escrow, balances, payouts; no catalog access | Accountant, CFO |
| **Seller Logistics** | `Seller Logistics` | Manage shipments, tracking, carriers; no financial access | Logistics coordinator |

**Replaces**: `Seller` -> split into 5 granular roles, `Satici Admin` -> `Seller Owner`, `Satici` -> `Seller Staff`, `Marka Sahibi` -> `Seller Owner` (for brand owners)

#### Group 3: Buyer Roles

These roles are held by users within buyer organizations.

| Role Name | Frappe Role ID | Description | Typical Users |
|-----------|---------------|-------------|---------------|
| **Buyer Procurement** | `Buyer Procurement` | Browse catalog, create orders, manage cart, send RFQs | Procurement officer |
| **Buyer Approver** | `Buyer Approver` | Approve orders above spending limit, manage approval chains | Department head, VP |
| **Buyer Viewer** | `Buyer Viewer` | Read-only access to catalog, orders, and reports | Analyst, intern |
| **Buyer Finance** | `Buyer Finance` | Manage payments, view invoices, handle escrow releases | Accountant, CFO |

**Replaces**: `Buyer` -> split into 4 granular roles, `Alici Admin` -> `Buyer Approver`, `Alici Editor` -> `Buyer Procurement`

#### 3.3 Critical Permissions Separation

Certain operations must be separated from general roles for security and compliance:

| Critical Permission | Requires Role | Additional Gate | Rationale |
|--------------------|--------------|----------------|-----------|
| **Escrow release** | Platform Admin | Multi-approval (2 admins) | Financial risk |
| **Tenant subscription change** | System Manager | Audit log | Revenue impact |
| **Seller payout execution** | Platform Admin + Seller Finance | Amount threshold check | Financial risk |
| **KYC verification override** | Platform Admin | Reason required + audit log | Compliance risk |
| **Sanctions/AML override** | System Manager only | Cannot be delegated | Regulatory risk |
| **Contract template modification** | Platform Admin | Version increment required | Legal risk |
| **Commission rate change** | System Manager | `permlevel: 1` (already enforced) | Revenue impact |
| **User role assignment** | Platform Admin | Cannot self-assign Platform Admin | Privilege escalation |
| **Data export (bulk)** | Explicit `export` flag per role | KVKK/GDPR consent check via ABAC | Data privacy |
| **Moderation case resolution** | Platform Moderator | Cannot resolve own-created cases | Conflict of interest |

### 4. Approval & Delegation Architecture

#### 4.1 Spending Limit Approval Chains

B2B procurement often requires multi-level approval based on order value:

```
Order Value          Required Approvals
--------------------------------------------------
< 10,000 TRY       Buyer Procurement (self-approve)
10,000 - 50,000     Buyer Procurement -> Buyer Approver
50,000 - 250,000    Buyer Procurement -> Buyer Approver -> Org Admin
> 250,000 TRY       Buyer Procurement -> Buyer Approver -> Org Admin -> CFO
```

**Implementation approach**: New `Spending Approval Rule` DocType:

```
DocType: Spending Approval Rule
Fields:
  - organization: Link(Organization)
  - tenant: Link(Tenant)
  - min_amount: Currency
  - max_amount: Currency
  - currency: Link(Currency)
  - required_approver_role: Select(Buyer Approver/Org Admin/Buyer Finance)
  - approval_sequence: Int (ordering)
  - is_active: Check
```

The `has_permission` hook for `Marketplace Order` and `Order` would check spending limits:

```python
# Pseudocode for approval chain in has_permission
def has_order_permission(doc, ptype, user):
    if ptype == "submit":
        order_amount = doc.grand_total
        org = doc.organization
        rules = get_spending_rules(org, order_amount)
        for rule in rules:
            if not has_approval(doc, rule.required_approver_role):
                return False  # Block submit until all approvals received
    return has_tenant_permission(doc, ptype, user)
```

#### 4.2 Temporary Delegation DocType

For vacations, sick leave, or cross-functional coverage:

```
DocType: Permission Delegation
Fields:
  - delegator: Link(User) -- who is delegating
  - delegate: Link(User) -- who receives temporary permissions
  - delegation_type: Select(Full Role/Specific DocType/Specific Document)
  - delegated_role: Link(Role) -- if Full Role type
  - delegated_doctype: Link(DocType) -- if Specific DocType type
  - delegated_document: Dynamic Link -- if Specific Document type
  - start_date: Date
  - end_date: Date
  - reason: Small Text
  - status: Select(Draft/Active/Expired/Revoked)
  - approved_by: Link(User)
  - tenant: Link(Tenant)

Constraints:
  - Cannot delegate System Manager role
  - Cannot delegate to users outside own tenant
  - Maximum delegation period: 90 days
  - Requires approval from user's manager or Platform Admin
  - Auto-expires via scheduler task
```

**Integration point**: The `has_permission` hook would check active delegations:

```python
def has_delegated_permission(doc, ptype, user):
    active_delegations = frappe.get_all("Permission Delegation", filters={
        "delegate": user,
        "status": "Active",
        "start_date": ["<=", today()],
        "end_date": [">=", today()]
    })
    for delegation in active_delegations:
        if delegation.delegation_type == "Full Role":
            # Temporarily treat user as having the delegated role
            if role_has_permission(delegation.delegated_role, doc.doctype, ptype):
                return True
    return None  # Fall through to normal permission check
```

#### 4.3 Multi-Approval Scenarios

For high-risk operations, multiple independent approvals may be required:

| Operation | Approvals Required | Conflict Rule |
|-----------|-------------------|---------------|

```python
def compliance_gated_permission(doc, ptype, user):
    # Get user's KYC profile
    kyc = get_user_kyc_profile(user)

    # Financial operations require cleared AML
    if doc.doctype in FINANCIAL_DOCTYPES:
        if not kyc or kyc.aml_check_status != "Cleared":
            frappe.throw(_("AML clearance required for financial operations"))
            return False

    # Seller operations require verified KYC
    if doc.doctype in SELLER_DOCTYPES:
        if not kyc or kyc.status != "Verified":
            return False

    # High-risk users have restricted operations
    if kyc and kyc.risk_level in ("High", "Very High"):
        if ptype in ("write", "submit", "cancel"):
            # Log for review, but allow with flag
            log_high_risk_operation(user, doc, ptype)

    return True
```

#### 6.2 Extended ABAC Permission Handler

Beyond tenant isolation, the `has_permission` hook should be extended with ABAC attributes:

```python
# tradehub_core/permissions.py -- Extended has_permission
def has_extended_permission(doc, ptype=None, user=None):
    """
    Extended permission check combining RBAC + ABAC.

    ABAC attributes checked:
    1. Tenant isolation (existing)
    2. Organization membership
    3. Seller profile ownership
    4. KYC/AML compliance status
    5. Spending limits
    6. Country-specific regulations
    """
    user = user or frappe.session.user
    ptype = ptype or "read"

    # Layer 1: Tenant isolation (existing logic)
    if not has_tenant_permission(doc, ptype, user):
        return False

    # Layer 2: Organization-scoped access
    if hasattr(doc, "organization") and doc.organization:
        if not is_organization_member(user, doc.organization):
            # User is in same tenant but different organization
            # Allow read for public-facing DocTypes, deny for private ones
            if doc.doctype in ORGANIZATION_PRIVATE_DOCTYPES:
                return False

    # Layer 3: Seller profile ownership
    if hasattr(doc, "seller") and doc.seller:
        if not is_seller_user(user, doc.seller):
            if doc.doctype in SELLER_PRIVATE_DOCTYPES and ptype != "read":
                return False

    # Layer 4: KYC/AML compliance gate
    if ptype in ("create", "write", "submit") and doc.doctype in COMPLIANCE_GATED_DOCTYPES:
        if not check_compliance_status(user):
            return False

    # Layer 5: Delegation check
    delegation_result = check_delegation(doc, ptype, user)
    if delegation_result is not None:
        return delegation_result

    return True
```

```python
def get_extended_query_conditions(user=None):
    """
    Build WHERE clause combining tenant + organization + seller filtering.

    user = user or frappe.session.user
    conditions = []

    # Tenant condition
    tenant_cond = get_tenant_permission_query_conditions(user)

    user_orgs = get_user_organizations(user)
    if user_orgs and "System Manager" not in frappe.get_roles(user):
        org_list = ", ".join(f"'{frappe.db.escape(o)}'" for o in user_orgs)
        conditions.append(f"`organization` IN ({org_list})")

    # Seller condition (for seller-scoped DocTypes)

    if user_seller and "System Manager" not in frappe.get_roles(user):

    return " AND ".join(conditions) if conditions else ""
```

#### 8.1 Alibaba B2B (3-Tier Model)

Alibaba's B2B platform uses a 3-tier permission model:

| Tier | Description | TradeHub Equivalent |
|------|-------------|-------------------|
| **Platform** | Alibaba staff managing marketplace operations | Platform Admin/Moderator/Support |
| **Supplier** | Seller companies with sub-accounts for different functions | Seller Owner/Manager/Staff/Finance |

- **Trust Levels**: Gold Supplier, Verified Supplier, Trade Assurance -> maps to TradeHub's Seller Tier/Level system

Key Amazon patterns adopted:
- **Guided buying**: Restrict catalog visibility by buyer role/department

2. **Organization Member -> Role Profile sync**: A mechanism must be built to automatically assign Frappe Role Profiles when Organization Member records are created/updated.

5. **Task 106 (i18n) coordination**: New role names must be included in translation CSVs for all target languages (tr, en, ar, de).

6. **`permission_query_conditions` limitation**: This hook only affects `frappe.db.get_list()`. API endpoints using `frappe.db.get_all()` must be individually audited to add manual tenant filtering via `apply_tenant_filter()`.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Cross-tenant data leakage** | Medium | Critical | Comprehensive testing with multi-tenant setup; `permission_query_conditions` on ALL tenant-aware DocTypes; code review for `frappe.db.get_all()` usage |
| **Permission hook performance** | Medium | High | Cache tenant context (already implemented); minimize DB queries in `has_permission`; use `permission_query_conditions` for list filtering (SQL-level) |
| **`All` role removal breaking existing users** | High | High | Phase removal: first add new roles, then remove `All`; test with staging data; provide migration script |
| **Organization Member role sync lag** | Low | Medium | Real-time sync via doc_events on Organization Member; fallback to periodic scheduler task |
| **Spending limit bypass via API** | Medium | High | Enforce limits in both `has_permission` hook AND `validate` doc_event; audit API endpoints |
| **Delegation abuse** | Low | Medium | Maximum 90-day delegation; require manager approval; cannot delegate System Manager; audit logging |
| **Country regulation mismatch** | Medium | High | Configurable per-tenant compliance rules; default to strictest regulation; legal review before each country launch |
| **`permission_query_conditions` vs `get_all` gap** | High | High | Audit all codebase for `frappe.db.get_all()` calls that should respect tenant isolation; consider monkey-patching or wrapping `get_all` |
| **Turkish role name migration** | Medium | Medium | Create migration script that maps old roles to new; run in staging first; keep old roles temporarily as aliases |
| **Hook startup timing** | Low | High | Use Option A (manual registration) instead of Option B (dynamic); test thoroughly during `bench restart` |
