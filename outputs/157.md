# Task 157 - Link Field Filtering: Complete Audit & Implementation Plan

## 1. Overview

### 1.1 Purpose

This document provides a comprehensive audit and implementation plan for filtering all Link fields across the TradeHub Marketplace platform. Unfiltered Link fields degrade user experience by presenting irrelevant options (e.g., sellers from other tenants, inactive categories, products from unrelated brands), increase data entry errors, and create potential data integrity issues in a multi-tenant B2B marketplace.

### 1.2 Scope

- **7 TradeHub apps** audited: `tradehub_core`, `tradehub_seller`, `tradehub_catalog`, `tradehub_commerce`, `tradehub_logistics`, `tradehub_marketing`, `tradehub_compliance`
- **737 total Link fields** identified across all DocTypes
- **139 fields (18.9%)** already have `set_query` filters
- **598 fields (81.1%)** currently unfiltered
- **~407 editable unfiltered fields** need filter rules (excluding read-only/hidden fields)
- **~95 required fields** among unfiltered editable fields (critical priority)

### 1.3 Key Findings

| Metric | Count |
|--------|-------|
| Total Link fields across all apps | 737 |
| Already filtered via `set_query` | 139 |
| Unfiltered (all) | 598 |
| Unfiltered + editable + visible | 407 |
| Unfiltered required fields | ~95 |

| Child table fields filtered from parent JS | 12 |

### 1.4 Filter Priority Classification

| Priority | Criteria | Count (est.) |
|----------|----------|------|
| **P0 - Critical** | Tenant isolation (multi-tenancy leak risk), required fields with no filter | ~120 |
| **P1 - High** | Cross-entity scoping (Seller, Buyer, Organization), cascading dependencies | ~100 |
| **P2 - Medium** | Domain-specific filters (UOM categories, active-only, country scoping) | ~120 |
| **P3 - Low** | System/admin fields (User, DocType, Workflow), rarely edited fields | ~67 |

### 2.3 Existing `set_query` Patterns in Codebase

The codebase already implements several well-established filtering patterns across 18 JS files:

**Pattern A: Tenant Isolation** (most common, 40+ instances)
```javascript
// Used in: order.js, cart.js, marketplace_order.js, seller_profile.js, etc.
frm.set_query('field_name', function() {
    if (frm.doc.tenant) {
        return {
            filters: {
                'tenant': frm.doc.tenant
            }
        };
    }
    return {};
});
```

**Pattern B: Cascading Parent-Child** (category -> subcategory, city -> district -> neighborhood)
```javascript
// Used in: listing.js, seller_profile.js, district.js, neighborhood.js
frm.set_query('subcategory', function() {
    let filters = { 'is_active': 1 };
    if (frm.doc.category) {
        filters['parent_category'] = frm.doc.category;
    }
    return { filters: filters };
});
```

**Pattern C: Child Table Cascading** (child table fields filtered by row values)
```javascript
// Used in: seller_profile.js (addresses, locations), cart.js (items)
frm.set_query('district', 'addresses', function(doc, cdt, cdn) {
    let row = locals[cdt][cdn];
    return {
        filters: {
            city: row.city || '',
            is_active: 1
        }
    };
});
```

**Pattern D: Custom Query** (server-side filtering with complex logic)
```javascript
// Used in: cart.js for buyer filtering
frm.set_query('buyer', function() {
    if (frm.doc.tenant) {
        return {
            query: 'tr_tradehub.tr_tradehub.doctype.cart.cart.get_buyers_for_tenant',
            filters: { 'tenant': frm.doc.tenant }
        };
    }
    return {};
});
```

**Pattern E: Restricted DocType List** (limit Link to specific DocType names)
```javascript
// Used in: moderation_case.js
frm.set_query('content_type', function() {
    return {
        filters: {
            'name': ['in', ['Listing', 'Review', 'Storefront', 'Message',
                           'Seller Profile', 'Certificate']]
        }
    };
});
```

## 4. Context-Aware Filter Rules

### 4.1 Rule Category: Tenant Isolation (P0 - Critical)

**Rule:** Every Link field targeting a tenant-aware DocType MUST be filtered by the current document's `tenant` value. This is the foundational multi-tenancy requirement.

**Affected fields:** ~79 `tenant` fields + ~41 fields on tenant-aware targets (Seller Profile, Organization, Category, etc.)

**Filter logic:**
```javascript
// Standard tenant isolation pattern
frm.set_query('field_name', function() {
    if (frm.doc.tenant) {
        return {
            filters: {
                'tenant': frm.doc.tenant
            }
        };
    }
    return {};
});
```

**Special cases:**
- If the target DocType does not have a `tenant` field (e.g., `Currency`, `Country`, `UOM`), tenant filtering is not applicable
- If the document itself is the tenant record, skip self-filtering
- `Tenant` fields themselves don't need filtering (they ARE the tenant selector)

### 4.2 Rule Category: Entity Scoping (P1 - High)

**Rule:** Link fields that reference entities belonging to a parent entity must be scoped to that parent.

| Scope Pattern | Fields | Filter |
|---------------|--------|--------|
| Seller -> Tenant | Seller Profile links when tenant is set | `tenant: frm.doc.tenant, status: 'Active'` |
| Buyer -> Tenant | Buyer Profile links when tenant is set | `tenant: frm.doc.tenant` |
| Organization -> Tenant | Organization links when tenant is set | `tenant: frm.doc.tenant` |
| Listing -> Seller + Tenant | Listing links when seller/tenant is set | `seller: frm.doc.seller, tenant: frm.doc.tenant, status: 'Active'` |
| Listing Variant -> Listing | Listing Variant links when listing is set | `listing: frm.doc.listing` or `listing: row.listing` |
| SKU -> Seller | SKU links when seller is set | `seller: frm.doc.seller` |
| Sub Order -> Marketplace Order | Sub Order links | `marketplace_order: frm.doc.marketplace_order` |
| Contract -> Tenant | Contract links | `tenant: frm.doc.tenant` |

### 4.3 Rule Category: Status/Active Filtering (P2 - Medium)

**Rule:** Link fields should filter out disabled, inactive, or irrelevant records where applicable.

| Target DocType | Status Filter | Condition |
|----------------|---------------|-----------|
| Currency | `enabled: 1` | Always |
| Country | `enabled: 1` | Always |
| UOM | Category-based (see Section 4.4) | Always |
| Category | `is_active: 1` | Always |
| Brand | `is_active: 1` | Always |
| Seller Profile | `status: 'Active'` | When selecting sellers for operations |
| Commission Plan | `is_active: 1` | Always |
| Subscription Package | `is_active: 1` | Always |
| Contract Template | `is_active: 1` | Always |
| Consent Topic | `is_active: 1` | Always |
| Warehouse | `is_group: 0, disabled: 0` | Always (leaf warehouses only) |
| Item Group | `is_group: 0` | When selecting leaf groups |
| Supplier | `disabled: 0` | Always |
| User | `enabled: 1` | Always |

### 4.4 Rule Category: UOM Category Filtering (P2 - Medium)

**Rule:** UOM Link fields must be filtered by their measurement category. This prevents nonsensical selections like using "kg" for a dimension field.

| UOM Context | Category Filter | Applicable Fields |
|-------------|----------------|-------------------|
| Weight/Mass | `uom_category: 'Mass'` | weight_uom in Listing, SKU, Shipment, etc. |
| Dimensions/Length | `uom_category: 'Length'` | dimension_uom in Listing, SKU, PIM Product, etc. |
| Stock/Quantity | `uom_category: 'Unit'` | stock_uom in Listing, Order Item, etc. |
| Volume | `uom_category: 'Volume'` | volume_uom in Shipment |
| General | No category filter | uom/unit fields in RFQ (user picks any) |

> **Note:** UOM category filtering depends on the `uom_category` custom field being added to the ERPNext UOM DocType. See Task 122 (outputs/122.md) for the full UOM category implementation plan.

| Message Thread | reference_doctype | `Listing`, `Order`, `RFQ`, `Sample Request` |

| SEO Meta | reference_doctype | `PIM Product`, `Category`, `Brand`, `Storefront` |
| ECA Rule | event_doctype | `istable: 0, issingle: 0, is_virtual: 0` |
| ECA Rule Action | target_doctype | `istable: 0, issingle: 0` |
| Product Class | workflow_template / approval_workflow | `document_type: 'PIM Product'` |
| Buyer Category | approval_workflow | `document_type: 'Buyer Category'` |

### 4.7 Rule Category: Admin/System User Filtering (P3 - Low)

**Rule:** User Link fields that represent admin actions (assigned_to, reviewed_by, approved_by, escalated_to) should filter for System Users.

```javascript
    return {

            'enabled': 1,
            'user_type': 'System User'
        }
    };

```

**Affected fields:** assigned_to, reviewed_by, approved_by, escalated_to, action_by, awarded_by, account_manager, released_by, initiated_by, created_by (~25 fields across compliance, seller, commerce apps)

## 5. Implementation Method Per Field

### 5.1 Method Selection Criteria

| Method | When to Use | Pros | Cons |
|--------|-------------|------|------|
| **`set_query` (dynamic)** | Filter depends on current form values (tenant, seller, category) | Reactive to user input, cascading filters | Requires JS file, slightly more code |
| **`link_filters` (static JSON)** | Filter is always the same regardless of form state | No JS needed, simple DocType JSON property | Cannot reference `frm.doc` values |
| **Custom Query (server-side)** | Complex filtering requiring joins, permissions, or computed logic | Full SQL power, complex authorization | Requires Python API endpoint |

### 5.2 Implementation Method Assignment

| Filter Category | Method | Rationale |
|-----------------|--------|-----------|
| Tenant isolation | `set_query` (dynamic) | Depends on `frm.doc.tenant` |
| Entity scoping (Seller, Buyer, Organization) | `set_query` (dynamic) | Depends on current entity selection |
| Cascading parent-child | `set_query` (dynamic) | Depends on parent field value |
| Active/enabled status | `set_query` (dynamic) or `link_filters` (static) | Static for simple `enabled: 1`; dynamic if combined with tenant |
| UOM category | `set_query` (dynamic) | Will use `uom_category` field (requires Task 122) |
| Self-reference prevention | `set_query` (dynamic) | Depends on `frm.doc.name` |
| Restricted DocType lists | `set_query` (dynamic) or `link_filters` (static) | Static if always the same list; dynamic if combined with tenant |
| Admin user filtering | `set_query` (dynamic) | Combined with `enabled: 1` check |

### 5.3 Fields Eligible for Static `link_filters` (JSON)

These fields always apply the same filter regardless of form context and could use the DocType JSON `link_filters` property (Frappe v15+):

| DocType | Field | Static Filter |
|---------|-------|---------------|
| Buyer Level | threshold_currency | `{"enabled": 1}` |
| Phone Code | city | `{"is_active": 1}` |
| Product | brand | `{"is_active": 1}` |
| Sales Channel | default_currency | `{"enabled": 1}` |

> **Recommendation:** Use `set_query` for all fields to maintain consistency and allow future filter expansion. Static `link_filters` is acceptable for simple status filters on fields with no tenant context.

```javascript

// Cascading clear on change
frappe.ui.form.on('Address Item', {
    city: function(frm, cdt, cdn) {
        let row = locals[cdt][cdn];
        if (!row.city) {
            frappe.model.set_value(cdt, cdn, 'district', '');
            frappe.model.set_value(cdt, cdn, 'neighborhood', '');
        } else {
            // Check if current district belongs to new city
            if (row.district) {
                frappe.db.get_value('District', row.district, 'city', function(r) {
                    if (r && r.city !== row.city) {
                        frappe.model.set_value(cdt, cdn, 'district', '');
                        frappe.model.set_value(cdt, cdn, 'neighborhood', '');
                    }
                });
            }
        }
    },

    district: function(frm, cdt, cdn) {
        let row = locals[cdt][cdn];
        if (!row.district) {
            frappe.model.set_value(cdt, cdn, 'neighborhood', '');
        } else {
            // Check if current neighborhood belongs to new district
            if (row.neighborhood) {
                frappe.db.get_value('Neighborhood', row.neighborhood, 'district', function(r) {
                    if (r && r.district !== row.district) {
                        frappe.model.set_value(cdt, cdn, 'neighborhood', '');
                    }
                });
            }
        }
    }
```

## 8. Edge Cases

### 8.1 Empty Filter Results

**Problem:** Filters may be too restrictive, returning zero results, which confuses users.

**Scenarios:**
1. Tenant has no active sellers yet -> Seller Profile filters return empty
2. Category has no attribute sets assigned -> Attribute Set filter returns nothing
3. Listing has no variants created -> Listing Variant filter is empty
4. New tenant with no data yet -> Most tenant-scoped filters empty

**Mitigation strategies:**

```javascript
// Strategy 1: Graceful fallback with user guidance
frm.set_query('seller', function() {
    if (frm.doc.tenant) {
        return {
            filters: {
                'tenant': frm.doc.tenant,
                'status': 'Active'
            }
        };
    }
    // No tenant selected - show all (with warning via description)
    return {};
});

// Strategy 2: Show alert when no options available
frm.fields_dict['attribute_set'].df.onchange_after_query = function() {
    if (frm.fields_dict['attribute_set'].$input &&
        !frm.fields_dict['attribute_set'].$input.val()) {
        // If user clicks the field and no results come back
        // The Link field's awesomplete will show "No results"
        // No additional code needed - this is Frappe's default behavior
    }
};

// Strategy 3: Pre-check and disable field if no options
if (frm.doc.tenant) {
    frappe.call({
        method: 'frappe.client.get_count',
        args: {
            doctype: 'Seller Profile',
            filters: { 'tenant': frm.doc.tenant, 'status': 'Active' }
        },
        callback: function(r) {
            if (r && r.message === 0) {
                frm.set_df_property('seller', 'description',
                    __('No active sellers found for this tenant. Create a Seller Profile first.'));
            }
        }
    });
}
```

### 8.2 Circular Dependencies

**Problem:** Field A's filter depends on Field B, while Field B's filter depends on Field A.

**Identified circular risks:**

| Field A | Field B | Dependency |
|---------|---------|------------|
| seller | tenant | Seller filtered by tenant; tenant can auto-set from seller |
| organization | tenant | Organization filtered by tenant; tenant auto-set from org |
| category | attribute_set | Category filters attribute_set; attribute_set can belong to category |

**Resolution pattern (already implemented in listing.js and seller_profile.js):**

```javascript
// Resolution: Establish a primary direction (tenant -> seller, not reverse)
// 1. User selects tenant first (or it auto-populates from parent)
// 2. Seller is filtered by tenant
// 3. If seller is selected first, auto-populate tenant FROM seller
// 4. Make tenant read-only when seller is set

frm.set_query('seller', function() {
    if (frm.doc.tenant) {
        return {
            filters: { 'tenant': frm.doc.tenant }
        };
    }
    return {};
});

// Handle reverse direction
seller: function(frm) {
    if (frm.doc.seller) {
        // Auto-populate tenant from seller
        frappe.db.get_value('Seller Profile', frm.doc.seller, 'tenant', function(r) {
            if (r && r.tenant) {
                if (!frm.doc.tenant || frm.doc.tenant !== r.tenant) {
                    frm.set_value('tenant', r.tenant);
                }
            }
        });
        // Lock tenant since it's derived from seller
        frm.set_df_property('tenant', 'read_only', 1);
    } else {
        frm.set_df_property('tenant', 'read_only', 0);
    }
}
```

**Circular dependency resolution rules:**
1. **Tenant is always the root** - Never filter tenant based on child entities
2. **Use `read_only` to prevent conflicts** - When a child sets the parent, lock the parent field
3. **Clear-on-change handlers** - When a parent changes, validate and clear invalid children
4. **One-directional cascade** - Always cascade DOWN (tenant -> organization -> seller), never UP

### 8.3 Permission-Based Filter Visibility

**Problem:** Some filters may expose records the user shouldn't see (permission bypass through search).

**Mitigation:**
- Frappe's Link field search automatically respects DocType-level permissions
- `frappe.has_permission()` is checked server-side when searching
- No additional client-side permission filtering needed for standard cases
- For sensitive DocTypes, use `ignore_permissions=False` in custom queries

### 8.4 Race Conditions in Cascading Filters

**Problem:** Rapid field changes (tenant -> seller -> listing) can cause race conditions where `frappe.db.get_value` callbacks execute out of order.

**Mitigation:**
```javascript
// Use the built-in Frappe debounce behavior for Link field searches
// Frappe's awesomplete already debounces search requests

// For cascading clears, always check current value before clearing
seller: function(frm) {
    if (frm.doc.listing) {
        frappe.db.get_value('Listing', frm.doc.listing, 'seller', function(r) {
            // Re-check frm.doc.seller hasn't changed during async call
            if (r && r.seller !== frm.doc.seller && frm.doc.listing) {
                frm.set_value('listing', null);
            }
        });
    }
}
```

### 8.5 Bulk Data Import Compatibility

**Problem:** `set_query` filters only apply in the UI. Data imports via CSV or API bypass these filters.

**Mitigation:**
- Server-side `validate()` must also enforce the same constraints
- `set_query` is for UX guidance; `validate()` is for data integrity
- Plan: Add corresponding Python validations for P0 (tenant isolation) filters

### 8.6 Performance with Large Datasets

**Problem:** Some DocTypes (User, Country, Currency) may have many records, causing slow Link field searches.

**Mitigation:**
- Frappe Link fields use AJAX search with LIMIT (default 20 results)
- Filters reduce the search space, improving performance
- For very large tables, ensure indexed fields are used in filters
- `tenant`, `status`, `is_active`, `enabled` should all be indexed
