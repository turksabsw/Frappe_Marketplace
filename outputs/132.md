# Task 132 - Buyer & Seller Scoring System Plan

## 1. Overview

This document defines the **scoring engine** for the TradeHub marketplace platform. The scoring system consumes raw metrics defined in `119.md` and produces normalized, weighted scores for both buyers and sellers. It provides:

- **Admin-configurable buyer metrics**: On-time payment, return rate, order frequency, dispute rate, total spend, feedback rate, average order value, account age, and last activity.
- **Admin-configurable seller metrics**: Order Defect Rate (ODR), shipping performance (late shipment rate, on-time delivery), response time, rating, return rate, cancellation rate, compliance, and engagement.
- **Automatic scoring formula system**: Configurable scoring methods (Linear, Step, Logarithmic, Exponential, Binary, Custom) applied per metric using the existing `KPI Template` and `KPI Template Item` DocTypes.
- **Tier/badge auto-assignment**: Automatic tier promotion/demotion based on score thresholds using the existing `Seller Tier` and `Buyer Level` DocTypes.
- **Six system integration points**: Search ranking, Buy Box weight, commission rates, segmentation, trust badges, and risk management.

### Existing DocTypes Referenced

| DocType | App | Role in Scoring System |
|---------|-----|----------------------|
| `Seller Score` | tradehub_seller | Stores per-period seller score records with component scores, weights, penalties, bonuses |
| `KPI Template` | tradehub_seller | Defines the configurable template for scoring metrics (target_type: Seller/Buyer/Both) |
| `KPI Template Item` | tradehub_seller | Child table of `KPI Template` defining individual metrics: name, code, weight, formula, thresholds, scoring method |
| `Seller Tier` | tradehub_seller | Tier definitions (Bronze-Top Rated) with minimum score requirements and benefits |
| `Buyer Level` | tradehub_core | Level definitions (New-Premium) with qualification thresholds and benefits |
| `Seller Metrics` | tradehub_seller | Raw seller metric data (ODR, return_rate, cancellation_rate, etc.) consumed by scoring |
| `Buyer Profile` | tradehub_core | Raw buyer metric data (payment_on_time_rate, return_rate, dispute_rate, etc.) consumed by scoring |

### Design Principles

1. **Separation of Metrics vs. Scores**: `119.md` defines *what* is measured (raw metrics). This document defines *how* those measurements are scored, weighted, and applied.
2. **Admin Configurability**: All metric weights, thresholds, and scoring methods are configurable via the `KPI Template` system -- no code changes needed to adjust scoring.
3. **Dual Scoring**: Buyer and seller scoring use the same engine (`KPI Template`) with different templates (`target_type = "Buyer"` vs `target_type = "Seller"`).
4. **Score Isolation**: A seller's score is independent of buyer scoring and vice versa. Each has its own template, weights, and tier system.

## 2. Reference Models

### 2.1 Amazon Account Health & Performance Score

Amazon uses a composite **Account Health Score (AHS)** of 0-1000 with weighted components:

| Component | Amazon Weight | TradeHub Mapping |
|-----------|-------------|------------------|
| Order Defect Rate | Critical (binary gate) | `order_defect_rate` metric, weight configurable via KPI Template |
| Late Shipment Rate | High | `late_shipment_rate` metric |
| Valid Tracking Rate | Medium | `valid_tracking_rate` metric |
| Pre-Fulfillment Cancel Rate | High | `cancellation_rate` metric |
| Policy Violations | Critical | `compliance_score` component |

**Key Amazon pattern adopted**: Binary gate metrics -- if ODR > 1%, seller is flagged regardless of other scores. TradeHub implements this via `critical_threshold` on `KPI Template Item`.

### 2.2 eBay Seller Performance Standards

eBay evaluates sellers on a **periodic basis** (monthly/quarterly) with tiered outcomes:

| eBay Tier | TradeHub Tier | Key Criteria |
|-----------|--------------|-------------|
| Below Standard | Bronze | Defect rate > 2%, late shipment > 10% |
| Standard | Silver | Meets minimum on all metrics |
| Above Standard | Gold/Platinum | Exceeds expectations on most metrics |
| Top Rated | Top Rated | Exceptional performance across all metrics |

**Key eBay pattern adopted**: Periodic evaluation with automatic tier changes. TradeHub uses the `Seller Score.score_type` options (Daily/Weekly/Monthly/Quarterly) and the daily `tier_tasks` scheduled job from `119.md`.

### 2.3 Alibaba Trade Assurance Score

Alibaba scores sellers for **trade assurance** eligibility with multiple dimensions:

| Dimension | Alibaba Approach | TradeHub Component Score |
|-----------|-----------------|------------------------|
| Transaction history | GMV-based levels | `engagement_score` (via period_sales_amount, repeat_customer_rate) |
| Response rate | > 90% target | `service_score` (via avg_response_time_hours) |
| Dispute resolution | Resolution rate tracking | `quality_score` (via complaint_rate, dispute resolution) |
| Delivery performance | On-time shipping | `delivery_score` (via on_time_delivery_rate, late_shipment_rate) |

### 2.4 Buyer Scoring Reference (Unique to TradeHub)

Most marketplaces do not publicly score buyers. TradeHub adopts a **seller-visible buyer reliability score** inspired by:

| Source | Feature | TradeHub Adoption |
|--------|---------|-------------------|
| Alibaba | Buyer credit levels | `Buyer Level` (New-Premium) based on combined spend + frequency |
| eBay | Buyer feedback score | `buyer_score` on `Buyer Profile` (0-100 composite) |
| B2B platforms | Payment reliability score | `payment_on_time_rate` as a primary buyer metric |
| Internal analytics | Customer Lifetime Value | Weighted combination of `total_spent`, `total_orders`, `average_order_value` |

## 4. Business Logic & Formulas

### 4.1 Score Calculation Pipeline

The scoring pipeline runs in a fixed sequence for each entity (seller or buyer):

```
Step 1: COLLECT        Gather raw metrics from data sources
         |
Step 2: NORMALIZE      Convert raw values to 0-100 scale per metric
         |
Step 3: APPLY WEIGHTS  Multiply normalized scores by metric weights
         |
Step 4: AGGREGATE      Sum weighted scores to get raw overall score
         |
Step 5: APPLY CURVE    Apply scoring curve (Linear/Bell/Exponential/Step)
         |
Step 6: PENALTIES      Subtract penalty deductions
         |
Step 7: BONUSES        Add bonus points (capped at max_score)
         |
Step 8: FINALIZE       Clamp to 0-100, set trend, update tier
```

### 4.2 Metric Normalization Formulas

Each metric value is normalized to a 0-100 score based on its `scoring_method` and `threshold_type`:

#### Linear Scoring (Higher is Better)

```python
def linear_higher_is_better(value, target, critical):
    """
    value: actual metric value
    target: target_value from KPI Template Item
    critical: critical_threshold from KPI Template Item
    """
    if value >= target:
        return 100.0
    if value <= critical:
        return 0.0
    return ((value - critical) / (target - critical)) * 100.0
```

Example: `on_time_delivery_rate` = 96%, target = 98%, critical = 90%
- Score = ((96 - 90) / (98 - 90)) * 100 = 75.0

#### Linear Scoring (Lower is Better, with invert_score = 1)

```python
def linear_lower_is_better(value, target, critical):
    """
    For metrics like ODR, return_rate where lower is better.
    target = 0 (ideal), critical = maximum acceptable.
    """
    if value <= target:
        return 100.0
    if value >= critical:
        return 0.0
    return ((critical - value) / (critical - target)) * 100.0
```

Example: `order_defect_rate` = 0.5%, target = 0%, critical = 2%
- Score = ((2.0 - 0.5) / (2.0 - 0.0)) * 100 = 75.0

#### Logarithmic Scoring

```python
import math

def logarithmic_score(value, target, critical, higher_is_better=True):
    """
    Logarithmic curve: rapid initial gains, diminishing returns near target.
    Used for metrics like total_spend, order_frequency, account_age.
    """
    if higher_is_better:
        if value <= critical:
            return 0.0
        if value >= target:
            return 100.0
        normalized = (value - critical) / (target - critical)
    else:
        if value >= critical:
            return 0.0
        if value <= target:
            return 100.0
        normalized = (critical - value) / (critical - target)
    return min(100.0, math.log(1 + normalized * 99) / math.log(100) * 100)
```

#### Step Scoring

```python
def step_score(value, steps):
    """
    Steps: [(threshold, score), ...] sorted by threshold ascending.
    Example: [(0, 0), (25, 25), (50, 50), (75, 75), (100, 100)]
    """
    result = 0
    for threshold, score in steps:
        if value >= threshold:
            result = score
        else:
            break
    return result
```

#### Binary Scoring

```python
def binary_score(value, target, higher_is_better=True):
    """Pass/fail: 100 if meets target, 0 otherwise."""
    if higher_is_better:
        return 100.0 if value >= target else 0.0
    else:
        return 100.0 if value <= target else 0.0
```

### 4.3 Overall Score Calculation

```python
def calculate_overall_score(metric_scores, template):
    """
    metric_scores: dict of {kpi_code: normalized_score}
    template: KPI Template with items
    """
    total_weight = 0
    weighted_sum = 0

    for item in template.items:
        if not item.is_active:
            continue
        kpi_code = item.kpi_code
        weight = item.weight
        score = metric_scores.get(kpi_code, 0)

        weighted_sum += score * weight
        total_weight += weight

    if total_weight == 0:
        return 0.0

    raw_score = weighted_sum / total_weight

    # Apply scoring curve
    if template.scoring_curve == "Bell Curve":
        raw_score = apply_bell_curve(raw_score)
    elif template.scoring_curve == "Exponential":
        raw_score = apply_exponential_curve(raw_score)
    elif template.scoring_curve == "Step":
        raw_score = apply_step_curve(raw_score)
    # Linear: no transformation needed

    # Normalize to 0-100 if enabled
    if template.normalize_scores:
        raw_score = max(0, min(100, raw_score))

    return round(raw_score, 2)
```

For sellers, the scoring system also populates the 6 component scores on `Seller Score`:

```python
    Full seller score calculation pipeline.
    """
    # Step 1: Collect raw metrics from Seller Metrics
    metrics = frappe.get_doc("Seller Metrics", {"seller": seller})

        "ODR": metrics.order_defect_rate,

        "CANCEL_RATE": metrics.cancellation_rate,
        "AVG_RATING": metrics.avg_rating,
        "RESPONSE_TIME": metrics.avg_response_time_hours,
        "COMPLAINT_RATE": metrics.complaint_rate,
        "REPEAT_RATE": metrics.repeat_customer_rate,
        "POS_FEEDBACK": metrics.positive_feedback_pct
    }

    # Step 2: Normalize each metric
    normalized = {}
    for item in template.items:
        if not item.is_active:
            continue
        value = raw_values.get(item.kpi_code, 0)
        normalized[item.kpi_code] = normalize_metric(
            value, item.scoring_method, item.threshold_type,
            item.target_value, item.warning_threshold,
            item.critical_threshold, item.invert_score
        )

    # Step 3-4: Calculate weighted overall score
    overall = calculate_overall_score(normalized, template)

    # Step 5: Calculate component scores (averages of mapped metrics)
    components = {
        "fulfillment_score": avg_scores(normalized, ["ODR", "CANCEL_RATE"]),
        "delivery_score": avg_scores(normalized, ["ON_TIME_DELIVERY", "LSR"]),
        "quality_score": avg_scores(normalized, ["RETURN_RATE", "AVG_RATING"]),

        "compliance_score": avg_scores(normalized, ["COMPLAINT_RATE"]),
        "engagement_score": avg_scores(normalized, ["REPEAT_RATE", "POS_FEEDBACK"])
    }

    # Step 6-7: Apply penalties and bonuses
    penalties = calculate_penalties(seller, period)
    bonuses = calculate_bonuses(seller, period)

    return final_score, components
```

### 4.6 Penalty Calculation

Penalties are deductions applied after the weighted score calculation:

| Penalty Type | Points Deducted | Source Field | Condition |
|-------------|----------------|-------------|-----------|
| Policy violation | 5 per violation | `Seller Score.policy_violations` | Any policy violation in period |
| Late shipment spike | 3 per occurrence above threshold | `Seller Score.late_shipments` | `late_shipments > 5` in period |
| Seller-cancelled order | 2 per cancellation | `Seller Score.cancelled_orders` | Seller-initiated cancellations |
| Dispute loss | 4 per lost dispute | `Seller Score.disputed_orders` | Disputes resolved against seller |

```python
def calculate_penalties(seller, period):
    penalty_points = 0
    penalty_points += policy_violations * 5
    penalty_points += max(0, late_shipments - 5) * 3
    penalty_points += cancelled_orders * 2
    penalty_points += lost_disputes * 4
    return min(penalty_points, 30)  # Cap penalty deduction at 30 points
```

### 4.7 Bonus Calculation

Bonuses reward exceptional performance:

| Bonus Type | Points Added | Condition |
|-----------|-------------|-----------|
| Perfect rating streak | 5 | `avg_rating >= 4.8` for 3+ consecutive periods |
| Zero defect period | 3 | `order_defect_rate = 0` for the period |
| High engagement | 2 | `repeat_customer_rate > 40%` |

```python
def calculate_bonuses(seller, period):
```

### 4.8 Tier Auto-Assignment Logic

After score calculation, the system evaluates tier eligibility:

#### Seller Tier Auto-Assignment

| Tier | Level | Min Score | Additional Requirements (from `Seller Tier` DocType) |
|------|-------|-----------|-----------------------------------------------------|
| Bronze | 1 | 0 | `min_total_orders >= 0`, `min_overall_score >= 0` |
| Silver | 2 | 40 | `min_total_orders >= 50`, `min_overall_score >= 40` |
| Gold | 3 | 60 | `min_total_orders >= 200`, `min_overall_score >= 60`, `min_avg_rating >= 4.0` |
| Platinum | 4 | 75 | `min_total_orders >= 500`, `min_overall_score >= 75`, `min_avg_rating >= 4.5` |
| Top Rated | 5 | 90 | `min_total_orders >= 1000`, `min_overall_score >= 90`, `min_avg_rating >= 4.8`, `max_return_rate <= 2%` |

```python
def assign_seller_tier(seller, score):
    """
    Assigns the highest qualifying tier to the seller.
    Uses grace_period_days from Seller Tier for demotion protection.
    """
    tiers = frappe.get_all("Seller Tier",
        fields=["name", "tier_level", "min_overall_score", "min_total_orders",
                "min_avg_rating", "max_return_rate", "grace_period_days"],
        order_by="tier_level DESC"
    )
    metrics = frappe.get_doc("Seller Metrics", {"seller": seller})
    current_tier = frappe.db.get_value("Seller Profile", seller, "seller_tier")

    for tier in tiers:
        if (score >= tier.min_overall_score
            and metrics.total_orders >= tier.min_total_orders
            and metrics.avg_rating >= (tier.min_avg_rating or 0)
            and metrics.return_rate <= (tier.max_return_rate or 100)):

            if tier.name == current_tier:
                return  # No change
            elif tier.tier_level > get_tier_level(current_tier):
                # Promotion - immediate
                update_seller_tier(seller, tier.name)
                send_tier_notification(seller, "promotion", tier.name)
                return
            else:
                # Demotion - check grace period
                if is_within_grace_period(seller, tier.grace_period_days):
                    return  # Still within grace
                update_seller_tier(seller, tier.name)
                send_tier_notification(seller, "demotion", tier.name)
                return

    # Default to Bronze if no tier matches
    update_seller_tier(seller, "Bronze")
```

### 5.1 Seller Score Calculation Job

```python
def calculate_seller_scores():
    """Weekly seller score calculation job."""
    lock_key = "seller_score_calc_lock"
    if frappe.cache.get_value(lock_key):
        frappe.log_error("Seller score calculation already running", "Scheduler Lock")
        return
    frappe.cache.set_value(lock_key, 1, expires_in_sec=7200)

    try:
        template = get_default_template("Seller")
        sellers = frappe.get_all("Seller Profile",
            filters={"status": "Active"},
            pluck="name"
        )
        period = get_current_period(template.evaluation_period)

        success_count = 0
        error_count = 0

        for seller in sellers:
            try:
                can_eval, reason = check_minimum_requirements(seller, template)
                if not can_eval:
                    continue

                score, components = calculate_seller_score(seller, template, period)
                create_seller_score_record(seller, score, components, template, period)
                assign_seller_tier(seller, score)
                calculate_percentile_rank(seller, period)

                success_count += 1
                frappe.db.commit()
            except Exception:
                error_count += 1
                frappe.db.rollback()
                frappe.log_error(
                    f"Seller score calculation failed for {seller}",
                    "Seller Score Error"
                )

        frappe.logger().info(
            f"Seller score calculation complete: {success_count} success, {error_count} errors"
        )
    finally:
        frappe.cache.delete_value(lock_key)
```

## 6. System Integration Points

This section defines the six areas where scores impact platform behavior. Each integration point is configurable via the `Score Integration Rule` child table on `KPI Template`.

### 6.1 Integration Point 1: Search Ranking

**Purpose**: Higher-scoring sellers appear higher in marketplace search results.

**Mechanism**: Score contributes a weighted factor to the search ranking algorithm.

| Score Range | Search Boost Factor | Effect |
|-------------|-------------------|--------|
| 90-100 (Top Rated) | 1.30 | 30% ranking boost |
| 75-89 (Platinum) | 1.15 | 15% ranking boost |
| 60-74 (Gold) | 1.05 | 5% ranking boost |
| 40-59 (Silver) | 1.00 | No adjustment |
| 0-39 (Bronze) | 0.90 | 10% ranking penalty |

**Integration formula**:

```python
def apply_search_score_boost(base_relevance, seller_score):
    """
    Applied during search result ranking.
    base_relevance: text/category relevance score from search engine
    seller_score: overall_score from latest Seller Score
    """
    rules = get_integration_rules("Search Ranking", seller_score)
    boost_factor = 1.0
    for rule in rules:
        if rule.action_type == "Multiplier":
            boost_factor *= rule.action_value
    return base_relevance * boost_factor
```

**Admin Configuration**: Marketplace admin can adjust boost factors via `Score Integration Rule` entries on the seller KPI Template without code changes.

### 6.2 Integration Point 2: Buy Box Weight

**Purpose**: Seller score is a factor in determining which seller wins the Buy Box for a shared product listing.

**Mechanism**: The existing `buybox_rotation` hourly task (from `119.md`) already rotates Buy Box winners. Score adds a weight factor.

| Buy Box Factor | Weight | Source |
|---------------|--------|--------|
| Price competitiveness | 35% | Product price vs. competitors |
| Seller score | 25% | `Seller Score.overall_score` |
| Fulfillment speed | 20% | `Seller Metrics.on_time_delivery_rate` |
| Stock availability | 10% | Listing inventory level |
| Seller tier bonus | 10% | Tier-based multiplier |

```python
def calculate_buybox_score(seller, product):
    """
    Called by buybox_rotation task.
    Returns a weighted score used for Buy Box selection.
    """
    price_score = calculate_price_competitiveness(seller, product)
    seller_score = get_latest_seller_score(seller)
    fulfillment_score = get_fulfillment_speed_score(seller)
    stock_score = get_stock_availability_score(seller, product)
    tier_bonus = get_tier_buybox_bonus(seller)

    buybox_score = (
        price_score * 0.35 +
        seller_score * 0.25 +
        fulfillment_score * 0.20 +
        stock_score * 0.10 +
        tier_bonus * 0.10
    )
    return buybox_score
```

### 6.3 Integration Point 3: Commission Rates

**Purpose**: Higher-scoring sellers earn lower commission rates as a reward, incentivizing quality.

**Mechanism**: Commission rate discounts applied per tier/score.

| Tier | Base Commission | Discount | Effective Rate |
|------|----------------|----------|---------------|
| Bronze | 15% | 0% | 15.0% |
| Silver | 15% | 2% | 14.7% |
| Gold | 15% | 5% | 14.25% |
| Platinum | 15% | 10% | 13.5% |
| Top Rated | 15% | 15% | 12.75% |

```python
def get_effective_commission_rate(seller, base_rate):
    """
    Applied during commission calculation in payout processing.
    Uses Score Integration Rules with action_type = "Set Rate".
    """
    rules = get_integration_rules("Commission Rate", get_seller_score(seller))
    discount = 0
    for rule in rules:
        if rule.action_type == "Multiplier":
            discount = max(discount, (1 - rule.action_value) * 100)
    effective_rate = base_rate * (1 - discount / 100)
    return round(effective_rate, 2)
```

**Admin Configuration**: Commission discount percentages are set via `Score Integration Rule` entries. Admin can create tier-specific or score-range-specific discount rules.

### 6.6 Integration Point 6: Risk Management

**Purpose**: Low-scoring sellers and buyers are automatically flagged for review and may have restrictions applied.

**Mechanism**: Risk flags and automated actions based on score thresholds.

#### Seller Risk Management

| Risk Level | Score Range | Automated Actions |
|-----------|------------|-------------------|

```python
def evaluate_seller_risk(seller, score):

    Applies risk management actions based on score.
    Called after score calculation.
```
