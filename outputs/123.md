# 123 - Seller-Grouped Checkout System Plan

## Overview

This document defines the Seller-Grouped Checkout system for TradeHub B2B Marketplace. The system automatically groups cart items by seller at checkout, allowing buyers to pay for items from each seller independently or all at once. Each seller group produces a separate Marketplace Order (and its corresponding Sub Order), linked together by a shared Checkout Session ID for traceability and unified buyer experience.

**Module:** TradeHub Commerce
**Reference Platforms:** AliExpress (seller-grouped cart with independent checkout), Amazon (multi-seller cart, unified checkout), Alibaba (per-supplier order separation), eBay (seller-specific carts)

### Key Design Decisions

1. **Build on existing Cart infrastructure** - The existing `Cart` DocType already has a `seller_summary_table` (child table: `Cart Seller Summary`) and a deprecated `seller_summary` JSON field. The checkout system leverages the `Cart Seller Summary` rows as the foundation for Seller Cart Groups.
2. **Each Seller Cart Group creates one Marketplace Order** - Rather than a single master Marketplace Order with Sub Orders (the current `create_sub_orders()` pattern), the seller-grouped checkout creates separate Marketplace Orders per seller from the start. Each order functions independently for payment, fulfillment, and tracking.
3. **Shared Checkout Session ID** - All Marketplace Orders from a single checkout share a `checkout_session_id`, enabling the buyer to view them as a unified transaction and the platform to track multi-seller checkout analytics.
4. **Checkout Session DocType** - A new `Checkout Session` DocType serves as the orchestrator, tracking the overall state of a multi-seller checkout including payment status per seller group and rollback/recovery capabilities.
5. **Per-seller payment independence** - Each seller group can use a different payment method. Payment failure for one seller does not block payment for others (unless "Pay All" mode is used with a unified payment intent).

## 1. Cart Grouping Structure (Seller Cart Group)

### Concept

A **Seller Cart Group** is a logical grouping of cart items that share the same seller. During the checkout transition (Cart status: `Active` -> `Checkout`), the system reads the `Cart Seller Summary` table and the `Cart Line` items, then partitions them into discrete seller groups. Each group has independent subtotals, shipping calculations, tax computations, and payment term resolution.

### Grouping Algorithm

```
Input:  Cart with N items from M distinct sellers
Output: M Seller Cart Groups, each containing only items from one seller

Steps:
1. Load all Cart Line items from the Cart
2. For each Cart Line, read the `seller` field (Link to Seller Profile)
3. Group Cart Lines by `seller` value
4. For each group:
   a. Calculate group_subtotal = SUM(line_total) for all lines in group
   b. Calculate group_tax = SUM(tax_amount) for all lines in group
   c. Resolve shipping via seller's shipping rules (see Section 2)
   d. Resolve payment terms via Seller Payment Terms (see 117.md)
   e. Calculate group_grand_total = group_subtotal - group_discount + group_shipping + group_tax
5. Update the Cart Seller Summary table with per-group totals
6. Return the M groups to the checkout UI
```

### Data Structure

The Seller Cart Group is a **virtual/computed grouping**, not a new DocType. It is derived from existing Cart data at checkout time and represented in the API response. The existing `Cart Seller Summary` child table stores the persisted summary data.

## 2. Per-Seller Subtotal, Shipping, and Tax Calculation

### Subtotal Calculation

Per-seller subtotal is computed as the sum of all line totals for items belonging to that seller:

```
seller_subtotal = SUM(cart_line.line_total)
    WHERE cart_line.seller = <seller_id>
    AND cart_line.parent = <cart_name>
```

Each `Cart Line` item already stores `unit_price`, `qty`, `line_total`, `discount_amount`, and `tax_amount`. The seller group aggregates these.

### Shipping Calculation

Shipping is calculated **independently per seller group** because:
- Each seller may ship from a different warehouse/city
- Different sellers may use different carriers with different rate structures
- B2B shipments often have seller-specific freight agreements

**Shipping Resolution Logic:**

```
For each Seller Cart Group:
  1. Look up seller's shipping rules (Seller Profile -> linked Shipping Rules)
  2. Calculate total weight/volume for items in this group
  3. Determine destination zone from buyer's shipping address
  4. Apply the seller's applicable shipping rule:
     a. Free shipping threshold: if group_subtotal >= seller.free_shipping_threshold -> shipping = 0
     b. Flat rate: shipping = seller.flat_shipping_rate
     c. Weight-based: shipping = base_rate + (weight * per_kg_rate)
     d. Volume-based: shipping = base_rate + (volume * per_m3_rate)
     e. Carrier-specific: query carrier API for real-time rate
  5. Store result in Cart Seller Summary.shipping_amount
```

### Tax Calculation (KDV - Turkish VAT)

Tax is calculated per item based on the item's tax category, then aggregated per seller group:

```
For each Cart Line in a Seller Cart Group:
  tax_rate = item.tax_rate OR cart.default_tax_rate (default: 18% KDV)
  if cart.price_includes_tax:
    tax_amount = line_total - (line_total / (1 + tax_rate / 100))
  else:
    tax_amount = line_total * (tax_rate / 100)

seller_group_tax = SUM(cart_line.tax_amount)
    WHERE cart_line.seller = <seller_id>
```

### Grand Total Formula

```
seller_group_grand_total = seller_group_subtotal
                         - seller_group_discount
                         + seller_group_shipping
                         + (seller_group_tax if NOT price_includes_tax else 0)
```

### Calculation Trigger Points

| Event | Action |
|-------|--------|
| Item added to cart | Recalculate affected seller group totals |
| Item quantity changed | Recalculate affected seller group totals |
| Item removed from cart | Recalculate affected seller group totals; remove seller group if empty |
| Coupon code applied | Apply to eligible seller groups, recalculate |
| Shipping address changed | Recalculate shipping for all seller groups |
| Checkout initiated | Full recalculation of all seller groups with latest prices/stock |
| Payment method selected | Update `payment_method` on the specific seller group row |

## 5. Payment Flow

### 5.1 Per-Seller "Pay" Button

Each Seller Cart Group in the checkout UI displays an independent "Pay" button. When the buyer clicks "Pay" for a specific seller group:

```
Buyer clicks "Pay" for Seller Group X:

1. [Frontend] Validate:
   - Payment method is selected for this group
   - Shipping address is complete
   - Stock is still available (re-check via API)
   - Prices haven't changed since checkout started

2. [Frontend -> API] Call `initiate_seller_group_payment`:
   Input: checkout_session_id, seller (Seller Profile name), payment_method

3. [Backend] `initiate_seller_group_payment()`:
   a. Lock: Acquire lock on Checkout Session to prevent concurrent modification
   b. Validate stock availability for items in this seller group
   c. Validate prices match (compare with current listing prices)
   d. Reserve stock for this seller group (see 127.md integration)
   e. Create Payment Intent for this seller group:
      - amount = seller_group.grand_total
      - payment_method = selected method
      - metadata = { checkout_session_id, seller, items }
   f. If payment method requires redirect (card gateway):
      - Return payment gateway URL
   g. If payment method is offline (bank transfer, COD):
      - Create Marketplace Order immediately with status "Await Payment"
      - Return order confirmation
   h. Update Checkout Seller Group row: payment_status = "Processing"
   i. Release lock

4. [Payment Gateway Callback] `on_payment_complete`:
   a. Verify payment signature/hash
   b. Update Checkout Seller Group: payment_status = "Paid", paid_at = now()
   c. Create Marketplace Order (see Section 6)
   d. Update Checkout Session totals (groups_paid += 1, total_paid += amount)
   e. Check if all groups are paid -> if yes, set session status = "Fully Paid"
   f. Notify buyer: "Payment successful for {seller_name}"
   g. Notify seller: "New order received"

5. [Payment Failure]:
   a. Update Checkout Seller Group: payment_status = "Failed"
   b. Release reserved stock for this seller group
   c. Update Checkout Session: groups_failed += 1
   d. Notify buyer: "Payment failed for {seller_name}. Please retry."
   e. Buyer can retry payment for this group without affecting others
```

### 5.2 "Pay All" Button

The "Pay All" option allows the buyer to pay for all seller groups in a single transaction. This is a convenience feature that processes payments sequentially (or in parallel where the payment gateway supports batch).

```
Buyer clicks "Pay All":

1. [Frontend] Validate:
   - All seller groups have a payment method selected
   - Shared payment method check: if different methods per group, use sequential processing
   - If same payment method for all groups: may use single aggregated payment intent

2. [Frontend -> API] Call `initiate_pay_all`:
   Input: checkout_session_id, payment_methods (dict of seller -> payment_method)

3. [Backend] `initiate_pay_all()`:
   a. Acquire lock on Checkout Session
   b. Set checkout_session.payment_mode = "Pay All"
   c. For each Seller Cart Group (ordered by grand_total DESC for fail-fast on large amounts):
      i.   Validate stock and prices
      ii.  Reserve stock
      iii. Create Payment Intent
   d. Strategy selection:
      - SAME payment method for all groups:
        Create single aggregated Payment Intent with total = checkout_grand_total
        On success: mark all groups as "Paid", create all Marketplace Orders
        On failure: mark all groups as "Failed", release all stock
      - DIFFERENT payment methods per group:
        Process sequentially (group by payment method for efficiency)
        Each group payment is independent
        Partial success is possible (see Section 8.2 edge case)
   e. Release lock

4. [Sequential Processing - Different Payment Methods]:
   For each Seller Cart Group:
     a. Process payment for this group
     b. If SUCCESS:
        - Update group: payment_status = "Paid"
        - Create Marketplace Order for this group
        - Continue to next group
     c. If FAILURE:
        - Update group: payment_status = "Failed"
        - Release stock for this group
        - Mark Checkout Session: status = "Partially Failed"
        - Continue to next group (do NOT stop)
        - After all groups: notify buyer of partial failure with retry options
```

## 8. Edge Cases and Error Handling

### 8.1 Mixed Seller Cart

**Scenario:** A single product listing is sold by multiple sellers (not applicable in TradeHub's current model where each Listing has exactly one seller). However, if a buyer adds the same product type from two different sellers:

**Handling:**
- Each listing has exactly one `seller` field (Link to Seller Profile)
- Two listings for the same product from different sellers are treated as two separate Cart Line items with different `seller` values
- The grouping algorithm naturally separates them into different Seller Cart Groups
- No special handling is required

**Validation during checkout:**
```python
def validate_seller_groups(cart):
    """Ensure every item has a valid seller."""
    for item in cart.items:
        if not item.seller:
            listing = frappe.get_doc("Listing", item.listing)
            item.seller = listing.seller
        if not item.seller:
            frappe.throw(
                _("Item {0} has no seller assigned. Cannot proceed with checkout.").format(item.title)
            )
```

### 8.2 Partial Payment Failure in "Pay All"

**Scenario:** Buyer clicks "Pay All" with 3 seller groups. Payment succeeds for sellers A and B but fails for seller C.

**Handling Strategy: Commit-and-Continue (not all-or-nothing)**

Rationale: In a B2B marketplace, each seller group is an independent transaction. Reversing successful payments because an unrelated seller group's payment failed would be a poor user experience and could cause financial complications with payment gateways.

```
"Pay All" with partial failure:

1. Seller A payment: SUCCESS
   -> Create Marketplace Order for Seller A (status: Confirmed)
   -> Checkout Seller Group A: payment_status = "Paid"

2. Seller B payment: SUCCESS
   -> Create Marketplace Order for Seller B (status: Confirmed)
   -> Checkout Seller Group B: payment_status = "Paid"

3. Seller C payment: FAILURE
   -> Release reserved stock for Seller C
   -> Checkout Seller Group C: payment_status = "Failed"
   -> Checkout Session: status = "Partially Failed"

4. Post-failure actions:
   a. Notify buyer: "2 of 3 orders placed successfully. Payment failed for {Seller C}."
   b. Display retry UI for Seller C with options:
      - "Retry Payment" (same or different method)
      - "Remove from Cart" (abandon this seller group)
   c. Already-created orders for Sellers A and B proceed independently
   d. Cart remains in "Checkout" status (not "Converted") until Seller C is resolved

5. Retry scenario:
   - Buyer retries payment for Seller C with a different card
   - If success: Create Marketplace Order, Cart -> "Converted"
   - If failure again: Buyer can keep retrying or abandon

6. Timeout scenario (see 127.md for timer details):
   - If Seller C's stock reservation expires before retry:
     - Release stock, mark group as "Cancelled"
     - Checkout Session: status = "Partially Paid" (final state)
     - Cart remains in "Checkout" with the failed group removed
```

### 8.3 Seller Becomes Inactive During Checkout

**Scenario:** Between checkout initiation and payment, a seller deactivates their store or a listing becomes unavailable.

**Handling:**
```python
def validate_seller_group_before_payment(checkout_session, seller):
    """Re-validate seller and items before processing payment."""

    # Check seller is still active
    seller_status = frappe.db.get_value("Seller Profile", seller, "status")
    if seller_status != "Active":
        frappe.throw(
            _("Seller {0} is no longer active. Please remove these items from your cart.").format(seller)
        )

    # Check all items are still available
    for item in get_seller_group_items(checkout_session, seller):
        listing = frappe.get_doc("Listing", item.listing)
        if listing.status not in ["Active", "Published"]:
            frappe.throw(
                _("Item '{0}' is no longer available.").format(item.title)
            )
        if listing.available_qty < item.qty:
            frappe.throw(
                _("Insufficient stock for '{0}'. Available: {1}, Requested: {2}").format(
                    item.title, listing.available_qty, item.qty
                )
            )
```

### 8.4 Price Changes During Checkout

**Scenario:** A seller updates a listing price after the buyer initiated checkout but before payment.

**Handling:**
```
At checkout initiation:
  - Snapshot current prices in the Cart Line items
  - Store checkout_started_at timestamp

Before payment processing:
  - Compare Cart Line prices with current Listing prices
  - If any price increased:
    -> Block payment
    -> Notify buyer: "Price changed for {item}. Old: {old}, New: {new}"
    -> Buyer must acknowledge and recalculate
  - If any price decreased:
    -> Option A (recommended): Auto-apply lower price, notify buyer of savings
    -> Option B: Show notification, let buyer confirm
  - Update Cart Seller Summary totals after price adjustment
```

### 8.5 Concurrent Checkout from Same Cart

**Scenario:** Buyer opens checkout in two browser tabs simultaneously.

**Handling:**
- Only one active Checkout Session per Cart is allowed
- When `initiate_checkout()` is called:
  - Check for existing non-expired, non-cancelled Checkout Session for this Cart
  - If found: return existing session (idempotent)
  - If not found: create new session
- Cart's `session_id` field (existing) is updated to the active session
- All checkout API calls validate that the `session_id` matches the active session

### 8.6 Empty Seller Group After Item Removal

**Scenario:** During checkout, buyer removes all items from one seller group (e.g., removes the only item from Seller A).

**Handling:**
```
When item is removed during checkout:
  1. Remove Cart Line item
  2. Recalculate affected Seller Cart Group totals
  3. If seller group item_count == 0:
     a. Remove the Checkout Seller Group row
     b. Remove the Cart Seller Summary row
     c. Update Checkout Session totals (total_sellers -= 1)
     d. If total_sellers == 0:
        - Cancel Checkout Session
        - Revert Cart to "Active" status
  4. If seller group item_count > 0:
     - Recalculate group totals
     - Update Checkout Session totals
```

## 10. Integration with Existing Systems

### 10.1 Integration with 117.md (Payment Terms)

During checkout, each Seller Cart Group resolves its applicable payment terms:

```
For each Seller Cart Group:
  1. Look up Seller Payment Terms for this seller (from 117.md)
  2. Evaluate conditional rules against:
     - Order amount (seller_group.grand_total)
     - Buyer type (Individual / Organization)
     - Buyer region (from shipping address)
     - Product category (from listing categories)
  3. Apply the matching Payment Terms Template
  4. Calculate:
     - Due date (based on term_type and due_days)
     - Deposit amount (if Deposit-Balance type)
     - Installment schedule (if Installment type)
  5. Store resolved template in Checkout Seller Group.payment_terms_template
  6. Pass through to Marketplace Order on creation
```

### 10.2 Integration with Cart DocType (Existing)

The existing Cart DocType requires these changes for seller-grouped checkout:

**New fields on Cart:**

| Field Name | Field Type | Options / Details | Required | Default | Description |
|------------|-----------|-------------------|----------|---------|-------------|
| `checkout_session` | Link | options: Checkout Session; read_only: 1 | No | - | Active checkout session for this cart |

**Updated `seller_summary_table`:** Extend `Cart Seller Summary` child table with new fields as defined in Section 1.

**Status transition update:** Add validation that Cart cannot transition to `Checkout` if there is already an active (non-expired, non-cancelled) Checkout Session.

### 10.3 Integration with 127.md (Temporal Protection)

Checkout session integrates with the Cart Protection system:

- **Stock reservation:** When `initiate_checkout()` is called, stock is reserved for all items in the cart. The reservation timer starts based on `Cart Protection Settings.checkout_reservation_duration` (default: 30 minutes).
- **Checkout expiry:** `checkout_session.expires_at` is set based on the reservation duration. If the session expires before payment, stock is released.
- **Per-group reservation:** When "Pay" is clicked for a specific seller group, that group's stock reservation is refreshed. Other groups' reservations continue their countdown.
- **Payment deadline:** After a Marketplace Order is created with status "Await Payment" (offline payment methods), the payment deadline from 127.md applies.

## 12. Checkout Session Lifecycle

### Complete Flow

```
1. CART ACTIVE
   - Buyer browses and adds items from multiple sellers
   - Cart automatically maintains seller_summary_table
   - No stock reservation, no payment processing

2. INITIATE CHECKOUT
   - Buyer clicks "Proceed to Checkout"
   - API: initiate_checkout(cart_name)
   - Creates Checkout Session (status: Initiated)
   - Transitions Cart status: Active -> Checkout
   - Reserves stock for all items (127.md integration)
   - Sets checkout expiry timer
   - Returns checkout session data with seller groups

3. SELECT PAYMENT METHODS
   - Buyer selects payment method per seller group
   - API: update_seller_group_payment_method(session, seller, method)
   - Payment terms auto-resolved from 117.md
   - Checkout Session status: In Progress

4a. PAY PER SELLER
   - Buyer clicks "Pay" for each seller group individually
   - API: initiate_seller_group_payment(session, seller)
   - Each payment processed independently
   - Marketplace Order created on success
   - Other groups unaffected by one group's failure

4b. PAY ALL
   - Buyer clicks "Pay All"
   - API: initiate_pay_all(session)
   - All groups processed (single or sequential)
   - Partial failure handled per Section 8.2

5. ORDERS CREATED
   - Each successful payment produces a Marketplace Order
   - Order linked to Checkout Session via checkout_session_id
   - Sub Order created per existing create_sub_orders() logic
   - Seller notified of new order

6. CHECKOUT COMPLETE
   - All seller groups paid -> Checkout Session: Completed
   - Cart: Converted
   - Buyer redirected to order confirmation page
   - Shows all orders from this checkout session

7. POST-CHECKOUT
   - Each Marketplace Order follows independent lifecycle
   - Seller processes their order independently
   - Buyer tracks each order separately (or via session view)
   - Escrow, fulfillment, invoicing per existing system
```

## 15. Post-Implementation Steps

After implementing this plan, the following bench commands must be executed:

```bash
# Create new DocTypes and add new fields
bench --site <site_name> migrate

# Clear cache for new DocTypes to be available
bench --site <site_name> clear-cache

# Rebuild frontend assets (for checkout UI)
bench build

# Optional: Import default data
bench --site <site_name> execute tradehub_commerce.setup.install.create_default_checkout_settings
```

### Migration Notes

1. **Cart Seller Summary extension:** Adding new fields to an existing child table requires a `bench migrate` to add columns to the database table.
2. **Marketplace Order field addition:** Adding `checkout_session_id` to existing Marketplace Orders will default to `NULL` for historical orders (which were created before seller-grouped checkout existed).
3. **Existing orders:** Historical Marketplace Orders created via the old `create_sub_orders()` pattern will not have a `checkout_session_id`. The system should handle `checkout_session_id = None` gracefully in all queries and API responses.
4. **Cart Seller Summary data:** Existing Cart Seller Summary rows will have `NULL` for the new fields. These are populated only when a cart enters the checkout flow after migration.

## 16. Glossary

| Term | Definition |
|------|-----------|
| Seller Cart Group | Logical grouping of cart items belonging to the same seller, used during checkout for independent payment processing |
| Checkout Session | DocType that orchestrates a multi-seller checkout, linking the source cart to all resulting Marketplace Orders |
| Checkout Session ID | Unique identifier (e.g., CS-2026-00015) shared across all Marketplace Orders created from the same buyer checkout |
| Pay All | Checkout mode where the buyer pays for all seller groups in a single action (may process sequentially if different payment methods) |
| Per-Seller Pay | Checkout mode where the buyer pays for each seller group independently |
| Commit-and-Continue | Error handling strategy where successful payments are not reversed due to unrelated failures in other seller groups |
| Stock Reservation | Temporary hold on inventory during checkout to prevent overselling (see 127.md) |
| Seller Cart Group Grand Total | Calculated as: subtotal - discount + shipping + tax (if not price-inclusive) for one seller's items |
