## 1. Overview

### 1.1 Problem Statement

When a B2B marketplace order is fulfilled through multiple partial shipments, each shipment requires its own packing slip document that:

- Shows only the items and quantities included **in that specific shipment**, not the full order.
- Provides clear context about the overall order: order reference, total ordered quantities, previously shipped quantities, and remaining balance.
- Includes shipping and billing addresses, company information, and shipment date for warehouse and logistics use.
- Supports both PDF download (for digital records, email attachments) and direct printing (for warehouse staff inserting slips into packages).

Without a purpose-built packing slip, sellers must manually create packing documentation or rely on the generic Delivery Note print, which lacks partial shipment context (e.g., "Shipment 2 of 3") and order-level balance information.

### 1.2 Scope

This document defines:

1. **Print button placement** - "Print Packing Slip" button on each Delivery Note record
2. **Per-shipment quantity display** - Showing only this shipment's items and quantities with balance context
3. **PDF/Print support** - PDF download and direct printer send using Frappe's built-in PDF generation
4. **Jinja2 Print Format template design** - Complete template with order reference, shipment quantities, dates, addresses, company info
5. **Multi-item partial shipment handling** - Correct display when some items are fully shipped, others partially shipped, and some not included in this shipment
6. **Print Format registration** - How to register the custom Print Format in Frappe

## 2. Print Button Placement

### 2.1 Button Location

The "Print Packing Slip" button is placed on the **Delivery Note form view** in ERPNext, visible only when the Delivery Note is linked to a Marketplace Order.

#### 2.1.1 Button Visibility Conditions

| Condition | Required | Reason |
|-----------|----------|--------|
| `doc.docstatus >= 0` | Yes | Button available in Draft (preview) and Submitted states |
| `doc.custom_marketplace_order` | Yes | Only show for marketplace-linked Delivery Notes |
| `doc.items.length > 0` | Yes | At least one item must exist |
| User has `read` permission on Delivery Note | Yes | Standard Frappe permission check |

#### 2.1.2 Client-Side JS Implementation

The button is added in the Delivery Note's client script (custom script or app override):

```javascript
// File: tradehub_commerce/public/js/delivery_note_packing_slip.js
// Loaded via hooks.py: doctype_js = {"Delivery Note": "public/js/delivery_note_packing_slip.js"}

frappe.ui.form.on("Delivery Note", {
    refresh(frm) {
        // Only show for marketplace-linked Delivery Notes
        if (!frm.doc.custom_marketplace_order) return;

        // --- Print Packing Slip Button ---
        frm.add_custom_button(
            __("Packing Slip"),
            function () {
                // Open Print Format in new window
                const print_url = frappe.urllib.get_full_url(
                    "/api/method/frappe.utils.print_format.download_pdf"
                    + "?doctype=Delivery Note"
                    + "&name=" + encodeURIComponent(frm.doc.name)
                    + "&format=Marketplace Packing Slip"
                    + "&no_letterhead=0"
                    + "&letterhead=" + encodeURIComponent(frappe.boot.default_print_letterhead || "")
                    + "&settings={}"
                );

                // Open in new tab for preview / download
                window.open(print_url, "_blank");
            },
            __("Print")
        );

        // --- Direct Print Button (sends to default printer) ---
        frm.add_custom_button(
            __("Print Packing Slip (Direct)"),
            function () {
                frm.print_doc("Marketplace Packing Slip");
            },
            __("Print")
        );
    }
});
```

## 3. Per-Shipment Quantity Display

### 3.1 Display Philosophy

The packing slip must clearly answer these questions for the warehouse team:

1. **What is in THIS box?** - Items and quantities being shipped now
2. **What was the full order?** - Total ordered quantity per item for context
3. **What was already sent?** - Previously shipped quantities
4. **What remains after this?** - Balance remaining for future shipments

### 3.2 Quantity Data Sources

For each item row on the Delivery Note, the packing slip displays:

| Column | Source Field | Description |
|--------|-------------|-------------|
| SKU | `Delivery Note Item.custom_sku` | Product SKU from marketplace listing |
| Item | `Delivery Note Item.item_name` | Item display name |
| This Shipment | `Delivery Note Item.qty` | Quantity included in THIS Delivery Note |
| Total Ordered | `Delivery Note Item.custom_ordered_qty` | Original order quantity |
| Previously Shipped | `Delivery Note Item.custom_previously_shipped_qty` | Qty shipped in prior Delivery Notes |
| Remaining After This | `Delivery Note Item.custom_remaining_after_this` | Balance remaining: `custom_ordered_qty - custom_previously_shipped_qty - qty` |
| UOM | `Delivery Note Item.uom` | Unit of measure |

### 3.3 Example: Multi-Shipment Scenario

**Order MO-2026-00042:**
- Item A (Industrial Valve): 1,000 units ordered
- Item B (Gasket Set): 500 units ordered
- Item C (Pipe Fitting): 200 units ordered

#### Shipment 1 Packing Slip (DN-00089)

```
+------+------------------+----------+---------+----------+-----------+-----+
| SKU  | Item             | This     | Total   | Previous | Remaining | UOM |
|      |                  | Shipment | Ordered | Shipped  | After     |     |
+------+------------------+----------+---------+----------+-----------+-----+
| VLV1 | Industrial Valve |      600 |   1,000 |        0 |       400 | Pcs |
| GSK1 | Gasket Set       |      500 |     500 |        0 |         0 | Pcs |
+------+------------------+----------+---------+----------+-----------+-----+
```

- Item B shows "Remaining: 0" indicating it is **fully shipped** in this first shipment.
- Item C is **not included** in this shipment (not listed at all).

#### Shipment 2 Packing Slip (DN-00094)

```
+------+------------------+----------+---------+----------+-----------+-----+
| SKU  | Item             | This     | Total   | Previous | Remaining | UOM |
|      |                  | Shipment | Ordered | Shipped  | After     |     |
+------+------------------+----------+---------+----------+-----------+-----+
| VLV1 | Industrial Valve |      400 |   1,000 |      600 |         0 | Pcs |
```

- Item C is partially shipped (100 of 200).

- Final shipment with the remaining 100 pipe fittings.

### 3.4 Visual Indicators

The packing slip template uses visual cues for shipment status:

| Condition | Visual Indicator |
|-----------|-----------------|
| Remaining = 0 (fully shipped with this shipment) | Green background or checkmark icon |
| Remaining > 0 (partial, more to come) | Amber text or "Partial" label |
| Previously Shipped > 0 (follow-up shipment) | Shows prior shipped qty in lighter text |
| Single-shipment order (not partial) | Balance columns hidden for simplicity |

## 4. PDF/Print Support

### 4.1 Frappe PDF Generation Architecture

Frappe provides built-in PDF generation through `wkhtmltopdf` (or `weasyprint` in newer versions). The packing slip leverages this infrastructure without custom PDF libraries.

#### 4.1.1 PDF Generation Flow

```
User clicks "Print > Packing Slip"
        |
        v
Browser requests: GET /api/method/frappe.utils.print_format.download_pdf
    ?doctype=Delivery Note
    &name=DN-00089
    &format=Marketplace Packing Slip
        |
        v
Frappe Server:
    1. Load Delivery Note doc (DN-00089)
    2. Find "Marketplace Packing Slip" Print Format
    3. Render Jinja2 template with doc context
    4. Apply letterhead (if configured)
    5. Convert HTML to PDF via wkhtmltopdf
    6. Return PDF binary with Content-Disposition: attachment
        |
        v
Browser: Opens PDF in new tab / downloads
```

#### 4.1.2 Key Frappe APIs Used

| API | Purpose | Usage |
|-----|---------|-------|
| `frappe.utils.print_format.download_pdf()` | Generate and download PDF | Called from JS button click |
| `frappe.utils.pdf.get_pdf(html, options)` | Convert HTML string to PDF bytes | Used internally by download_pdf |
| `frappe.get_print(doctype, name, print_format)` | Render Print Format to HTML string | Used for email attachments |
| `frappe.attach_print(doctype, name, print_format)` | Attach PDF to document / email | Used when emailing packing slip |
| `frm.print_doc(print_format)` | Open print dialog with selected format | Called from "Direct Print" button |
| `frappe.utils.print_format.report_to_pdf(html)` | Alternative PDF generation | Fallback for complex layouts |

### 4.3 Direct Printer Support

#### 4.3.1 Browser Print Dialog

The "Print Packing Slip (Direct)" button uses Frappe's built-in `frm.print_doc()` which:

1. Opens the Frappe print preview modal
2. Pre-selects "Marketplace Packing Slip" as the Print Format

For warehouse environments with network printers, the standard browser print dialog handles printer selection. Frappe does not provide native network printer integration; this relies on:

### 4.4 Email Integration

The packing slip PDF can be attached to email notifications sent to the buyer when a shipment is dispatched:

```python
def send_shipment_notification(delivery_note_name):
    """
    Send shipment notification email with packing slip PDF attached.

    Called from on_delivery_note_submit() (defined in 126.md Section 4.3.1).
    """
    dn = frappe.get_doc("Delivery Note", delivery_note_name)

    if not dn.custom_marketplace_order:
        return

    order = frappe.get_doc("Marketplace Order", dn.custom_marketplace_order)

    # Generate packing slip PDF
    pdf_content = frappe.get_print(
        "Delivery Note",
        delivery_note_name,
        "Marketplace Packing Slip",
        as_pdf=True
    )

    # Compose email
    frappe.sendmail(
        recipients=[order.buyer_email],
        subject=_("Shipment {0} of {1} for Order {2}").format(
            dn.custom_shipment_number,
            dn.custom_total_shipments,
            order.order_id
        ),
        message=_("Your order {0} has been partially shipped. Please find the packing slip attached.").format(
            order.order_id
        ),
        attachments=[{
            "fname": "Packing-Slip-{0}-Shipment-{1}.pdf".format(
                order.order_id,
                dn.custom_shipment_number
            ),
            "fcontent": pdf_content
        }]
    )
```

### 4.5 Batch PDF Generation

For sellers shipping multiple orders simultaneously, a batch PDF generation utility combines multiple packing slips into a single downloadable PDF:

```python
@frappe.whitelist()
def download_batch_packing_slips(delivery_notes):
    """
    Generate a combined PDF containing packing slips for multiple Delivery Notes.

    Args:
        delivery_notes (str): JSON-encoded list of Delivery Note names
                              e.g., '["DN-00089", "DN-00094", "DN-00101"]'

    Returns:
        PDF file download response with all packing slips concatenated.
    """
    import json
    dn_list = json.loads(delivery_notes)

    if not dn_list:
        frappe.throw(_("No Delivery Notes specified"))

    if len(dn_list) > 50:
        frappe.throw(_("Maximum 50 packing slips per batch"))

    combined_html = ""

    for dn_name in dn_list:
        dn = frappe.get_doc("Delivery Note", dn_name)

        # Permission check
        if not frappe.has_permission("Delivery Note", "read", dn):
            frappe.throw(_("Not permitted to access {0}").format(dn_name))

        # Verify marketplace link
        if not dn.custom_marketplace_order:
            continue

        # Render Print Format to HTML
        html = frappe.get_print(
            "Delivery Note",
            dn_name,
            "Marketplace Packing Slip"
        )

        # Add page break between slips (except last)
        combined_html += html
        combined_html += '<div style="page-break-after: always;"></div>'

    # Remove trailing page break
    if combined_html.endswith('<div style="page-break-after: always;"></div>'):
        combined_html = combined_html[:-len('<div style="page-break-after: always;"></div>')]

    # Generate combined PDF
    pdf = frappe.utils.pdf.get_pdf(combined_html)

    # Return as download
    frappe.local.response.filename = "Packing-Slips-Batch-{0}.pdf".format(
        frappe.utils.now_datetime().strftime("%Y%m%d-%H%M%S")
    )
    frappe.local.response.filecontent = pdf
    frappe.local.response.type = "download"
```

## 6. Multi-Item Partial Shipment Handling

### 6.1 Scenarios

The packing slip template must correctly handle all combinations of item fulfillment states within a single Delivery Note:

| Scenario | Description | Template Behavior |
|----------|-------------|-------------------|
| **All items fully shipped** | Every item's remaining = 0 | All items show "Done" status |
| **All items partially shipped** | Every item has remaining > 0 | All items show "Partial" status |
| **Mixed: some full, some partial** | Some items remaining = 0, others > 0 | Per-item status indicators (Done/Partial) |
| **Items not in this shipment** | Ordered items with qty=0 in this DN | Not listed (only items with qty > 0 appear in DN) |
| **Single-item order** | Only one item in the Delivery Note | Table displays with one row; balance columns shown only if partial |
| **First shipment (not yet partial)** | First DN, no previous shipments | `custom_is_partial_shipment = 0`, balance columns hidden. If subsequent shipments are created, the first DN is retroactively marked as partial (per 126.md Section 4.3.4). |
| **Different UOMs** | Items with different units (Pcs, Kg, Liters) | UOM column shows each item's unit; totals row sums qty but notes "mixed UOM" |

### 6.2 Item Filtering Logic

The Delivery Note only contains items being shipped in THIS delivery. Items not included in this particular shipment are simply absent from the Delivery Note's items child table. The template iterates only over `doc.items`, so excluded items are automatically omitted.

```
Order MO-2026-00042:
  - Item A: 1,000 units  -> DN-00089: 600 units, DN-00094: 400 units
  - Item B: 500 units    -> DN-00089: 500 units (fully in first shipment)
  - Item C: 200 units    -> DN-00094: 100 units, DN-00101: 100 units

DN-00089 items table contains ONLY Item A (600) and Item B (500).
Item C does not appear because it is not being shipped in this DN.
```

### 6.3 Handling Mixed UOMs in Totals

When items have different units of measure, the totals footer handles this gracefully:

```html
{# In the tfoot section, check for mixed UOMs #}
{%- set uom_set = doc.items | map(attribute='uom') | unique | list -%}
{%- if uom_set | length > 1 -%}
    {# Mixed UOMs - show total count only, note mixed units #}
    <td class="qty-cell" style="font-weight: bold;">
        {{ ns.total_this_shipment }} <span style="font-size: 7pt;">(mixed units)</span>
    </td>
{%- else -%}
    {# Single UOM - show total with unit #}
    <td class="qty-cell" style="font-weight: bold;">
        {{ ns.total_this_shipment }} {{ uom_set[0] if uom_set else "" }}
    </td>
{%- endif -%}
```

### 6.4 Balance Column Visibility

The balance columns (Total Ordered, Previously Shipped, Remaining) are conditionally displayed:

| Condition | Balance Columns | Reason |
|-----------|----------------|--------|
| `custom_is_partial_shipment = 1` | Shown | Partial shipment needs balance context |
| `custom_is_partial_shipment = 0` | Hidden | Single shipment - balance info is redundant |

This is handled in the template by the `{% if is_partial %}` conditional around the extra `<th>` and `<td>` elements in the items table.

### 6.5 Permission and Data Security

The Print Format runs server-side with the permissions of the requesting user. Key security considerations:

| Concern | Mitigation |
|---------|-----------|
| **Seller viewing other seller's packing slip** | Frappe's standard `has_permission` check on Delivery Note read. The `custom_sub_order` link and Sub Order's seller field enable permission filtering. |
| **Buyer viewing packing slip** | Buyers can view packing slips for their own orders. The Marketplace Order's buyer field validates ownership. |
| **Price information on packing slip** | The template intentionally omits `rate` and `amount` columns. Packing slips are operational documents, not commercial invoices. |
| **Cross-tenant data leak** | Frappe's standard document permission framework prevents cross-tenant access in multi-tenant setups (TradeHub Core tenant isolation). |

## 7. Edge Cases

### 7.1 Multi-Item Partial Shipment - Mixed Fulfillment States

**Scenario:** A Delivery Note contains 3 items: Item A is fully shipped (remaining = 0), Item B is partially shipped (remaining = 50), and Item C is a new item being shipped for the first time.

**Handling:**
- Each item row displays its own status indicator (Done/Partial)
- The summary section calculates overall fulfillment percentage across all ordered items (not just items in this DN)
- Items not present in this DN do not affect the displayed totals for "This Shipment"

### 7.2 Cancelled Delivery Note

**Scenario:** A submitted Delivery Note is later cancelled (docstatus = 2).

**Handling:**
- The Print Format is still accessible on cancelled Delivery Notes for record-keeping
- A visual indicator ("CANCELLED" watermark or banner) should be added if `doc.docstatus == 2`
- The balance data may be stale (since cancellation recalculates balances on subsequent DNs)

```html
{# Add cancelled watermark if applicable #}
{% if doc.docstatus == 2 %}
    <div style="position: fixed; top: 50%; left: 50%;
                transform: translate(-50%, -50%) rotate(-45deg);
                font-size: 72pt; color: rgba(255,0,0,0.1);
                font-weight: bold; z-index: 1000;
                pointer-events: none;">
        CANCELLED
    </div>
{% endif %}
```

### 7.3 Draft Delivery Note Preview

**Scenario:** User wants to preview the packing slip before submitting the Delivery Note.

**Handling:**
- The Print button is available in Draft state (`docstatus = 0`)
- A "DRAFT" watermark is displayed to distinguish from submitted documents
- The `custom_shipment_number` and other metadata fields are populated during `set_missing_values()` (126.md Section 3.4), so they are available even in draft

```html
{# Add draft watermark if applicable #}
{% if doc.docstatus == 0 %}
    <div style="position: fixed; top: 50%; left: 50%;
                transform: translate(-50%, -50%) rotate(-45deg);
                font-size: 72pt; color: rgba(0,0,255,0.08);
                font-weight: bold; z-index: 1000;
                pointer-events: none;">
        DRAFT
    </div>
{% endif %}
```

### 7.4 Non-Marketplace Delivery Note

**Scenario:** A Delivery Note that is not linked to a Marketplace Order (regular ERPNext DN).

**Handling:**
- The Print button is hidden (visibility condition requires `custom_marketplace_order`)
- If someone directly accesses the Print Format URL with a non-marketplace DN, the template degrades gracefully:
  - Order reference section shows "N/A"
  - Balance columns are hidden (no partial shipment context)
  - Basic DN data (items, quantities, addresses) still renders correctly

### 7.5 Very Large Order (Many Items)

**Scenario:** An order with 50+ line items, with a partial shipment containing 30 items.

**Handling:**
- The items table uses CSS `page-break-inside: avoid` on table rows where possible
- If the table spans multiple pages, the header row repeats on each page using `thead { display: table-header-group; }`
- The summary section always appears after the items table (on the last page)

```css
/* Multi-page table support */
.packing-slip table.items-table thead {
    display: table-header-group;
}

.packing-slip table.items-table tbody tr {
    page-break-inside: avoid;
}

.packing-slip .summary-section {
    page-break-before: auto;
    page-break-inside: avoid;
}
```

### 7.6 Missing Custom Fields (Backward Compatibility)

**Scenario:** Delivery Notes created before the marketplace custom fields were added (pre-migration).

**Handling:**
- All custom field references in the template use Jinja's `or` operator for fallback values
- `item.custom_ordered_qty or item.qty` - falls back to the DN item's own qty
- `item.custom_previously_shipped_qty or 0` - defaults to 0
- `item.custom_remaining_after_this or 0` - defaults to 0
- When all custom fields are missing, the template renders as a standard packing slip without balance context

### 7.7 Multiple Sellers in One Order (Multi-Seller)

**Scenario:** Marketplace Order with items from 3 different sellers, each creating separate Delivery Notes.

**Handling:**
- Each seller creates their own Delivery Note linked to their Sub Order
- The packing slip shows only that seller's items (the DN only contains their items)
- The `custom_sub_order` field identifies the seller scope
- Seller name is displayed in the Order Information section
- The shipment counter is per-order (not per-seller), so "Shipment 2 of 5" reflects overall order progress

### 7.8 Returned Items After Partial Shipment

**Scenario:** Shipment 1 (600 units) is delivered, buyer returns 100 units, then Shipment 2 (400 units) is created.

**Handling:**
- Returns do not affect the packing slip for Shipment 2
- The packing slip shows shipping quantities only, not return adjustments
- `custom_previously_shipped_qty` still shows 600 (gross shipped, not net of returns)
- Return quantities are tracked separately on the Marketplace Order Item (`returned_qty` field)
- A note may be added to the packing slip if returns are pending: "Note: 100 units returned from Shipment 1"

## 11. Summary

This document defines the complete plan for the Partial Shipment Packing Slip Print Format, the final piece in the partial shipment workflow chain:

```
125.md (Create Delivery Note) -> 126.md (Track Balances) -> 148.md (Print Packing Slip)
```

**Key design decisions:**

1. **Frappe-native approach** - Uses Frappe's built-in Print Format engine (Jinja2 + wkhtmltopdf) rather than custom PDF libraries, ensuring compatibility with Frappe upgrades and the existing print infrastructure.

2. **Per-shipment focus** - The packing slip shows only what is in THIS box, with optional balance context for partial shipments. Warehouse staff can immediately see what to pack.

3. **Conditional complexity** - Balance columns are hidden for single-shipment orders (clean, simple output) and only appear when `custom_is_partial_shipment` is flagged (detailed context for multi-shipment fulfillment).

4. **No pricing information** - The packing slip intentionally omits rates and amounts. It is an operational document for warehouse use, not a commercial invoice.

5. **Seller panel integration** - The same Print Format serves both admin and seller panel users, with Frappe's permission system controlling access.

6. **Data from 126.md** - All balance-related data (`custom_ordered_qty`, `custom_previously_shipped_qty`, `custom_remaining_after_this`) comes from the Delivery Note custom fields defined in 126.md, which are populated during `make_delivery_note()`.
