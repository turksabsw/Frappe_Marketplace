# Task 130 - Dynamic User Segmentation Plan

## 1. Overview

This document defines the **Dynamic User Segmentation System** for the TradeHub marketplace platform. The system enables platform administrators to create rule-based, automatically maintained user segments for both buyers and sellers. It provides:

- **User Segment DocType**: A master document defining a named segment with target type (Buyer/Seller/Both), status (Active/Inactive), description, and ownership metadata.
- **Segment Rule Group child DocType**: A child table defining logical groups of conditions with group-level AND/OR logic, following the same nested group pattern used by the existing `Seller Tag Rule` DocType.
- **Segment Rule Condition child DocType**: A child table defining individual conditions within a group, specifying data source, field name, comparison operator, value, and row-level AND/OR logic.
- **Segment Member DocType**: A link table tracking which users belong to which segments, with join/leave timestamps for audit.
- **Five data source categories**: Profile, Transactions, Behavior, Product, Financial -- each mapping to specific fields from existing DocTypes.
- **Score-based segmentation**: Integration with the scoring system defined in `132.md`, allowing `buyer_score`, `overall_score`, `buyer_level`, `seller_tier`, and `score_trend` to be used as segment criteria.
- **Live member count preview**: Real-time count displayed when saving a segment definition.
- **Scheduled membership refresh**: Periodic job to re-evaluate all active segments and update membership.

### Key Design Decisions

1. **Admin-configurable, not hardcoded**: All segments are defined by administrators via the UI. No segment definitions exist in code. This follows the spec requirement for dynamic, rule-based segmentation.
2. **Child table conditions (not JSON)**: Unlike `Seller Tag Rule` which stores conditions as a JSON blob in `rule_json`, this system uses structured child table rows for conditions. This enables Frappe's built-in form UI, validation, filtering, and reporting on individual conditions -- making the rule builder more accessible to non-technical admins.
3. **Grouped conditions with AND/OR**: Conditions are organized into logical groups (via `Segment Rule Group`), with AND/OR logic both within groups and between groups. This provides the same expressive power as the `Seller Tag Rule` JSON structure while using Frappe's native child table pattern.
4. **Separate Segment Member table**: Membership is materialized in a `Segment Member` DocType rather than computed on-the-fly. This enables fast lookups ("which segments does user X belong to?") and historical tracking.
5. **Coexistence with Seller Tags**: Segments are complementary to tags. Tags are badge-like labels (achievement, certification, warning) assigned via `Seller Tag Rule`. Segments are dynamic groupings used for bulk operations, campaigns, and analytics.

## 2. Reference Models

### 2.1 Amazon - Customer Segmentation

| Aspect | Amazon Model | TradeHub Adaptation |
|--------|-------------|-------------------|
| Segmentation Type | Behavioral + demographic (internal) | Admin-defined rules combining profile, transaction, behavior, product, financial data |
| Segment Criteria | Purchase history, browsing, returns, Prime status | Order count, revenue, return rate, payment behavior, level/grade |
| Dynamic Updates | Near real-time based on activity | Scheduled refresh (configurable: hourly/daily/weekly) with on-save preview |
| Usage | Personalized recommendations, marketing, pricing | Campaigns, commission tiers, communication, analytics, risk grouping |
| Visibility | Internal only (not visible to customers) | Admin-only; segments can drive automated actions but are not shown to users |

### 2.2 eBay - Buyer/Seller Categorization

| Aspect | eBay Model | TradeHub Adaptation |
|--------|-----------|-------------------|
| Buyer Groups | Power Buyers, Occasional, New | Admin-defined via rule conditions (e.g., order_count > 50 AND total_spent > 100000) |
| Seller Groups | Top Rated, Above Standard, Standard, Below Standard | Seller tier-based segments (seller_tier = Top Rated) or score-based (overall_score >= 80) |
| Rule Logic | Internal criteria per group | Exposed AND/OR rule builder for admin configuration |
| Bidirectional | Separate buyer and seller categorization | `target_type` field: Buyer, Seller, or Both |

### 2.3 Alibaba - Trade Assurance Segmentation

| Aspect | Alibaba Model | TradeHub Adaptation |
|--------|--------------|-------------------|
| Supplier Tiers | Gold Supplier, Verified, Assessed, Trade Assurance | Segment rules based on verification_status, seller_tier, overall_score |
| Geographic Segmentation | By country/region/city | Profile data source with city, region fields |
| Transaction Tiers | By annual GMV volume | Transaction data source with total_sales_amount, order_count |
| Verification-Based | Verification status as primary segment driver | `verification_status` available as profile data source field |

### 2.4 B2B Marketplace Segmentation Patterns

| Pattern | Description | TradeHub Implementation |
|---------|-------------|----------------------|
| RFM Segmentation | Recency, Frequency, Monetary value | Behavior (last_login, last_order_date) + Transactions (order_count, total_spent) |
| Lifecycle Stages | Onboarding, Active, At-Risk, Churned | Behavior (days_since_last_order, activity_score) + Financial (score_trend) |
| Value Tiers | High/Medium/Low value customers | Transactions (total_spent) + Scoring (buyer_score, buyer_level) |
| Engagement Cohorts | Highly engaged, Passive, Dormant | Behavior (last_login, activity_score) + Transactions (order_frequency) |

### 3.2 Segment Rule Group Child DocType

**App**: `tradehub_core`
**Module**: `TradeHub Core`
**Is Table**: Yes (child table of `User Segment`, parentfield: `rule_groups`)
**Naming Rule**: Random
**Track Changes**: No

This child table defines **logical groups** of conditions. Each group contains one or more conditions (via `Segment Rule Condition` child table) and specifies the logic for combining conditions within the group.

**Design Note**: The two-level hierarchy (Groups + Conditions) mirrors the nested group pattern from `Seller Tag Rule.rule_json`:

```
Seller Tag Rule JSON structure:
{
  "logic": "AND",           -->  User Segment.group_logic
  "groups": [               -->  Segment Rule Group (child rows)
    {
      "group_logic": "AND", -->  Segment Rule Group.condition_logic
      "conditions": [...]   -->  Segment Rule Condition (child rows)
    }
  ]
}
```

#### Field Definitions

| Field Name | Field Type | Label | Options | Required | Default | Description |
|------------|-----------|-------|---------|----------|---------|-------------|
| `group_name` | Data | Group Name | - | No | - | Optional label for this group (e.g., "Spend Criteria", "Activity Criteria") |
| `condition_logic` | Select | Condition Logic | AND\nOR | Yes | AND | How conditions within this group are combined: AND = all must match; OR = any matches |
| `conditions` | Table | Conditions | Segment Rule Condition | No | - | Child table of individual conditions for this group |
| `enabled` | Check | Enabled | - | No | 1 | Disable to temporarily exclude this group from evaluation |

#### List View (Editable Grid)

- **in_list_view**: `group_name`, `condition_logic`, `enabled`
- **editable_grid**: Yes

### 3.3 Segment Rule Condition Child DocType

**App**: `tradehub_core`
**Module**: `TradeHub Core`
**Is Table**: Yes (child table of `Segment Rule Group`, parentfield: `conditions`)
**Naming Rule**: Random
**Track Changes**: No

This child table defines **individual conditions** that evaluate a single field against a value using a comparison operator. The structure follows the same pattern as `Seller Tag Rule Condition` (field, operator, value, logic) but adds `data_source` to support multiple source categories.

#### Field Definitions

| Field Name | Field Type | Label | Options | Required | Default | Description |
|------------|-----------|-------|---------|----------|---------|-------------|

| `field_name` | Select | Field | *(dynamic options based on data_source -- see Section 4.1)* | Yes | - | Specific field to evaluate from the selected data source |

**Activity Score Formula**:
```
activity_score = (
    login_recency_score * 0.30 +      # Based on days_since_last_login
    order_recency_score * 0.30 +       # Based on days_since_last_order
    interaction_frequency_score * 0.20 + # Based on actions per week
    feedback_participation_score * 0.20  # Based on feedback_rate
)

Where each sub-score is normalized to 0-100:
  login_recency_score = max(0, 100 - (days_since_last_login * 3.33))  # 0 at 30+ days
  order_recency_score = max(0, 100 - (days_since_last_order * 1.11))  # 0 at 90+ days
  interaction_frequency_score = min(100, actions_per_week * 10)         # 100 at 10+ per week
  feedback_participation_score = feedback_rate                          # 0-100 directly
```

### 4.2 Condition Evaluation Logic

#### 4.2.1 Operator Definitions

Each operator defines how the condition's `field_name` value is compared against the `value` field:

| Operator | SQL Equivalent | Value Format | Example | Description |
|----------|---------------|-------------|---------|-------------|
| `=` | `field = value` | Single value | `city = Istanbul` | Exact match |
| `!=` | `field != value` | Single value | `status != Inactive` | Not equal |
| `>` | `field > value` | Numeric | `total_orders > 100` | Greater than |
| `<` | `field < value` | Numeric | `return_rate < 5` | Less than |
| `>=` | `field >= value` | Numeric | `buyer_score >= 70` | Greater than or equal |
| `<=` | `field <= value` | Numeric | `avg_response_time_hours <= 24` | Less than or equal |
| `between` | `field BETWEEN a AND b` | Two values, comma-separated | `total_spent between 5000,50000` | Inclusive range (both bounds included) |
| `in` | `field IN (a, b, c)` | Comma-separated list | `city in Istanbul,Ankara,Izmir` | Value is one of the listed values |
| `not in` | `field NOT IN (a, b, c)` | Comma-separated list | `buyer_level not in New` | Value is not one of the listed values |
| `is set` | `field IS NOT NULL AND field != ''` | *(not required)* | `last_order_date is set` | Field has a non-empty value |
| `is not set` | `field IS NULL OR field = ''` | *(not required)* | `verification_status is not set` | Field is empty or null |
| `contains` | `field LIKE '%value%'` | String | `category contains Electronics` | Substring match |
| `not contains` | `field NOT LIKE '%value%'` | String | `sector not contains Test` | Negative substring match |

#### 4.2.2 Condition Combination Logic

Conditions within a group are combined using the `logic` field on each condition row. The logic field specifies how to combine with the **next** condition (last row's logic is ignored):

```
Group: "Spend & Activity Criteria" (condition_logic = AND)
  Condition 1: total_spent > 10000      logic: AND
  Condition 2: order_count >= 5         logic: AND
  Condition 3: days_since_last_login < 30  logic: -

Evaluation: (total_spent > 10000) AND (order_count >= 5) AND (days_since_last_login < 30)
```

Groups are combined using the segment-level `group_logic`:

```
User Segment: "VIP Buyers" (group_logic = AND)
  Group 1: "Spend Criteria" (condition_logic = AND)
    total_spent > 50000 AND order_count >= 10
  Group 2: "Behavior Criteria" (condition_logic = OR)
    feedback_rate >= 80 OR avg_response_time_hours <= 12

Evaluation: (Group 1 result) AND (Group 2 result)
         = ((total_spent > 50000) AND (order_count >= 10))
           AND
           ((feedback_rate >= 80) OR (avg_response_time_hours <= 12))
```

#### 4.2.3 Mixed AND/OR Within Groups

The per-condition `logic` field allows mixing AND/OR within a single group, following standard boolean precedence (AND binds tighter than OR):

```
Group: "Risk Indicators" (condition_logic = AND)
  Condition 1: return_rate > 20         logic: OR
  Condition 2: dispute_rate > 10        logic: AND
  Condition 3: buyer_score < 30         logic: -

Evaluation: (return_rate > 20) OR ((dispute_rate > 10) AND (buyer_score < 30))
```

**Implementation Note**: The `condition_logic` field on the group provides the default logic for combining conditions. The per-row `logic` field overrides this for the specific combination between that row and the next. The evaluation engine processes conditions left-to-right with AND having higher precedence than OR (standard boolean algebra).

#### 4.2.4 Dynamic Field Options

The `field_name` Select field in `Segment Rule Condition` dynamically updates its options based on the selected `data_source` and the parent segment's `target_type`. This is implemented via a client-side `frappe.call` on `data_source` change:

### 4.3 Segment Evaluation Engine

The core evaluation engine processes segments by:
1. Loading all active segments ordered by priority
2. For each segment, loading all enabled rule groups
3. For each group, loading conditions and evaluating them against user data
4. Combining group results using the segment's `group_logic`
5. Updating `Segment Member` records based on results

```python
def evaluate_segment(segment_name):
    """
    Evaluate a single segment for all applicable users.

    Returns:
        dict with total_evaluated, added, removed, unchanged counts
    """
    segment = frappe.get_doc("User Segment", segment_name)
    if segment.status != "Active":
        return {"skipped": True, "reason": "Segment not active"}

    # Determine which users to evaluate
    users = get_target_users(segment.target_type)

    # Build the evaluation query from rule groups
    rule_groups = [g for g in segment.rule_groups if g.enabled]
    if not rule_groups:
        return {"skipped": True, "reason": "No enabled rule groups"}

    results = {"total_evaluated": 0, "added": 0, "removed": 0, "unchanged": 0}

    for user in users:
        results["total_evaluated"] += 1
        user_data = collect_user_data(user, segment.target_type)
        matches = evaluate_groups(rule_groups, segment.group_logic, user_data)

        current_member = is_segment_member(segment_name, user)

        if matches and not current_member:
            add_segment_member(segment_name, user, segment.target_type)
            results["added"] += 1
            execute_segment_actions(segment, "On Enter", user)
        elif not matches and current_member:
            remove_segment_member(segment_name, user)
            results["removed"] += 1
            execute_segment_actions(segment, "On Leave", user)
        else:
            results["unchanged"] += 1

    # Update segment statistics
    segment.member_count = frappe.db.count(
        "Segment Member",
        filters={"segment": segment_name, "is_active": 1}
    )
    segment.last_refreshed = now_datetime()
    segment.refresh_status = "Completed"
    segment.save(ignore_permissions=True)
    frappe.db.commit()

    return results


def evaluate_groups(rule_groups, group_logic, user_data):
    """
    Evaluate all rule groups using the specified group_logic.

    Args:
        rule_groups: List of Segment Rule Group records
        group_logic: "AND" or "OR"
        user_data: Dict of user field values

    Returns:
        bool - whether the user matches the segment criteria
    """
    group_results = []

    for group in rule_groups:
        if not group.enabled:
            continue
        group_result = evaluate_conditions(group.conditions, user_data)
        group_results.append(group_result)

    if not group_results:
        return False

    if group_logic == "AND":
        return all(group_results)
    else:  # OR
        return any(group_results)


def evaluate_conditions(conditions, user_data):
    """
    Evaluate conditions within a single group.
    Processes left-to-right with AND having higher precedence than OR.

    Args:
        conditions: List of Segment Rule Condition records
        user_data: Dict of user field values

    Returns:
        bool - whether the conditions are satisfied
    """
    if not conditions:
        return True

    # Build expression tree respecting AND/OR precedence
    # Split into OR-groups (groups of AND-connected conditions)
    or_groups = []
    current_and_group = []

    for i, condition in enumerate(conditions):
        result = evaluate_single_condition(condition, user_data)
        current_and_group.append(result)

        # Check logic connector to next condition
        logic = condition.logic if i < len(conditions) - 1 else None
        if logic == "OR" or logic is None:
            or_groups.append(all(current_and_group))
            current_and_group = []

    return any(or_groups)


def evaluate_single_condition(condition, user_data):
    """
    Evaluate a single condition against user data.

    Args:
        condition: Segment Rule Condition record
        user_data: Dict of user field values

    Returns:
        bool - whether the condition is satisfied
    """
    field_value = user_data.get(condition.field_name)
    compare_value = condition.value

    # Handle null/empty for is set / is not set operators
    if condition.operator == "is set":
        return field_value is not None and str(field_value).strip() != ""
    elif condition.operator == "is not set":
        return field_value is None or str(field_value).strip() == ""

    # Handle null field values for other operators
    if field_value is None:
        return False

    # Type coercion based on field type
    field_value, compare_value = coerce_types(field_value, compare_value, condition.operator)

    operator_map = {
        "=": lambda a, b: a == b,
        "!=": lambda a, b: a != b,
        ">": lambda a, b: a > b,
        "<": lambda a, b: a < b,
        ">=": lambda a, b: a >= b,
        "<=": lambda a, b: a <= b,
        "between": lambda a, b: b[0] <= a <= b[1],
        "in": lambda a, b: str(a) in [str(v).strip() for v in b],
        "not in": lambda a, b: str(a) not in [str(v).strip() for v in b],
        "contains": lambda a, b: str(b).lower() in str(a).lower(),
        "not contains": lambda a, b: str(b).lower() not in str(a).lower(),
    }

    evaluator = operator_map.get(condition.operator)
    if not evaluator:
        frappe.log_error(
            f"Unknown operator: {condition.operator}",
            "Segment Evaluation"
        )
        return False

    try:
        return evaluator(field_value, compare_value)
    except (TypeError, ValueError) as e:
        frappe.log_error(
            f"Condition evaluation error: {condition.field_name} "
            f"{condition.operator} {condition.value}: {str(e)}",
            "Segment Evaluation"
        )
        return False
```

### 4.4 Live Member Count Preview

When an administrator saves a segment, the system calculates a preview count before persisting changes. This provides immediate feedback on the segment's scope.

**Mechanism**: The `validate` method on the `User Segment` controller triggers a preview evaluation:

```python
class UserSegment(Document):
    def before_insert(self):
        if not self.created_by:
            self.created_by = frappe.session.user
        if not self.created_at:
            self.created_at = now_datetime()

    def validate(self):
        self.validate_segment_code()
        self.validate_rule_groups()
        self.validate_weight_of_conditions()
        self.update_preview_count()

    def validate_segment_code(self):
        """Ensure segment code is uppercase with underscores."""
        if self.segment_code:
            self.segment_code = self.segment_code.upper().replace(" ", "_")
            import re
            if not re.match(r'^[A-Z][A-Z0-9_]*$', self.segment_code):
                frappe.throw(
                    _("Segment Code must start with a letter and contain only "
                      "uppercase letters, numbers, and underscores.")
                )

    def validate_rule_groups(self):
        """Ensure at least one enabled rule group exists for active segments."""
        if self.status == "Active":
            enabled_groups = [g for g in (self.rule_groups or []) if g.enabled]
            if not enabled_groups:
                frappe.throw(
                    _("Active segments must have at least one enabled rule group.")
                )

    def validate_weight_of_conditions(self):
        """Ensure conditions have valid data sources and operators."""
        for group in (self.rule_groups or []):
            for condition in (group.conditions or []):
                if condition.operator not in [
                    "=", "!=", ">", "<", ">=", "<=",
                    "between", "in", "not in",
                    "is set", "is not set",
                    "contains", "not contains"
                ]:
                    frappe.throw(
                        _("Invalid operator '{0}' in group '{1}'").format(
                            condition.operator, group.group_name or group.idx
                        )
                    )

    def update_preview_count(self):
        """Calculate preview member count on save."""
        try:
            rule_groups = [g for g in (self.rule_groups or []) if g.enabled]
            if not rule_groups:
                self.member_count_preview = 0
                return

            # Build SQL query from conditions for fast preview
            query = build_preview_query(self.target_type, rule_groups, self.group_logic)
            count = frappe.db.sql(query, as_dict=False)
            self.member_count_preview = count[0][0] if count else 0

            # Update HTML preview with first 20 members
            members = frappe.db.sql(
                query.replace("COUNT(*)", "*") + " LIMIT 20",
                as_dict=True
            )
            self.member_preview_html = render_member_preview(members, self.member_count_preview)

            frappe.msgprint(
                _("This segment contains {0} {1}(s)").format(
                    self.member_count_preview,
                    self.target_type.lower()
                ),
                indicator="blue",
                alert=True
            )
        except Exception as e:
            frappe.log_error(
                f"Preview count error for segment {self.segment_name}: {str(e)}",
                "Segment Preview"
            )
            self.member_count_preview = 0
            frappe.msgprint(
                _("Could not calculate preview count: {0}").format(str(e)),
                indicator="orange",
                alert=True
            )
```

### 4.5 Edge Cases

#### 4.5.1 Empty Segments

A segment with zero matching members is valid. The system handles this by:

- Setting `member_count = 0` and `member_count_preview = 0`
- Displaying "This segment contains 0 buyer(s)" in the preview message
- Not creating any `Segment Member` records
- Logging a warning if an active segment has had zero members for 7+ consecutive refreshes
- Segment actions (On Enter/On Leave) are never triggered for empty segments

#### 4.5.2 New Users with No Data

Users with no transaction history, no orders, and no behavioral data are handled with safe defaults:

| Condition Type | Default for New Users | Matches? |
|---------------|----------------------|----------|
| `total_orders > 10` | `total_orders = 0` | No (0 is not > 10) |
| `total_spent between 100,500` | `total_spent = 0` | No (0 is not between 100 and 500) |
| `last_order_date is set` | `last_order_date = NULL` | No |
| `buyer_score >= 50` | `buyer_score = NULL` | No (NULL fails all comparisons except `is not set`) |
| `registration_date > 2025-01-01` | Registration date always exists | Yes if registered after 2025-01-01 |
| `verification_status is not set` | `verification_status = NULL` | Yes |

**Rule**: All numeric fields with NULL/None values return `False` for any comparison operator except `is set`/`is not set`. This prevents division-by-zero and ensures new users are not accidentally included in performance-based segments.

#### 4.5.3 Segment with Overlapping Criteria

A user can belong to multiple segments simultaneously. This is by design:

- A buyer with `buyer_score = 85` and `total_spent = 200000` may be in both "High Score Buyers" and "High Spend Buyers"
- Segment priority determines conflict resolution when segment-driven actions conflict (lower priority number wins)
- The admin dashboard shows overlap analysis between segments

#### 4.5.4 Rule Changes on Active Segments

When an admin modifies rules on an active segment:

1. Preview count updates immediately on save
2. Actual membership is **not** changed until the next scheduled refresh or a manual "Refresh Now" action
3. This prevents disruptive mid-cycle membership changes
4. The admin sees both `member_count` (current actual) and `member_count_preview` (projected after refresh)

#### 4.5.5 Target Type Mismatch

If a segment with `target_type = "Buyer"` contains conditions using seller-only fields (e.g., `product_count`), the system:

- Filters out inapplicable fields in the `get_available_fields()` API
- Validates on save that all conditions use fields applicable to the target type
- Throws a validation error: "Field 'product_count' is not available for Buyer segments"

def refresh_segment_membership():
    """
    Periodic job to refresh segment membership for all active segments.

    Runs hourly but respects each segment's refresh_frequency setting:
    - Hourly: Evaluated every run
    - Daily: Evaluated if last_refreshed was more than 24 hours ago
    - Weekly: Evaluated if last_refreshed was more than 7 days ago
    - Manual: Skipped by the scheduled job (admin must click "Refresh Now")
    """
    segments = frappe.get_all(
        "User Segment",
        filters={"status": "Active", "auto_refresh": 1},
        fields=["name", "segment_name", "refresh_frequency", "last_refreshed"],
        order_by="priority asc"
    )

    for segment in segments:
        try:
            if not should_refresh(segment):
                continue

            frappe.db.set_value(
                "User Segment", segment.name,
                "refresh_status", "Running",
                update_modified=False
            )
            frappe.db.commit()

            result = evaluate_segment(segment.name)

            frappe.logger().info(
                f"Segment '{segment.segment_name}' refreshed: "
                f"evaluated={result.get('total_evaluated', 0)}, "
                f"added={result.get('added', 0)}, "
                f"removed={result.get('removed', 0)}"
            )

        except Exception as e:
            frappe.db.set_value(
                "User Segment", segment.name,
                "refresh_status", "Failed",
                update_modified=False
            )
            frappe.log_error(
                f"Segment refresh failed for '{segment.segment_name}': {str(e)}",
                "Segment Refresh"
            )
            frappe.db.commit()

    frappe.db.commit()


def should_refresh(segment):
    """
    Check if a segment needs to be refreshed based on its frequency setting.

    Args:
        segment: Segment record with refresh_frequency and last_refreshed

    Returns:
        bool - True if the segment should be evaluated
    """
    if segment.refresh_frequency == "Manual":
        return False

    if not segment.last_refreshed:
        return True  # Never refreshed, always refresh

    from frappe.utils import now_datetime, time_diff_in_hours

    hours_since = time_diff_in_hours(now_datetime(), segment.last_refreshed)

    frequency_hours = {
        "Hourly": 1,
        "Daily": 24,
        "Weekly": 168
    }

    required_hours = frequency_hours.get(segment.refresh_frequency, 24)
    return hours_since >= required_hours
```

**Data Sources Read**:
- `User Segment` (active segments and their rule groups/conditions)
- `Buyer Profile` (buyer profile, transaction, financial, scoring fields)
- `Seller Profile` (seller profile fields)
- `Seller Metrics` (seller transaction, behavior, product, financial fields)
- `Seller Score` (seller scoring fields)
- `User` (last_login for behavior data source)

**Data Targets Written**:
- `Segment Member` (create/update membership records)
- `User Segment` (update member_count, last_refreshed, refresh_status)

**Estimated Runtime**: For 10,000 users and 50 active segments, estimated 2-5 minutes depending on condition complexity. The hourly schedule provides sufficient headroom.

**Error Handling**:
- Per-segment try/except: A failing segment does not block others
- `refresh_status` set to "Failed" on error with error details in Error Log
- `frappe.db.commit()` after each segment to prevent partial transaction rollback
- Locking: Uses `frappe.db.set_value` with `update_modified=False` to avoid concurrency issues with admin edits

def cleanup_segment_members():
    """
    Weekly cleanup job for segment membership records.

    - Archives inactive member records older than 90 days
    - Removes members of deleted/inactive segments
    - Recalculates member_count for all active segments
    """
    # Remove members of inactive segments
    inactive_segments = frappe.get_all(
        "User Segment",
        filters={"status": "Inactive"},
        pluck="name"
    )

    if inactive_segments:
        frappe.db.sql("""
            UPDATE `tabSegment Member`
            SET is_active = 0, left_at = NOW()
            WHERE segment IN %(segments)s AND is_active = 1
        """, {"segments": inactive_segments})

    # Archive old inactive records (soft delete)
    frappe.db.sql("""
        DELETE FROM `tabSegment Member`
        WHERE is_active = 0
        AND left_at < DATE_SUB(NOW(), INTERVAL 90 DAY)
    """)

    # Recalculate member counts
    active_segments = frappe.get_all(
        "User Segment",
        filters={"status": "Active"},
        pluck="name"
    )

    for seg_name in active_segments:
        count = frappe.db.count(
            "Segment Member",
            filters={"segment": seg_name, "is_active": 1}
        )
        frappe.db.set_value(
            "User Segment", seg_name,
            "member_count", count,
            update_modified=False
        )

    frappe.db.commit()
```

## 7. Permission Model

### 7.1 User Segment Permissions

| Role | Read | Write | Create | Delete | Export | Report |
|------|------|-------|--------|--------|--------|--------|
| System Manager | Yes | Yes | Yes | Yes | Yes | Yes |
| Marketplace Manager | Yes | Yes | Yes | No | Yes | Yes |
| All | No | No | No | No | No | No |

**Notes**:
- Only System Manager can delete segments (to prevent accidental removal of segments driving automated actions)
- Marketplace Manager can create and edit but not delete
- Segments are not visible to buyers or sellers

### 7.2 Segment Member Permissions

| Role | Read | Write | Create | Delete | Export | Report |
|------|------|-------|--------|--------|--------|--------|
| System Manager | Yes | Yes | Yes | Yes | Yes | Yes |
| Marketplace Manager | Yes | No | No | No | Yes | Yes |
| All | No | No | No | No | No | No |

**Notes**:
- Marketplace Manager can view and export membership data but cannot manually modify
- Only the scheduled job (running as Administrator) and System Manager can modify membership
- Manual membership additions by System Manager use `source = "Manual"`

Child table -- inherits from parent `User Segment` permissions.

### 7.4 Segment Rule Condition Permissions

Child table -- inherits from parent `Segment Rule Group` (which inherits from `User Segment`).

Child table -- inherits from parent `User Segment` permissions.
