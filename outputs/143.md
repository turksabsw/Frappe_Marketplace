# Task 143 — Platform Satin Alma Talebi Sistemi (Platform Purchase Request System)

## 1. Genel Bakis (Overview)

Bu belge, B2B pazar yeri platformunda **Platform Satin Alma Talebi (Platform Purchase Request)** sistemini kapsamli olarak planlar. Bu sistem, platform yoneticisinin (admin) toplu satin alma talepleri olusturmasini, uygun saticilara yayinlamasini, saticilardan teklifler toplamasini ve otomatik puanlama algoritmasi ile en uygun teklifi secmesini saglar.

**Referans Modeller:**
- **Amazon Vendor Central** — Platform tarafindan olusturulan satin alma talepleri, satici davet sistemi
- **Alibaba RFQ** — Coklu satici teklif toplama, otomatik karsilastirma tablosu
- **Mevcut RFQ Sistemi** — `tradehub_commerce/doctype/rfq/rfq.py` ile benzer workflow yapisi (Draft -> Published -> Quoting -> Accepted/Closed)

**Problem:** Platform yoneticileri, belirli urunler/hizmetler icin toplu satin alma ihtiyaclarini merkezi bir sistem uzerinden yonetemez. Mevcut RFQ sistemi alici-odaklidir; platform-odakli bir satin alma talebi mekanizmasi bulunmamaktadir.

**Hedef:**
- Platform yoneticisinin satin alma talepleri olusturabilecegi merkezi bir DocType tanimlamak
- Saticilardan yapilandirilmis teklifler toplayan bir teklif sistemi kurmak
- Otomatik puanlama algoritmasi ile teklif degerlendirme surecini hizlandirmak
- Satici portalinda "Platform Talepleri" sekmesi ile satici deneyimini zenginlestirmek

**Kapsam:**
- 5 yeni DocType tanimlama (Platform Purchase Request, Purchase Request Item, Pricing Tier Template, Seller Offer, Seller Offer Item)
- 2 child table tanimlama (Pricing Tier, Seller Offer Item)
- Agirlikli otomatik puanlama algoritmasi
- Kapsamli bildirim matrisi
- Satici portal entegrasyonu
- Tam workflow diyagrami

**Mevcut Sistemle Iliski:**
Mevcut RFQ DocType (`rfq.json`) su alanlari icerir: `rfq_code`, `title`, `buyer`, `status`, `category`, `budget_min`, `budget_max`, `delivery_date`, `items` (Table: RFQ Item), `target_type` (Public/Category/Selected). Yeni Platform Purchase Request bu pattern'i takip eder ancak **platform-odakli** (alici degil admin tarafindan olusturulan) bir yaklasim benimser.

```python
def _generate_request_code(self):
    """
    Format: PPR-YYYYMMDD-XXXX
    Mevcut RFQ pattern'i: RFQ-YYYYMMDD-XXXX
    """
    import random
    import string
    date_part = today().replace("-", "")
    random_part = ''.join(random.choices(string.ascii_uppercase + string.digits, k=4))
    return f"PPR-{date_part}-{random_part}"
```

```python
# Pseudocode — Pricing Tier Template validasyonu
def validate_tiers(self):
    """Kademelerin mantikli sirada oldugunu dogrula."""
    if not self.tiers:
        frappe.throw(_("En az bir fiyat kademesi tanimlanmalidir"))

    for i in range(1, len(self.tiers)):
        if self.tiers[i].min_quantity <= self.tiers[i-1].min_quantity:
            frappe.throw(_("Kademe miktarlari artan sirada olmalidir"))

    for tier in self.tiers:
        if flt(tier.discount_percentage) < 0 or flt(tier.discount_percentage) > 100:
            frappe.throw(_("Indirim yuzdeleri 0-100 arasinda olmalidir"))

def validate_default(self):
    """Yalnizca bir varsayilan sablon olabilir."""
    if cint(self.is_default):
        existing = frappe.db.exists("Pricing Tier Template", {
            "is_default": 1,
            "is_active": 1,
            "name": ["!=", self.name]
        })
        if existing:
            frappe.db.set_value("Pricing Tier Template", existing, "is_default", 0)
            frappe.msgprint(_("Onceki varsayilan sablon deaktive edildi"), indicator="blue")
```

### 6.2 Toplam Hesaplama

```python
# Pseudocode — Seller Offer total_offered_amount hesaplama
def calculate_totals(self):
    """
    Teklifteki tum kalemlerin toplamini hesaplar.
    Mevcut Commission Plan pattern'i ile uyumlu.
    """
    total = 0
    for item in self.items:
        item.total_price = flt(item.offered_quantity) * flt(item.unit_price)
        if flt(item.discount_percentage) > 0:
            item.total_price = item.total_price * (1 - flt(item.discount_percentage) / 100)
        total += item.total_price

    self.total_offered_amount = total
```

## 7. Teklif Degerlendirme — Otomatik Puanlama Algoritmasi (Auto-Scoring Mechanism)

### 7.1 Puanlama Agirliklari

Teklifler 4 ana kritere gore otomatik puanlanir. Agirliklar admin tarafindan yapilandirilabilir:

| Kriter | Varsayilan Agirlik | Aciklama | Puan Kaynagi |
|--------|-------------------|----------|-------------|
| **Fiyat (Price)** | %40 | Dusuk fiyat = yuksek puan | Teklif tutari / en dusuk teklif tutari |
| **Teslimat (Delivery)** | %25 | Erken teslimat = yuksek puan | Teklif teslim suresi / en kisa teslim suresi |
| **Satici Puani (Rating)** | %20 | Yuksek puan = yuksek skor | Satici puani / 5.0 (maksimum) |
| **Odeme Kosullari (Payment Terms)** | %15 | Uzun vade = yuksek puan | Odeme vadesi puanlama tablosuna gore |

> **Admin-yapilandirilabilir:** Tum agirliklar (price_weight, delivery_weight, rating_weight, payment_weight) Marketplace Settings veya Platform Purchase Request formunda ayarlanabilir. Toplam agirlik %100 olmalidir.

### 7.2 Puanlama Algoritmasi

```python
# Pseudocode — Otomatik puanlama algoritmasi

def calculate_auto_score(offer, all_offers, weights=None):
    """
    Satici teklifini agirlikli kriterlere gore puanlar.

    Args:
        offer: Seller Offer DocType instance
        all_offers: Ayni purchase_request icin tum teklifler listesi
        weights: Yapilandirilabilir agirliklar dict'i

    Returns:
        dict: {auto_score, price_score, delivery_score, rating_score, payment_score}
    """
    # 1. Agirliklari al (admin-yapilandirilabilir)
    if not weights:
        weights = get_scoring_weights()  # Marketplace Settings'ten

    # Varsayilan agirliklar
    w_price = weights.get("price_weight", 0.40)
    w_delivery = weights.get("delivery_weight", 0.25)
    w_rating = weights.get("rating_weight", 0.20)
    w_payment = weights.get("payment_weight", 0.15)

    # Agirlik toplami kontrolu
    total_weight = w_price + w_delivery + w_rating + w_payment
    if abs(total_weight - 1.0) > 0.01:
        frappe.throw(_("Puanlama agirliklari toplami %100 olmalidir"))

    # 2. FIYAT PUANI (0-100)
    # En dusuk fiyat = 100 puan
    prices = [flt(o.total_offered_amount) for o in all_offers if flt(o.total_offered_amount) > 0]
    if prices:
        min_price = min(prices)
        if flt(offer.total_offered_amount) > 0:
            price_score = (min_price / flt(offer.total_offered_amount)) * 100
        else:
            price_score = 0
    else:
        price_score = 50  # Tek teklif varsa ortalama puan

    # 3. TESLIMAT PUANI (0-100)
    # En kisa teslim suresi = 100 puan
    delivery_days_list = [o.proposed_delivery_days or 999 for o in all_offers]
    min_delivery = min(delivery_days_list) if delivery_days_list else 30
    offer_delivery = offer.proposed_delivery_days or 999
    if offer_delivery > 0:
        delivery_score = (min_delivery / offer_delivery) * 100
    else:
        delivery_score = 50

    # 4. SATICI PUANI (0-100)
    # 5.0 uzerinden rating, 100'e normalize
    seller_rating = flt(offer.seller_rating_snapshot) or 0
    rating_score = (seller_rating / 5.0) * 100

    # 5. ODEME KOSULLARI PUANI (0-100)
    payment_scores = {
        "Advance": 100,    # Platform icin en avantajli
        "30 Days": 75,
        "60 Days": 50,
        "90 Days": 25,
        "Custom": 50       # Varsayilan orta puan
    }
    payment_score = payment_scores.get(offer.proposed_payment_terms, 50)

    # 6. AGIRLIKLI TOPLAM PUAN
    auto_score = (
        (price_score * w_price) +
        (delivery_score * w_delivery) +
        (rating_score * w_rating) +
        (payment_score * w_payment)
    )

    return {
        "auto_score": round(auto_score, 2),
        "price_score": round(price_score, 2),
        "delivery_score": round(delivery_score, 2),
        "rating_score": round(rating_score, 2),
        "payment_score": round(payment_score, 2)
    }


def get_scoring_weights():
    """
    Marketplace Settings'ten puanlama agirliklarini okur.
    Admin-yapilandirilabilir.
    """
    settings = frappe.get_single("Marketplace Settings")
    return {
        "price_weight": flt(settings.price_weight) or 0.40,
        "delivery_weight": flt(settings.delivery_weight) or 0.25,
        "rating_weight": flt(settings.rating_weight) or 0.20,
        "payment_weight": flt(settings.payment_weight) or 0.15
    }
```

| Platform Purchase Request | `idx_target_type` | (`target_type`, `status`) | Hedefleme bazli sorgular |
| Seller Offer | `idx_seller_pr` | (`seller`, `purchase_request`) | Unique constraint |
| Seller Offer | `idx_pr_status_score` | (`purchase_request`, `status`, `auto_score`) | Puanlama bazli siralama |
| Seller Offer | `idx_seller_status` | (`seller`, `status`) | Satici bazli listeler |

## 12. Edge Cases (Ozel Durumlar)

### 12.1 Ayni Satici Birden Fazla Teklif

**Senaryo:** Satici zaten teklif gondermis bir talep icin tekrar teklif gondermek ister.

**Cozum:**
- `validate_unique_offer()` ile engellenir
- Satici mevcut teklifini "Withdrawn" yaparak yeni teklif gonderebilir (eger talep hala "Receiving Offers" ise)
- Alternatif: Satici "Draft" durumdaki mevcut teklifini guncelleyebilir

### 12.2 Teklif Suresi Dolduktan Sonra Teklif Gonderimi

**Senaryo:** Satici `closing_date` gectikten sonra teklif gondermek ister.

**Cozum:**
- `validate()` metodunda `closing_date < now_datetime()` kontrolu yapilir
- Hata mesaji: "Bu talep icin teklif gonderme suresi dolmustur."
- Admin zorunlu durumlarda sureyi uzatabilir (closing_date guncelleme yetkisi)

### 12.3 Teklif Toplama Sirasinda Talep Iptali

**Senaryo:** "Receiving Offers" durumunda 5 teklif alinmis, admin talebi iptal eder.

**Cozum:**
- Tum "Submitted" ve "Under Review" teklifler otomatik olarak "Rejected" yapilir
- `rejection_reason = "Talep admin tarafindan iptal edildi"`
- N7 bildirimi tum saticicilara gonderilir
- Saticilar iptal bildiriminde neden gorur

### 12.4 Tum Teklifler Butce Asimi

**Senaryo:** Tum alinan teklifler `budget_max` degerini asiyordur.

**Cozum:**
- Otomatik puanlama budge kontrolu yapmaz (puanlama, karsilastirmayla ilgilidir)
- Admin karsilastirma tablosunda butce asim uyarisi gosterilir (turuncu isaret)
- Admin talebi kapatabilir veya butce limitini yukseltebilir
- Admin bilgilendirmesi: "Tum teklifler butce limitini asmaktadir"

### 12.5 Tekrarlayan Talep (Recurring Request)

**Senaryo:** Aylik tekrarlayan bir talep olusturulmus, her ay otomatik kopyalanmali.

**Cozum:**
- `create_recurring_purchase_requests` scheduled task'i kontrol eder
- Mevcut donem talebi "Closed" veya "Awarded" ise yeni kopya olusturulur
- Yeni kopya "Draft" durumunda olusturulur (admin onay bekler)
- Onceki donem teklifleri ve puanlama gecmisi referans olarak saklanir
- Recurring interval'a gore (Weekly/Bi-Weekly/Monthly/Quarterly) sonraki tarih hesaplanir

### 12.6 Satici Aboneligi Askida (Suspended)

**Senaryo:** Satici teklif gonderirken, Task 142'deki abonelik durumu "Suspended" veya "Cancelled" olur.

**Cozum:**
- Teklif gonderme sirasinda saticinin abonelik durumu kontrol edilir
- Askida/iptal abonelik ile teklif gonderilemez
- Zaten gonderilmis teklifler gecerli kalir (geri cekme olmaz) ama yeni teklif veya guncelleme yapilamaz
- Task 142 Etki Matrisi ile uyumlu: Suspended durumda "Teklif Verme" **Engel**

### 12.7 Esit Puan Durumu

**Senaryo:** Iki veya daha fazla teklif esit auto_score puan alir.

**Cozum:**
- Birincil siralama: `auto_score` (azalan)
- Ikincil siralama: `submitted_at` (artan — once gelen teklif oncelikli)
- Ucuncul siralama: `seller_rating_snapshot` (azalan)
- Admin karsilastirma tablosunda esit puanli teklifler ozel isaret ile gosterilir
- Final karar admin'e biirakilir (otomatik kabul yapilmaz)

### 12.8 Kismi Teklif (Partial Offer)

**Senaryo:** Talep 5 kalem icerir ancak satici yalnizca 3 kaleme teklif vermek ister.

**Cozum:**
- `allow_partial_offers` flagi kontrol edilir
- Aktif ise: Satici istedigi kalemlere teklif verebilir, diger kalemler bos birakilir
- Deaktif ise: Tum kalemlere teklif verilmek zorundadir
- Kismi teklifler puanlamada cezalandirilmaz ancak karsilastirma tablosunda "Kismi Teklif" etiketi ile isaretlenir

### 12.9 Maksimum Teklif Sayisina Ulasilmasi

**Senaryo:** `max_offers` alani 10 olarak ayarlanmis ve 10 teklif alinmis.

**Cozum:**
- 10. teklif alindiginda yeni teklif kabul edilmez
- Satici portali "Bu talep icin maksimum teklif sayisina ulasilmistir" mesaji gosterir
- Admin `max_offers` degerini yukseltebilir
- `max_offers = 0` ise sinirsiz teklif kabul edilir (varsayilan)

### 12.10 Puanlama Agirliklari Degistirilmesi

**Senaryo:** Admin, degerlendirme surecinde puanlama agirliklarini degistirir.

**Cozum:**
- Agirlik degisikligi aninda "Evaluation" durumundaki tum talepler icin yeniden puanlama yapilir
- Onceki puanlamalar Activity Log'a kaydedilir
- Admin'e "Puanlama agirliklari degistirildi, X talep icin yeniden hesaplama yapildi" bildirimi gosterilir
