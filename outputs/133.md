# Task 133 - Buyer KPI Template System Plan

## 1. Overview

This document defines the **Buyer KPI Template System** for the TradeHub marketplace platform. The system provides a customizable, template-based approach to buyer performance evaluation using weighted metrics. It extends the existing `KPI Template` and `KPI Template Item` DocTypes (from `tradehub_seller`) with buyer-specific configurations and introduces a dedicated `Buyer KPI Template` DocType with a `Buyer KPI Metric` child table tailored for buyer evaluation.

Key capabilities:

- **Buyer KPI Template DocType**: A buyer-focused KPI template that extends the existing `KPI Template` pattern with buyer-specific fields (customer group assignment, radar chart config, clone source tracking).
- **Buyer KPI Metric child DocType**: A child table defining individual buyer metrics with `metric_type` options of Currency, Percent, Number, Rating, and Duration; threshold levels (poor, fair, good, excellent); and weight percentages.
- **Default template**: A locked, undeletable template (`is_default = 1`) with exactly 9 metrics summing to 100% weight.
- **Customization**: Clone & edit flow for creating derivative templates, add/remove metrics, weight validation ensuring sum equals 100%.
- **Template assignment**: Assign different templates to different customer groups for differentiated evaluation.
- **Radar/spider chart visualization**: Visual KPI scorecard rendering all metrics on a single radar chart.
- **Template comparison view**: Side-by-side comparison of two templates showing metric and weight differences.

### Design Principles

1. **Extend, Don't Duplicate**: The `Buyer KPI Template` follows the same structural pattern as the existing `KPI Template` but is a separate DocType to allow buyer-specific features (customer group assignment, radar chart config) without polluting the seller-focused original.
2. **Default Template Protection**: The default template (`is_default = 1`) cannot be deleted or have its `is_default` flag cleared. It serves as the fallback for all buyers without a specific template assignment.
3. **Weight Integrity**: All active metrics in a template must have weights summing to exactly 100%. This is validated on save and enforced programmatically.
4. **Metric Naming Consistency**: All metric field names and KPI codes align with the naming contract established in `119.md` (Section 8.3) and the buyer scoring template defined in `132.md` (Section 3.1).
5. **Clone-Based Customization**: Custom templates are created by cloning the default (or another template), not by building from scratch. This ensures structural consistency and reduces configuration errors.

## 2. Reference Models

### 2.1 Amazon - Seller Performance KPI Templates

Amazon uses a fixed, non-customizable set of seller KPIs with strict thresholds:

| Aspect | Amazon Approach | TradeHub Adaptation |
|--------|----------------|---------------------|
| Template flexibility | Single fixed template per marketplace | Multiple templates assignable per customer group |
| Metric weights | Fixed, not disclosed publicly | Configurable via admin UI, transparent weights |
| Default metrics | ODR, LSR, PFCR, Valid Tracking | 9 buyer-specific metrics (spend, payment, returns, etc.) |
| Threshold system | Binary pass/fail per metric | Four-level thresholds (poor/fair/good/excellent) |
| Customization | None for sellers | Clone & edit for admin-created variants |

**Key Amazon pattern adopted**: Undeletable default template that serves as baseline. All accounts are evaluated against it unless a specific override is assigned.

### 2.2 eBay - Detailed Seller Ratings (DSR) Template

eBay's DSR system provides a template-like evaluation structure:

| Aspect | eBay Approach | TradeHub Adaptation |
|--------|--------------|---------------------|
| Rating categories | 4 fixed categories (item accuracy, communication, shipping time, shipping cost) | 9 configurable buyer metrics with adjustable weights |
| Weighting | Equal weight per category | Configurable weights per metric (must sum to 100%) |
| Visualization | Star bars per category | Radar/spider chart for holistic view |
| Customization | No customization | Admin can clone and create category-specific templates |
| Comparison | Top Rated vs Standard requirements | Template comparison view for side-by-side analysis |

**Key eBay pattern adopted**: Multi-category evaluation with per-category scoring, displayed visually for quick comprehension.

### 2.3 Alibaba - Supplier Assessment Templates

Alibaba uses tiered assessment templates for different supplier categories:

| Aspect | Alibaba Approach | TradeHub Adaptation |
|--------|-----------------|---------------------|
| Template per category | Different assessment criteria for Gold vs Verified suppliers | Different templates per customer group (New, Regular, VIP, Premium) |
| Weight distribution | Category-specific weights | Admin-configurable weights per template |
| Radar chart | Supplier capability radar chart | Buyer KPI radar chart with all 9 metrics |
| Threshold levels | Pass/fail with recommendations | Four levels: poor, fair, good, excellent |

**Key Alibaba pattern adopted**: Category-specific templates with radar chart visualization and multi-level threshold assessment.

## 4. Business Logic & Formulas

### 4.1 Weight Validation

Weights must sum to exactly 100% for all active metrics. This validation runs on every save operation.

```python
class BuyerKPITemplate(Document):
    def validate(self):
        self.validate_weights()
        self.validate_default_uniqueness()
        self.validate_metric_codes_unique()
        self.update_total_weight()

    def validate_weights(self):
        """Ensure active metric weights sum to exactly 100%."""
        active_metrics = [m for m in self.metrics if m.is_active]

        if not active_metrics:
            frappe.throw(
                _("At least one active metric is required."),
                title=_("No Active Metrics")
            )

        total_weight = sum(m.weight_pct for m in active_metrics)

        if abs(total_weight - 100.0) > 0.01:
            frappe.throw(
                _("Active metric weights must sum to exactly 100%. "
                  "Current total: {0}%").format(total_weight),
                title=_("Invalid Weight Sum")
            )

    def validate_default_uniqueness(self):
        """Only one template can be the default."""
        if self.is_default:
            existing_default = frappe.db.get_value(
                "Buyer KPI Template",
                {"is_default": 1, "name": ("!=", self.name)},
                "name"
            )
            if existing_default:
                frappe.throw(
                    _("Template '{0}' is already the default. "
                      "Clear its default flag first.").format(existing_default),
                    title=_("Duplicate Default")
                )

    def validate_metric_codes_unique(self):
        """Ensure no duplicate metric codes within this template."""
        codes = [m.metric_code for m in self.metrics if m.metric_code]
        duplicates = [c for c in codes if codes.count(c) > 1]
        if duplicates:
            frappe.throw(
                _("Duplicate metric codes found: {0}").format(
                    ", ".join(set(duplicates))
                ),
                title=_("Duplicate Metric Codes")
            )

    def update_total_weight(self):
        """Auto-calculate total_weight from active metrics."""
```

### 4.4 Metric Score Normalization

Each metric raw value is normalized to a 0-100 scale based on its threshold levels:

```python
def normalize_metric_score(raw_value, metric):
    """
    Normalize a raw metric value to a 0-100 score using four-level thresholds.

    Threshold mapping:
        - Poor:      0 - 25 normalized
        - Fair:     25 - 50 normalized
        - Good:     50 - 75 normalized
        - Excellent: 75 - 100 normalized

    Args:
        raw_value: The actual metric value.
        metric: Buyer KPI Metric child record.

    Returns:
        Normalized score (0-100).
    """
    poor = metric.threshold_poor or 0
    fair = metric.threshold_fair or 0
    good = metric.threshold_good or 0
    excellent = metric.threshold_excellent or 0

    higher_is_better = (metric.threshold_type == "Higher is Better")

    if higher_is_better:
        # For higher-is-better: poor < fair < good < excellent
        if raw_value <= poor:
            return 0.0
        elif raw_value <= fair:
            return 25.0 * (raw_value - poor) / max(fair - poor, 0.001)
        elif raw_value <= good:
            return 25.0 + 25.0 * (raw_value - fair) / max(good - fair, 0.001)
        elif raw_value <= excellent:
            return 50.0 + 25.0 * (raw_value - good) / max(excellent - good, 0.001)
        else:
            # Above excellent -> 75-100, capped at 100
            bonus = min(25.0, 25.0 * (raw_value - excellent) / max(excellent, 0.001))
            return min(100.0, 75.0 + bonus)
    else:
        # For lower-is-better: poor > fair > good > excellent
        if raw_value >= poor:
            return 0.0
        elif raw_value >= fair:
            return 25.0 * (poor - raw_value) / max(poor - fair, 0.001)
        elif raw_value >= good:
            return 25.0 + 25.0 * (fair - raw_value) / max(fair - good, 0.001)
        elif raw_value >= excellent:
            return 50.0 + 25.0 * (good - raw_value) / max(good - excellent, 0.001)
        else:
            # Below excellent (best possible) -> 75-100
            bonus = min(25.0, 25.0 * (excellent - raw_value) / max(excellent + 0.001, 0.001))
            return min(100.0, 75.0 + bonus)
```

### 4.5 Overall Score Calculation

The overall buyer KPI score is calculated by combining weighted, normalized metric scores:

```python
def calculate_buyer_kpi_score(buyer_name, template_name=None):
    """
    Calculate overall buyer KPI score using a specific template.

    Args:
        buyer_name: Buyer Profile name.
        template_name: Buyer KPI Template name. If None, uses default.

    Returns:
        dict with overall_score, per_metric_scores, grade
    """
    # Step 1: Resolve template
    if not template_name:
        template_name = frappe.db.get_value(
            "Buyer KPI Template",
            {"is_default": 1, "status": "Active"},
            "name"
        )
    if not template_name:
        frappe.throw(_("No active default Buyer KPI Template found."))

    template = frappe.get_doc("Buyer KPI Template", template_name)
    profile = frappe.get_doc("Buyer Profile", buyer_name)

    # Step 2: Check eligibility
    if profile.total_orders < template.min_orders_required:
        return {"overall_score": None, "status": "insufficient_orders"}
    if days_since(profile.joined_at) < template.min_days_active:
        return {"overall_score": None, "status": "insufficient_activity"}

    # Step 3: Collect raw values per metric
    raw_values = collect_raw_metric_values(profile, template)

    # Step 4: Normalize each metric (0-100)
    normalized_scores = {}
    for metric in template.metrics:
        if not metric.is_active:
            continue
        raw = raw_values.get(metric.metric_code, 0)
        normalized_scores[metric.metric_code] = normalize_metric_score(raw, metric)

    # Step 5: Apply weights
    weighted_sum = 0.0
    for metric in template.metrics:
        if not metric.is_active:
            continue
        score = normalized_scores.get(metric.metric_code, 0)
        weighted_sum += score * (metric.weight_pct / 100.0)

    # Step 6: Apply scoring curve
    overall_score = apply_scoring_curve(weighted_sum, template.scoring_curve)

    # Step 7: Clamp to 0-100
    overall_score = max(0.0, min(100.0, round(overall_score, 2)))

    return {
        "overall_score": overall_score,
        "per_metric_scores": normalized_scores,
        "raw_values": raw_values,
        "template_name": template_name,
        "status": "calculated"
    }


def collect_raw_metric_values(profile, template):
    """
    Collect raw metric values from buyer profile and related DocTypes.

    Returns dict mapping metric_code -> raw_value.
    """
    values = {}

    for metric in template.metrics:
        if not metric.is_active:
            continue

        code = metric.metric_code

        if code == "TOTAL_SPEND":
            values[code] = profile.total_spent or 0
        elif code == "ORDER_FREQ":
            values[code] = calculate_order_frequency(
                profile.name, months=6
            )
        elif code == "PAYMENT_ON_TIME":
            values[code] = profile.payment_on_time_rate or 0
        elif code == "BUYER_RETURN_RATE":
            values[code] = (profile.return_rate or 0)
        elif code == "AVG_ORDER_VAL":
            values[code] = profile.average_order_value or 0
        elif code == "BUYER_DISPUTE_RATE":
            values[code] = profile.dispute_rate or 0
        elif code == "FEEDBACK_RATE":
            values[code] = profile.feedback_rate or 0
        elif code == "ACCOUNT_AGE":
            values[code] = days_since(profile.joined_at)
        elif code == "LAST_ACTIVITY":
            values[code] = days_since(profile.last_order_date)
        else:
            # Custom metric: use formula or source_field
            values[code] = evaluate_custom_metric(profile, metric)

    return values


def days_since(date_value):
    """Return number of days since a given date. Returns 0 if date is None."""
    if not date_value:
        return 0
    from frappe.utils import date_diff, today
    return max(0, date_diff(today(), date_value))


def calculate_order_frequency(buyer_name, months=6):
    """
    Calculate average orders per month over the last N months.
    Consistent with 129.md calculation (Section 4.5).
    """
    from_date = frappe.utils.add_months(frappe.utils.today(), -months)

    order_count = frappe.db.count("Sales Order", filters={
        "buyer": buyer_name,
        "status": ("in", ["Completed", "Delivered"]),
        "transaction_date": (">=", from_date),
    })

    return round(order_count / months, 2) if months > 0 else 0.0
```

```python
def resolve_template_for_buyer(buyer_name):
```

### 4.7 Weight Redistribution Helper

When adding or removing metrics, the admin may need help redistributing weights:

```python
@frappe.whitelist()
def redistribute_weights(template_name, method="equal"):
    """
    Redistribute metric weights across all active metrics.

    Args:
        template_name: Buyer KPI Template name.
        method: "equal" for equal distribution, "proportional" to maintain ratios.

    Returns:
        Updated weight values.
    """
    template = frappe.get_doc("Buyer KPI Template", template_name)
    active_metrics = [m for m in template.metrics if m.is_active]

    if not active_metrics:
        frappe.throw(_("No active metrics to distribute weights to."))

    if method == "equal":
        base_weight = round(100.0 / len(active_metrics), 2)
        remainder = round(100.0 - base_weight * len(active_metrics), 2)

        for i, metric in enumerate(active_metrics):
            metric.weight_pct = base_weight
            if i == 0:
                metric.weight_pct += remainder  # Add remainder to first metric

    elif method == "proportional":
        current_total = sum(m.weight_pct for m in active_metrics)
        if current_total > 0:
            for metric in active_metrics:
                metric.weight_pct = round(
                    (metric.weight_pct / current_total) * 100.0, 2
                )
            # Fix rounding to ensure exact 100%
            diff = 100.0 - sum(m.weight_pct for m in active_metrics)
            if active_metrics:
                active_metrics[0].weight_pct = round(
                    active_metrics[0].weight_pct + diff, 2
                )

    template.save()
    return {m.metric_code: m.weight_pct for m in active_metrics}
```

### 4.8 Scoring Curve Application

```python
def apply_scoring_curve(raw_score, curve_type):
    """
    Apply a scoring curve to the weighted sum.

    Args:
        raw_score: Weighted sum of normalized metric scores (0-100).
        curve_type: One of Linear, Bell Curve, Exponential, Step.

    Returns:
        Adjusted score (0-100).
    """
    import math

    if curve_type == "Linear":
        return raw_score

    elif curve_type == "Bell Curve":
        # Gaussian bell: rewards scores near 70-80, compresses extremes
        mean, std = 70, 15
        z = (raw_score - mean) / std
        bell = math.exp(-0.5 * z * z)
        return raw_score * (0.7 + 0.3 * bell)

    elif curve_type == "Exponential":
        # Rewards high scores disproportionately
        return 100 * (raw_score / 100) ** 0.8

    elif curve_type == "Step":
        # Discrete step function
        if raw_score >= 85:
            return 95
        elif raw_score >= 70:
            return 80
        elif raw_score >= 55:
            return 65
        elif raw_score >= 40:
            return 50
        elif raw_score >= 25:
            return 35
        else:
            return 15

    return raw_score
```

### 4.9 Edge Cases

#### New Users with No Data

Buyers who have not met the `min_orders_required` or `min_days_active` thresholds are not evaluated. Their KPI score is `null` (not 0), and they receive a "Pending Evaluation" status:

```python
if profile.total_orders < template.min_orders_required:
    return {
        "overall_score": None,
        "status": "insufficient_orders",
        "message": f"Buyer needs {template.min_orders_required - profile.total_orders} "
                   f"more orders before evaluation."
    }
```

#### Division by Zero

All metric calculations guard against division by zero:
- `order_frequency`: Returns 0 if `months == 0`
- `return_rate`, `dispute_rate`, `feedback_rate`: Return 0 if `total_orders == 0`
- `average_order_value`: Returns 0 if `total_orders == 0`

#### Weight Rounding

When redistributing weights, floating-point rounding may cause the sum to be 99.99 or 100.01. The system:
1. Rounds all weights to 2 decimal places.
2. Calculates the difference from 100.00.
3. Adds/subtracts the difference to/from the first active metric.
4. Validates that `abs(total_weight - 100.0) <= 0.01`.

#### Metric Type Validation

Each `metric_type` constrains the expected value range:

| metric_type | Value Range | Normalization Notes |
|------------|------------|---------------------|
| Currency | 0 to unlimited | Logarithmic normalization recommended for large ranges |
| Percent | 0 to 100 | Direct mapping or linear interpolation between thresholds |
| Number | 0 to unlimited | Context-dependent normalization |
| Rating | 0 to 5 | Scale to 0-100 (multiply by 20) |
| Duration | 0 to unlimited (days) | Logarithmic for long durations, linear for short |

### 5.1 Template Usage Statistics Update

The buyer KPI template system does not run its own scoring job -- that responsibility belongs to the scoring engine described in `132.md` and the dashboard scheduled job in `135.md`. However, the template system maintains usage statistics:

**Job Name**: `tradehub_core.tradehub_core.tasks.update_buyer_kpi_template_stats`
**Frequency**: Weekly (Sunday 06:00 UTC)
**Cron Expression**: `0 6 * * 0`

```python
def update_buyer_kpi_template_stats():
    """
    Update usage_count, average_score, and last_evaluated_at
    for all active Buyer KPI Templates.

    Reads from: Buyer KPI Score Log (defined in 135.md)
    Writes to: Buyer KPI Template (usage_count, average_score, last_evaluated_at)
    """
    templates = frappe.get_all(
        "Buyer KPI Template",
        filters={"status": "Active"},
        fields=["name"]
    )

    for template in templates:
        try:
            stats = frappe.db.sql("""
                SELECT
                    COUNT(*) as usage_count,
                    AVG(overall_score) as avg_score,
                    MAX(evaluation_date) as last_eval
                FROM `tabBuyer KPI Score Log`
                WHERE template = %s
                  AND docstatus = 1
            """, template.name, as_dict=True)[0]

            frappe.db.set_value(
                "Buyer KPI Template", template.name,
                {
                    "usage_count": stats.usage_count or 0,
                    "average_score": round(stats.avg_score or 0, 2),
                    "last_evaluated_at": stats.last_eval,
                },
                update_modified=False
            )
            frappe.db.commit()

        except Exception:
            frappe.log_error(
                f"Failed to update stats for template {template.name}",
                "Buyer KPI Template Stats Update"
            )
            frappe.db.rollback()
```

**Runtime Estimate**: < 5 seconds (one aggregate query per active template; typically 1-10 templates).
**Error Handling**: Per-template try/except with rollback. Errors logged to Error Log. Subsequent templates continue processing.

```python
            "metric_name": (a.metric_name if a else b.metric_name),
            "weight_a": a.weight_pct if a else None,
            "weight_b": b.weight_pct if b else None,
            "difference": (
                (b.weight_pct if b else 0) - (a.weight_pct if a else 0)
            ),
            "in_a": bool(a),
            "in_b": bool(b),
            "threshold_diff": {
                "poor": (a.threshold_poor if a else None, b.threshold_poor if b else None),
                "fair": (a.threshold_fair if a else None, b.threshold_fair if b else None),
                "good": (a.threshold_good if a else None, b.threshold_good if b else None),
                "excellent": (a.threshold_excellent if a else None, b.threshold_excellent if b else None),
            }
        })

    return {
        "template_a": {"name": doc_a.name, "display_name": doc_a.display_name or doc_a.template_name},
        "template_b": {"name": doc_b.name, "display_name": doc_b.display_name or doc_b.template_name},
        "header_diff": {
            "passing_score": (doc_a.passing_score, doc_b.passing_score),
            "evaluation_period": (doc_a.evaluation_period, doc_b.evaluation_period),
            "evaluation_frequency": (doc_a.evaluation_frequency, doc_b.evaluation_frequency),
            "min_orders_required": (doc_a.min_orders_required, doc_b.min_orders_required),
        },
        "metrics": comparison,
    }
```

### 8.4 Weight Comparison Across Documents

The default template weights in this document differ from the grading weights in `129.md` by design:

| Metric | 133.md (KPI Template) | 129.md (Grading) | 132.md (Scoring) | Rationale for Difference |
|--------|----------------------|-------------------|-------------------|------------------------|
| Total Spend | 20% | 25% | 10% | KPI balances spend against behavior; grading prioritizes high-value customers |
| Order Frequency | 15% | 20% | 15% | Consistent between KPI and scoring; grading values loyalty more |
| On-Time Payment | 15% | 15% | 20% | Consistent with grading; scoring weighs payment reliability highest |
| Return/Cancel Rate | 15% | 15% | 15% | Consistent across all three systems |
| Avg Order Value | 10% | - | 5% | KPI includes AOV; grading rolls it into spend; scoring uses lower weight |
| Dispute Rate | 10% | 10% | 15% | Consistent with grading; scoring treats disputes more severely |
| Feedback Rate | 5% | 5% | 10% | Consistent with grading; scoring values engagement more |
| Account Age | 5% | 10% | 5% | KPI and scoring use lower weight; grading values tenure more |
| Last Activity | 5% | - | 5% | KPI and scoring include recency; grading doesn't track separately |

**Note**: Different weight distributions serve different purposes:
- **133.md (KPI Template)**: Balanced evaluation across 9 dimensions for buyer performance monitoring.
- **129.md (Grading)**: 7 criteria focused on customer value and reliability for letter grade assignment.
- **132.md (Scoring)**: 9 metrics focused on platform risk and reliability for system integration (search, commission, etc.).
