# Task 120: Mutual Transparent Data Sharing Planning Document

## Overview

This document specifies the Mutual Transparent Data Sharing system for the TradeHub marketplace platform. The system enables trust-building between buyers and sellers by providing structured, consent-based visibility into each party's reliability metrics, verification status, and transaction history. It introduces **two new DocTypes** (Seller Transparency Profile, Buyer Transparency Profile), a **child DocType** (Seller Certificate), and a **Data Sharing Preference** DocType for granular privacy control. The design is fully compliant with **KVKK** (6698 Sayili Kisisel Verilerin Korunmasi Kanunu) and **GDPR** (General Data Protection Regulation), with progressive disclosure stages, data masking rules, and consent withdrawal handling. Existing fields on Buyer Profile (e.g., `total_orders`, `payment_on_time_rate`, `verification_status`) and Seller Profile (e.g., `seller_score`, `average_rating`, `return_rate`) are referenced -- not duplicated -- via `fetch_from` to maintain a single source of truth.

### 5.6 Buyer Tier Computation Logic

```python
def compute_buyer_tier(total_orders, payment_on_time_rate):
    """
    Tier is primarily based on order count, with a payment reliability gate.
    If payment_on_time_rate < 70%, tier is capped at Bronze regardless of orders.
    """
    if total_orders <= 5:
        return "New"
    elif total_orders <= 20:
        tier = "Bronze"
    elif total_orders <= 50:
        tier = "Silver"
    elif total_orders <= 100:
        tier = "Gold"
    else:
        tier = "Platinum"

    # Payment reliability gate
    if payment_on_time_rate < 70 and tier in ("Silver", "Gold", "Platinum"):
        return "Bronze"

    return tier
```

### 8.2 Data Masking Rules

Personal data is masked based on the relationship stage between buyer and seller. Masking rules apply to both API responses and UI rendering.

**Masking Function (Server-Side):**

```python
def mask_email(email):
    """a*****z@f***a.com pattern"""
    if not email:
        return ""
    local, domain = email.split("@")
    domain_name, domain_ext = domain.rsplit(".", 1)
    masked_local = local[0] + "*" * (len(local) - 2) + local[-1] if len(local) > 2 else local[0] + "*"
    masked_domain = domain_name[0] + "*" * (len(domain_name) - 2) + domain_name[-1] if len(domain_name) > 2 else domain_name
    return f"{masked_local}@{masked_domain}.{domain_ext}"

def mask_phone(phone):
    """+90 *** *** **67 pattern - show country code and last 2 digits"""
    if not phone:
        return ""
    digits = ''.join(filter(str.isdigit, phone))
    if len(digits) < 4:
        return "***"
    country_code = phone[:3] if phone.startswith("+") else ""
    return f"{country_code} *** *** **{digits[-2:]}"

def mask_tax_id(tax_id, tax_id_type="VKN"):
    """VKN: 12****7890, TCKN: 123*****901"""
    if not tax_id:
        return ""
    if tax_id_type == "TCKN" and len(tax_id) >= 11:
        return tax_id[:3] + "*" * 5 + tax_id[-3:]
    elif len(tax_id) >= 10:
        return tax_id[:2] + "*" * 4 + tax_id[-4:]
    return "***"
```

### 8.3 Progressive Disclosure Stages

Data visibility changes based on the transaction relationship between buyer and seller. The progressive disclosure model defines three stages.

#### Stage 1: Pre-Order (No Transaction History)

The buyer and seller have no active or completed orders between them.

| Data Category | Buyer Sees (about Seller) | Seller Sees (about Buyer) |
|--------------|--------------------------|--------------------------|
| Company display name | Yes | No (buyer is anonymous per Task 121) |
| Company legal name | No | No |
| Verification badges | Tax Verified/Not Verified badge | Buyer Tier badge only |
| Seller score / rating | Yes | N/A |
| Performance metrics | Aggregated only (fulfillment %, delivery %) | N/A |
| Contact info | No | No |
| Certificates | Public certificates only | N/A |
| Order statistics | Visible if toggled on | Visible if toggled on |
| Capacity info | Visible if toggled on | N/A |

#### Stage 2: Active Order (Order Placed but Not Delivered)

There is at least one active (not yet delivered) order between the parties.

| Data Category | Buyer Sees (about Seller) | Seller Sees (about Buyer) |
|--------------|--------------------------|--------------------------|
| Company display name | Yes | Anonymous Buyer ID + Region |
| Company legal name | No | No |
| Verification badges | Full verification status details | Buyer Tier + Verification badge |
| Seller score / rating | Yes + trend indicator | N/A |
| Performance metrics | Full details (all rates) | Buyer's payment on-time rate, cancellation rate |
| Contact info | Masked email/phone | No (platform messaging only) |
| Tax ID | Masked (partial) | No |
| Certificates | All (including non-public) | N/A |
| Messaging | Full (through platform) | Full (buyer identity masked) |
| Order status | Real-time | Real-time |

#### Stage 3: Post-Delivery (Order Completed/Delivered)

At least one order has been successfully delivered between the parties.

| Data Category | Buyer Sees (about Seller) | Seller Sees (about Buyer) |
|--------------|--------------------------|--------------------------|
| Company display name | Yes | Anonymous Buyer ID + Region (per Task 121) |
| Company legal name | Yes (if consented) | No |
| Verification badges | Full details with dates | Buyer Tier + Full verification details |
| All performance metrics | Yes | Buyer's full reliability metrics (if consented) |
| Contact info | Full (if consented) | No (platform messaging only, per Task 121) |
| Tax ID | Full (if consented, for B2B invoicing) | No |
| Certificates & Audits | All | N/A |
| Messaging history | Full historical | Full historical (buyer identity masked) |
| Dispute history | Mutual disputes visible | Mutual disputes visible (buyer ID masked) |

**Stage Determination Logic:**

```python
def get_disclosure_stage(buyer, seller):
    """
    Determine progressive disclosure stage between a buyer and seller.
    Returns: 'pre_order', 'active_order', or 'post_delivery'
    """
    from frappe import db

    # Check for completed/delivered orders
    delivered = db.exists("Marketplace Order", {
        "buyer": buyer,
        "seller": seller,
        "status": ["in", ["Delivered", "Completed"]]
    })
    if delivered:
        return "post_delivery"

    # Check for active orders
    active = db.exists("Marketplace Order", {
        "buyer": buyer,
        "seller": seller,
        "status": ["in", ["Placed", "Confirmed", "Shipped", "Processing"]]
    })
    if active:
        return "active_order"

    return "pre_order"
```

### 8.4 Data Sharing Preferences Mechanism

#### 8.4.1 Onboarding Flow

### 8.5 Consent Withdrawal Handling

When a user withdraws consent (changes a Data Sharing Preference from "Opted In" to "Opted Out"), the following process executes:

#### 8.5.1 Immediate Actions (Within Seconds)

1. Set the corresponding privacy toggle to `0` (hidden)
2. Update the Data Sharing Preference record with `opted_out_at = now()`, `sharing_status = "Opted Out"`
3. Create/update the Consent Record with `status = "Revoked"`, `revoked_at = now()`
4. Log the withdrawal to Consent Audit Log

#### 8.5.2 Propagation Actions (Within 5 Minutes)

1. All cached API responses containing the withdrawn data category are invalidated
2. Any active session displaying the data receives a real-time update (via Frappe's socket) to mask the withdrawn data
3. Ongoing conversations continue but the withdrawn data is no longer accessible in the profile view

#### 8.5.3 Cleanup Actions (Within 72 Hours)

1. If `Contact Information` is withdrawn: mask email/phone in all non-essential records (historical messages retain masked versions)
2. If `Company Information` is withdrawn: remove legal name from public profile; display name switches to anonymized form
3. Anonymization is logged in the Consent Audit Log for regulatory compliance

#### 8.5.4 Mid-Transaction Consent Withdrawal

When a user withdraws consent while an active order exists:

| Scenario | Behavior |
|----------|----------|
| Seller withdraws company info during active order | Company **display** name remains visible to buyer for the active order (contractual obligation per KVKK Madde 5/2-c); masked for all future browsing |
| Seller withdraws contact info during active order | Contact info remains accessible through platform messaging only; direct contact masked immediately |
| Buyer withdraws metrics during active order | Buyer tier badge remains visible to seller for the active order; detailed metrics masked immediately |
| Either party withdraws during dispute | All data relevant to the dispute is preserved per legal retention requirements (KVKK Madde 5/2-e); masked from general profile view |

**Legal Basis for Retention During Active Transactions:**

Per KVKK Madde 5, paragraf 2:
- **(c)** Processing is necessary for the performance of a contract (active order)
- **(e)** Processing is necessary for the establishment, exercise, or defence of legal claims (active dispute)
- **(f)** Processing is necessary for the legitimate interests of the data controller (fraud prevention during active order)

## 12. Edge Cases

### 12.1 New Seller with Zero Metrics

**Scenario:** A newly registered seller has no transaction history. All metrics (seller_score, average_rating, fulfillment rates) are at their defaults.

**Handling:**
- Display a "New Seller" badge instead of numerical metrics
- `seller_score` defaults to `0`, displayed as "Not yet rated"
- `average_rating` defaults to `0`, displayed as "No reviews yet"
- `order_fulfillment_rate` defaults to `100`, but hidden until at least 5 orders completed
- `membership_duration_days` is computed normally (may be 0 or 1)
- `active_product_count` may be 0 if no listings yet -- display "Setting up store"
- A threshold of **minimum 5 completed orders** is required before performance percentages are shown publicly
- The Seller Transparency Profile is still created but with a "New Seller" indicator

```python
MINIMUM_ORDERS_FOR_METRICS_DISPLAY = 5

def should_show_metrics(seller_profile):
    return seller_profile.total_sales_count >= MINIMUM_ORDERS_FOR_METRICS_DISPLAY
```

### 12.2 Consent Withdrawal Mid-Transaction

**Scenario:** A seller opts out of sharing "Performance Metrics" while a buyer has an active order with them.

**Handling:**
1. **Active order context:** The buyer retains access to the performance metrics that were visible at the time of order placement (snapshot principle). This is legally justified under KVKK Madde 5/2-c (contractual necessity).
2. **Snapshot mechanism:** When an order is placed, the system takes a point-in-time snapshot of the seller's transparency data visible to the buyer. This snapshot is stored as a JSON field on the Marketplace Order record.
3. **New browsing:** If the same buyer browses the seller's profile outside the order context, the metrics are immediately hidden.
4. **Future orders:** Any new order will show the updated (reduced) transparency level.
5. **Audit trail:** The consent withdrawal and its interaction with the active order are logged in the Consent Audit Log.

### 12.3 Buyer Transparency with Zero Orders

**Scenario:** A newly registered buyer who has never placed an order.

**Handling:**
- `buyer_tier` = "New"
- All rate metrics (`payment_on_time_rate`, `cancellation_rate`, `return_rate`, `feedback_rate`) display as "N/A" or "No history"
- `membership_duration_days` is computed normally
- `verification_status_display` shows the current verification state
- Sellers see a "New Buyer" badge alongside the anonymous buyer ID

### 12.4 Seller Verification Status Changes

**Scenario:** A seller's tax verification changes from "Verified" to "Failed" after an audit.

**Handling:**
1. The `tax_verification_status` on Seller Transparency Profile is updated by admin
2. All cached transparency API responses for this seller are invalidated
3. Buyers with active orders are notified via Notification Log
4. The change is logged in the Consent Audit Log
5. If the seller had `show_tax_verification` enabled, buyers immediately see the updated badge

### 12.5 Data Sharing Preference Reset

**Scenario:** A user requests a complete reset of all Data Sharing Preferences to defaults.

**Handling:**
1. All Data Sharing Preference records for the user are set to `sharing_status = "Not Set"`
2. All privacy toggles on the Transparency Profile are reset to defaults
3. Consent Records for the affected categories are updated to "Revoked"
4. New Consent Records are not created until the user explicitly opts in again
5. The reset is logged as a single batch operation in the Consent Audit Log

### 12.6 Concurrent Profile Access During Metric Refresh

**Scenario:** A buyer requests a seller's transparency profile while the daily metric refresh is running.

**Handling:**
- Use the `last_refreshed` timestamp to serve the most recently completed snapshot
- Do not serve partially refreshed data
- If refresh is in progress, serve the previous snapshot
- Implement optimistic locking: `before_save` checks `last_refreshed` to prevent stale overwrites

### 12.7 Deleted or Suspended Seller/Buyer Profile

**Scenario:** A Seller Profile or Buyer Profile is suspended or deleted while a Transparency Profile exists.

**Handling:**
- **Suspended:** Transparency Profile status changes to "Suspended"; API returns empty data with `"status": "suspended"` message
- **Deleted:** Transparency Profile is archived (never hard-deleted for audit trail); `status = "Archived"`; Data Sharing Preferences are set to "Opted Out"
- In both cases, Consent Records remain for regulatory compliance (KVKK retention requirements)

## 13. Cross-References

### 13.1 Task 121 (Asymmetric Visibility) Dependencies

- **Buyer anonymity:** This document's Buyer Transparency Profile is always presented to sellers through the anonymization layer defined in Task 121. Sellers never see the buyer's real name, email, or phone.
- **Masked messaging:** The messaging history described in Section 7.2 uses the masked messaging system from Task 121.
- **Disclosure stages:** The progressive disclosure stages in Section 8.3 must be consistent with Task 121's visibility rules. Specifically:
  - Pre-order: Seller sees nothing about buyer (consistent with Task 121's "anonymous until order" rule)
  - Active-order: Seller sees buyer tier + anonymous ID (consistent with Task 121's order-level visibility)
  - Post-delivery: Seller still uses anonymous buyer ID (Task 121 rule: seller NEVER sees buyer identity)

### 13.2 Task 118 (Favorites & Wishlist) Dependencies

- The `active_product_count` on Seller Transparency Profile may be cross-referenced with wishlist/favorite counts from Task 118 for social proof aggregation.
- Sellers can see aggregated wishlist save counts (e.g., "Products saved to wishlists: 1,240") without knowing which buyers saved them.

### 13.3 Task 150 (Buy Box Algorithm) Dependencies

- The Seller Transparency Profile metrics (`seller_score`, `average_rating`, `return_rate`, `cancellation_rate`, `on_time_delivery_rate`) are the same metrics used by the Buy Box scoring algorithm in Task 150.
- Both the Transparency Profile and Buy Box Entry source their data from Seller Profile fields, ensuring consistency.
- The `active_product_count` reflects the same listing data used for Buy Box eligibility checks.
