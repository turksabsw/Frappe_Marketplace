# 147 - Seller Panel Order Document Creation Plan

## Overview

This document defines the complete **Seller Panel Create dropdown** action system for the Sub Order DocType in TradeHub. It specifies how sellers, operating through their panel, can create linked downstream documents (invoices, delivery notes, shipments, etc.) scoped to their own orders, with role-based visibility and ownership validation.

**This document explicitly extends 125.md** (Marketplace Order Actions / Create Button Plan). While 125.md defines the platform-wide Create actions on the Marketplace Order, this document (147.md) defines the seller-scoped equivalent on the **Sub Order** DocType with the following key differences:

| Aspect | 125.md (Marketplace Order) | 147.md (Seller Panel / Sub Order) |
|--------|---------------------------|-----------------------------------|
| **Source DocType** | Marketplace Order | Sub Order |
| **Actors** | System Manager, Marketplace Admin, Accounts/Stock roles | Seller role (linked to Seller Profile) |
| **Scope** | All orders, all sellers | Only seller's own Sub Orders |
| **Create Options** | 7 (SI, DN, PE, PR, Shipment, Return, Dispute) | 8 (adds **Proforma Invoice**; removes Dispute) |
| **Permission Model** | Role-based (Frappe standard roles) | Seller field-based filtering + role-based |
| **Ownership Check** | `frappe.has_permission` standard | Seller Profile ownership validation in every `@frappe.whitelist()` method |
| **Form Pre-fill** | Maps from Marketplace Order | Maps from Sub Order (seller's items only) |
| **Dashboard** | Platform-wide linked docs | Seller-scoped linked docs with filtered counts |

**Reference Codebase:**
- `marketplace_order.py` - Existing `get_seller_orders()` API demonstrates seller-scoped filtering pattern
- `sub_order.json` - Sub Order DocType with seller field, fulfillment tracking, invoice/payment sections
- `seller_profile.py` - Seller role assignment (`assign_seller_role`), User Permission for tenant isolation (`create_user_permission_for_tenant`)
- `marketplace_order.js` - Existing `frm.add_custom_button` patterns

**Cross-References:**
- **125.md** (Marketplace Order Actions) - This document is the seller-panel extension of 125.md
- **126.md** (Partial Shipment Tracking) - Delivery Note creation here follows 126.md's partial shipment rules
- **123.md** (Seller-Grouped Checkout) - Sub Orders are created per seller during checkout
- **117.md** (Payment Terms) - Proforma Invoice and Payment Request integrate with payment terms

## 1. Seller Panel Context

### 1.1 Sub Order as the Seller's Primary Order View

In TradeHub, when a buyer places a Marketplace Order containing items from multiple sellers, the system automatically creates a **Sub Order** for each seller (see `create_sub_orders()` in `marketplace_order.py`). The Sub Order is the seller's primary order document:

- **Marketplace Order** = buyer's view (all items from all sellers)
- **Sub Order** = seller's view (only their items from the parent order)

The Sub Order already has its own:
- `seller` field (Link to Seller Profile, required)
- `status` field (with same lifecycle statuses)
- Fulfillment tracking (`fulfillment_status`, `items_packed`, `items_shipped`, `items_delivered`, `items_returned`)
- Invoice tracking (`requires_invoice`, `invoice_type`, `e_invoice_enabled`, `e_invoice_status`, `e_invoice_uuid`, `e_invoice_number`)
- ERPNext links (`erpnext_sales_order`, `erpnext_delivery_note`, `erpnext_sales_invoice`, `erpnext_payment_entry`)
- Payment tracking (`payment_status`, `paid_at`, `escrow_status`, `escrow_release_date`)
- Commission tracking (`commission_rate`, `commission_amount`, `seller_payout`, `payout_status`)

### 1.2 Seller Role and Identification

Sellers are identified through the chain:

```
User (frappe.session.user)
    -> Seller Profile (user field = session user)
        -> Sub Order (seller field = Seller Profile name)
```

The existing `seller_profile.py` already:
1. Assigns the `"Seller"` role to the user (`assign_seller_role()`)
2. Creates User Permission for tenant isolation (`create_user_permission_for_tenant()`)
3. Tracks seller capabilities (`can_sell`, `can_withdraw`, `can_create_listings`)

### 1.3 What 147.md Adds Over 125.md

| Addition | Description |
|----------|-------------|
| **Proforma Invoice** | New Create option not in 125.md; seller can generate a proforma before formal invoicing |
| **Seller Ownership Validation** | Every `@frappe.whitelist()` method validates that `session.user` owns the Sub Order's seller |
| **Seller-Scoped Field Filtering** | Link field queries filter by seller's own Sub Orders only |
| **Role-Based Button Visibility** | Buttons only appear when user has `"Seller"` role AND owns the Sub Order |
| **Seller-Specific Pre-fill** | Forms pre-filled with seller's company info, address, bank details |
| **Commission-Aware Invoicing** | Invoice amounts reflect seller's payout (excluding commission) |
| **Dispute Removed** | Sellers cannot file disputes against their own orders; disputes are buyer-initiated (see 125.md) |

## 3. Create Dropdown Options (Seller Panel)

### 3.1 Complete Create Button List

The following actions appear in the **Create** dropdown button group on the Sub Order form when the logged-in user is the owning seller. This is a **superset** of 125.md's options, adding Proforma Invoice and removing Dispute.

| # | Create Action | Target DocType | Conditional Visibility | Status Auto-Advance | Added in 147 (vs 125) |
|---|---------------|----------------|------------------------|---------------------|------------------------|
| 1 | **Proforma Invoice** | Proforma Invoice | `status` in `[Accepted, Processing]` AND no existing Proforma | No auto-advance | **NEW** (not in 125.md) |
| 2 | Sales Invoice | Sales Invoice | `status` in `[Accepted, Processing, Packed]` AND no existing SI | `Accepted` -> `Processing` | Same as 125.md #1 |
| 3 | Delivery Note | Delivery Note | `status` in `[Processing, Packed]` AND SI exists AND not fully delivered | No auto-advance (see 126.md) | Same as 125.md #2 |
| 4 | Payment Request | Payment Request | `status` in `[Accepted, Processing]` AND `payment_status` in `[Pending, Partially Paid]` | No auto-advance | Same as 125.md #4 |
| 5 | Shipment | Shipment | `status` in `[Processing, Packed]` AND Delivery Note exists | `Processing/Packed` -> `Shipped` | Same as 125.md #5 |

**Removed from seller panel (vs 125.md):**
- **Payment Entry** (#3 in 125.md) - Sellers do not record buyer payments; this is a platform/admin action

### 3.2 Conditional Visibility Rules Detail

All visibility checks include the **seller ownership guard** in addition to status checks:

```javascript
// Base seller ownership check (applies to ALL buttons)
function is_seller_owner(frm) {
    return frm.doc.__seller_is_owner === true;
    // Set by server via has_permission or a custom flag
}
```

#### 3.2.1 Proforma Invoice (NEW - Not in 125.md)

**When Visible:**
- User is the owning seller (`is_seller_owner(frm)`)
- Sub Order status is `Accepted` or `Processing`
- No existing Proforma Invoice linked
- Seller's profile has `can_sell == 1`

**When Hidden:**
- Proforma Invoice already exists
- Status is beyond `Processing` (items already being fulfilled)
- Seller is suspended or blocked

**Purpose:**
Proforma Invoices are common in B2B Turkish commerce. Sellers issue them as a formal price quotation / pre-invoice before the actual Sales Invoice. This is especially important for:
- Large B2B orders requiring management approval
- International buyers needing import documentation
- Orders with custom payment terms (ref: 117.md)

```javascript
let can_create_proforma = is_seller_owner(frm)
    && ['Accepted', 'Processing'].includes(frm.doc.status)
    && !frm.doc.proforma_invoice
    && frm.doc.__seller_can_sell;
```

#### 3.2.2 Sales Invoice

**When Visible:**
- User is the owning seller
- Status is `Accepted`, `Processing`, or `Packed`
- No Sales Invoice already linked (`!frm.doc.erpnext_sales_invoice`)
- Payment status is not `Cancelled` or `Refunded`

**Difference from 125.md:** Operates on Sub Order (seller's items only), not full Marketplace Order. Invoice amount is the seller's subtotal, not the full order total.

```javascript
let can_create_si = is_seller_owner(frm)
    && ['Accepted', 'Processing', 'Packed'].includes(frm.doc.status)
    && !frm.doc.erpnext_sales_invoice
    && !['Cancelled', 'Refunded'].includes(frm.doc.payment_status);
```

#### 3.2.3 Delivery Note

**When Visible:**
- User is the owning seller
- Status is `Processing` or `Packed`
- Sales Invoice exists
- Not fully delivered (`!frm.doc.fully_fulfilled`)

**Difference from 125.md:** Only includes items belonging to this seller's Sub Order. Follows same partial shipment rules as 126.md.

```javascript
let can_create_dn = is_seller_owner(frm)
    && ['Processing', 'Packed'].includes(frm.doc.status)
    && frm.doc.erpnext_sales_invoice
    && !frm.doc.fully_fulfilled;
```

#### 3.2.4 Payment Request

**When Visible:**
- User is the owning seller
- Payment status is `Pending` or `Partially Paid`
- Status is `Accepted` or `Processing`
- No active (unsettled) Payment Request exists

**Difference from 125.md:** Seller sends payment request to the buyer for their portion of the order. Amount is the Sub Order grand total, not the full Marketplace Order total.

```javascript
let can_create_pr = is_seller_owner(frm)
    && ['Pending', 'Partially Paid'].includes(frm.doc.payment_status)
    && ['Accepted', 'Processing'].includes(frm.doc.status);
```

#### 3.2.5 Shipment

**When Visible:**
- User is the owning seller
- Status is `Processing` or `Packed`
- At least one Delivery Note exists and is submitted

**Difference from 125.md:** Shipment is created from the seller's Sub Order, pre-filled with seller's pickup address.

```javascript
let can_create_shipment = is_seller_owner(frm)
    && ['Processing', 'Packed'].includes(frm.doc.status)
    && frm.doc.erpnext_delivery_note;
```

#### 3.2.6 Return / Credit Note

**When Visible:**
- User is the owning seller
- Status is `Delivered` or `Completed`
- Original Sales Invoice exists
- Return window has not expired

**Difference from 125.md:** Seller processes the return for their own items. Credit Note amount reflects seller's items only.

```javascript
let can_create_return = is_seller_owner(frm)
    && ['Delivered', 'Completed'].includes(frm.doc.status)
    && frm.doc.erpnext_sales_invoice;
```

## 4. Seller-Scoped Permission Rules

### 4.1 Permission Model Architecture

The seller panel uses a **three-layer permission model**:

```
Layer 1: Frappe Role Permissions (DocType-level)
    -> "Seller" role has read/write on Sub Order with conditions

Layer 2: User Permission (record-level via seller field)
    -> User Permission links User to their Seller Profile
    -> Frappe automatically filters Sub Orders where seller = their Seller Profile

Layer 3: Explicit Ownership Validation (in @frappe.whitelist methods)
    -> Every API method re-validates seller ownership before proceeding
```

### 4.2 Layer 1: DocType Role Permissions

Permissions defined in `sub_order.json`:

| Role | Read | Write | Create | Delete | Submit | Cancel | Amend | If Owner |
|------|:----:|:-----:|:------:|:------:|:------:|:------:|:-----:|:--------:|
| System Manager | Yes | Yes | Yes | Yes | Yes | Yes | Yes | No |
| Marketplace Admin | Yes | Yes | Yes | No | Yes | Yes | No | No |
| **Seller** | Yes | Yes | No | No | No | No | No | No |
| Accounts Manager | Yes | Yes | No | No | No | No | No | No |
| Stock User | Yes | Yes | No | No | No | No | No | No |

**Key:** Sellers have read + write but **not** create, submit, cancel, or delete. Sub Orders are system-created (by `create_sub_orders()` on Marketplace Order submit). Sellers can only update within allowed transitions.

```python
# NEW: Create User Permission for Seller Profile itself
def create_user_permission_for_seller(self):
    """
    Create User Permission linking the user to their Seller Profile.

    This enables Frappe's automatic record-level filtering:
    when a user with "Seller" role views Sub Order list, only
    Sub Orders where seller = their Seller Profile will be shown.
    """
    if not self.user:
        return

    if self.user == "Administrator":
        return

    if frappe.db.exists("User Permission", {
        "user": self.user,
        "allow": "Seller Profile",
        "for_value": self.name
    }):
        return

    try:
        perm = frappe.get_doc({
            "doctype": "User Permission",
            "user": self.user,
            "allow": "Seller Profile",
            "for_value": self.name,
            "apply_to_all_doctypes": 0,
            "applicable_for": "Sub Order",
            "is_default": 1,
        })
        perm.flags.ignore_permissions = True
        perm.insert()
    except Exception as e:
        frappe.log_error(
            f"Failed to create Seller Profile User Permission: {str(e)}",
            "Seller Permission Error"
        )
```

### 4.4 Layer 3: Explicit Ownership Validation

Every `@frappe.whitelist()` method in the seller panel includes this validation:

```python
def _validate_seller_ownership(sub_order_name):
    """
    Validate that the current user owns the Sub Order via their Seller Profile.

    This is the critical security gate. Even if a seller somehow gets the
    sub_order_name of another seller's order, this check prevents unauthorized
    access.

    Args:
        sub_order_name (str): Sub Order document name

    Returns:
        tuple: (sub_order Document, seller_profile str)

    Raises:
        frappe.PermissionError: If user is not the owning seller
    """
    sub_order = frappe.get_doc("Sub Order", sub_order_name)

    # Get seller profile for current user
    seller_profile = frappe.db.get_value(
        "Seller Profile",
        {"user": frappe.session.user},
        "name"
    )

    if not seller_profile:
        frappe.throw(
            _("You do not have a Seller Profile"),
            frappe.PermissionError
        )

    # Validate ownership
    if sub_order.seller != seller_profile:
        frappe.throw(
            _("You do not have permission to perform this action on this Sub Order"),
            frappe.PermissionError
        )

    # Validate seller is active
    seller_status = frappe.db.get_value("Seller Profile", seller_profile, "status")
    if seller_status not in ["Active"]:
        frappe.throw(
            _("Your seller account is not active. Current status: {0}").format(seller_status),
            frappe.PermissionError
        )

    return sub_order, seller_profile
```

On the Sub Order form, buttons are conditionally shown based on the user's role:


## 5. Python Whitelisted Methods (Seller-Scoped)

All methods are defined in a new file: `sub_order_seller_actions.py` in the Sub Order doctype package.

**Module Path:** `tradehub_commerce.tradehub_commerce.doctype.sub_order.sub_order_seller_actions`

### 5.1 check_seller_ownership

```python
@frappe.whitelist()
def check_seller_ownership(sub_order_name):
    """
    Check if the current user is the owner of the given Sub Order.

    Called from client-side JS to determine button visibility.

    Args:
        sub_order_name (str): Sub Order document name

    Returns:
        dict: {
            "is_owner": bool,
            "can_sell": bool,
            "seller_profile": str or None,
            "seller_name": str or None
        }
    """
    seller_profile = frappe.db.get_value(
        "Seller Profile",
        {"user": frappe.session.user},
        ["name", "seller_name", "can_sell", "status"],
        as_dict=True
    )

    if not seller_profile:
        return {"is_owner": False, "can_sell": False}

    sub_order_seller = frappe.db.get_value("Sub Order", sub_order_name, "seller")

    is_owner = (sub_order_seller == seller_profile.name)

    return {
        "is_owner": is_owner,
        "can_sell": cint(seller_profile.can_sell) if is_owner else False,
        "seller_profile": seller_profile.name if is_owner else None,
        "seller_name": seller_profile.seller_name if is_owner else None,
    }
```

### 5.2 seller_make_proforma_invoice (NEW - Not in 125.md)

```python
@frappe.whitelist()
def seller_make_proforma_invoice(source_name, target_doc=None):
    """
    Create a Proforma Invoice from a Sub Order (seller panel).

    THIS IS NEW IN 147.md - NOT PRESENT IN 125.md.

    Proforma Invoices are pre-invoices used in B2B Turkish commerce for:
    - Price quotation confirmation before formal invoicing
    - Import/export documentation
    - Orders with custom payment terms (ref: 117.md)

    Args:
        source_name (str): Sub Order name
        target_doc (Document, optional): Existing Proforma Invoice to map onto

    Returns:
        Document: Mapped Proforma Invoice document (unsaved)

    Conditions:
        - Current user must be the owning seller
        - Sub Order status must be in [Accepted, Processing]
        - No existing Proforma Invoice linked
        - Seller must be active (can_sell == 1)

    Status Effect:
        No automatic status change. Proforma Invoice is informational.
        Seller manually proceeds to create Sales Invoice after buyer confirms.

    Cross-Reference:
        - 125.md does NOT include Proforma Invoice (platform admin doesn't need it)
        - 117.md Payment Terms may dictate Proforma requirements
    """
    from frappe.model.mapper import get_mapped_doc

    # SELLER OWNERSHIP VALIDATION (Layer 3)
    sub_order, seller_profile = _validate_seller_ownership(source_name)

    # Validate status
    if sub_order.status not in ["Accepted", "Processing"]:
        frappe.throw(
            _("Proforma Invoice can only be created for Accepted or Processing orders")
        )

    # Check for existing proforma
    if sub_order.proforma_invoice:
        frappe.throw(
            _("Proforma Invoice {0} already exists for this order").format(
                sub_order.proforma_invoice
            )
        )

    def set_missing_values(source, target):
        """Set seller-specific values on the Proforma Invoice."""
        target.flags.ignore_permissions = True

        # Set seller's company info
        seller_doc = frappe.get_doc("Seller Profile", seller_profile)
        target.seller = seller_profile
        target.seller_name = seller_doc.seller_name
        target.seller_company = seller_doc.company_name or seller_doc.seller_name
        target.seller_tax_id = seller_doc.tax_id
        target.seller_address = _get_seller_address(seller_doc)

        # Set buyer info
        target.buyer = source.buyer
        target.buyer_name = source.buyer_name
        target.buyer_email = source.buyer_email
        target.buyer_tax_id = source.buyer_tax_id

        # Set amounts (seller's portion, excluding commission)
        target.subtotal = source.subtotal
        target.tax_amount = source.tax_amount
        target.shipping_amount = source.shipping_amount
        target.grand_total = source.grand_total
        target.currency = source.currency or "TRY"

        # Set dates
        target.posting_date = frappe.utils.nowdate()
        target.valid_until = frappe.utils.add_days(frappe.utils.nowdate(), 30)

        # Set payment terms if available (ref: 117.md)
        if source.payment_terms_template:
            target.payment_terms_template = source.payment_terms_template

        # Link back to sub order
        target.sub_order = source.name
        target.marketplace_order = source.marketplace_order
        target.marketplace_order_number = source.marketplace_order_number

        # Set proforma-specific flags
        target.is_proforma = 1
        target.proforma_status = "Draft"

    def update_item(source_doc, target_doc, source_parent):
        """Map Sub Order items to Proforma Invoice items."""
        target_doc.item_name = source_doc.title
        target_doc.description = source_doc.title
        target_doc.qty = source_doc.qty
        target_doc.rate = source_doc.unit_price
        target_doc.amount = source_doc.line_total
        target_doc.uom = source_doc.stock_uom or "Nos"
        target_doc.custom_listing = source_doc.listing
        target_doc.custom_sku = source_doc.sku

    return get_mapped_doc(
        "Sub Order",
        source_name,
        {
            "Sub Order": {
                "doctype": "Proforma Invoice",
                "field_map": {
                    "name": "sub_order",
                    "marketplace_order": "marketplace_order",
                    "seller": "seller",
                    "seller_name": "seller_name",
                    "buyer": "buyer",
                    "buyer_name": "buyer_name",
                    "currency": "currency",
                    "grand_total": "grand_total",
                },
                "validation": {
                    "status": ["in", ["Accepted", "Processing"]]
                }
            },
            "Sub Order Item": {
```

```python
def _get_or_create_customer_from_sub_order(sub_order):
    """
    Get or create ERPNext Customer from Sub Order buyer info.

    Delegates to parent Marketplace Order's customer lookup.

    Args:
        sub_order: Sub Order document

    Returns:
        str: Customer name/ID
    """
    # Try to get from parent marketplace order
    if sub_order.marketplace_order:
        mo_customer = frappe.db.get_value(
            "Marketplace Order",
            sub_order.marketplace_order,
            "erpnext_customer"
        )
        if mo_customer:
            return mo_customer

    # Fallback: use buyer info to find/create customer
    existing = frappe.db.get_value(
        "Customer",
        {"custom_marketplace_user": sub_order.buyer},
        "name"
    )
    if existing:
        return existing

    # Create minimal customer
    from tradehub_core.tradehub_core.utils.erpnext_sync import _get_or_create_customer_for_order
    mo = frappe.get_doc("Marketplace Order", sub_order.marketplace_order)
    return _get_or_create_customer_for_order(mo)


def _get_erpnext_item_code(order_item):
    """
    Get ERPNext Item Code for an order item.

    Same logic as 125.md Section 3.8 _get_erpnext_item_code.

    Args:
        order_item: Sub Order Item row

    Returns:
        str: ERPNext Item Code
    """
    if hasattr(order_item, 'erpnext_item_code') and order_item.erpnext_item_code:
        return order_item.erpnext_item_code

    if order_item.listing:
        listing = frappe.get_doc("Listing", order_item.listing)
        if listing.sku_product:
            sku = frappe.get_doc("SKU Product", listing.sku_product)
            if sku.erpnext_item_code:
                return sku.erpnext_item_code

    return order_item.sku or order_item.listing

        seller_doc.country if hasattr(seller_doc, 'country') else "Turkey",
    ]
    return "\n".join(filter(None, parts))

def _format_address_from_sub_order(sub_order):
    """
    Format shipping address from Sub Order as a display string.

    Args:
        sub_order: Sub Order document

    Returns:
        str: Formatted address string
    """
    parts = [
        sub_order.shipping_address_line1,
        sub_order.shipping_address_line2,
        sub_order.shipping_city,
        sub_order.shipping_state,
        sub_order.shipping_postal_code,
        sub_order.shipping_country,
    ]
    return "\n".join(filter(None, parts))


def _get_delivered_qty_for_sub_order(sub_order_name, sub_order_item_name):
    """
    Get total qty already delivered for a Sub Order item.

    Similar to 125.md Section 3.8 _get_delivered_qty but scoped to Sub Order.

    Args:
        sub_order_name: Sub Order name
        sub_order_item_name: Sub Order Item name (child row)

    Returns:
        float: Total delivered qty
    """
    result = frappe.db.sql("""
        SELECT IFNULL(SUM(dni.qty), 0) as delivered_qty
        FROM `tabDelivery Note Item` dni
        JOIN `tabDelivery Note` dn ON dn.name = dni.parent
        WHERE dn.custom_sub_order = %(sub_order_name)s
        AND dni.custom_sub_order_item = %(sub_order_item_name)s
        AND dn.docstatus = 1
    """, {
        "sub_order_name": sub_order_name,
        "sub_order_item_name": sub_order_item_name
    }, as_dict=True)

    return flt(result[0].delivered_qty) if result else 0
```

```python
def on_sales_invoice_submit_for_sub_order(doc, method):
    """
    When a Sales Invoice linked to a Sub Order is submitted:
    1. Update Sub Order's erpnext_sales_invoice field
    2. Advance Sub Order status (Accepted -> Processing)
    3. Update parent Marketplace Order's aggregated invoice status
    """
    if not doc.custom_sub_order:
        return

    sub_order = frappe.get_doc("Sub Order", doc.custom_sub_order)

    # Set link
    sub_order.db_set("erpnext_sales_invoice", doc.name)

    # Advance status
    if sub_order.status == "Accepted":
        sub_order.db_set("status", "Processing")
        sub_order.db_set("processing_started_at", frappe.utils.now_datetime())

    # Update e-invoice fields if applicable
    if doc.custom_e_invoice_enabled:
        sub_order.db_set("e_invoice_status", "Generated")
        sub_order.db_set("e_invoice_number", doc.name)
        sub_order.db_set("e_invoice_date", doc.posting_date)


def on_sales_invoice_cancel_for_sub_order(doc, method):
    """When a linked Sales Invoice is cancelled, clear the Sub Order link."""
    if not doc.custom_sub_order:
        return

    current_link = frappe.db.get_value(
        "Sub Order", doc.custom_sub_order, "erpnext_sales_invoice"
    )
    if current_link == doc.name:
        frappe.db.set_value("Sub Order", doc.custom_sub_order, "erpnext_sales_invoice", None)

        # Revert status if it was advanced due to this invoice
        sub_status = frappe.db.get_value("Sub Order", doc.custom_sub_order, "status")
        if sub_status == "Processing":
            frappe.db.set_value("Sub Order", doc.custom_sub_order, "status", "Accepted")


def on_delivery_note_submit_for_sub_order(doc, method):
    """
    When a Delivery Note linked to a Sub Order is submitted:
    1. Update Sub Order's erpnext_delivery_note field
    2. Update fulfillment counts on Sub Order
    3. Propagate fulfillment status to parent Marketplace Order
    """
    if not doc.custom_sub_order:
        return

    sub_order = frappe.get_doc("Sub Order", doc.custom_sub_order)

    # Set link (latest DN)
    sub_order.db_set("erpnext_delivery_note", doc.name)

    # Increment delivery note count
    current_count = frappe.db.get_value(
        "Sub Order", doc.custom_sub_order, "delivery_note_count"
    ) or 0
    sub_order.db_set("delivery_note_count", cint(current_count) + 1)

    # Update fulfillment tracking on Sub Order
    _update_sub_order_fulfillment(doc.custom_sub_order)

    # Propagate to parent Marketplace Order
    if sub_order.marketplace_order:
        mo = frappe.get_doc("Marketplace Order", sub_order.marketplace_order)
        mo.update_fulfillment_counts()


def on_delivery_note_cancel_for_sub_order(doc, method):
    """When a linked Delivery Note is cancelled, update Sub Order fulfillment."""
    if not doc.custom_sub_order:
        return

    # Recalculate fulfillment
    _update_sub_order_fulfillment(doc.custom_sub_order)

    # Update DN link to previous DN
    previous_dn = frappe.db.get_value(
        "Delivery Note",
        {
            "custom_sub_order": doc.custom_sub_order,
            "docstatus": 1,
            "name": ["!=", doc.name]
        },
        "name",
        order_by="creation desc"
    )
    frappe.db.set_value("Sub Order", doc.custom_sub_order, "erpnext_delivery_note", previous_dn)

    # Decrement count
    current_count = frappe.db.get_value(
        "Sub Order", doc.custom_sub_order, "delivery_note_count"
    ) or 0
    frappe.db.set_value(
        "Sub Order", doc.custom_sub_order, "delivery_note_count",
        max(0, cint(current_count) - 1)
    )


def _update_sub_order_fulfillment(sub_order_name):
    """
    Update fulfillment counts on a Sub Order based on linked Delivery Notes.

    Mirrors the Marketplace Order's update_fulfillment_counts() logic
    but scoped to the Sub Order's items.
    """
    sub_order = frappe.get_doc("Sub Order", sub_order_name)

    packed = 0
    shipped = 0
    delivered = 0
    returned = 0
    total_items = len(sub_order.items) if hasattr(sub_order, 'items') and sub_order.items else 0

    if total_items == 0:
        return

    for item in sub_order.items:
        status = item.fulfillment_status if hasattr(item, 'fulfillment_status') else "Pending"
        if status in ["Packed", "Shipped", "In Transit", "Out for Delivery", "Delivered"]:
            packed += 1
        if status in ["Shipped", "In Transit", "Out for Delivery", "Delivered"]:
            shipped += 1
        if status == "Delivered":
            delivered += 1
        if status == "Returned":
            returned += 1

    sub_order.db_set("items_packed", packed)
    sub_order.db_set("items_shipped", shipped)
    sub_order.db_set("items_delivered", delivered)
    sub_order.db_set("items_returned", returned)

    if delivered == total_items:
        sub_order.db_set("fully_fulfilled", 1)
        sub_order.db_set("fulfillment_status", "Delivered")
    elif delivered > 0:
        sub_order.db_set("fulfillment_status", "Partially Delivered")
    elif shipped == total_items:
        sub_order.db_set("fulfillment_status", "Shipped")
    elif shipped > 0:
        sub_order.db_set("fulfillment_status", "Partially Shipped")
    elif packed == total_items:
        sub_order.db_set("fulfillment_status", "Packed")
    elif packed > 0:
        sub_order.db_set("fulfillment_status", "Partially Packed")
```

### 9.3 Seller Status Update Method

```python
@frappe.whitelist()
def seller_update_status(sub_order_name, action, reason=None):
    """
    Update Sub Order status from seller panel.

    Validates seller ownership and allowed transitions before updating.

    Args:
        sub_order_name (str): Sub Order name
        action (str): Action (accept, process, pack, ship, deliver, complete, cancel)
        reason (str, optional): Reason for status change (required for cancel)

    Returns:
        dict: {"status": "success", "order_status": str, "message": str}
    """
    # SELLER OWNERSHIP VALIDATION
    sub_order, seller_profile = _validate_seller_ownership(sub_order_name)

    action_map = {
        "accept": {
            "from": ["Pending"],
            "to": "Accepted",
            "timestamp_field": "accepted_at",
        },
        "process": {
            "from": ["Accepted"],
            "to": "Processing",
            "timestamp_field": "processing_started_at",
        },
        "pack": {
            "from": ["Processing"],
            "to": "Packed",
            "timestamp_field": "packed_at",
        },
        "ship": {
            "from": ["Packed"],
            "to": "Shipped",
            "timestamp_field": "shipped_at",
        },
        "deliver": {
            "from": ["Shipped", "In Transit", "Out for Delivery"],
            "to": "Delivered",
            "timestamp_field": "delivered_at",
        },
        "complete": {
            "from": ["Delivered"],
            "to": "Completed",
            "timestamp_field": "completed_at",
        },
        "cancel": {
            "from": ["Pending", "Accepted"],
            "to": "Cancelled",
            "timestamp_field": "cancelled_at",
        },
    }

    if action not in action_map:
        frappe.throw(_("Invalid action: {0}").format(action))

    config = action_map[action]

    if sub_order.status not in config["from"]:
        frappe.throw(
            _("Cannot {0} order in {1} status").format(action, sub_order.status)
        )

    # Perform the status update
    sub_order.db_set("status", config["to"])
    sub_order.db_set(config["timestamp_field"], frappe.utils.now_datetime())

    # Handle cancellation specifics
    if action == "cancel":
        sub_order.db_set("cancellation_requested", 1)
        sub_order.db_set("cancellation_reason", reason or "Cancelled by seller")
        sub_order.db_set("cancellation_requested_at", frappe.utils.now_datetime())

    # Notify buyer of status change
    _notify_buyer_status_change(sub_order, config["to"])

    # Propagate to parent Marketplace Order if needed
    _propagate_status_to_marketplace_order(sub_order)

    return {
        "status": "success",
        "order_status": config["to"],
        "message": _("Order status updated to {0}").format(config["to"])
    }


def _notify_buyer_status_change(sub_order, new_status):
    """Send notification to buyer about Sub Order status change."""
    try:
        frappe.publish_realtime(
            "sub_order_status_change",
            {
                "sub_order": sub_order.name,
                "marketplace_order": sub_order.marketplace_order,
                "seller_name": sub_order.seller_name,
                "new_status": new_status,
            },
            user=sub_order.buyer
        )
    except Exception:
        pass  # Non-critical


def _propagate_status_to_marketplace_order(sub_order):
    """
    Update parent Marketplace Order status based on all Sub Order statuses.

    Rules:
    - If ALL Sub Orders are Completed -> Marketplace Order = Completed
    - If ANY Sub Order is Shipped -> Marketplace Order >= Shipped
    - If ALL Sub Orders are Cancelled -> Marketplace Order = Cancelled
    """
    if not sub_order.marketplace_order:
        return

    all_sub_orders = frappe.get_all(
        "Sub Order",
        filters={"marketplace_order": sub_order.marketplace_order},
        fields=["status"]
    )

    statuses = [so.status for so in all_sub_orders]

    if all(s == "Completed" for s in statuses):
        frappe.db.set_value(
            "Marketplace Order", sub_order.marketplace_order,
            "status", "Completed"
        )
    elif all(s == "Cancelled" for s in statuses):
        frappe.db.set_value(
            "Marketplace Order", sub_order.marketplace_order,
            "status", "Cancelled"
        )
```

## 11. Concurrency & Edge Cases

### 11.1 Concurrent Seller Actions

**Problem:** Seller has two browser tabs open and clicks "Create Sales Invoice" in both simultaneously.

**Solution:** Same locking pattern as 125.md Section 8.1:
1. `frappe.lock_doc("Sub Order", source_name)` in every `seller_make_*` method
2. Double-check link field after acquiring lock
3. Release lock in `finally` block

### 11.2 Seller Creates Invoice While Admin Creates on Marketplace Order

**Problem:** Seller creates a Sales Invoice on their Sub Order while an admin simultaneously creates one on the parent Marketplace Order (via 125.md's `make_sales_invoice`).

**Solution:**
- Sub Order invoice and Marketplace Order invoice are separate documents for separate scopes
- The `custom_sub_order` field distinguishes seller-created invoices from admin-created ones
- No conflict: both can coexist (Sub Order invoice is seller's perspective, MO invoice is platform's)

### 11.3 Seller Account Suspended Mid-Action

**Problem:** Seller's account is suspended between clicking the Create button and the server processing the request.

**Solution:** `_validate_seller_ownership()` checks `seller_status` in real-time:
```python
seller_status = frappe.db.get_value("Seller Profile", seller_profile, "status")
if seller_status not in ["Active"]:
    frappe.throw(_("Your seller account is not active"))
```

### 11.4 Multiple Partial Delivery Notes (Ref: 126.md)

Same as 125.md Section 8.2. The `erpnext_delivery_note` field on Sub Order stores only the latest DN. All DNs are tracked via `custom_sub_order` link field on each Delivery Note.

### 11.5 Seller Attempting to Access Other Seller's Sub Order

**Problem:** A seller manually constructs a URL to another seller's Sub Order form.

**Solution:** Three-layer defense:
1. **User Permission** (Layer 2) prevents Sub Order from loading
2. **`_validate_seller_ownership()`** (Layer 3) in every `@frappe.whitelist()` method
3. **`check_seller_ownership()`** returns `is_owner: false`, so no Create buttons are rendered

### 11.6 Return After Partial Delivery on Sub Order

If only some items were delivered:
- The Credit Note should only include items with `fulfillment_status == "Delivered"`
- The `seller_make_return_credit_note` method should add a `condition` to filter by delivered items
- Partial returns create a Credit Note with only the returned item quantities
