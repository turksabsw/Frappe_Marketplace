# Task 109 - PWA Compliance Plan

## Executive Summary

TradeHub's headless frontend currently has **no Progressive Web App (PWA) infrastructure** -- there is no `manifest.json`, no service worker file, and no PWA-related HTML meta tags. This plan provides a complete specification for making the TradeHub buyer/seller-facing frontend fully PWA-compliant, capable of passing the Lighthouse PWA audit, and installable on desktop and mobile devices. The plan covers the Web App Manifest, service worker lifecycle, per-resource cache strategies, offline fallback, Safari/iOS compatibility, HTTPS enforcement, and a full Lighthouse PWA checklist mapping. Given TradeHub's B2B context (Turkey-first, expanding to MENA and Turkic states), PWA compliance is strategically important: many users in target markets rely on mobile devices with intermittent connectivity, and an installable app-like experience reduces friction for repeat B2B buyers and sellers.

## Current State Analysis

### Existing PWA Infrastructure

A scan of the TradeHub codebase reveals **zero** PWA-related files:

| Artifact | Expected Location | Status |
|----------|------------------|--------|
| `manifest.json` / `manifest.webmanifest` | Frontend root (`/`) | **Missing** |
| Service Worker (`sw.js` / `service-worker.js`) | Frontend root (`/`) | **Missing** |
| `<link rel="manifest">` | `index.html` / base template | **Missing** |
| `<meta name="theme-color">` | `index.html` / base template | **Missing** |
| Apple mobile web app meta tags | `index.html` / base template | **Missing** |
| Offline fallback page (`offline.html`) | Frontend root or assets | **Missing** |
| PWA icons (192x192, 512x512) | `/icons/` or `/img/` | **Missing** |

### Frappe PWA App (Third-Party)

A "PWA" app exists on the Frappe Marketplace, but it is designed for the Frappe Desk (admin) interface, not for headless frontends. It adds a service worker to the Frappe Desk shell and is **not suitable** for TradeHub's custom buyer/seller-facing frontend. The TradeHub PWA implementation must be built from scratch for the headless frontend architecture.

### Frontend Architecture Context

TradeHub uses a **headless frontend** architecture:
- The backend is Frappe v15/ERPNext v15 serving a REST/JSON API
- The buyer/seller-facing frontend is a separate application (not Frappe Desk)
- The frontend has its own HTML entry point, build system, and asset pipeline
- PWA artifacts must be integrated into the frontend build, not into Frappe's asset system

#### Manifest Field Rationale

| Field | Value | Rationale |
|-------|-------|-----------|
| `name` | "TradeHub - B2B Wholesale Marketplace" | Full name shown in install dialog and app info (max ~45 chars recommended) |
| `short_name` | "TradeHub" | Shown on homescreen/launcher where space is limited (max 12 chars recommended) |
| `start_url` | "/?source=pwa" | Root URL with analytics parameter to track PWA launches vs browser visits |
| `scope` | "/" | Entire site is within PWA scope; navigating outside scope opens in browser |
| `display` | "standalone" | App-like experience without browser chrome (address bar, tabs). Alternatives: `fullscreen` (no status bar -- too immersive for B2B), `minimal-ui` (shows back/reload -- fallback for older browsers) |
| `orientation` | "any" | Allow both portrait and landscape; B2B users may use tablets in landscape |
| `theme_color` | "#1a56db" | Brand blue -- colors the status bar on Android and title bar on desktop |
| `background_color` | "#ffffff" | White -- shown during app launch splash screen before CSS loads |
| `lang` | "tr" | Default language is Turkish (primary market). Can be dynamically served per locale in future |
| `dir` | "ltr" | Left-to-right text direction. Must switch to "rtl" for Arabic locale variant |
| `categories` | ["business", "shopping"] | W3C standardized categories for app store discovery |
| `icons` | 8 sizes + 2 maskable | Covers all platforms: 72-512px for Android/Chrome, maskable for adaptive icons, 152px for iPad, 180px apple-touch-icon |
| `prefer_related_applications` | false | Do not prompt users to install a native app instead of the PWA |

#### Multi-Language Manifest Consideration

For Arabic (`ar`) locale, the manifest must serve `"dir": "rtl"` and translated `name`/`short_name`. Two approaches:

1. **Server-side dynamic manifest**: Serve manifest.json from a server route that detects `Accept-Language` header and returns localized content
2. **Multiple manifest files**: `manifest-tr.json`, `manifest-ar.json`, etc., with the HTML `<link rel="manifest">` pointing to the correct file based on user locale

**Recommendation**: Use approach 1 (dynamic manifest) since it requires no changes to HTML and centralizes locale logic. This aligns with the i18n plan (Task 106).

### 2. Service Worker Implementation Plan

The service worker (`sw.js`) must be placed at the frontend root (`/`) to have maximum scope over all pages. It handles three lifecycle events: `install`, `activate`, and `fetch`.

#### 2a. Cache Naming Convention

All caches use a versioned naming scheme to enable atomic cache invalidation during updates:

```javascript
const CACHE_VERSION = 'v1';
const STATIC_CACHE = `tradehub-static-${CACHE_VERSION}`;
const DYNAMIC_CACHE = `tradehub-dynamic-${CACHE_VERSION}`;
const API_CACHE = `tradehub-api-${CACHE_VERSION}`;
const OFFLINE_CACHE = `tradehub-offline-${CACHE_VERSION}`;
```

When `CACHE_VERSION` is incremented, the `activate` event cleans up all caches with old version prefixes.

#### 2b. Install Event

The `install` event fires when the browser detects a new or updated service worker. It pre-caches the "app shell" -- the minimum set of assets needed to render the application:

```javascript
const APP_SHELL_ASSETS = [
  '/',
  '/index.html',
  '/offline.html',
  '/manifest.json',
  '/css/main.css',
  '/js/app.js',
  '/js/vendor.js',
  '/icons/icon-192x192.png',
  '/icons/icon-512x512.png',
  '/fonts/inter-var.woff2'
];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(STATIC_CACHE)
      .then((cache) => {
        return cache.addAll(APP_SHELL_ASSETS);
      })
      .then(() => {
        // Skip waiting to activate immediately (useful for critical updates)
        // Only enable if update UX handles the transition gracefully
        // return self.skipWaiting();
      })
  );
});
```

**Key decisions:**
- `APP_SHELL_ASSETS` must be kept minimal to avoid install failures (if any single asset fails to cache, the entire install fails)
- `skipWaiting()` is commented out by default -- it should only be enabled when paired with a "New version available, refresh" prompt in the main app
- Font files (`.woff2`) are included in the app shell because they are critical for first render and rarely change

#### 2c. Activate Event

The `activate` event fires when a new service worker takes control. It cleans up old caches to prevent storage bloat:

```javascript
self.addEventListener('activate', (event) => {
  const currentCaches = [STATIC_CACHE, DYNAMIC_CACHE, API_CACHE, OFFLINE_CACHE];

  event.waitUntil(
    caches.keys()
      .then((cacheNames) => {
        return Promise.all(
          cacheNames
            .filter((name) => name.startsWith('tradehub-'))
            .filter((name) => !currentCaches.includes(name))
            .map((name) => caches.delete(name))
        );
      })
      .then(() => {
        // Take control of all open tabs immediately
        return self.clients.claim();
      })
  );
});
```

**Key decisions:**
- Only deletes caches prefixed with `tradehub-` to avoid interfering with other service workers or browser caches
- `clients.claim()` ensures the new service worker controls existing tabs without requiring a page reload
- Combined with cache version rotation, this ensures users always get fresh assets after an update

#### 2d. Fetch Event

The `fetch` event intercepts all network requests and applies the appropriate cache strategy based on resource type:

```javascript
self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // Skip non-GET requests (POST, PUT, DELETE should always go to network)
  if (request.method !== 'GET') {
    return;
  }

  // Skip cross-origin requests that are not CDN assets
  if (url.origin !== self.location.origin && !isTrustedCDN(url)) {
    return;
  }

  // Route to appropriate cache strategy
  if (isAPIRequest(url)) {
    event.respondWith(networkFirst(request, API_CACHE));
  } else if (isStaticAsset(url)) {
    event.respondWith(cacheFirst(request, STATIC_CACHE));
  } else if (isNavigationRequest(request)) {
    event.respondWith(staleWhileRevalidate(request, DYNAMIC_CACHE));
  } else {
    event.respondWith(cacheFirst(request, DYNAMIC_CACHE));
  }
});

// --- Helper functions ---

function isAPIRequest(url) {
  return url.pathname.startsWith('/api/') ||
         url.pathname.startsWith('/method/') ||
         url.pathname.includes('/frappe.client.');
}

function isStaticAsset(url) {
  return /\.(css|js|woff2?|ttf|eot|png|jpe?g|gif|svg|ico|webp|avif)$/i.test(url.pathname);
}

function isNavigationRequest(request) {
  return request.mode === 'navigate' ||
         (request.method === 'GET' && request.headers.get('accept').includes('text/html'));
}

function isTrustedCDN(url) {
  const trustedHosts = [
    'fonts.googleapis.com',
    'fonts.gstatic.com',
    'cdn.jsdelivr.net'
  ];
  return trustedHosts.includes(url.hostname);
}
```

### 3. Cache Strategies (Per Resource Type)

Each resource type requires a different caching strategy to balance freshness against availability:

| Resource Type | Strategy | Cache Name | Rationale |
|---------------|----------|------------|-----------|
| **Static assets** (CSS, JS, fonts, images) | Cache-First | `tradehub-static-v1` | Fingerprinted (hashed) filenames guarantee immutability; cache hit is fastest |
| **API responses** (`/api/*`, `/method/*`) | Network-First | `tradehub-api-v1` | B2B data (prices, inventory, orders) must be fresh; cached fallback only for offline |
| **HTML pages** (navigation requests) | Stale-While-Revalidate | `tradehub-dynamic-v1` | Show cached page instantly, fetch update in background; balances speed and freshness |
| **Offline fallback** (`offline.html`) | Pre-cached (install) | `tradehub-offline-v1` | Always available; served when both network and cache miss for navigation |
| **CDN resources** (fonts, libraries) | Cache-First | `tradehub-static-v1` | External CDN resources rarely change; cache for performance |

#### 3a. Cache-First Strategy (Static Assets)

```javascript
async function cacheFirst(request, cacheName) {
  const cachedResponse = await caches.match(request);
  if (cachedResponse) {
    return cachedResponse;
  }

  try {
    const networkResponse = await fetch(request);
    if (networkResponse.ok) {
      const cache = await caches.open(cacheName);
      cache.put(request, networkResponse.clone());
    }
    return networkResponse;
  } catch (error) {
    // For images, return a placeholder; for other assets, return 503
    if (isImageRequest(request)) {
      return caches.match('/icons/placeholder.svg');
    }
    return new Response('Offline', { status: 503, statusText: 'Service Unavailable' });
  }
}
```

**When to use**: CSS bundles, JS bundles, font files, product images, icons. These files are typically content-addressed (hashed filenames like `app.a1b2c3.js`) and never change once deployed.

**Cache size management**: Implement a max-entries limit for the dynamic cache to prevent unbounded storage growth:

```javascript
async function trimCache(cacheName, maxEntries) {
  const cache = await caches.open(cacheName);
  const keys = await cache.keys();
  if (keys.length > maxEntries) {
    await cache.delete(keys[0]);
    return trimCache(cacheName, maxEntries);
  }
}
```

#### 3b. Network-First Strategy (API Responses)

```javascript
async function networkFirst(request, cacheName) {
  try {
    const networkResponse = await fetch(request);
    if (networkResponse.ok) {
      const cache = await caches.open(cacheName);
      cache.put(request, networkResponse.clone());
    }
    return networkResponse;
  } catch (error) {
    const cachedResponse = await caches.match(request);
    if (cachedResponse) {
      return cachedResponse;
    }
    return new Response(
      JSON.stringify({
        error: 'offline',
        message: 'You are offline. Please check your connection.'
      }),
      {
        status: 503,
        headers: { 'Content-Type': 'application/json' }
      }
    );
  }
}
```

**When to use**: All Frappe API calls (`/api/resource/*`, `/api/method/*`, `/method/*`). B2B operations (pricing, inventory, order status) require the freshest data possible. The cached fallback is only used when the network is completely unavailable, with a clear offline indicator returned.

**Selective API caching**: Not all API responses should be cached. Sensitive endpoints should be excluded:

```javascript
const NO_CACHE_API_PATTERNS = [
  '/api/method/login',
  '/api/method/logout',
  '/api/method/frappe.client.get_password',
  '/api/method/.*payment.*',
  '/api/method/.*escrow.*'
];

function shouldCacheAPIResponse(url) {
  return !NO_CACHE_API_PATTERNS.some(pattern =>
    new RegExp(pattern).test(url.pathname)
  );
}
```

#### 3c. Stale-While-Revalidate Strategy (HTML Pages)

```javascript
async function staleWhileRevalidate(request, cacheName) {
  const cachedResponse = await caches.match(request);

  const fetchPromise = fetch(request)
    .then((networkResponse) => {
      if (networkResponse.ok) {
        const cache = caches.open(cacheName).then((cache) => {
          cache.put(request, networkResponse.clone());
        });
      }
      return networkResponse;
    })
    .catch(() => {
      // Network failed; if we had no cached response, serve offline page
      if (!cachedResponse) {
        return caches.match('/offline.html');
      }
      return null;
    });

  // Return cached immediately if available; otherwise wait for network
  return cachedResponse || fetchPromise;
}
```

**When to use**: HTML page navigations (catalog browsing, order lists, seller dashboard). The user sees the cached version immediately (fast perceived load), while the service worker fetches the latest version in the background for the next visit. This is ideal for B2B catalog pages where product listings change periodically but not in real-time.

#### 3d. Cache Strategy Decision Flowchart

```
Incoming GET Request
  |
  |-- Is it a POST/PUT/DELETE? --> Pass through to network (no caching)
  |
  |-- Is it an API request (/api/*, /method/*)? --> Network-First
  |     |-- Network available? --> Return network response, cache it
  |     |-- Network unavailable? --> Return cached response (if any)
  |     |-- No cache? --> Return offline JSON error
  |
  |-- Is it a static asset (.css, .js, .png, .woff2)? --> Cache-First
  |     |-- In cache? --> Return cached response
  |     |-- Not in cache? --> Fetch from network, cache it, return
  |     |-- Network unavailable + not cached? --> Return fallback/503
  |
  |-- Is it a navigation request (HTML page)? --> Stale-While-Revalidate
  |     |-- In cache? --> Return cached, fetch update in background
  |     |-- Not in cache? --> Fetch from network
  |     |-- Network unavailable + not cached? --> Return offline.html
  |
  |-- Default --> Cache-First (DYNAMIC_CACHE)
```

#### Tag-by-Tag Rationale

| Tag | Purpose | Notes |
|-----|---------|-------|
| `<link rel="manifest">` | Links to the Web App Manifest | `crossorigin="use-credentials"` needed if manifest is served from a different origin or requires auth cookies |
| `<meta name="theme-color">` | Colors browser UI (address bar on Android, title bar on desktop) | Must match `theme_color` in manifest.json |
| `apple-mobile-web-app-capable` | Enables standalone mode on iOS Safari | Without this, iOS always opens in Safari with browser chrome |
| `apple-mobile-web-app-status-bar-style` | Controls iOS status bar appearance | Options: `default` (black text, white bg), `black` (black bg), `black-translucent` (transparent overlay) |
| `apple-mobile-web-app-title` | App name shown on iOS homescreen | Falls back to `<title>` if not set |
| `apple-touch-icon` | Homescreen icon for iOS | Must be 180x180px; iOS does NOT read manifest.json icons |

### 6. Service Worker Registration

The service worker must be registered from the main application JavaScript. Registration should be deferred to avoid blocking initial page load:

```javascript
// In main app entry point (e.g., app.js or main.js)
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => {
    navigator.serviceWorker.register('/sw.js', { scope: '/' })
      .then((registration) => {
        // Check for updates periodically (every hour)
        setInterval(() => {
          registration.update();
        }, 60 * 60 * 1000);

        // Listen for new service worker waiting to activate
        registration.addEventListener('updatefound', () => {
          const newWorker = registration.installing;
          newWorker.addEventListener('statechange', () => {
            if (newWorker.state === 'installed' && navigator.serviceWorker.controller) {
              // New version available - prompt user
              showUpdateNotification();
            }
          });
        });
      })
      .catch((error) => {
        // SW registration failed - app still works without it
        // Do not throw; degrade gracefully
      });
  });
}

function showUpdateNotification() {
  // Display a non-blocking notification bar:
  // "A new version of TradeHub is available. [Refresh]"
  const banner = document.createElement('div');
  banner.className = 'update-banner';
  banner.innerHTML = `
    <span>Yeni bir surum mevcut.</span>
    <button onclick="window.location.reload()">Yenile</button>
  `;
  document.body.appendChild(banner);
}
```

**Key considerations:**
- Registration happens on `load` event, not on DOMContentLoaded, to avoid competing with critical resource loading
- Periodic update checks (hourly) ensure users get fixes without waiting for a page close/reopen cycle
- The update notification follows the "prompt, don't force" pattern -- the user decides when to reload
- Registration failure is silently handled -- the app degrades gracefully to a standard web experience

### 7. Service Worker Update Strategy

Service worker updates require careful handling to avoid the "update race condition" (old cached pages conflicting with new API response formats):

#### Update Lifecycle

```
1. Browser detects byte difference in sw.js
2. New SW enters "installing" state
3. Install event: pre-caches new APP_SHELL_ASSETS with new CACHE_VERSION
4. New SW enters "waiting" state (old SW still controls tabs)
5. User prompted: "New version available. Refresh to update."
6. User clicks refresh OR all tabs close
7. New SW enters "active" state
8. Activate event: deletes old versioned caches
9. clients.claim() takes control of open tabs
```

#### Cache Versioning Strategy

```javascript
// In sw.js - increment CACHE_VERSION with each deployment
const CACHE_VERSION = 'v2';  // Was 'v1' in previous deployment
```

**Build integration**: The CACHE_VERSION should be injected by the build system (e.g., webpack, vite) using the build hash or deployment timestamp:

```javascript
// Generated by build tool
const CACHE_VERSION = '__BUILD_HASH__';  // Replaced at build time
```

This ensures every deployment creates new cache buckets and cleans up old ones.

### Safari-Specific Implementation Concerns

#### 1. Storage Eviction (7-Day Rule)

Safari/WebKit evicts all website data (including service worker caches, IndexedDB, localStorage) after **7 days of non-use** unless the PWA is added to the homescreen. For B2B users who may not visit daily:

- **Mitigation**: Prompt frequent users to install the PWA to homescreen (installed PWAs are exempt from the 7-day rule)
- **Mitigation**: Store critical user preferences server-side (via Frappe API), not only in client-side storage
- **Mitigation**: Design the app to function correctly when caches are cold (first-visit experience must be smooth)

#### 2. Custom Install Prompt for iOS

Since Safari does not fire `beforeinstallprompt`, a custom "Add to Homescreen" guide must be shown:

```javascript
function isIOSSafari() {

  return isIOS && isWebkit && isNotChrome;
}

function isInStandaloneMode() {
  return window.matchMedia('(display-mode: standalone)').matches ||
         window.navigator.standalone === true;
}

function showIOSInstallGuide() {
  if (isIOSSafari() && !isInStandaloneMode()) {
    // Show overlay with instructions:
    // "TradeHub'u ana ekranina ekle:

    //  2. 'Ana Ekrana Ekle' secenegini sec
    //  3. 'Ekle' butonuna dokun"
```

#### 3. Apple Touch Icon Specification

iOS requires a separate 180x180px icon that is NOT read from the manifest:

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Service worker update race condition** | Medium | High | Use cache versioning with `CACHE_VERSION` prefix. Old caches deleted during activate. Never use `skipWaiting()` without user confirmation prompt. Test update flow thoroughly. |
| **Safari storage eviction (7-day rule)** | High (for infrequent users) | Medium | Prompt users to install PWA to homescreen (exempts from eviction). Store critical data server-side. Design for cold-cache resilience. |
| **HTTPS not configured on all environments** | Medium | Critical | PWA features are completely blocked on HTTP. Document HTTPS as a hard prerequisite. Verify SSL in CI. |
| **Mixed content blocking** | Medium | High | Audit all resource URLs for HTTP references. Use protocol-relative or HTTPS-only URLs. Content-Security-Policy header to enforce. |
| **Large app shell causing install failure** | Low | High | If any single pre-cached asset fails to fetch, the entire install fails. Keep APP_SHELL_ASSETS minimal (< 15 files, < 2MB total). Use error handling in install event. |
| **Cache storage quota exceeded** | Low | Medium | Implement `trimCache()` to limit dynamic cache entries. Monitor `navigator.storage.estimate()`. Clear old API caches proactively. |
| **Stale cached data misleading B2B users** | Medium | High | Network-first for API data ensures freshness. Add "Last updated" timestamps to cached data displays. Show clear "offline" indicators when serving cached data. |
| **iOS splash screen fragmentation** | High | Low | Many iOS device sizes require individual splash images. Use `pwa-asset-generator` to automate. Accept that some device sizes may not have perfect splash. |
| **Service worker scope issues** | Low | High | Place `sw.js` at frontend root (`/`). If frontend is served from a subpath (e.g., `/app/`), adjust `scope` in both registration and manifest. |
| **Build hash injection failure** | Low | Medium | If `__BUILD_HASH__` is not replaced at build time, cache versioning breaks. Add build verification step. Fallback to timestamp-based versioning. |
| **Cross-origin API requests** | Medium | Medium | If Frappe API is on a different origin than the frontend, service worker cannot intercept cross-origin requests by default. Ensure same-origin setup or use CORS with `mode: 'cors'`. |
| **Arabic RTL support in manifest** | Low | Low | Arabic locale needs `"dir": "rtl"` in manifest. Plan for dynamic manifest serving (Section 1, Multi-Language Manifest Consideration). Not blocking for initial Turkish-only launch. |
