# Task 114 - Smart Mapping / Auto-Match System Plan

## Executive Summary

The TradeHub B2B marketplace's Import Job system currently relies on **manual, one-by-one column mapping** stored as a flat JSON blob (`column_mapping` field). Meanwhile, Frappe's native Data Import uses `build_fields_dict_for_column_matching()` which performs **exact-match-only** lookups -- matching column headers against field labels, fieldnames, and translated labels with zero tolerance for variation. Neither system offers fuzzy matching, historical learning, or reusable mapping templates. This creates a significant pain point for sellers who repeatedly import files with similar but non-identical column names (e.g., "Urun Adi" vs "urun_adi" vs "Product Name").

This plan designs a **Smart Mapping / Auto-Match System** that introduces: (1) a new `Mapping Template` DocType for reusable, shareable mapping configurations, (2) a multi-strategy matching algorithm with four priority levels (Historical -> Exact -> Normalized -> Fuzzy), (3) Turkish locale-aware string normalization addressing the I/i problem, (4) a user workflow for review/confirmation/save, and (5) seamless integration with the existing Import Job DocType's `column_mapping` JSON field.

## Current State Analysis

### 1. Frappe Native Data Import -- Column Matching

The Frappe v15 `importer.py` (located at `frappe/core/doctype/data_import/importer.py`) uses a purely dictionary-based exact lookup for column-to-field matching:

**`build_fields_dict_for_column_matching()` (lines 1069-1209)**

This function builds a Python dict where keys are possible column header values and values are the corresponding `df` (docfield) objects. The matching keys include:

| Key Type | Example | Scope |
|----------|---------|-------|
| Label (exact) | `"Customer"` | Parent DocType fields |
| Translated label | `_("Customer")` -> `"Musteri"` | Parent DocType fields |
| Fieldname (exact) | `"customer"` | Parent DocType fields |
| Label + fieldname | `"Customer (customer)"` | Parent DocType fields |
| Child: fieldname | `"items.item_code"` | Child table fields |
| Child: label + table ref | `"Item Code (Sales Invoice Item)"` | Child table fields |

**`get_df_for_column_header()` (lines 1212-1219)**

```python
def get_df_for_column_header(doctype, header):
    def build_fields_dict_for_doctype():
        return build_fields_dict_for_column_matching(doctype)

    df_by_labels_and_fieldname = frappe.cache.hget(
        "data_import_column_header_map", doctype, generator=build_fields_dict_for_doctype
    )
    return df_by_labels_and_fieldname.get(header)
```

This is a simple `dict.get(header)` -- **zero tolerance** for case differences, whitespace, Turkish characters, or synonyms. If the header is `"Urun Kodu"` but the field label is `"Urun Kodu"` (with Turkish dotless i/I), the match fails silently.

**`Column.parse()` (lines 875-942)**

When no match is found, the column is marked `skip_import = True` and a warning is issued:
```python
elif header_title and not df:
    self.warnings.append({
        "col": column_number,
        "message": _("Cannot match column {0} with any field").format(frappe.bold(header_title)),
        "type": "info",
    })
```

This means unmatched columns are silently skipped, and users must manually assign them via the `column_to_field_map` in `template_options`. There is **no suggestion mechanism**.

```python
def map_row_to_fields(self, row, mapping=None, defaults=None):
    mapping = mapping or self.get_column_mapping()
    defaults = defaults or self.get_field_defaults()
    result = dict(defaults)

    if isinstance(row, dict):
        # Dict rows: {"source_column_name": "target_fieldname"}
        for source_col, target_field in mapping.items():
            if source_col in row:
                value = row[source_col]
                if value is not None and value != "":
                    result[target_field] = value
    elif isinstance(row, list):
        # List rows: {"column_index": "target_fieldname"}
        for i, target_field in mapping.items():
            try:
                idx = int(i)
                if idx < len(row):
                    value = row[idx]
                    if value is not None and value != "":
                        result[target_field] = value
            except (ValueError, IndexError):
                pass

    return result
```

**Key observations about Import Job:**

| Aspect | Current State | Limitation |
|--------|--------------|------------|
| Mapping storage | Flat JSON: `{"source": "target"}` | No confidence scores, no match strategy metadata |
| Mapping creation | Manual -- user must type/select each mapping | No auto-suggestion, no historical learning |
| Mapping reuse | None -- mapping is per-job only | Cannot apply same mapping to similar future imports |
| Validation | `validate_column_mapping()` checks JSON validity only | No semantic validation of field references |
| Import types | `Listing`, `Listing Variant`, `SKU`, `Category`, `Media Asset`, `Inventory Update`, `Price Update` | Each type needs its own mapping, but no template system exists |
| Encoding support | `utf-8`, `utf-16`, `iso-8859-1`, `iso-8859-9`, `windows-1254`, `windows-1252` | Turkish encodings (iso-8859-9, windows-1254) are supported for file reading, but not for matching logic |
| Tenant/seller scoping | Import Job has `tenant` (Link to Tenant) and `seller` (Link to Seller Profile) | Mappings are not scoped to tenant/seller |

**Import Job Error child table** (`import_job_error.json`) includes:
- `row_number` (Int), `error_type` (Select), `error_message` (Small Text)
- `field_name` (Data), `field_value` (Data), `severity` (Select)
- `row_data` (JSON), `stack_trace` (Long Text)

The error types include `Missing Required Field` and `Reference Not Found` which commonly result from unmapped or incorrectly mapped columns.

## Implementation Details

### 1. Multi-Strategy Matching Algorithm

The auto-match engine processes each source column header through four strategies in priority order. The first strategy to produce a match above the confidence threshold wins.

```
Priority 0: Historical Match (confidence: 100%)
    |
    v  (no match found)
Priority 1: Exact Match (confidence: 100%)
    |
    v  (no match found)
Priority 2: Normalized Match (confidence: 90%)
    |
    v  (no match found)
Priority 3: Fuzzy Match (confidence: 70-89%)
    |
    v  (no match found)
Result: Unmatched (requires manual mapping)
```

#### Strategy 0: Historical Match (Priority 0)

**Logic**: Look up the source column header in `Mapping History` for this target DocType, import type, and tenant/seller context.

```python
def find_historical_match(source_column, target_doctype, import_type, tenant=None, seller=None):
    """
    Priority 0: Historical exact match from previously confirmed mappings.
    Returns the most-confirmed target fieldname or None.
    """
    filters = {
        "source_column_normalized": normalize_turkish(source_column),
        "target_doctype": target_doctype
    }

    if import_type:
        filters["import_type"] = ["in", [import_type, ""]]

    # Prioritize seller-specific, then tenant-specific, then global
    results = []

    if seller:
        filters["seller"] = seller
        results = frappe.get_all(
            "Mapping History",
            filters=filters,
            fields=["target_fieldname", "confirmation_count"],
            order_by="confirmation_count desc",
            limit=1
        )

    if not results and tenant:
        filters.pop("seller", None)
        filters["tenant"] = tenant
        results = frappe.get_all(
            "Mapping History",
            filters=filters,
            fields=["target_fieldname", "confirmation_count"],
            order_by="confirmation_count desc",
            limit=1
        )

    if not results:
        # Global fallback (no tenant/seller filter)
        filters.pop("seller", None)
        filters.pop("tenant", None)
        results = frappe.get_all(
            "Mapping History",
            filters=filters,
            fields=["target_fieldname", "confirmation_count"],
            order_by="confirmation_count desc",
            limit=1
        )

    if results:
        return {
            "target_fieldname": results[0].target_fieldname,
            "confidence": 100.0,
            "strategy": "Historical"
        }

    return None
```

**Scoping priority**: Seller-specific > Tenant-specific > Global. A seller who always maps "Urun Kodu" to `item_code` will see that suggestion first, even if another seller in the same tenant maps it differently.

#### Strategy 1: Exact Match (Priority 1)

**Logic**: Match the source column header exactly against field labels, fieldnames, and translated labels. This mirrors Frappe's native `build_fields_dict_for_column_matching()` but is encapsulated in the auto-match engine.

```python
def find_exact_match(source_column, target_doctype):
    """
    Priority 1: Exact string match against field labels and fieldnames.
    Mirrors Frappe's native build_fields_dict_for_column_matching().
    """
    fields_dict = build_fields_dict_for_target(target_doctype)

    if source_column in fields_dict:
        df = fields_dict[source_column]
        return {
            "target_fieldname": df.fieldname,
            "confidence": 100.0,
            "strategy": "Exact"
        }

    return None


def build_fields_dict_for_target(doctype):
    """
    Build a dict mapping possible column headers to docfield objects.
    Keys: label, translated_label, fieldname, "label (fieldname)".
    """
    meta = frappe.get_meta(doctype)
    out = {}

    for df in meta.fields:
        if df.fieldtype in frappe.model.no_value_fields:
            continue

        label = (df.label or "").strip()
        translated_label = _(label)

        # Register all possible exact-match keys
        for key in (label, translated_label, df.fieldname):
            if key and key not in out:
                out[key] = df

        # Also register "Label (fieldname)" pattern
        composite_key = f"{label} ({df.fieldname})"
        out[composite_key] = df

    # Add ID/name field
    out["ID"] = frappe._dict({"fieldname": "name", "label": "ID", "fieldtype": "Data"})
    out["name"] = out["ID"]

    return out
```

#### Strategy 2: Normalized Match (Priority 2)

**Logic**: Normalize both source column and target labels using Turkish-locale-aware rules, then compare.

```python
def find_normalized_match(source_column, target_doctype):
    """
    Priority 2: Normalized match using Turkish-locale-aware normalization.
    Handles case differences, whitespace, underscores, and Turkish I/i.
    """
    normalized_source = normalize_turkish(source_column)
    fields_dict = build_fields_dict_for_target(target_doctype)

    for key, df in fields_dict.items():
        normalized_key = normalize_turkish(key)
        if normalized_source == normalized_key:
            return {
                "target_fieldname": df.fieldname,
                "confidence": 90.0,
                "strategy": "Normalized"
            }

    return None
```

#### Strategy 3: Fuzzy Match (Priority 3)

**Logic**: Use `difflib.SequenceMatcher` from Python's standard library (zero external dependency) to find the best fuzzy match above a configurable threshold.

```python
import difflib

# Configurable threshold -- recommended 0.7 (70%)
FUZZY_THRESHOLD = 0.7

def find_fuzzy_match(source_column, target_doctype):
    """
    Priority 3: Fuzzy match using difflib.SequenceMatcher.
    Returns the best match above FUZZY_THRESHOLD, or None.
    """
    normalized_source = normalize_turkish(source_column)
    fields_dict = build_fields_dict_for_target(target_doctype)

    best_match = None
    best_ratio = 0.0

    for key, df in fields_dict.items():
        normalized_key = normalize_turkish(key)
        ratio = difflib.SequenceMatcher(
            None, normalized_source, normalized_key
        ).ratio()

        if ratio > best_ratio and ratio >= FUZZY_THRESHOLD:
            best_ratio = ratio
            best_match = {
                "target_fieldname": df.fieldname,
                "confidence": round(ratio * 100, 1),
                "strategy": "Fuzzy",
                "matched_against": key
            }

    return best_match
```

**Why `difflib.SequenceMatcher`?**

| Library | Pros | Cons | Recommendation |
|---------|------|------|----------------|
| `difflib.SequenceMatcher` (stdlib) | Zero-dependency, available on every Python install, well-tested | Slower than C-based alternatives for large datasets | **Recommended for v1** |
| `rapidfuzz` (PyPI) | 10-100x faster (C++ implementation), MIT license, compatible API | Requires `pip install rapidfuzz`, adds external dependency | Consider for v2 if performance is an issue |
| `fuzzywuzzy` (PyPI) | Popular, good API | GPL license (problematic), depends on python-Levenshtein for speed | Avoid due to license |
| `thefuzz` (PyPI) | Fork of fuzzywuzzy, MIT license | Still slower than rapidfuzz | Alternative if rapidfuzz is unavailable |

For a typical import with 20-50 columns and 20-100 target fields, `difflib.SequenceMatcher` executes in < 50ms -- well within acceptable UX latency for an interactive mapping review screen.

#### Combined Auto-Match Orchestrator

```python
def auto_match_columns(source_columns, target_doctype, import_type=None,
                       tenant=None, seller=None):
    """
    Run multi-strategy auto-matching for all source columns.

    Args:
        source_columns: list of column header strings from the import file
        target_doctype: the DocType to match against
        import_type: optional Import Job type (Listing, SKU, etc.)
        tenant: optional Tenant name for scoping
        seller: optional Seller Profile name for scoping

    Returns:
        list of dicts with keys:
            source_column, target_fieldname, confidence,
            strategy, is_confirmed, alternatives
    """
    results = []
    used_targets = set()  # prevent duplicate target assignments

    for source_col in source_columns:
        match = None

        # Priority 0: Historical
        match = find_historical_match(
            source_col, target_doctype, import_type, tenant, seller
        )

        # Priority 1: Exact
        if not match:
            match = find_exact_match(source_col, target_doctype)

        # Priority 2: Normalized
        if not match:
            match = find_normalized_match(source_col, target_doctype)

        # Priority 3: Fuzzy
        if not match:
            match = find_fuzzy_match(source_col, target_doctype)

        if match and match["target_fieldname"] not in used_targets:
            used_targets.add(match["target_fieldname"])
            results.append({
                "source_column": source_col,
                "target_fieldname": match["target_fieldname"],
                "confidence": match["confidence"],
                "strategy": match["strategy"],
                "is_confirmed": match["strategy"] == "Historical",
                "alternatives": get_alternative_matches(
                    source_col, target_doctype, exclude=match["target_fieldname"]
                )
            })
        else:
            results.append({
                "source_column": source_col,
                "target_fieldname": None,
                "confidence": 0,
                "strategy": "Unmatched",
                "is_confirmed": False,
                "alternatives": get_all_unmapped_fields(
                    target_doctype, used_targets
                )
            })

    return results


def get_alternative_matches(source_column, target_doctype, exclude=None):
    """
    Get top-3 alternative fuzzy matches for a source column,
    excluding the primary match. Used in the review UI dropdown.
    """
    normalized_source = normalize_turkish(source_column)
    fields_dict = build_fields_dict_for_target(target_doctype)

    alternatives = []
    for key, df in fields_dict.items():
        if df.fieldname == exclude:
            continue
        normalized_key = normalize_turkish(key)
        ratio = difflib.SequenceMatcher(None, normalized_source, normalized_key).ratio()
        if ratio >= 0.5:  # lower threshold for alternatives
            alternatives.append({
                "target_fieldname": df.fieldname,
                "target_label": df.label,
                "confidence": round(ratio * 100, 1)
            })

    # Sort by confidence, return top 3
    alternatives.sort(key=lambda x: x["confidence"], reverse=True)
    return alternatives[:3]
```

### 2. Turkish Locale-Aware String Normalization

Turkish has four "I" variants that break standard Python `str.lower()`/`str.upper()`:

| Character | Unicode | Name | Standard `lower()` | Turkish `lower()` |
|-----------|---------|------|--------------------|--------------------|
| I | U+0049 | Latin Capital I | i (U+0069) | i (U+0131, dotless) |
| i | U+0069 | Latin Small I | i (stays) | i (stays) |
| I | U+0130 | Latin Capital I with Dot Above | i with dot above (wrong) | i (U+0069) |
| i | U+0131 | Latin Small Dotless I | stays | stays |

**Python's `str.lower()` is locale-unaware and always uses English rules.** This means `"ISTANBUL".lower()` produces `"istanbul"` (with dotted i), not `"istanbul"` (with dotless i) which would be correct in Turkish.

**Normalization function:**

```python
# Turkish-specific character mappings
TURKISH_LOWER_MAP = str.maketrans({
    "I": "\u0131",      # I -> i (dotless small i) -- Turkish rule
    "\u0130": "i",       # I -> i (regular small i) -- Turkish rule
    "\u00dc": "\u00fc",  # U -> u
    "\u00d6": "\u00f6",  # O -> o
    "\u00c7": "\u00e7",  # C -> c
    "\u015e": "\u015f",  # S -> s
    "\u011e": "\u011f",  # G -> g
})

def normalize_turkish(text):
    """
    Turkish-locale-aware string normalization for column matching.

    Steps:
    1. Apply Turkish-specific lowercase (I -> i, I -> i)
    2. Apply standard lowercase for remaining characters
    3. Strip leading/trailing whitespace
    4. Replace underscores with spaces
    5. Collapse multiple spaces into one
    6. Remove common filler words and punctuation
    """
    if not text:
        return ""

    # Step 1: Turkish-specific lowercasing
    result = text.translate(TURKISH_LOWER_MAP)

    # Step 2: Standard lowercase for remaining characters
    result = result.lower()

    # Step 3: Strip whitespace
    result = result.strip()

    # Step 4: Replace underscores and hyphens with spaces
    result = result.replace("_", " ").replace("-", " ")

    # Step 5: Collapse multiple spaces
    import re
    result = re.sub(r"\s+", " ", result)

    # Step 6: Remove common filler words (optional, for improved fuzzy matching)
    # These are stripped only for comparison, not for display
    filler_words = {"the", "a", "an", "no", "nr", "#"}
    words = result.split()
    words = [w for w in words if w not in filler_words]
    result = " ".join(words)

    return result
```

#### Recording Confirmed Mappings

When a user confirms a mapping (either in the review dialog or by saving a template), each confirmed source-target pair is recorded in `Mapping History`:

```python
def record_mapping_history(confirmed_mappings, target_doctype, import_type=None,
                           tenant=None, seller=None):
    """
    Record confirmed mapping pairs in Mapping History for future suggestions.

    Args:
        confirmed_mappings: list of dicts with source_column and target_fieldname
        target_doctype: target DocType name
        import_type: Import Job type
        tenant: Tenant name
        seller: Seller Profile name
    """
    for mapping in confirmed_mappings:
        source_normalized = normalize_turkish(mapping["source_column"])

        # Check if this exact mapping already exists
        existing = frappe.db.get_value(
            "Mapping History",
            {
                "source_column_normalized": source_normalized,
                "target_fieldname": mapping["target_fieldname"],
                "target_doctype": target_doctype,
                "import_type": import_type or "",
                "tenant": tenant or "",
                "seller": seller or ""
            },
            "name"
        )

        if existing:
            # Increment confirmation count
            frappe.db.set_value(
                "Mapping History", existing,
                {
                    "confirmation_count": frappe.db.get_value(
                        "Mapping History", existing, "confirmation_count"
                    ) + 1,
                    "last_used_at": frappe.utils.now_datetime()
                }
            )
        else:
            # Create new history entry
            frappe.get_doc({
                "doctype": "Mapping History",
                "source_column": mapping["source_column"],
                "source_column_normalized": source_normalized,
                "target_fieldname": mapping["target_fieldname"],
                "target_doctype": target_doctype,
                "import_type": import_type or "",
                "tenant": tenant or "",
                "seller": seller or "",
                "confirmation_count": 1,
                "last_used_at": frappe.utils.now_datetime()
            }).insert(ignore_permissions=True)

    frappe.db.commit()
```

#### Suggestion Ranking

When multiple historical mappings exist for the same normalized source column (e.g., the same seller mapped "Urun Kodu" to `item_code` 5 times and to `sku_code` once), the system ranks by `confirmation_count` descending. The most-confirmed mapping becomes the primary suggestion.

## Risks & Mitigations

### Risk 1: Fuzzy Match False Positives

**Risk**: A similarity threshold of 0.7 may produce incorrect matches for short column names. For example, "ad" (Turkish for "name") and "id" have a SequenceMatcher ratio of 0.5, but short strings like "adi" and "adı" could incorrectly match or fail to match.

**Mitigation**:
- Auto-matched fuzzy results are **never auto-confirmed** -- they always require user review (yellow indicator)
- Display the `matched_against` field label so users can see what the algorithm matched against
- Provide a configurable threshold in Frappe's `site_config.json`:
  ```json
  {"tradehub_fuzzy_threshold": 0.75}
  ```
- Consider applying a minimum string length filter (e.g., skip fuzzy matching for headers shorter than 3 characters)

### Risk 2: Historical Mapping Data Pollution

**Risk**: If a user incorrectly confirms a mapping, it pollutes the history and affects future suggestions.

**Mitigation**:
- Add a "Reset Mapping History" action on the Mapping Template form
- Mapping History entries with `confirmation_count = 1` that haven't been used in 90 days are automatically cleaned up via a scheduled task
- System Managers can view and delete Mapping History records
- Consider adding a "Report incorrect suggestion" button in the review dialog

### Risk 3: Performance Degradation with Large Mapping History

**Risk**: As `Mapping History` grows (thousands of entries), the historical lookup query could slow down.

**Mitigation**:
- Add database indexes on `(source_column_normalized, target_doctype, import_type, tenant)` -- the primary lookup combination
- The query already filters by `target_doctype` and optionally `import_type`, `tenant`, `seller`, which limits the result set
- Implement a cache layer: `frappe.cache.hget("mapping_history", cache_key)` with a 5-minute TTL
- Periodically aggregate/deduplicate entries with very low confirmation counts

### Risk 4: Turkish Normalization Edge Cases

**Risk**: Some Turkish words change meaning when I/i is normalized incorrectly. For example, "ile" (with) vs "ile" (could be a normalization artifact).

**Mitigation**:
- The normalization is only used for **matching**, not for display or storage of the original column name
- The original `source_column` is always preserved in Mapping History alongside `source_column_normalized`
- Fuzzy matching provides a secondary safety net -- even if normalization produces an incorrect equality, the confidence score reflects the actual similarity
- Edge cases are documented with test coverage

### Risk 5: Breaking Change to Import Job

**Risk**: Adding a `mapping_template` field to Import Job JSON could cause migration issues.

**Mitigation**:
- The new field is optional (no `reqd: 1`) and has no default value
- It only appears when `status == 'Pending'` (via `depends_on`)
- The existing `column_mapping` JSON field remains unchanged -- it continues to be the runtime data source
- The `map_row_to_fields()` method requires zero changes
- Migration is additive only: `bench migrate` adds the new field without affecting existing data

### Risk 6: Multi-Tenant Data Leakage

**Risk**: Mapping templates or history from one tenant could be visible to another tenant.

**Mitigation**:
- `Mapping Template` has a `tenant` Link field. Tenant isolation can be enforced via the existing `has_tenant_permission` / `get_tenant_permission_query_conditions` hooks in `tradehub_core/permissions.py`
- `Mapping History` also has a `tenant` field, and the historical lookup function explicitly filters by tenant
- Global templates (`is_global = True`) are only creatable by System Managers
- The scoping priority (seller > tenant > global) ensures that tenant-specific data is preferred over global data

### Risk 7: Encoding Mismatch in Header Extraction

**Risk**: Turkish characters in CSV headers may be garbled if the wrong encoding is used, leading to normalization failures.

**Mitigation**:
- Import Job already supports `iso-8859-9` and `windows-1254` (both common Turkish encodings) alongside `utf-8`
- The header extraction step must respect the Import Job's `encoding` setting
- Consider adding automatic encoding detection using Python's `chardet` library as a fallback
- The normalization function handles Unicode Turkish characters natively, so properly decoded headers will match correctly
