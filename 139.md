## 1. Executive Summary

This document defines the redesign of the TradeHub Seller Tag/Badge system within the `tradehub_seller` module. The current system provides basic tag assignment through a JSON-based rule engine (`rule_json` field) with AND/OR logic. The redesign introduces:

- **5 tag categories** (Verification, Performance, Program, Promotional, Identity) replacing the current flat `tag_type` field
- **Weakest-link scoring** (Alibaba SOPI model) where the lowest individual metric determines tag eligibility
- **Enhanced rule conditions** via a redesigned child table with weighted metrics
- **Scheduled evaluation** with grace periods and warning notifications
- **Status-managed assignments** with expiry, revocation, and score snapshot tracking

The current DocTypes — `Seller Tag`, `Seller Tag Rule`, `Seller Tag Rule Condition`, `Seller Tag Assignment`, and `Seller Metrics` — will be extended, not replaced. The existing `rule_engine.py`, `seller_metrics.py`, and `tasks.py` modules will be enhanced to support the new functionality.

### 2.2 Existing Rule Engine Architecture

The current `rule_engine.py` implements a recursive AND/OR evaluator:

```
rule_json = {
    "logic": "AND",           // top-level combinator
    "groups": [
        {
            "group_logic": "AND",  // group-level combinator
            "conditions": [
                {"field": "total_orders", "operator": ">=", "value": 100},
                {"field": "avg_rating",   "operator": ">=", "value": 4.5}
            ]
        }
    ]
}
```

**Supported operators:** `==`, `!=`, `>`, `>=`, `<`, `<=`, `in`, `not_in`, `contains`, `not_contains`, `is_set`, `is_not_set`, `starts_with`, `ends_with`

**Key Observation:** The `Seller Tag Rule Condition` child table exists with fields (`field`, `operator`, `value`, `logic`) but the `SellerTagRule` controller **exclusively uses `rule_json`** (JSON blob). The child table is dead code.

## 3. Tag Category Taxonomy (5 Categories)

### 3.1 Category Overview

| # | Category | Assignment Type | Evaluation | Stackable | Description |
|---|----------|----------------|------------|-----------|-------------|
| 1 | **Verification** | Manual (Admin) | On admin action | No (one per seller) | Admin-verified identity/business status |
| 2 | **Performance** | Automatic (Rule) | Daily scheduled | Yes | Data-driven KPI badges |
| 3 | **Program** | Hybrid (Opt-in + Rule) | On enrollment + periodic | Yes | Voluntary program memberships |
| 4 | **Promotional** | Manual (Admin) | Time-limited auto-expire | Yes | Temporary campaign badges |
| 5 | **Identity** | Application + Document | On application approval | No (one per type) | Document-backed certifications |

### 3.2 Category 1 — Verification Tags

Tags assigned by platform administrators after manual review. Only one verification-level tag can be active at a time (non-stackable).

| Tag Code | Tag Name | Criteria | Icon | Color |
|----------|----------|----------|------|-------|
| `VERIFIED_SELLER` | Verified Seller | Identity + business documents verified by admin | `fa-check-circle` | `#27ae60` |
| `TRUSTED_SELLER` | Trusted Seller | 6+ months verified + clean track record | `fa-shield-alt` | `#2980b9` |
| `VERIFIED_MANUFACTURER` | Verified Manufacturer | Factory/production facility verified | `fa-industry` | `#8e44ad` |
| `OFFICIAL_STORE` | Official Store | Brand owner or exclusive authorized distributor | `fa-store` | `#e67e22` |

**Assignment Flow:**
1. Admin reviews seller documentation in Seller Profile
2. Admin manually assigns tag via `Seller Tag Assignment` with `source = "Manual"`
3. `override_state = "ForcedOn"` prevents automatic removal
4. No expiry unless admin sets one

### 3.3 Category 2 — Performance Tags

Automatically assigned/removed by the rule engine based on Seller Metrics data. These use the weakest-link scoring algorithm (Section 6).

| Tag Code | Tag Name | Key Thresholds | Evaluation Period |
|----------|----------|----------------|-------------------|
| `TOP_RATED` | Top Rated Seller | avg_rating ≥ 4.7, total_reviews ≥ 50, positive_review_rate ≥ 90% | Last 90 days |
| `FAST_SHIPPER` | Fast Shipper | on_time_delivery_rate ≥ 95%, total_orders ≥ 20 | Last 60 days |
| `HIGH_VOLUME` | High Volume Seller | total_orders ≥ 500, total_sales_amount ≥ 100,000 | Last 180 days |
| `LOW_RETURN` | Low Return Rate | return_rate ≤ 2%, total_orders ≥ 50 | Last 90 days |
| `RESPONSIVE` | Responsive Seller | avg_response_time_hours ≤ 2, active_days ≥ 30 | Last 30 days |
| `RISING_STAR` | Rising Star | total_orders ≥ 20, avg_rating ≥ 4.5, active_days ≤ 180 | Since registration |
| `REPEAT_MAGNET` | Repeat Customer Magnet | repeat_customer_rate ≥ 30%, total_orders ≥ 100 | Last 180 days |

**Assignment Flow:**
1. Daily cron evaluates all performance rules
2. Seller Metrics refreshed → Rules evaluated → Tags assigned/removed
3. Grace period applies before removal (see Section 7)

### 3.4 Category 3 — Program Tags

Require seller opt-in (enrollment) plus automatic condition verification.

| Tag Code | Tag Name | Program Requirements | Ongoing Conditions |
|----------|----------|---------------------|--------------------|
| `PREMIUM_SELLER` | Premium Seller | Enrolled in Premium program, subscription active | Maintain cancellation_rate ≤ 5%, avg_rating ≥ 4.0 |
| `FREE_SHIPPING_PARTNER` | Free Shipping Partner | Opted into free shipping program | All active listings offer free shipping |
| `EXPRESS_DELIVERY` | Express Delivery | Signed express delivery agreement | on_time_delivery_rate ≥ 98% for express orders |
| `TRADEHUB_GUARANTEED` | TradeHub Guaranteed | Accepted guarantee terms | complaint_rate ≤ 1%, return_rate ≤ 3% |

**Assignment Flow:**
1. Seller enrolls in a program (creates a Program Enrollment record)
2. System assigns tag with `source = "Rule"` + enrollment check
3. Ongoing evaluation verifies conditions still met
4. If conditions fail → warning → grace period → tag removal + program notification

### 3.5 Category 4 — Promotional Tags

Time-limited tags for seasonal campaigns, promotions, or events. Admin-created with mandatory expiry.

| Tag Code | Tag Name | Typical Duration | Use Case |
|----------|----------|-----------------|----------|
| `FLASH_SALE_PARTICIPANT` | Flash Sale Participant | 1–7 days | Sellers in an active flash sale event |
| `SEASONAL_PICK` | Seasonal Pick | 1–3 months | Curated seasonal seller selections |
| `NEW_ARRIVAL` | New on TradeHub | 30 days | Automatic for newly registered sellers |
| `CAMPAIGN_PARTNER` | Campaign Partner | Campaign duration | Sellers participating in marketing campaigns |

**Assignment Flow:**
1. Admin creates assignment with mandatory `expiry_date`
2. Tag automatically expires on `expiry_date` (status → Expired)
3. `NEW_ARRIVAL` auto-assigned on Seller Profile creation with 30-day expiry

### 3.6 Category 5 — Identity Tags

Require formal application with document submission. Admin reviews and approves/rejects.

| Tag Code | Tag Name | Required Documents | Review SLA |
|----------|----------|--------------------|------------|
| `AUTHORIZED_DISTRIBUTOR` | Authorized Distributor | Distribution agreement, brand authorization letter | 5 business days |
| `CERTIFIED_ORGANIC` | Certified Organic | Organic certification document | 7 business days |
| `HALAL_CERTIFIED` | Halal Certified | Halal certificate from recognized body | 7 business days |
| `ISO_CERTIFIED` | ISO Certified | ISO certificate (9001, 22000, etc.) | 7 business days |
| `EXPORT_LICENSED` | Export Licensed | Export license from trade ministry | 5 business days |

**Assignment Flow:**
1. Seller submits application with required documents
2. Admin reviews documents, verifies authenticity
3. Approved → Tag assigned with `source = "Manual"`, expiry = certificate expiry date
4. Rejected → Seller notified with reason, can re-apply after 30 days

### 3.7 Mapping to Existing `tag_type` Field

The current `Seller Tag` DocType has `tag_type` with options: Achievement, Badge, Certification, Warning, Special.

**Migration Strategy:**

| Current `tag_type` | New `tag_category` | Mapping Logic |
|--------------------|--------------------|--------------|
| Achievement | Performance | Automatically earned via metrics |
| Badge | Program | Earned through program participation |
| Certification | Identity | Document-backed verification |
| Warning | *(remove — see below)* | Handled by negative tags system |
| Special | Promotional | Time-limited special designations |

**Decision:** Add new `tag_category` field (Select) to `Seller Tag` DocType. Keep `tag_type` as a secondary display classifier (deprecated, read-only) for backward compatibility. New code references `tag_category` only. A one-time data migration script maps existing tags.

## 5. Redesigned Seller Tag Rule DocType

### 5.1 Decision: `rule_json` vs Child Table

#### Analysis

| Approach | Pros | Cons |
|----------|------|------|
| **`rule_json` (current)** | Supports nested AND/OR groups, flexible structure, easy to export/import | Not queryable by SQL, no field validation by Frappe, hard to report on, admin must write JSON |
| **Child table `Seller Tag Rule Condition`** | SQL-queryable, Frappe form validation, admin-friendly UI, standard reporting | Cannot express nested groups natively, limited to flat AND/OR logic |
| **Hybrid (recommended)** | Best of both: child table for common rules, JSON override for complex rules | Slightly more complex code path |

#### Decision: **Hybrid Approach**

1. **Primary path:** Redesigned `Seller Tag Rule Condition` child table for 90% of use cases (simple threshold-based rules)
2. **Advanced path:** `rule_json` field retained (optional) for complex nested AND/OR group rules
3. **Evaluation priority:** If `rule_json` is populated and non-empty, it takes precedence over child table conditions
4. **Migration:** Existing `rule_json` values remain. New rules can be created via either method.

**Rule Engine Change:**
```python
def evaluate_rule(self, rule_doc, metrics):
    # Priority 1: Use rule_json if present
    if rule_doc.rule_json and rule_doc.rule_json.strip() not in ('', '{}', 'null'):
        rule_config = json.loads(rule_doc.rule_json)
        return evaluate_conditions(rule_config, metrics)

    # Priority 2: Use child table conditions
    if rule_doc.conditions:
        return self._evaluate_child_conditions(rule_doc, metrics)

    return False  # No conditions defined
```

## 8. Weakest-Link Scoring Algorithm (Alibaba SOPI Model)

### 8.1 Concept

The **weakest-link** model is inspired by Alibaba's SOPI (Service and Product Information) score. The core principle:

> **A seller's eligibility for a performance tag is determined by their lowest-performing metric relative to the threshold, not the average of all metrics.**

If a seller excels in every KPI except one, that single failing metric disqualifies them — just as a chain is only as strong as its weakest link.

### 8.2 Algorithm Definition

For a rule with N conditions, each condition `i` has:
- `metric_value_i` — seller's actual metric value
- `threshold_i` — required threshold
- `weight_i` — importance weight (0.0–1.0)

**Step 1: Calculate per-condition score**

For each condition with numeric comparison (`>`, `>=`, `<`, `<=`):

```
                  metric_value_i
score_i = clamp( ────────────── , 0.0, 1.0 )
                  threshold_i
```

For "less-is-better" metrics (cancellation_rate, return_rate, complaint_rate, avg_response_time_hours):

```
                  threshold_i
score_i = clamp( ────────────── , 0.0, 1.0 )
                  metric_value_i
```

For equality/boolean operators (`==`, `is_set`, `in`):
```
score_i = 1.0 if condition passes, else 0.0
```

**Step 2: Apply weakest-link**

```
weakest_score = min(score_1, score_2, ..., score_N)
```

**Step 3: Determine result**

```
PASS if weakest_score >= 1.0 (all conditions fully met)
WARNING if weakest_score >= (1.0 - warning_threshold_pct / 100)
FAIL otherwise
```

### 8.3 Scoring Modes

The `scoring_mode` field on `Seller Tag Rule` supports three modes:

| Mode | Formula | Use Case |
|------|---------|----------|
| **All Must Pass** (default) | Every condition must individually pass (boolean AND) | Simple threshold rules |
| **Weakest Link** | `min(score_1, score_2, ..., score_N)` must be ≥ 1.0 to pass | Performance tags — prevents gaming |
| **Weighted Average** | `Σ(score_i × weight_i) / Σ(weight_i)` must be ≥ threshold | Lenient composite scoring |

### 8.4 Concrete Example Calculation

**Scenario:** Seller "AnadoluTextile" evaluated for `TOP_RATED` tag.

**Rule conditions:**

| # | metric_field | operator | threshold | weight |
|---|-------------|----------|-----------|--------|
| 1 | avg_rating | >= | 4.7 | 0.40 |
| 2 | total_reviews | >= | 50 | 0.30 |
| 3 | positive_review_rate | >= | 90 | 0.30 |

**Seller's actual metrics:**

| Metric | Value |
|--------|-------|
| avg_rating | 4.6 |
| total_reviews | 120 |
| positive_review_rate | 94.2 |

**Step 1 — Per-condition scores:**

```
score_1 = 4.6 / 4.7  = 0.9787    ← BELOW 1.0 (fails threshold)
score_2 = 120 / 50   = 2.4 → clamped to 1.0  ← PASS
score_3 = 94.2 / 90  = 1.0467 → clamped to 1.0  ← PASS
```

**Step 2 — Weakest Link scoring:**

```
weakest_score = min(0.9787, 1.0, 1.0) = 0.9787
```

**Step 3 — Result determination (warning_threshold_pct = 10%):**

```
PASS threshold: weakest_score >= 1.0 → 0.9787 < 1.0 → NOT PASS
WARNING threshold: weakest_score >= (1.0 - 0.10) = 0.90 → 0.9787 >= 0.90 → WARNING
```

**Result:** `WARNING` — Seller is within 10% of losing the tag. A warning notification is sent. The `avg_rating` (4.6 vs required 4.7) is the weakest link.

**Contrast with Weighted Average mode:**

```
weighted_avg = (0.9787 × 0.40 + 1.0 × 0.30 + 1.0 × 0.30) / (0.40 + 0.30 + 0.30)
            = (0.3915 + 0.30 + 0.30) / 1.0
            = 0.9915
```

Under Weighted Average, this seller would have a composite score of 0.9915 — much closer to passing because the strong metrics compensate for the weak one. This is why **Weakest Link is recommended for Performance tags** — it prevents sellers from gaming the system by excelling in easy metrics while neglecting harder ones.

### 8.5 Second Example: Grace Period Trigger

**Scenario:** Seller "IstanbulElectro" had `FAST_SHIPPER` tag with status `Active`.

**Rule conditions:**

| # | metric_field | operator | threshold | weight |
|---|-------------|----------|-----------|--------|
| 1 | on_time_delivery_rate | >= | 95 | 0.60 |
| 2 | total_orders | >= | 20 | 0.40 |

**Metrics on Day 1 (previous evaluation):**
- on_time_delivery_rate = 96.1 → score = 96.1/95 = 1.011 → 1.0 ✓
- total_orders = 85 → score = 85/20 = 4.25 → 1.0 ✓
- **Result: PASS** — weakest = 1.0

**Metrics on Day 2 (current evaluation):**
- on_time_delivery_rate = 93.8 → score = 93.8/95 = 0.9874 → **FAIL**
- total_orders = 87 → score = 1.0 ✓
- **Result: FAIL** — weakest = 0.9874

**Action taken:**
1. `evaluation_passed = 0`
2. `consecutive_failures += 1` (now = 1)
3. Since `grace_period_days = 7` on the rule:
   - `status` changes from `Active` → `Grace Period`
   - `grace_period_start = 2026-03-03`
   - `grace_period_end = 2026-03-10`
4. Warning notification sent to seller: "Your Fast Shipper badge is at risk. On-time delivery rate dropped to 93.8% (required: 95%). You have 7 days to recover."

**On Day 5 (next evaluation during grace period):**
- on_time_delivery_rate = 95.2 → score = 1.0 ✓
- **Result: PASS** → Status reverts `Grace Period` → `Active`, `consecutive_failures` reset to 0.

```python
def process_grace_periods():
    """
    Daily job: Process grace period expirations.

    Assignments in 'Grace Period' status whose grace_period_end
    has passed without recovery are moved to 'Expired'.
    """
    today = frappe.utils.today()

    # Find all assignments in grace period that have expired
    expired_assignments = frappe.get_all(
        "Seller Tag Assignment",
        filters={
            "status": "Grace Period",
            "grace_period_end": ["<=", today]
        },
        fields=["name", "seller", "tag"]
    )

    for assignment in expired_assignments:
        doc = frappe.get_doc("Seller Tag Assignment", assignment.name)
        doc.status = "Expired"
        doc.notes = f"Grace period expired on {doc.grace_period_end}. Tag removed."
        doc.save(ignore_permissions=True)

        # Send notification to seller
        _notify_tag_expired(doc.seller, doc.tag)

    frappe.db.commit()
```

### 9.4 Warning Notification Logic

```python
def send_warning_notifications():
    """
    Daily job: Send warning notifications for at-risk tags.

    Finds assignments in 'Warning' status where warning has not yet been sent.
    """
    warning_assignments = frappe.get_all(
        "Seller Tag Assignment",
        filters={
            "status": "Warning",
            "warning_sent": 0
        },
        fields=["name", "seller", "tag", "score_snapshot"]
    )

    for assignment in warning_assignments:
        doc = frappe.get_doc("Seller Tag Assignment", assignment.name)

        # Parse score snapshot to identify weak metrics
        snapshot = json.loads(doc.score_snapshot or '{}')
        weak_metrics = [
            c for c in snapshot.get("condition_results", [])
            if not c.get("passed") or c.get("score", 1.0) < 1.0
        ]

        # Send notification via Frappe Notification system
        _send_tag_warning(doc.seller, doc.tag, weak_metrics)

        doc.warning_sent = 1
        doc.warning_sent_at = frappe.utils.now_datetime()
        doc.save(ignore_permissions=True)

    frappe.db.commit()
```

## 11. Edge Cases & Conflict Resolution

### 11.1 Tag Conflict Resolution

**Problem:** Seller qualifies for conflicting tags (e.g., `TOP_RATED` and `WARNING_LOW_RATING` — if warning tags are implemented).

**Resolution:**
1. `is_stackable` flag determines if multiple tags of same category can coexist
2. For non-stackable categories (Verification, Identity), only the highest-priority tag is active
3. When a new non-stackable tag is assigned, existing same-category tag moves to `Expired`

### 11.2 Seasonal Performance Dips

**Problem:** Seller temporarily drops below threshold during seasonal slowdown (e.g., Ramadan, summer holidays).

**Resolution:**
1. Grace period prevents immediate removal (`grace_period_days` on rule, default 7)
2. Extended grace periods possible per rule (up to 30 days)
3. Seasonal campaign tags (Promotional category) unaffected by performance dips
4. Future enhancement: seasonal adjustment factors for thresholds

### 11.3 New Seller Cold Start

**Problem:** New sellers have insufficient data for Performance tag evaluation.

**Resolution:**
1. Performance rules require minimum `total_orders` threshold (prevents false qualification with 1 perfect order)
2. `RISING_STAR` tag specifically targets new sellers (active_days ≤ 180)
3. `NEW_ARRIVAL` promotional tag auto-assigned for 30 days
4. Metrics with insufficient data default to null → `is_set` / `is_not_set` operators handle this

### 11.4 Manual Override vs Rule Conflict

**Problem:** Admin manually forces a tag on, but rule evaluation later says seller doesn't qualify.

**Resolution (existing pattern, preserved):**
1. `override_state = "ForcedOn"` prevents rule engine from removing the tag
2. `override_state = "ForcedOff"` prevents rule engine from assigning the tag
3. Manual assignments (`source = "Manual"`) are never overwritten by rules
4. Admin can clear override to re-enable rule-based evaluation

### 11.5 Concurrent Rule Evaluation

**Problem:** Multiple daily jobs evaluate the same seller simultaneously.

**Resolution:**
1. Use Frappe's `frappe.locks.lock()` for per-seller locking during evaluation
2. Evaluation order: rules sorted by `priority` (lower first)
3. First rule to assign a non-stackable tag wins
4. All evaluations use the same metrics snapshot (calculated once per cycle)

### Key Rules

1. **Seller Tag**: Can be global (`is_global=1`) or tenant-scoped. System-wide performance tags (e.g., `TOP_RATED`) are typically global; tenant-specific program tags are scoped.
2. **Seller Tag Rule**: Always inherits tenant from its `target_tag`. A rule targeting a global tag is also global.
3. **Seller Tag Assignment**: Always inherits tenant from the `seller`. The rule engine only evaluates rules matching the seller's tenant (or global rules).
4. **Seller Tag Rule Condition**: Child table — no tenant field needed; isolated via parent `Seller Tag Rule`.
