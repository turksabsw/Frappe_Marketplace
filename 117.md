# 117 - Payment Terms (Odeme Kosullari) System Plan

## Overview

This document defines the Payment Terms system for TradeHub B2B Marketplace. The system enables sellers to define flexible payment terms (advance payment, net days, deposit-balance, installments, COD) and associate them with their products/orders. During order creation, the applicable payment terms are auto-populated based on seller configuration and buyer/order context, and due dates, deposit amounts, and balance calculations are computed automatically.

**Module:** TradeHub Commerce
**Reference Platforms:** Alibaba (Trade Assurance terms), Amazon (Net-30/60 B2B), AliExpress (installment plans)

### Key Design Decisions

1. **Payment Method DocType already exists** - This plan does NOT redefine it. Instead, we plan a new `Seller Payment Method` DocType that links sellers to existing Payment Methods with conditional restrictions.
2. **Seller Profile is the anchor** - All seller-specific payment configuration links to the existing `Seller Profile` DocType (autoname: `SELLER-.#####`, module: TradeHub Seller).
3. **Child tables for conditional rules** - Both `Seller Payment Terms` and `Seller Payment Method` use child tables to define conditional applicability rules (min order amount, region, customer group, product category).
4. **Priority-based conflict resolution** - When multiple conditional rules match, the system uses explicit priority ordering to resolve conflicts.

## 5. Auto-Population Logic for Order Creation

### Overview

When a Marketplace Order (or Sub Order) is created from a Cart, the system must automatically determine and populate the applicable payment terms and allowed payment methods based on the seller's configuration and the order context.

### Trigger Points

| Trigger | Event | Action |
|---------|-------|--------|
| Cart to Order conversion | `marketplace_order.py` -> `before_insert` | Resolve and populate payment terms for each seller group |
| Sub Order creation | `sub_order.py` -> `before_insert` | Inherit payment terms from parent or resolve independently |
| Checkout page load | API call `get_checkout_payment_options()` | Return available payment methods + terms per seller group |
| Payment method selection | Checkout JS -> `on_payment_method_change()` | Recalculate due dates and amounts based on selected method + terms |

### Auto-Population Workflow

```
1. Order Created (from Cart)
   |
   v
2. For each seller group in the order:
   |
   v
3. Load Seller Payment Terms for this seller
   |-- If no Seller Payment Terms exists -> use platform defaults
   |
   v
4. Evaluate conditional rules (ordered by priority ASC):
   |-- Check min_order_amount <= order.grand_total <= max_order_amount
   |-- Check buyer_type matches
   |-- Check region matches buyer's shipping_country
   |-- Check product_category matches (any item in seller group)
   |-- Check buyer_level matches
   |-- Check requires_buyer_verification (if set, buyer must be verified)
   |-- Check is_b2b_only (if set, order must be B2B)
   |
   v
5. First matching rule (lowest priority number) determines the Payment Term
   |-- If no rules match -> use default_payment_term
   |
   v
6. Load Seller Payment Method for this seller
   |-- Filter accepted_methods by order context (amount, region, buyer_type)
   |-- If use_platform_defaults -> return all active platform Payment Methods
   |
   v
7. Populate order fields:
   |-- payment_terms_template (Link to Payment Terms Template)
   |-- payment_due_date (calculated)
   |-- deposit_amount (calculated)
   |-- deposit_due_date (calculated)
   |-- balance_amount (calculated)
   |-- balance_due_date (calculated)
   |-- installment_schedule (JSON, if installment type)
   |-- available_payment_methods (populated for checkout display)
```

## 6. Due Date / Deposit / Balance Calculation Formulas

### 6.1 Due Date Calculation

```python
def calculate_payment_due_date(order, payment_terms_template):
    """Calculate the payment due date based on term configuration."""
    base_event = payment_terms_template.due_date_based_on  # e.g., "Order Date"

    # Determine base date from the specified event
    if base_event == "Order Date":
        base_date = order.order_date
    elif base_event == "Invoice Date":
        base_date = order.e_invoice_date or order.order_date
    elif base_event == "Shipment Date":
        base_date = order.shipped_at.date() if order.shipped_at else None
    elif base_event == "Delivery Date":
        base_date = order.delivered_at.date() if order.delivered_at else None
    else:
        base_date = order.order_date

    if not base_date:
        # If event hasn't occurred yet, return None (will be calculated later)
        return None

    due_days = payment_terms_template.due_days or 0

    # For Advance and COD, due date is immediate
    if payment_terms_template.term_type == "Advance":
        return base_date  # Due immediately
    elif payment_terms_template.term_type == "Cash on Delivery":
        return None  # Due at delivery, not a fixed date

    return frappe.utils.add_days(base_date, due_days)
```

### 6.2 Deposit Amount Calculation

```python
def calculate_deposit_amount(order, payment_terms_template):
    """Calculate the deposit amount for Deposit-Balance terms."""
    if payment_terms_template.term_type != "Deposit-Balance":
        if payment_terms_template.term_type == "Advance":
            return order.grand_total  # Full amount due as advance
        return 0

    deposit_pct = payment_terms_template.deposit_percentage or 0
    deposit_amount = flt(order.grand_total * deposit_pct / 100, 2)

    return deposit_amount
```

### 6.3 Balance Amount Calculation

```python
def calculate_balance_amount(order, payment_terms_template):
    """Calculate the remaining balance after deposit."""
    deposit = calculate_deposit_amount(order, payment_terms_template)
    balance = flt(order.grand_total - deposit, 2)

    return max(balance, 0)
```

### 6.4 Balance Due Date Calculation

```python
def calculate_balance_due_date(order, payment_terms_template):
    """Calculate when the remaining balance is due."""
    if payment_terms_template.term_type != "Deposit-Balance":
        return None

    event = payment_terms_template.balance_due_event or "Shipment Date"

    if event == "Order Date":
        base_date = order.order_date
    elif event == "Shipment Date":
        base_date = order.shipped_at.date() if order.shipped_at else None
    elif event == "Delivery Date":
        base_date = order.delivered_at.date() if order.delivered_at else None
    elif event == "Invoice Date":
        base_date = order.e_invoice_date or None
    else:
        base_date = order.order_date

    if not base_date:
        return None  # Will be calculated when event occurs

    balance_days = payment_terms_template.balance_due_days or 0
    return frappe.utils.add_days(base_date, balance_days)
```

### 6.5 Installment Schedule Calculation

```python
def calculate_installment_schedule(order, payment_terms_template):
    """Generate installment schedule as JSON array."""
    if payment_terms_template.term_type != "Installment":
        return None

    count = payment_terms_template.installment_count or 1
    interval = payment_terms_template.installment_interval_days or 30
    first_pct = payment_terms_template.first_installment_percentage or 0
    total = flt(order.grand_total, 2)

    schedule = []
    base_date = order.order_date

    if first_pct > 0:
        # First installment is a custom percentage
        first_amount = flt(total * first_pct / 100, 2)
        remaining = flt(total - first_amount, 2)
        equal_amount = flt(remaining / (count - 1), 2) if count > 1 else 0

        schedule.append({
            "installment_number": 1,
            "due_date": str(base_date),
            "amount": first_amount,
            "status": "Pending"
        })

        for i in range(2, count + 1):
            due_date = frappe.utils.add_days(base_date, interval * (i - 1))
            # Last installment gets the rounding remainder
            amount = equal_amount if i < count else flt(remaining - equal_amount * (count - 2), 2)
            schedule.append({
                "installment_number": i,
                "due_date": str(due_date),
                "amount": amount,
                "status": "Pending"
            })
    else:
        # Equal installments
        equal_amount = flt(total / count, 2)

        for i in range(1, count + 1):
            due_date = frappe.utils.add_days(base_date, interval * (i - 1))
            # Last installment gets the rounding remainder
            amount = equal_amount if i < count else flt(total - equal_amount * (count - 1), 2)
            schedule.append({
                "installment_number": i,
                "due_date": str(due_date),
                "amount": amount,
                "status": "Pending"
            })

    return schedule
```

### 6.6 Late Penalty Calculation

```python
def calculate_late_penalty(order, payment_terms_template):
    """Calculate late payment penalty amount."""
    if payment_terms_template.late_penalty_type == "None" or not payment_terms_template.late_penalty_type:
        return 0

    due_date = order.payment_due_date
    if not due_date:
        return 0

    today = frappe.utils.today()
    grace_days = payment_terms_template.grace_period_days or 0
    effective_due_date = frappe.utils.add_days(due_date, grace_days)

    if frappe.utils.getdate(today) <= frappe.utils.getdate(effective_due_date):
        return 0  # Not yet overdue (including grace period)

    overdue_days = frappe.utils.date_diff(today, effective_due_date)
    outstanding = flt(order.grand_total - order.paid_amount, 2)

    if outstanding <= 0:
        return 0

    if payment_terms_template.late_penalty_type == "Fixed":
        return flt(payment_terms_template.late_penalty_fixed_amount, 2)
    elif payment_terms_template.late_penalty_type == "Daily Percentage":
        daily_rate = flt(payment_terms_template.late_penalty_rate / 365, 6)
        return flt(outstanding * daily_rate / 100 * overdue_days, 2)
    elif payment_terms_template.late_penalty_type == "Monthly Percentage":
        monthly_rate = flt(payment_terms_template.late_penalty_rate / 12, 6)
        months_overdue = overdue_days / 30.0
        return flt(outstanding * monthly_rate / 100 * months_overdue, 2)
    elif payment_terms_template.late_penalty_type == "Annual Percentage":
        annual_rate = flt(payment_terms_template.late_penalty_rate, 6)
        years_overdue = overdue_days / 365.0
        return flt(outstanding * annual_rate / 100 * years_overdue, 2)

    return 0
```

### 6.7 Early Payment Discount Calculation

```python
def calculate_early_payment_discount(order, payment_terms_template):
    """Calculate early payment discount if payment is made within discount window."""
    discount_rate = payment_terms_template.early_payment_discount_rate or 0
    discount_days = payment_terms_template.early_payment_discount_days or 0

    if discount_rate <= 0 or discount_days <= 0:
        return 0

    discount_deadline = frappe.utils.add_days(order.order_date, discount_days)

    if order.paid_at and frappe.utils.getdate(order.paid_at) <= frappe.utils.getdate(discount_deadline):
        return flt(order.grand_total * discount_rate / 100, 2)

    return 0
```

## 8. Edge Cases

### 8.1 Conflicting Payment Terms (Multiple Rules Match)

**Scenario:** A seller has two conditional rules that both match the current order context.

**Resolution Strategy:**
1. Rules are evaluated in ascending `priority` order (lower number = higher priority)
2. The **first matching rule** wins
3. If two rules have the same priority, the one with the most specific conditions wins (more non-empty condition fields = more specific)
4. If still tied, the rule created earlier (lower `name`) wins
5. Platform administrators can review and adjust priorities in the Seller Payment Terms form

### 8.2 No Seller Payment Terms Configured

**Scenario:** A seller has not set up any Seller Payment Terms record.

**Resolution:** Fall back to platform defaults:
1. Check if a "Platform Default" Payment Terms Template exists (where `is_default = 1`)
2. If found, apply it
3. If not, treat the order as "Advance" (full payment required before order confirmation)

### 8.3 Payment Method + Payment Terms Incompatibility

**Scenario:** Buyer selects "Cash on Delivery" payment method but the resolved payment term is "Net 30 Days".

**Resolution:**
1. The `compatible_term_types` field on Payment Method filters incompatible combinations
2. At checkout, only show payment terms that are compatible with the selected payment method (and vice versa)
3. If the buyer changes the payment method, recalculate and re-display available payment terms
4. If no compatible combination exists, show a warning and fall back to the seller's default

### 8.4 Order Amount Change After Terms Applied

**Scenario:** Coupon or discount applied after payment terms were calculated, changing the grand_total.

**Resolution:**
1. Recalculate deposit_amount, balance_amount, and installment_schedule on every `grand_total` change
2. Hook into `validate()` method to ensure amounts are consistent
3. If order is already partially paid, adjust the balance_amount accordingly:
   ```
   balance_amount = grand_total - deposit_amount - paid_amount
   ```

### 8.5 Shipment/Delivery Triggers Balance Due Date

**Scenario:** Balance due date is based on "Shipment Date" but the order hasn't shipped yet.

**Resolution:**
1. `balance_due_date` remains `None` until the triggering event occurs
2. When the order status changes to "Shipped" (or "Delivered"), a `before_save` hook recalculates `balance_due_date`
3. The buyer is notified of the newly set balance due date via email/notification
4. If the trigger event is "Delivery Date" and partial deliveries occur, use the **first delivery date**

### 8.6 Currency Mismatch

**Scenario:** Payment terms template uses TRY but the order uses USD.

**Resolution:**
1. Amount-based conditions (min_order_amount, max_order_amount) are always evaluated in the order's currency
2. If the payment term's currency differs from order currency, convert using the platform exchange rate
3. Deposit/balance amounts are always calculated and stored in the order's currency

## 9. Integration Points

### 9.1 Integration with Checkout (123.md)

- During seller-grouped checkout, `get_checkout_payment_options()` is called per seller group
- Each seller group may have different payment terms and available methods
- The checkout UI displays per-group payment term info (due dates, deposit requirements)

### 9.2 Integration with Order Actions (125.md / 147.md)

- Payment terms affect when certain actions become available:
  - "Create Payment Entry" becomes available when deposit/installment is due
  - "Create Sales Invoice" may be triggered by payment term events
  - Invoice date may serve as the due date base for "Net Days" terms
- Order status transitions validate payment terms compliance

### 9.3 Integration with Partial Shipment (126.md)

- If `balance_due_event = "Shipment Date"`, partial shipment triggers balance due date calculation
- Each partial shipment Delivery Note does NOT independently trigger balance recalculation (only the first shipment does)

### 9.4 Integration with Temporal Protection (127.md)

- Payment term due dates integrate with the order payment deadline:
  - For "Advance" terms: checkout reservation timer applies
  - For "Net Days" terms: the payment_due_date serves as the deadline
  - Auto-cancellation may be triggered if payment_due_date passes and `is_payment_overdue = True`
