# Kategori Erişim Kısıtlama Planı (Category Access Restriction Plan)

## 1. Bağlam (Context)

### Mevcut Durum

TradeHub platformundaki `Category` DocType'ı şu anda geniş erişim izinlerine sahiptir ve doğrudan sidebar/search bar üzerinden herkes tarafından erişilebilir durumdadır. Bu, yönetici dışı kullanıcıların kategori yapısına doğrudan müdahale etmesini kolaylaştırır ve yalnızca yetkili modül/panel üzerinden yönetilmesi gereken bir kaynak için güvenlik riski oluşturur.

#### 1.1 Category DocType — Mevcut İzin Yapısı

- **Konum:** `tradehub_catalog/doctype/category/category.json`
- **Modül:** TradeHub Catalog
- **Ağaç Yapısı:** `is_tree: 1`, `nsm_parent_field: "parent_category"`
- **Web Görünüm:** `has_web_view: 1`, `index_web_pages_for_search: 1`
- **Global Arama:** `show_name_in_global_search: 1`
- **Sidebar Durumu:** Herhangi bir kısıtlama yok — sidebar'da, arama çubuğunda ve URL ile doğrudan erişilebilir

**Mevcut İzin Matrisi:**

| Rol | create | read | write | delete | email | export | print | report | share |
|-----|--------|------|-------|--------|-------|--------|-------|--------|-------|
| System Manager | 1 | 1 | 1 | 1 | 1 | 1 | 1 | 1 | 1 |
| All | 0 | 0 | 0 | 0 | 1 | 1 | 1 | 1 | 0 |

**"All" Rolü Sorunları:**
- `All` rolü Frappe'de **tüm giriş yapmış kullanıcıları** kapsar (Guest hariç)
- `read: 1` olduğu için herkes Category kayıtlarına erişebilir
- `email: 1`, `export: 1`, `report: 1` yetkileri gereksiz geniş
- Satıcılar ve alıcılar dahil tüm kullanıcılar kategori listesini doğrudan görüntüleyebilir
- Sidebar'da "Category" linki tüm kullanıcılara görünür

**Mevcut `is_active` Alanı:**

| fieldname | fieldtype | label | default | in_list_view | in_standard_filter | description |
|-----------|-----------|-------|---------|--------------|-------------------|-------------|
| is_active | Check | Is Active | 1 | 1 | 0 | Aktif/pasif durumu |

> **Not:** `is_active` alanı mevcut ancak `in_standard_filter: 0` olarak ayarlı ve varsayılan liste filtresinde `is_active = 1` koşulu uygulanmıyor. Bu nedenle pasif kategoriler de liste görünümünde gösterilmektedir.

#### 1.2 Product Category DocType — Referans İzin Yapısı

- **Konum:** `tradehub_catalog/doctype/product_category/product_category.json`
- **Modül:** TradeHub Catalog
- **Aktiflik Alanı:** `enabled` (Check, default: 1)

**Mevcut İzin Matrisi:**

| Rol | create | read | write | delete | email | export | print | report | share |
|-----|--------|------|-------|--------|-------|--------|-------|--------|-------|
| System Manager | 1 | 1 | 1 | 1 | 1 | 1 | 1 | 1 | 1 |
| Seller | 0 | 1 | 0 | 0 | 1 | 1 | 1 | 1 | 0 |
| Buyer | 0 | 1 | 0 | 0 | 0 | 0 | 1 | 0 | 0 |

> Product Category, `All` rolü yerine **açık Seller ve Buyer rolleri** kullanmaktadır. Bu, daha güvenli ve kontrollü bir yaklaşımdır. Category DocType da bu modele geçirilmelidir.

### Neden Erişim Kısıtlaması Gerekli?

1. **Güvenlik:** Kategori yapısı platform mimarisinin temelini oluşturur; yetkisiz değişiklikler ürün listeleme, komisyon hesaplama, vergi oranları ve SEO yapılarını bozabilir
2. **Veri Bütünlüğü:** Pasif kategorilerin yanlışlıkla kullanılması veya ürünlere atanması engellenmeli
3. **Kullanıcı Deneyimi:** Satıcılar ve alıcılar doğrudan kategori yönetim sayfasına erişmek yerine, ürün ekleme/listeleme arayüzleri üzerinden kategori seçimi yapmalı
4. **Tutarlılık:** Product Category DocType'ındaki rol bazlı izin modeliyle uyumlu hale getirilmeli
5. **Denetlenebilirlik:** Kategori değişikliklerinin yalnızca yetkili admin paneli üzerinden yapılması, denetim kaydı takibini kolaylaştırır

## 2. Analiz (Analysis)

### 2.1 Mevcut Erişim Noktaları Analizi

Category DocType'a şu anda aşağıdaki yollarla erişilebilir:

| Erişim Noktası | Mevcut Durum | Hedef Durum | Yöntem |
|----------------|-------------|-------------|--------|
| Sidebar menüsü (Sol panel) | ✅ Görünür (tüm kullanıcılar) | ❌ Gizli (yalnızca admin modülü) | Workspace yapılandırması |
| Arama çubuğu (Awesome Bar) | ✅ Erişilebilir | ⚠️ Yalnızca System Manager | `show_name_in_global_search: 0` + izin kısıtlaması |
| Doğrudan URL (`/app/category`) | ✅ Erişilebilir | ⚠️ Yalnızca System Manager yazma; Seller/Buyer salt okunur | İzin kontrolü |
| API erişimi (`/api/resource/Category`) | ✅ Erişilebilir | ⚠️ Rol bazlı kısıtlama | Permission tabanlı |
| Link alanları (Listing, PIM Product) | ✅ Category seçilebilir | ✅ Korunur (salt okunur erişim gerekli) | İzin korunur |
| Ağaç görünümü (`/app/category?view=Tree`) | ✅ Erişilebilir | ⚠️ Yalnızca System Manager | İzin kontrolü |
| Web görünümü (has_web_view: 1) | ✅ Aktif | ⚠️ Değerlendirmeye bağlı | `has_web_view` ayarı |

### 2.2 "All" Rolü Risk Analizi

`All` rolünün Category DocType'ında bulunması şu riskleri taşır:

| Risk | Açıklama | Etki Seviyesi |
|------|----------|---------------|
| Bilgi sızıntısı | Tüm kategorilerin yapısı, komisyon oranları, vergi ayarları herkese açık | Orta |
| Sidebar karmaşası | Satıcı/alıcı panelinde gereksiz Category linki görünür | Düşük |
| Export riski | `export: 1` ile tüm kategori verileri dışa aktarılabilir | Orta |
| Raporlama erişimi | `report: 1` ile kategori bazlı raporlar oluşturulabilir | Düşük |
| Doğrudan erişim | URL bilgisiyle herkes kategori detaylarına ulaşabilir | Düşük |

### 2.3 `is_active` Filtre Eksikliği Analizi

Mevcut durum:
- `is_active` alanı `in_standard_filter: 0` — filtreleme panelinde görünmüyor
- Liste görünümünde pasif kategoriler de gösterilmekte
- Link alanlarında (Listing.category, PIM Product.primary_category) pasif kategoriler de seçilebilir
- Ağaç görünümünde pasif kategoriler aktiflerle karışık görünür

Hedef:
- Liste görünümünde varsayılan olarak `is_active = 1` filtresi uygulanmalı
- Link alanlarında pasif kategoriler seçilemez olmalı
- Ağaç görünümünde pasif kategoriler görsel olarak ayrılmalı (soluk gösterim)
- `in_standard_filter: 1` ile manuel filtre ekleme imkanı sağlanmalı

### 2.4 Sidebar Engelleme Yöntemleri Karşılaştırması

| Yöntem | Açıklama | Avantajlar | Dezavantajlar | Uygunluk |
|--------|----------|------------|---------------|----------|
| **A) Workspace Yapılandırması** | Category'yi workspace shortcut'larından kaldır | Temiz, yönetilebilir, Frappe native | Sadece workspace'i etkiler, URL erişimini engellemez | ⭐⭐⭐ |
| **B) İzin Kısıtlaması** | "All" rolünü kaldır, spesifik roller tanımla | Kökte engeller, API dahil tüm erişimi kontrol eder | Link alanlarında okuma izni gerektiğinden dikkatli ayar şart | ⭐⭐⭐⭐ |
| **C) `restrict_to_domain` Kullanımı** | DocType'ı belirli bir domain'e kısıtla | Güçlü izolasyon | Esnek değil, overkill | ⭐ |
| **D) Module Restriction** | Modül erişimini rol bazlı kısıtla (Module Def permissions) | Modül seviyesinde kontrol | Aynı modüldeki diğer DocType'ları da etkiler | ⭐⭐ |
| **E) Hibrit: İzin + Workspace** | İzin değişikliği + workspace temizliği birlikte | En kapsamlı, tüm erişim noktalarını kontrol eder | İki yerde değişiklik gerekir | ⭐⭐⭐⭐⭐ |

**Önerilen Yaklaşım: E — Hibrit (İzin Kısıtlaması + Workspace Yapılandırması)**

Bu yaklaşım:
1. `All` rolünü kaldırıp spesifik roller (Seller, Buyer) tanımlar
2. Workspace'ten Category shortcut'ını kaldırır/gizler
3. Global arama görünürlüğünü kısıtlar
4. Yalnızca admin modülü/paneli üzerinden erişim sağlar

### 3.1 Category DocType — İzin ve Ayar Değişiklikleri

Bu bölümde doğrudan yeni DocType oluşturulmaz; mevcut `category.json` üzerinde yapılacak yapılandırma değişiklikleri planlanır.

#### 3.1.1 Kaldırılacak / Değiştirilecek Özellikler

| Özellik | Mevcut Değer | Hedef Değer | Açıklama |
|---------|-------------|-------------|----------|
| `show_name_in_global_search` | `1` | `0` | Arama çubuğunda Category sonuçlarını gizle — kategori yönetimi yalnızca admin panelinden yapılmalı |
| `has_web_view` | `1` | Değerlendirmeye bağlı | Web görünüm gerekli değilse `0` yaparak public erişimi kapat. Eğer frontend kategori sayfaları bu mekanizmayı kullanıyorsa `1` olarak bırak |
| `index_web_pages_for_search` | `1` | `has_web_view` ile uyumlu | Web görünüm kapatılırsa bu da `0` yapılmalı |
| `quick_entry` | `0` | `0` (değişiklik yok) | Hızlı giriş zaten kapalı |

#### 3.1.2 `is_active` Alanı Güncelleme

Mevcut `is_active` alanına `in_standard_filter` eklenmeli:

**Mevcut:**
```json
{
  "default": "1",
  "fieldname": "is_active",
  "fieldtype": "Check",
  "in_list_view": 1,
  "label": "Is Active"
}
```

**Hedef:**
```json
{
  "default": "1",
  "fieldname": "is_active",
  "fieldtype": "Check",
  "in_list_view": 1,
  "in_standard_filter": 1,
  "label": "Is Active"
}
```

| fieldname | fieldtype | label | default | in_list_view | in_standard_filter | Değişiklik |
|-----------|-----------|-------|---------|--------------|-------------------|------------|
| is_active | Check | Is Active | 1 | 1 | **1** | `in_standard_filter` eklendi |

### 3.2 Varsayılan Liste Filtresi (Default List Filter)

Category DocType'ının liste görünümünde varsayılan olarak yalnızca aktif kategorileri göstermek için `category_list.js` dosyası oluşturulacaktır:

**Dosya:** `tradehub_catalog/doctype/category/category_list.js`

```javascript
// category_list.js (planlama notu — kod oluşturulmayacak)
frappe.listview_settings['Category'] = {
    // Varsayılan filtre: yalnızca aktif kategorileri göster
    filters: [
        ["is_active", "=", 1]
    ],

    // Pasif kategorileri görsel olarak ayır
    get_indicator: function(doc) {
        if (doc.is_active) {
            return [__("Active"), "green", "is_active,=,1"];
        } else {
            return [__("Inactive"), "gray", "is_active,=,0"];
        }
    },

    // Liste görünümünde satır renklendirme
    formatters: {
        category_name: function(value, field, doc) {
            if (!doc.is_active) {
                return `<span style="color: #8d99a6; text-decoration: line-through;">${value}</span>`;
            }
            return value;
        }
    }
};
```

**Davranış:**
- Liste ilk açıldığında yalnızca `is_active = 1` olan kategoriler gösterilir
- Kullanıcı filtreyi kaldırarak pasif kategorileri de görebilir (yetkili ise)
- Pasif kategoriler üzeri çizili ve soluk renkte gösterilir
- Durum indikatörü: Aktif = yeşil, Pasif = gri

### 3.3 Link Alanlarında `is_active` Filtresi

Listing ve PIM Product DocType'larındaki kategori Link alanlarında yalnızca aktif kategorilerin seçilebilmesi için:

**Listing DocType — `listing.js`:**
```javascript
// listing.js'e eklenecek filtre (planlama notu)
frappe.ui.form.on("Listing", {
    setup: function(frm) {
        // Kategori seçiminde yalnızca aktif kategorileri göster
        frm.set_query("category", function() {
            return {
                filters: {
                    is_active: 1
                }
            };
        });

        // Alt kategori seçiminde yalnızca aktif alt kategorileri göster
        frm.set_query("subcategory", function() {
            return {
                filters: {
                    is_active: 1,
                    parent_category: frm.doc.category || undefined
                }
            };
        });
    }
});
```

**PIM Product DocType — `pim_product.js`:**
```javascript
// pim_product.js'e eklenecek filtre (planlama notu)
frappe.ui.form.on("PIM Product", {
    setup: function(frm) {
        // Ana kategori seçiminde yalnızca aktif kategorileri göster
        frm.set_query("primary_category", function() {
            return {
                filters: {
                    is_active: 1
                }
            };
        });
    }
});
```

### 4.4 Misafir (Guest) Erişimi

`All` rolü kaldırıldığında, `Guest` (giriş yapmamış) kullanıcılar Category'ye hiç erişemeyecektir. Eğer frontend kategori sayfalarında `has_web_view: 1` ile public erişim gerekiyorsa:

**Seçenek A — Web View Aktif Tutma:**
```json
{
  "role": "Guest",
  "read": 1,
  "print": 0,
  "export": 0,
  "report": 0
}
```
> Yalnızca web sayfaları için `read` izni verilir, diğer tüm yetkiler kapalı.

**Seçenek B — Web View Kapatma (Önerilen):**
- `has_web_view: 0` olarak değiştir
- Kategori sayfalarını özel web template veya API ile oluştur
- Guest izni gerekmez

> **Öneri:** Kategori web sayfaları artık Product Category üzerinden yönetileceği için (102.md planına göre), eski Category DocType'ın `has_web_view: 0` olarak ayarlanması önerilir.

## 5. Validasyon Kuralları (Validation Rules)

### 5.1 Sidebar Engelleme Kuralları

| Kural | Konum | Koşul | Davranış |
|-------|-------|-------|----------|
| Workspace kaldırma | TradeHub Catalog Workspace | Category shortcut mevcut | Workspace JSON'dan Category shortcut'ını kaldır |
| Global arama gizleme | category.json | `show_name_in_global_search: 1` | `0` olarak değiştir |
| "New Category" engelleme | Sidebar "New" butonu | "All" rolü read izni | İzin kaldırılınca otomatik gizlenir |

### 5.2 İzin Bazlı Erişim Kuralları

| Kural | Rol | Koşul | Hata Mesajı |
|-------|-----|-------|-------------|
| Yazma engeli | Seller, Buyer | Kategori düzenleme denemesi | "You do not have permission to write to Category" |
| Oluşturma engeli | Seller, Buyer | Yeni kategori oluşturma denemesi | "You do not have permission to create Category" |
| Silme engeli | Seller, Buyer | Kategori silme denemesi | "You do not have permission to delete Category" |
| Export engeli | Seller, Buyer | Export/rapor denemesi | "You do not have permission to export Category" |
| API yazma engeli | Seller, Buyer | API üzerinden PUT/POST/DELETE | HTTP 403 Forbidden |

### 5.3 `is_active` Validasyonları

| Kural | Alan | Koşul | Hata Mesajı / Davranış |
|-------|------|-------|------------------------|
| Aktif ürün kontrolü | is_active | `is_active = 0` yapılırken bu kategoride aktif Listing var | "Cannot deactivate category '{0}': {1} active listings exist. Please move or deactivate them first." |
| Alt kategori kontrolü | is_active | `is_active = 0` yapılırken aktif alt kategoriler var | "Cannot deactivate category '{0}': {1} active sub-categories exist. Please deactivate them first." |
| Varsayılan filtre | Liste görünümü | Sayfa ilk yüklendiğinde | `is_active = 1` filtresi otomatik uygulanır |
| Link filtresi | Listing.category, PIM Product.primary_category | Kategori seçim dropdown'u açıldığında | Yalnızca `is_active = 1` olan kategoriler gösterilir |

**Python Validasyon Mantığı (Planlama Notu — Kod Oluşturulmayacak):**

```python
# category.py - validate() metodu içine eklenecek kurallar (planlama)
def validate(self):
    self.validate_deactivation()

def validate_deactivation(self):
    """Aktif ürün veya alt kategori varken deaktive etmeyi engelle"""
    if self.has_value_changed("is_active") and not self.is_active:
        # Aktif listing kontrolü
        active_listings = frappe.db.count("Listing", {
            "category": self.name,
            "docstatus": ["!=", 2]  # İptal edilmemiş
        })
        if active_listings:
            frappe.throw(
                _("Cannot deactivate category '{0}': {1} active listings exist. "
                  "Please move or deactivate them first.").format(
                    self.category_name, active_listings
                )
            )

        # Aktif alt kategori kontrolü
        active_children = frappe.db.count("Category", {
            "parent_category": self.name,
            "is_active": 1
        })
        if active_children:
            frappe.throw(
                _("Cannot deactivate category '{0}': {1} active sub-categories exist. "
                  "Please deactivate them first.").format(
                    self.category_name, active_children
                )
            )
```

### 5.4 Cascading Deactivation Yaklaşımı

Bir üst kategorinin pasif yapılması durumunda alt kategorilerin durumunu yönetmek için iki yaklaşım:

| Yaklaşım | Açıklama | Avantaj | Dezavantaj |
|----------|----------|---------|------------|
| **Engelleme (Önerilen)** | Alt kategoriler aktifse üst kategoriyi pasif yapma | Veri güvenliği, kullanıcı bilinçli aksiyon alır | Toplu deaktivasyon için ekstra iş |
| **Cascading deactivation** | Üst kategori pasif olunca alt kategorileri de pasif yap | Tek adımda toplu işlem | Veri kaybı riski, geri alma zor |

> **Seçim:** Engelleme yaklaşımı — veri güvenliği öncelikli.

## 6. Frontend Planı (Frontend Plan)

### 6.1 Modül-Bazlı Erişim Mimarisi

Kategoriler yalnızca **TradeHub Catalog admin modülü** üzerinden yönetilecektir. Doğrudan sidebar erişimi engellenir.

**Admin Modülü Erişim Akışı:**

```
System Manager Girişi
    └── Desk (Ana Sayfa)
        └── TradeHub Catalog Modülü (Workspace)
```

### 6.3 Ağaç Görünümü (Tree View) Kısıtlama

Category'nin ağaç görünümüne (`/app/category?view=Tree`) erişim, izin değişikliğiyle otomatik olarak kısıtlanacaktır. Ancak ek olarak:

**Pasif Kategorilerin Ağaçta Gösterimi:**

```javascript
// category/category_tree.js (planlama notu)
frappe.treeview_settings['Category'] = {
    // Ağaç düğümü renklendirme
    get_tree_node_html: function(node) {
        const label = node.data.category_name || node.data.value;
        const is_active = node.data.is_active;

        if (!is_active) {
            return `<span class="tree-node-inactive" style="color: #8d99a6; font-style: italic;">
                ${label} <span class="text-muted">(Inactive)</span>
            </span>`;
        }
        return `<span>${label}</span>`;
    },

    // Ağaçta yalnızca aktif kategorileri varsayılan göster
    filters: [
        ["is_active", "=", 1]
    ]
};
```

| `/api/method/frappe.client.get_list` | Tüm kullanıcılar (Category filtresiyle) | Rol bazlı | Permission query conditions |
